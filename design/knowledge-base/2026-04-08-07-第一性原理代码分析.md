## Phase 1 — 拆解：假设清单

**问题原样**：knowledge-base skill 的代码实现和设计是否存在影响可靠性、正确性的问题？

### 显式假设

1. JSONL 是索引存储的合适格式
2. CLI (typer) 是 Agent 与知识库交互的正确方式
3. 两步编译流程（prepare → Agent 写文件 → finalize）是可靠的
4. MD5 前 8 位作为 content hash 足够
5. 相对路径存储解决了可移植性问题

### 隐式假设

6. `generate_id()` 的全局计数器在所有调用场景下不会产生重复 ID
7. 手写的 frontmatter 解析器能处理所有合法 frontmatter
8. JSONL 全量读写在并发场景下是安全的
9. `get_project_root()` 的 parent×4 回溯在所有安装场景下正确
10. 单份 `wiki.bak/` 备份足以应对回滚需求
11. LLM 编译输出的 frontmatter 格式始终正确
12. `to_relative_path` 的 `startswith` 检查不会误匹配

## Phase 3 — 重建：具体问题与修复方案

### Bug 1（P0）: 三套独立的 frontmatter 解析器行为不一致

**位置**：

| 文件 | 函数 | 输入 | 输出 |
|------|------|------|------|
| `compiler.py` | `_parse_frontmatter(md_file: Path)` | Path 对象 | `dict \| None` |
| `searcher.py` | `_parse_frontmatter(text: str, fallback_id)` | 文本字符串 | `dict`（始终有返回） |
| `browser.py` | `_check_stale_sources()` | 内联解析 | sources 列表 |

**风险**：同一文件在不同模块中解析出不同结果。compiler 使用正则 `^---\n(.*?)\n---`，searcher 使用逐行扫描，browser 使用 split+strip。三者对 YAML 列表（`[a, b]`）、多行值、嵌套结构的处理方式不同。

**修复方案**：在 `indexer.py` 或新建 `parser.py` 中定义统一的 `parse_frontmatter(text: str) -> dict` 函数，compiler/searcher/browser 统一调用。

### Bug 2（P0）: `to_relative_path` 的 startswith 误匹配

**位置**：`indexer.py:25-31`

```python
def to_relative_path(abs_path: str) -> str:
    root = str(get_project_root())
    if abs_path.startswith(root):  # Bug: /foo/bar 会匹配 /foo/bar2/...
        rel = os.path.relpath(abs_path, root)
        return rel
    return abs_path
```

**风险**：如果项目根是 `/Users/foo/project`，路径 `/Users/foo/project-backup/file.md` 也会被错误转为相对路径。

**修复方案**：

```python
def to_relative_path(abs_path: str) -> str:
    root = str(get_project_root())
    if abs_path == root or abs_path.startswith(root + os.sep):
        return os.path.relpath(abs_path, root)
    return abs_path
```

### Bug 3（P1）: `generate_id()` 全局计数器跨进程不安全

**位置**：`indexer.py:65-72`

```python
_id_counter = 0

def generate_id() -> str:
    global _id_counter
    now = datetime.now(timezone.utc)
    _id_counter += 1
    return f"kb-{now.strftime('%Y%m%d')}-{now.strftime('%H%M%S')}-{_id_counter:03d}"
```

**风险**：每个 Python 进程启动时 `_id_counter = 0`。用户在同一秒内执行两次 `kb add`（两个进程），两者都从 `001` 开始计数，生成相同 ID `kb-20260408-120000-001`。

**修复方案**：在 ID 中加入随机成分。

```python
import uuid

def generate_id() -> str:
    now = datetime.now(timezone.utc)
    rand = uuid.uuid4().hex[:4]
    return f"kb-{now.strftime('%Y%m%d')}-{now.strftime('%H%M%S')}-{rand}"
```

### Bug 4（P1）: `get_project_root()` 硬编码 parent 层级

**位置**：`indexer.py:19-22`

```python
def get_project_root() -> Path:
    if _project_root:
        return _project_root
    return Path(__file__).resolve().parent.parent.parent.parent
```

**风险**：
- 项目内安装（`skills/knowledge-base/src/indexer.py`）：parent×4 = 项目根 ✅
- 全局安装（`~/.cursor/skills/knowledge-base/src/indexer.py`）：parent×4 = `~/.cursor/` ❌

**修复方案**：使用 `_common.py:PROJECT_ROOT`（已定义），或在 CLI 入口处 `set_project_root(Path.cwd())` 将工作目录设为项目根。

### 架构改进 1（P2）: JSONL 写入无原子性保证

**位置**：`indexer.py:57-62`

```python
def write_index(data_dir: Path, entries: list):
    index_file = data_dir / "raw" / "index.jsonl"
    with open(index_file, "w", encoding="utf-8") as f:  # 直接覆盖
        for entry in entries:
            f.write(json.dumps(entry, ensure_ascii=False) + "\n")
```

**风险**：如果写入中途进程崩溃（或磁盘满），文件被截断，丢失全部索引数据。

**修复方案**：先写临时文件，再原子替换。

```python
import tempfile

def write_index(data_dir: Path, entries: list):
    index_file = data_dir / "raw" / "index.jsonl"
    index_file.parent.mkdir(parents=True, exist_ok=True)
    fd, tmp = tempfile.mkstemp(dir=str(index_file.parent), suffix=".tmp")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            for entry in entries:
                f.write(json.dumps(entry, ensure_ascii=False) + "\n")
        os.replace(tmp, str(index_file))  # 原子替换
    except:
        os.unlink(tmp)
        raise
```

### 架构改进 2（P3）: `_concept_merge` 未更新 frontmatter

**位置**：`graph.py:162-190`

`_concept_merge` 只是将 concept2 的文本追加到 concept1 末尾，没有合并两个概念的 `sources`、`related`、`tags` 字段。合并后的概念 frontmatter 不完整。

**修复方案**：解析两个文件的 frontmatter，合并 sources/related/tags 列表去重后更新 concept1 的 frontmatter。

## 总结

```
问题（原样）: knowledge-base skill 代码实现是否有问题
消除的伪约束:
  - parent×4 回溯是唯一确定 project root 的方式 → 可以从 CWD 或 _common.py 获取
  - 每个模块需要自己的 frontmatter 解析 → 应该统一
确立的地基事实:
  - CLI typer 是 Agent 交互的平台约束
  - MD5[:8] 对当前规模足够
  - 两步编译设计（prepare + finalize）本身合理
推荐修复:
  P0: 统一 frontmatter 解析器 + 修复 startswith 误匹配
  P1: generate_id 加随机成分 + project_root 使用 _common.py
  P2: JSONL 原子写入
  P3: concept_merge 更新 frontmatter
核心取舍: 所有修复都是低成本局部改动，不需要架构重构
规模化时最先崩溃的点: JSONL 全量读写 — 超过数千条目时性能下降
仍需验证的假设: LLM 编译输出质量（已有 validate 兜底）
```
