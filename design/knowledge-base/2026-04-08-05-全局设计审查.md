## Dimension B — 假设挑战

### B1: "只索引不复制"的路径脆性

当前 `index.jsonl` 存储**绝对路径**：

```json
{
  "path": "/Users/.../design/memory/2026-02-18-01-总览与设计理念.md"
}
```

**假设**: "用户不会移动/重命名项目目录"
**判定**: ❌ False Constraint

项目目录重命名、用户切换工作区、Git clone 到不同位置，都会导致全部路径失效。`kb check` 能检测但无法自动修复。

**改进方向**：
- 存储相对于项目根目录的相对路径
- 提供 `kb migrate --old-base --new-base` 批量路径修复命令

---

### B2: "不设 Hook 自动编译"导致知识永远陈旧

设计文档 §10.3：

> 不设 Hook 自动编译。原因：编译需要 Agent 参与智能步骤。

**假设**: "用户/Agent 会记得定期编译"
**判定**: ⚠️ Needs Validation

结果：源文件更新了，wiki 概念仍然是旧版。Agent 查询 KB 时看到的是过期知识，但无法感知到。

**改进方向**：
- browse/read 输出时检测 source 的 content_hash 是否变化，标注 `[⚠ 来源已更新，概念可能过期]`
- status 命令增加"有 N 个概念的来源已变更"提示

---

### B3: "Agent 编译质量总是足够好的"

编译流程是单向的：Agent 写入概念 → `finalize` 处理。没有任何验证环节：

- 不检查 frontmatter 格式是否合法（id/title/category 是否存在）
- 不验证 `sources` ID 是否存在于 index.jsonl 中
- 不检查 `related` 引用的概念是否存在
- Agent 可能写出格式错误的条目，`_parse_frontmatter` 静默跳过

**假设**: "Agent LLM 总是能产出格式正确、内容准确的概念条目"
**判定**: ⚠️ Needs Validation

**改进方向**：
- `finalize` 中增加 validate 步骤：检查 frontmatter 必填字段、source ID 有效性、related 引用有效性
- 对验证失败的条目输出警告（不阻塞流程，但告知用户）

## Dimension C — 视角盲点

### C1: finalize 每次全量重建，无增量

`_compile_finalize` 每次执行都遍历全部 `concepts/*.md` 重建：
- `backlinks.json`（反向链接）
- `graph.json`（知识图谱）
- `index.md`（知识地图）
- 所有 `categories/*.md`（分类索引）

当概念数量增长到 50+ 后，这个操作会越来越慢，且无法做局部更新。

**影响**: 当前规模（16 个概念）无感知，但不可扩展
**改进方向**: 增量 finalize（只更新受影响的 backlinks/graph 节点）

---

### C2: 没有"概念过期"机制

当 source 文件被修改（`content_hash` 变化），对应的概念条目没有任何标记表示"来源已变更，概念可能过期"。

Agent 读取时（`kb read`）看不出这个概念是否反映了最新内容。

**影响**: Agent 可能基于过期知识做出决策
**改进方向**:
- `kb read` 时检查概念关联的 sources 的 content_hash，如有变化输出 stale 标记
- `kb status` 增加"N 个概念来源已变更"统计

---

### C3: 孤立概念无清理

如果通过 `kb remove` 删除一个 source，该 source 编译出的概念仍然存在于 wiki 中。没有自动检测孤立概念（所有 sources 已删除）的机制。

**影响**: wiki 中积累无来源的"幽灵概念"
**改进方向**:
- `finalize` 中检测 orphan concepts（sources 全部不存在于 index.jsonl）
- 输出警告，不自动删除（防误删）

---

### C4: 分类推断逻辑过于简单

`infer_category` 只识别 `design/<name>/` 模式：

```python
def infer_category(path: str) -> str:
    parts = Path(path).parts
    for i, part in enumerate(parts):
        if part == "design" and i + 1 < len(parts):
            next_part = parts[i + 1]
            if not next_part.endswith(".md"):
                return next_part
    return "uncategorized"
```

项目外的文件、根目录文件、非 `design/` 目录的文件全部归为 `uncategorized`。

**影响**: 非 design 目录的资料分类不友好
**改进方向**: 扩展推断规则（如 `skills/<name>/` → skill 名、`docs/<name>/` → 文档类、`tests/<name>/` → 测试类）

---

## Dimension D — 风险压测

### D1: 绝对路径 + 无迁移工具 = 数据脆性高

**场景**: 用户 clone 项目到新路径、macOS 升级改变用户目录结构
**结果**: index.jsonl 中所有路径失效，需要逐条 `kb edit` 或重新 import
**发生概率**: 中（跨设备开发、重构目录都会触发）
**改进方向**: 存储相对路径 + 提供 `kb migrate` 批量修复

---

### D2: 编译产物无版本/无回滚

**场景**: Agent 执行 compile，产出质量差（幻觉、误解）
**结果**: 概念文件被覆盖，旧的好概念已丢失
**发生概率**: 中（LLM 编译质量不可控）
**改进方向**:
- compile 前自动备份 `wiki/` 到 `wiki.bak/`
- 或使用 Git 追踪 wiki 目录变更

## 总结：优先级排序

| 优先级 | 问题 | 来源 | 影响 | 修复难度 |
|--------|------|------|------|---------|
| **P0** | 编译无质量验证 | B3 | Agent 错误直接污染 wiki | 中 |
| **P1** | 绝对路径脆性 | B1, D1 | 项目迁移后全部失效 | 中 |
| **P1** | 知识陈旧无感知 | B2, C2 | Agent 使用过期知识决策 | 低 |
| **P2** | finalize 全量重建 | C1 | 随规模增长变慢 | 中 |
| **P2** | 孤立概念无清理 | C3 | wiki 积累无源概念 | 低 |
| **P2** | 编译产物无回滚 | D2 | 差编译不可恢复 | 低 |
| **P3** | 分类推断简单 | C4 | 非 design 文件分类差 | 低 |
