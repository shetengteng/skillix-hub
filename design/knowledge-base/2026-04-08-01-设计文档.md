# Knowledge Base Skill 设计文档

> **日期**: 2026-04-08
> **状态**: 已实现（Phase 1-5）
> **作者**: AI Agent

---

## 1. 背景与目标

### 1.1 问题

AI Agent 在跨会话工作时，面临一个核心问题：**之前积累的设计资料、技术文档、数据集、代码仓库散落在本地各处，无法高效发现和复用**。

现有的 Memory Skill 解决了"记住对话中的决策和偏好"，但没有解决"如何管理和利用结构化的知识资产"。

### 1.2 目标

构建一个 **Knowledge Base (KB) Skill**，实现：

1. **索引层**：注册和维护各种本地资料（md、图片、数据集、代码仓库等）的位置索引
2. **编译层**：LLM 读取索引内容，提取概念、建立关联，编译成结构化 Wiki
3. **查询层**：Agent 通过渐进式披露高效浏览和搜索知识库

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **只索引不复制** | 只存储资料的路径引用，不复制原始文件，避免数据膨胀 |
| **Agent 即编译器** | 利用 Cursor/Claude Code 自身的 LLM 能力编译 Wiki，不依赖外部 API |
| **增量编译** | 只处理新增/修改的部分，通过内容哈希检测变更 |
| **渐进式披露** | 按需加载知识，不浪费 Agent 上下文窗口 |
| **多 Agent 兼容** | 统一 CLI 接口，Cursor / Claude Code / Codex 均可使用 |

## 2. 核心概念

```
┌──────────────────────────────────────────────────────┐
│                  Knowledge Base Skill                 │
│                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐   │
│  │ Raw Index │ ──→│ Compiler │ ──→│     Wiki     │   │
│  │  (索引层) │    │ (编译层)  │    │   (知识层)   │   │
│  └──────────┘    └──────────┘    └──────────────┘   │
│       ↑                                ↓             │
│  ┌──────────┐                   ┌──────────────┐    │
│  │   用户    │                   │  Agent 查询  │    │
│  │ 注册资料  │                   │ (渐进式披露)  │    │
│  └──────────┘                   └──────────────┘    │
└──────────────────────────────────────────────────────┘
```

### 2.1 Raw Index（原始索引层）

不存储实际文件，只维护一份索引清单，记录"什么资料在哪里"。

### 2.2 Compiler（编译层）

读取索引指向的实际内容，经 LLM 分析后输出结构化 Wiki。分两步：
- **确定性步骤**（Python 脚本）：扫描索引、读取内容、提取元数据、更新链接
- **智能步骤**（Agent LLM）：概念提取、条目编写、关系建立

### 2.3 Wiki（知识层）

编译后的产物。一组按概念/分类组织的 Markdown 文件，包含反向链接、交叉引用和知识图谱。

### 2.4 Knowledge Graph（知识图谱）

Wiki 的结构化表示。节点是概念/资料，边是关联关系。支持 JSON 和 Mermaid 两种格式输出。

## 3. 架构设计

### 3.1 整体数据流

```
本地资料（散落各处）
    │
    │ kb add <path>
    ↓
┌──────────────────────┐
│   Raw Index (JSONL)   │  ← 资料注册：路径、类型、标签、摘要
└──────────┬───────────┘
           │ kb compile
           ↓
┌──────────────────────┐
│ Step 1: 元数据提取    │  ← Python 脚本（确定性）
│ - 扫描索引             │     读取内容、计算哈希、检测变更
│ - 生成待编译清单       │
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│ Step 2: 概念提取      │  ← Agent LLM（智能）
│ - 从内容中提取关键概念 │     输出概念列表 + 初步分类
│ - 识别概念间关系       │
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│ Step 3: 条目生成      │  ← Agent LLM（智能）
│ - 为每个概念编写条目   │     包含定义、关联、引用来源
│ - 建立交叉引用        │
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│ Step 4: 索引更新      │  ← Python 脚本（确定性）
│ - 更新 backlinks.json │     反向链接、知识图谱、目录
│ - 更新 graph.json     │
│ - 重建 index.md       │
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│      Wiki 产物        │  ← 结构化 Markdown + JSON
│  Agent 按需查询使用    │
└──────────────────────┘
```

### 3.2 查询数据流

Agent 查询知识库时的数据读取路径（渐进式披露）：

```
Agent 发起查询
    │
    ├─── kb browse ──────────→ 读取 wiki/index.md（知识地图）
    │                              ↓ 选择某分类
    │    kb browse <category> → 读取 wiki/categories/<category>.md
    │                              ↓ 选择某概念
    │    kb read <concept-id> → 读取 wiki/concepts/<concept-id>.md
    │                              ↓ 需要原始资料
    │    kb source <source-id> → 读取 raw/index.jsonl 匹配条目元数据
    │                              ↓ 需要完整内容
    │                           直接 Read 工具读取本地原始文件
    │
    ├─── kb search <query> ──→ 1. 遍历 raw/index.jsonl → 标题/标签/分类关键词匹配
    │                          2. 遍历 wiki/concepts/*.md → frontmatter 关键词匹配
    │                          3. 合并去重，按相关度排序输出
    │
    ├─── kb status ──────────→ 读取 raw/index.jsonl 统计：
    │                            - 总条目 / 已编译 / 待编译
    │                            - 路径有效 / 路径失效
    │                          读取 wiki/graph.json 检查存在性
    │                          读取 compile/history.jsonl 最近编译时间
    │
    ├─── kb check ───────────→ 遍历 raw/index.jsonl 每个条目：
    │                            - 检查路径是否存在（os.path.exists）
    │                            - 计算当前 content_hash 与索引中对比
    │                            - 分类为：有效 / 失效 / 变更
    │
    └─── kb graph ───────────→ 读取 wiki/graph.json（节点 + 边）
         kb graph --center X  → 从 graph.json 中提取以 X 为中心的子图
         kb graph --mermaid   → 转换为 Mermaid 文本格式
```

**查询涉及的数据文件**：

| 命令 | 读取的数据 |
|------|-----------|
| `browse` | `wiki/index.md` |
| `browse <cat>` | `wiki/categories/<cat>.md` |
| `read <id>` | `wiki/concepts/<id>.md` |
| `source <id>` | `raw/index.jsonl` + `raw/cache/<id>.txt` |
| `search` | `raw/index.jsonl` + `wiki/concepts/*.md` |
| `status` | `raw/index.jsonl` + `wiki/graph.json` + `compile/history.jsonl` |
| `check` | `raw/index.jsonl` + 各条目原始文件路径 |
| `graph` | `wiki/graph.json` |

### 3.3 模块划分

| 层次 | 模块 | 职责 |
|------|------|------|
| **命令层** | `commands/*` | typer 参数定义 + 路由（一个命令一个文件） |
| **业务层** | `indexer` | 管理 raw 索引（add/remove/edit/list/import） |
| | `scanner` | 扫描索引内容、计算哈希、检测变更、缓存 |
| | `compiler` | 编排编译流程、生成待编译清单、后处理 |
| | `browser` | 渐进式浏览 Wiki（L1-L4 按层级输出） |
| | `searcher` | 关键词搜索 + 状态统计 + 路径检查 |
| | `graph` | 知识图谱 + 概念管理 + 分类管理 |
| **入口层** | `main.py` | typer app 组装（~25 行） |

## 4. 数据结构设计

### 4.1 索引条目格式（JSONL）

每行一个 JSON 对象，存储在 `raw/index.jsonl`：

```json
{
  "id": "kb-20260408-001",
  "title": "Memory Skill 四层架构设计",
  "type": "markdown",
  "path": "/Users/.../design/memory/2026-02-18-01-总览与设计理念.md",
  "tags": ["memory", "architecture", "skill-design"],
  "category": "skill-design",
  "summary": "Memory Skill 采用 Hooks 确定性触发 + Agent 智能执行的协作架构",
  "content_hash": "a1b2c3d4",
  "created_at": "2026-04-08T10:00:00Z",
  "updated_at": "2026-04-08T10:00:00Z",
  "compiled": false
}
```

**支持的 type 值**：

| type | 说明 | 索引时行为 |
|------|------|-----------|
| `markdown` | Markdown 文档 | 读取全文，提取标题/摘要 |
| `image` | 图片文件 | 只索引路径，需手动填写描述 |
| `dataset` | 数据集文件 | 索引路径 + 文件大小 + 格式信息 |
| `repo` | 代码仓库 | 索引路径 + 读取 README + 目录结构 |
| `link` | 外部链接 | 存储 URL + 手动填写摘要 |
| `directory` | 目录（批量注册） | 递归扫描目录下的文件 |

### 4.2 Wiki 条目格式

每个概念一个 Markdown 文件，存储在 `wiki/concepts/`：

```markdown
---
id: concept-memory-architecture
title: Memory 四层架构
category: skill-design
tags: [memory, architecture]
sources: [kb-20260408-001, kb-20260408-003]
related: [concept-hook-design, concept-agent-llm]
updated_at: 2026-04-08T10:30:00Z
---

# Memory 四层架构

Memory Skill 采用四层存储架构实现记忆的渐进式沉淀...

## 核心要点

- Layer 1: Rules 主动存储（实时）
- Layer 2: preCompact 刷写（上下文压缩时）
- ...

## 关联概念

- [[Hook 详细设计]] — 触发时机设计
- [[Agent LLM 协作]] — 智能执行层

## 来源资料

- [总览与设计理念](source://kb-20260408-001)
- [Hook 详细设计文档](source://kb-20260408-003)
```

## 5. 命令接口设计

所有命令通过 `python3 <skill-path>/main.py <command> [args]` 执行。基于 typer CLI 框架，支持 `--help` 自动文档和 shell 自动补全。

### 5.1 索引管理

```bash
# 添加单个资料
kb add <path> [--title "标题"] [--tags "tag1,tag2"] [--type markdown] [--category "分类"]

# 批量添加目录
kb add <dir-path> --type directory [--pattern "*.md"] [--recursive]

# 列出索引
kb list [--type markdown] [--tag memory] [--category skill-design] [--pending]

# 移除索引
kb remove <id>

# 更新索引元数据
kb edit <id> [--title "新标题"] [--tags "新标签"] [--category "新分类"]

# 快捷导入项目目录
kb import-project [--dir design] [--pattern "*.md"]
```

### 5.2 编译

```bash
# 增量编译（只处理 compiled=false 的条目）
kb compile

# 全量重新编译
kb compile --full

# 预览待编译清单（不执行）
kb compile --dry-run

# 编译指定条目
kb compile --id kb-20260408-001
```

**编译输出**：编译命令本身由 Python 脚本执行 Step 1（元数据提取）和 Step 4（索引更新），Step 2-3 输出结构化 prompt 供 Agent 执行。

### 5.3 浏览（渐进式披露核心）

```bash
# L1: 知识地图概览
kb browse

# L2: 查看分类下的条目
kb browse <category>

# L3: 查看具体概念条目
kb read <concept-id>

# L4: 查看原始资料信息
kb source <source-id>
```

### 5.4 搜索

```bash
# 关键词搜索
kb search "memory architecture"

# 按标签搜索
kb search --tag memory

# 按分类筛选
kb search --category skill-design
```

### 5.5 知识图谱

```bash
# JSON 格式输出
kb graph

# Mermaid 格式（可粘贴到 Markdown 渲染）
kb graph --format mermaid

# 以某概念为中心的子图
kb graph --center concept-memory-architecture --depth 2
```

### 5.6 状态与管理

```bash
# 知识库健康状态
kb status

# 路径有效性检查
kb check

# 概念管理
kb concept list
kb concept remove <concept-id>
kb concept merge <id1> <id2>
kb concept rename <concept-id> "新标题"

# 分类管理
kb category list
kb category rename <old-name> <new-name>
```

## 6. 编译流程详细设计

### 6.1 确定性阶段（Python 脚本自动完成）

**Step 1: 元数据提取**

```python
def prepare_compilation():
    """扫描索引，读取内容，生成待编译清单"""
    pending = []
    for entry in read_index():
        if entry["compiled"] and not content_changed(entry):
            continue
        content = extract_content(entry)  # 根据 type 读取内容
        pending.append({
            "id": entry["id"],
            "title": entry["title"],
            "type": entry["type"],
            "content_preview": content[:2000],  # 前 2000 字
            "tags": entry["tags"]
        })
    write_pending_list(pending)
    return pending
```

**内容提取策略**：

| type | 提取方式 |
|------|---------|
| `markdown` | 读取全文，保留标题结构 |
| `image` | 读取文件名 + 用户填写的描述 |
| `dataset` | 读取前 N 行 + schema 推断 |
| `repo` | 读取 README + 目录树（深度 3） |
| `link` | 使用缓存的摘要 |
| `directory` | 列出文件清单 + 各文件摘要 |

### 6.2 智能阶段（Agent LLM 执行）

编译命令执行完 Step 1 后，输出一段结构化 prompt，Agent 根据此 prompt 完成 Step 2-3。

**Step 2 Prompt 模板**（`prompts/extract_concepts.md`）：

```
你是知识库编译器。请分析以下资料，提取关键概念并建立关系。

## 待分析资料

{{pending_list}}

## 已有概念（避免重复）

{{existing_concepts}}

## 输出格式

请输出 JSON：
{
  "concepts": [
    {
      "id": "concept-xxx",
      "label": "概念名称",
      "category": "分类",
      "definition": "一句话定义",
      "related_sources": ["来源 ID"],
      "related_concepts": ["关联概念 ID"]
    }
  ]
}
```

**Step 3 Prompt 模板**（`prompts/write_entry.md`）：

```
请为以下概念编写 Wiki 条目。

## 概念信息

{{concept}}

## 相关资料内容

{{source_contents}}

## 要求

- 用 Markdown 格式编写
- 包含 frontmatter（id, title, category, tags, sources, related）
- 正文包含：概要、核心要点、关联概念、来源资料
- 控制在 500 字以内
- 使用 [[概念名]] 语法建立内部链接
```

### 6.3 后处理阶段（Python 脚本自动完成）

**Step 4: 索引更新**

Agent 完成条目编写后，执行：

```bash
kb compile --finalize
```

脚本自动：
1. 解析所有 Wiki 条目的 frontmatter
2. 提取 `[[]]` 内部链接，构建反向链接索引
3. 更新 `graph.json` 知识图谱
4. 重建 `index.md` 知识地图目录
5. 标记已编译条目的 `compiled = true`

### 6.4 增量编译策略

```
已有索引条目
    │
    ├─ content_hash 未变 + compiled=true → 跳过
    ├─ content_hash 已变 → 重新提取 + 标记关联概念待更新
    └─ compiled=false → 新条目，加入待编译清单

已有概念条目
    │
    ├─ 所有 source 未变 → 保持
    ├─ 有 source 变更 → 标记待更新
    └─ 所有 source 被删除 → 标记为孤立概念（不自动删除）
```

## 7. AI Agent 交互设计（渐进式披露）

### 7.1 披露层级

这是让 Agent 高效使用知识库的关键设计。Agent 不需要一次加载所有内容，而是按需逐层深入：

```
L0 ─── SKILL.md ──────────────────────── ~200 tokens
        "这是什么" + 命令列表 + 快速开始

L1 ─── kb browse ─────────────────────── ~500 tokens
        知识地图概览：分类列表 + 每分类条目数 + 最近更新

L2 ─── kb browse <category> ──────────── ~800 tokens
        分类下所有概念：标题 + 一句话摘要

L3 ─── kb read <concept-id> ──────────── ~1000 tokens
        概念全文 + 关联推荐 + 来源列表

L4 ─── kb source <source-id> ─────────── ~500 tokens
        原始资料元数据 + 内容预览（前 500 字）

L5 ─── Read <实际文件路径> ───────────── 完整内容
        直接读取原始文件
```

### 7.2 输出格式设计

每个命令的输出都包含三部分：**结果 + 统计 + 下一步建议**

```
=== Knowledge Base: 知识地图 ===

📂 skill-design (12 个概念, 25 个资料)
📂 architecture  (8 个概念, 15 个资料)
📂 testing       (5 个概念, 10 个资料)

共 25 个概念 | 50 个资料 | 最近更新: 2026-04-08

💡 下一步:
- `kb browse skill-design` — 查看 skill-design 分类下的概念
- `kb search "关键词"` — 搜索知识库
- `kb graph --format mermaid` — 查看知识图谱
```

### 7.3 多 Agent 兼容方案

所有 Agent 通过同一个接口使用：

| Agent | 发现方式 | 使用方式 |
|-------|---------|---------|
| **Cursor** | 读取 SKILL.md（通过 skills 配置） | Shell 执行 `python3 main.py <cmd>` |
| **Claude Code** | 读取 SKILL.md（通过 .claude/skills/） | Shell 执行 `python3 main.py <cmd>` |
| **Codex** | 读取 SKILL.md（通过 .codex/skills/） | Shell 执行 `python3 main.py <cmd>` |

SKILL.md 开头需包含清晰的触发规则：

```markdown
当用户提到"知识库"、"之前的设计文档"、"找之前的资料"、
"相关设计"、"knowledge base"、"kb"时使用本 Skill。
```

### 7.4 典型交互流程

**场景：Agent 需要参考之前的设计**

```
Agent: (需要了解 Memory Skill 的架构)
  → 读取 SKILL.md → 发现 kb browse 命令
  → 执行 kb browse → 看到 skill-design 分类
  → 执行 kb browse skill-design → 找到 "Memory 四层架构" 概念
  → 执行 kb read concept-memory-architecture → 获取概念条目
  → 条目中引用了 source kb-20260408-001
  → 执行 kb source kb-20260408-001 → 获取原始文件路径
  → 用 Read 工具直接读取原始设计文档
```

**场景：用户让 Agent 注册新资料**

```
用户: "把这个设计文档加到知识库"
Agent:
  → kb add /path/to/doc.md --tags "api,design" --category "api-design"
  → kb compile --dry-run → 看到 1 个待编译条目
  → kb compile → Step 1 自动完成，输出编译 prompt
  → Agent 执行概念提取和条目编写
  → kb compile --finalize → 更新索引和图谱
```

## 8. 技术栈选型

### 8.1 语言与运行时

| 选项 | 选择 | 理由 |
|------|------|------|
| 主语言 | **Python 3.x** | 与项目其他 Skill 一致，生态丰富 |
| 数据存储 | **JSONL + JSON + Markdown** | 简单、LLM 友好、无需额外依赖 |
| 可选增强 | **SQLite** | 当条目数量大（>500）时，提供高效查询 |
| LLM 编译 | **Agent 自身能力** | 不依赖外部 API，与 Memory Skill 策略一致 |

### 8.2 为什么不用向量数据库？

知识库的规模通常在数十到数百条目，关键词搜索 + 标签筛选已经足够。如果未来需要语义搜索，可以复用 Memory Skill 的本地嵌入模型方案。

### 8.3 为什么用 JSONL 而不是 SQLite？

- 索引条目数量预期 <500，JSONL 完全够用
- JSONL 是纯文本，LLM 可直接读取和理解
- 不需要安装额外依赖
- 如果需要 SQLite，可以作为增量升级添加，不影响核心架构

### 8.4 依赖

通过 `pyproject.toml` 管理：

```toml
[project]
requires-python = ">=3.10"
dependencies = [
    "typer[all]>=0.12",     # CLI 框架（彩色输出、自动补全）
    "pydantic>=2.0.0",      # 数据模型验证
    "pyyaml>=6.0",          # Markdown frontmatter 解析
]
```

标准库依赖：json, hashlib, pathlib, shutil, re

## 10. 关键设计决策

### 10.1 为什么用 Agent 编译而不是纯脚本？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯 Python 脚本 | 全自动、可定时 | 需要外部 LLM API Key，质量有限 |
| 纯 Agent 手动 | 质量高、上下文感知 | 每次都需要 Agent 介入 |
| **混合模式（选用）** | 确定性步骤自动化 + 智能步骤由 Agent 完成 | 编译需要 Agent 在场 |

与 Memory Skill 的 `distill_refined.py` 策略一致：**Python 准备数据，Agent 执行智能部分**。

### 10.2 Wiki 条目的粒度

- **一个概念 = 一个文件**，控制在 500 字以内
- 概念粒度由 LLM 在编译时判断，但有约束：
  - 每个概念至少关联 1 个 source
  - 每个概念应该是"可独立理解的知识单元"
  - 避免过于宏观（如"软件工程"）或过于微观（如"某变量名"）

### 10.3 编译触发时机

不设 Hook 自动编译。原因：
- 编译需要 Agent 参与智能步骤
- 自动触发可能在不合适的时机干扰用户
- 用户或 Agent 显式执行 `kb compile` 更可控

但 `kb list --pending` 可以快速查看有多少待编译条目，Agent 可以在适当时机主动提议编译。

### 10.4 content_hash 变更检测

```python
import hashlib
from pathlib import Path

def compute_hash(path: str, entry_type: str) -> str:
    if entry_type == "markdown":
        content = Path(path).read_text(encoding="utf-8")
        return hashlib.md5(content.encode()).hexdigest()[:8]
    elif entry_type == "repo":
        # 对 repo 类型，hash README + 目录结构
        ...
    elif entry_type in ("image", "dataset"):
        # 对二进制文件，hash 文件大小 + mtime
        stat = Path(path).stat()
        return hashlib.md5(f"{stat.st_size}-{stat.st_mtime}".encode()).hexdigest()[:8]
```
