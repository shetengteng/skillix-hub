## Dimension A — 问题定义：cache 是做什么的？

### 代码证据

**写入点**（`compiler.py:_compile_prepare`）：

```python
for item in pending:
    cache_content(data_dir, item["id"], item.get("content_preview", ""))
```

编译准备阶段，将每个待编译条目的 `content_preview`（原始文件前 2000 字）写入 `raw/cache/<id>.txt`。

**读取点**（`scanner.py:get_cached_content`）：

```python
def get_cached_content(data_dir: Path, entry_id: str) -> str | None:
    cache_file = data_dir / "raw" / "cache" / f"{entry_id}.txt"
    if cache_file.exists():
        return cache_file.read_text(encoding="utf-8")
    return None
```

**调用情况**：

| 函数 | 定义位置 | 调用位置 | 被调用次数 |
|------|---------|---------|-----------|
| `cache_content()` | scanner.py:196 | compiler.py:98 | 1 次 |
| `get_cached_content()` | scanner.py:204 | **无** | **0 次** |

### 结论

**cache 目录是"只写不读"的死数据。** `get_cached_content` 是一个定义了但从未被调用的死函数。content_preview 写入 cache 文件后，在整个编译流程中再也没有被读取——编译 prompt 直接使用内存中的 `item["content_preview"]`，不需要从 cache 文件读取。

## Dimension B — 假设挑战

| # | 假设 | 挑战 | 判定 |
|---|------|------|------|
| 1 | cache/ 是有用的缓存 | `get_cached_content` 从未被调用，cache 是死数据 | ❌ 无效 |
| 2 | cache/ 应该放在 raw/ 下 | cache 是编译中间产物，不是原始数据。raw 应该只包含索引清单 | ❌ 语义错位 |
| 3 | pending.json 跟踪编译进度 | 每次 compile 覆盖，finalize 不读取也不清理，崩溃后变成垃圾 | ⚠️ 设计未完成 |
| 4 | history.jsonl 记录完整历史 | 只追加不旋转，且只在 status 中读取最后一行，前面所有行是浪费 | ⚠️ 可简化 |
| 5 | backlinks.json 独立于 graph.json | graph.json 的 edges 已包含所有反向引用关系，backlinks 是冗余计算 | ❌ 数据冗余 |
| 6 | index.jsonl 是纯索引清单 | 混合了注册信息（path/title/type）和编译状态（compiled/content_hash），finalize 需要全量读写 | ⚠️ 职责耦合 |
| 7 | wiki.bak/ 在设计文档中有记录 | 设计文档目录结构图中没有 wiki.bak/，这是后来添加的 | ❌ 文档未同步 |

## Dimension C — 视角盲点

### 设计文档与实际实现的差距

| 项 | 设计文档 | 实际代码 |
|----|---------|---------|
| cache 用途 | "内容摘要缓存" — 暗示会被读取 | 只写不读，get_cached_content 是死函数 |
| wiki.bak/ | 不在目录结构图中 | 存在于 compiler.py:_backup_wiki |
| pending.json 生命周期 | "当前待编译清单" | finalize 后不清理，残留过期数据 |
| pyyaml 依赖 | pyproject.toml 列出 pyyaml | 实际用手写 key:value 分割，未使用 pyyaml |
| data_dir_name | 设计文档未说明如何配置 | config.json 中 data_dir_name 字段决定 |

### 数据流断裂点

```
compile --prepare
    ├─→ 写入 pending.json    ✅（但 finalize 不读取）
    ├─→ 写入 cache/<id>.txt  ❌（从未被读取）
    └─→ 输出 compile prompt  ✅

compile --finalize
    ├─→ 读取 wiki/concepts/  ✅
    ├─→ 写入 backlinks.json  ⚠️（与 graph.json 冗余）
    ├─→ 写入 graph.json      ✅
    ├─→ 重写 index.jsonl     ⚠️（仅为更新 compiled 字段）
    └─→ 追加 history.jsonl   ⚠️（只读最后一行）
```

## Dimension D — 风险压测

### 场景 1：编译中途 Agent 崩溃

- pending.json 残留（过期数据）
- wiki/concepts/ 可能有 Agent 写了一半的 md 文件
- index.jsonl 中对应条目仍为 `compiled=false`
- 下次 compile 重新处理，但半成品 concepts 不会被清理
- **风险**：概念重复（旧的半成品 + 新的完整版共存）

### 场景 2：条目超过 500

- `write_index` 全量读写 index.jsonl → O(n) 每次操作
- finalize 读取所有 entries 查找匹配的 source ID → O(concepts × entries)
- cache/ 目录积累大量 txt 文件，每次 compile 不清理旧文件
- **风险**：性能退化，磁盘浪费

### 场景 3：多次连续编译

- wiki.bak/ 只保留一份，第二次覆盖第一次
- history.jsonl 无限追加
- cache/ 文件不断累积（旧条目的 cache 永远不会被删除）
- **风险**：无法回滚到更早版本，磁盘浪费

## 最高优先级改进项 (Top 5)

| 优先级 | 问题 | 影响 | 改进方案 |
|--------|------|------|---------|
| **P0** | cache/ 只写不读 | 死数据，磁盘浪费 | 删除 `cache_content`、`get_cached_content`、`raw/cache/` 目录创建逻辑。content_preview 在编译 prompt 中已内联使用 |
| **P1** | pending.json finalize 后不清理 | 过期数据残留 | finalize 完成后删除 pending.json，或在 compile 开始时清理上次的 compile/artifacts |
| **P1** | backlinks.json 与 graph.json 冗余 | 两份数据可能不一致 | 保留 graph.json，backlinks 运行时从 graph.json 计算（在 `_load_backlinks` 中） |
| **P2** | history.jsonl 只追加不旋转 | 无限增长 | 限制保留最近 N 条记录（如 100 条），或 finalize 时截断 |
| **P2** | 设计文档未包含 wiki.bak/ | 实现与文档不一致 | 更新设计文档 9.2 节目录结构图 |

## 建议的精简数据结构

```
knowledge-base-data/
├── raw/
│   └── index.jsonl          ← 索引清单（唯一核心数据）
├── wiki/
│   ├── index.md             ← 知识地图
│   ├── graph.json           ← 知识图谱（含反向链接关系）
│   ├── concepts/            ← 概念条目
│   └── categories/          ← 分类索引
├── wiki.bak/                ← 编译前备份
└── compile/
    └── history.jsonl        ← 编译历史（限制 100 条）
```

**删除**：
- `raw/cache/` — 死数据
- `wiki/backlinks.json` — 从 graph.json 运行时计算
- `compile/pending.json` — finalize 后清理
