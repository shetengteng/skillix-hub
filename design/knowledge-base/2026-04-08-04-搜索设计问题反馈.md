---

## 1. 核心问题：谁在使用搜索？

Knowledge Base Skill 的设计原则明确写道：

> **Agent 即编译器** — 利用 Cursor/Claude Code 自身的 LLM 能力

> **渐进式披露** — 按需加载知识，不浪费 Agent 上下文窗口

但当前 `kb search` 的实现与这两条原则矛盾：

| 对比维度 | 渐进式披露设计（§7） | `kb search` 实现 |
|---------|---------------------|------------------|
| 核心用户 | LLM Agent | 像是给人用的 |
| 发现机制 | Agent 读 index.md → 理解 → 自主导航 | 程序遍历 JSONL + 概念文件 |
| 语义能力 | 利用 Agent 的 LLM 理解力 | 纯字符串 `in` 匹配 |
| 使用 Wiki 结构 | ✅ 利用分类/概念/图谱层级 | ❌ 绕过整个 Wiki 层，暴力扫描 |
| 中英文/近义词 | Agent 天然理解 | 完全不支持 |

---

## 2. 具体问题分析

### 2.1 关键词匹配精度不足

`_match_score` 的逻辑是 `query.lower() in title.lower()`：

- ❌ "记忆" 搜不到 "Memory 四层架构"
- ❌ "持久化" 搜不到 "存储架构"
- ❌ "LLM 编译" 搜不到 "Agent 即编译器模式"
- ❌ "自动化测试" 搜不到 "Playwright 浏览器自动化"

对于 Agent 来说，这些查询的意图是明确的，但程序化匹配无法理解。

### 2.2 绕过了 Wiki 结构

当前 search 的数据流：

```
kb search "query"
    → 遍历 raw/index.jsonl（原始索引）
    → 遍历 wiki/concepts/*.md（全文扫描）
    → 字符串匹配 → 排序输出
```

完全没有利用：
- `wiki/index.md`（知识地图 — 分类结构）
- `wiki/categories/*.md`（分类索引）
- `wiki/graph.json`（知识图谱 — 概念关联）
- `wiki/backlinks.json`（反向链接）

### 2.3 与 browse 形成矛盾的并行系统

设计文档 §7 的典型交互流程：

```
Agent 需要了解 Memory Skill 的架构
  → kb browse → 看到 skill-design 分类
  → kb browse skill-design → 找到 "Memory 四层架构"
  → kb read concept-memory-architecture → 获取概念条目
```

这个流程是 **Agent-native** 的 — Agent 用自己的理解力在层级中导航。

而 `kb search "memory architecture"` 是 **Human-native** 的 — 把 Agent 降级为一个接收搜索结果的被动角色。

---

## 3. 根因分析

设计文档 §8.2 的决策：

> **为什么不用向量数据库？** 知识库的规模通常在数十到数百条目，关键词搜索 + 标签筛选已经足够。

这个假设有两层问题：

1. **"足够"的标准是什么？** — 对人类用户可能足够，但对 Agent 来说，让 Agent 自己读 index.md 做语义匹配远比程序化关键词匹配更准确
2. **选择了错误的替代方案** — 放弃向量数据库的结论是合理的（规模小），但替代方案不应该是"关键词匹配"，而应该是"让 Agent 直接利用 Wiki 结构"

---

## 4. 改进方向

### 方案 A：Agent-native 搜索（推荐）

取消独立的程序化 search，让 Agent 通过 browse/read 自主导航：

```
Agent 需要搜索 →
  Step 1: 读取 wiki/index.md（知识地图，~500 tokens）
  Step 2: Agent 根据查询意图，选择相关分类
  Step 3: 读取 wiki/categories/<category>.md
  Step 4: Agent 选择具体概念 → kb read <concept-id>
```

**优势**：
- 利用 Agent 的语义理解力，精度远高于关键词匹配
- 复用已有的渐进式披露层级，不需要维护两套系统
- 中英文、近义词、模糊意图全部由 Agent 的 LLM 处理

**SKILL.md 中的引导**：
```
当需要查找特定知识时：
1. `kb browse` — 查看知识地图，了解有哪些分类和概念
2. `kb browse <category>` — 深入感兴趣的分类
3. `kb read <concept-id>` — 读取具体概念
4. `kb graph --center <id>` — 查看关联概念
```

### 方案 B：轻量辅助索引

保留 search 命令，但重新定位为"辅助索引"，输出不是最终结果，而是引导：

```
kb search "记忆" →
  输出: "根据索引元数据，以下分类/概念可能相关：
         - 分类 memory (4个概念)
         - 概念 hooks-agent-collaboration-pattern
         建议执行 kb browse memory 深入查看"
```

### 方案 C：保留但升级匹配算法

如果一定要保留程序化搜索，至少需要：
- 中文分词（jieba 等）
- 同义词映射
- TF-IDF 或 BM25 排序
- 利用 graph.json 做关联扩展

**不推荐** — 增加依赖和复杂度，且效果仍不如方案 A。

---

## 5. 总结

```
⚠️ Completeness: High
Uncovered blind spots: 无
最关键的未验证前提: "关键词匹配对 Agent 来说足够" — 这个前提不成立
如果这个前提是错的: search 功能需要重新设计，
                    应该从"程序化匹配"转向"Agent 自主导航"
```

**结论**：设计文档第7节的渐进式披露方案本身就是最好的"搜索"方案。`kb search` 应该被重新定义为引导 Agent 使用渐进式披露路径的入口，而非一个独立的程序化搜索引擎。
