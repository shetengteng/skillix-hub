# Browser Workflow Learner — 可行性分析

> 日期：2026-02-23
> 阶段：Phase 2 可行性分析
> 状态：待讨论

## 1. 需求摘要

用户在浏览器中执行重复性工作流（如：登录某系统 → 导航到某页面 → 填写表单 → 提交、部署配置、环境设置等），希望 AI Agent 能够**学习这些操作序列**，在下次执行相同任务时**自动重放**，同时支持**参数化**——即操作流程相同但输入值不同（如不同的用户名、不同的查询条件）。

**关键需求**：录制完成后，自动**生成一个独立的、可复用的 Skill**（包含 SKILL.md、tool.js 等完整结构），而非仅保存工作流 JSON。生成的 Skill 可以被安装到 `~/.cursor/skills/` 或项目的 `.cursor/skills/` 中，下次用户或 Agent 直接通过自然语言触发使用。

核心问题：如何录制用户的浏览器操作、抽象为可重放的工作流模板、并在重放时注入不同参数、最终生成独立 Skill。

## 2. 技术可行性

### 2.1 关键技术点分析

#### ① 操作录制：如何捕获用户的浏览器操作

| 方案 | 原理 | 成熟度 | 优缺点 |
|------|------|--------|--------|
| **CDP Input 域监听** | 通过 `Input.dispatchMouseEvent` / `Input.dispatchKeyEvent` 的反向监听，捕获用户的鼠标点击、键盘输入 | 中 | CDP 本身不提供 Input 事件的被动监听，需要注入 JS 监听 DOM 事件 |
| **DOM 事件注入监听** | 在页面中注入 JS，监听 `click`、`input`、`change`、`submit`、`keydown` 等 DOM 事件 | 高 | 成熟方案，Playwright Codegen 就是这个思路。可获取事件目标的 selector |
| **Playwright 调用层包装** | 在 Playwright Skill 的 tool.js 调用层记录每次命令和参数 | 高 | 最简单，但只能录制通过 Playwright CLI 发起的操作，无法录制用户手动操作 |
| **CDP Page 域 + DOM 域** | 通过 `Page.domContentEventFired`、`DOM.attributeModified` 等事件推断操作 | 低 | 信息不完整，难以还原具体用户操作 |

**结论**：方案 B（DOM 事件注入监听）是最可行的录制方式，也是 Playwright Codegen 和 Selenium IDE 等成熟工具的核心方案。

#### ② 元素定位：如何在重放时找到同一个元素

| 策略 | 稳定性 | 说明 |
|------|--------|------|
| CSS Selector | 中 | 页面结构变化时可能失效 |
| XPath | 中 | 同上 |
| 文本内容匹配 | 高 | `text="登录"` 比结构选择器更稳定 |
| `data-testid` 属性 | 最高 | 需要页面配合，不通用 |
| **多策略组合** | 高 | 同时记录 CSS、文本、aria-label，重放时按优先级尝试 |

**结论**：采用多策略组合定位。录制时为每个操作目标生成多个定位器（CSS selector、文本、aria-label、相对位置），重放时按稳定性优先级依次尝试。这也是 Playwright Codegen 的做法。

#### ③ 参数化：如何识别和替换可变输入

| 方案 | 说明 | 复杂度 |
|------|------|--------|
| **手动标记** | 用户录制后，手动指定哪些输入是参数 | 低 |
| **LLM 推断** | 录制完成后，LLM 分析操作序列，自动识别哪些是固定操作、哪些是可变参数 | 中 |
| **对比学习** | 录制同一流程两次（不同参数），自动 diff 出参数位置 | 中 |
| **模板语法** | 在重放时通过 `{{变量名}}` 语法注入参数 | 低 |

**结论**：推荐 **LLM 推断 + 模板语法** 组合。录制完成后，LLM 分析操作序列，自动将输入值抽象为参数（如用户名、密码、搜索词），生成模板。重放时用户提供参数值或由 LLM 根据上下文填充。

#### ④ 等待与同步：如何处理页面加载和异步操作

已有方案：Playwright Skill 的 `waitFor` 命令支持等待元素出现、消失、文本变化等。录制时记录每步操作前的页面状态（URL、关键元素存在性），重放时作为前置等待条件。

#### ⑤ Skill 生成：如何从工作流自动生成独立 Skill

| 维度 | 分析 |
|------|------|
| **目录结构** | 参考 skill-builder 的脚手架规范，生成 `SKILL.md` + `tool.js` + `lib/` + `package.json` |
| **SKILL.md** | LLM 根据工作流步骤自动生成描述、触发词、使用示例 |
| **tool.js** | 模板化生成，内嵌工作流 JSON，调用 Playwright Skill 执行重放 |
| **参数暴露** | 工作流中的 `{{param}}` 自动成为 Skill 的输入参数，写入 SKILL.md 和 CLI 帮助 |
| **安装** | 生成后自动复制到 `~/.cursor/skills/<name>/` 或 `.cursor/skills/<name>/` |

**结论**：完全可行。skill-builder 已有脚手架模板，生成的 Skill 本质是一个「工作流执行器」，内嵌录制的步骤 JSON，运行时调用 Playwright 命令逐步执行。LLM 负责生成人类可读的 SKILL.md 文档。

**生成的 Skill 结构示例**：

```
~/.cursor/skills/deploy-staging/
├── SKILL.md          # LLM 生成的文档（描述、触发词、参数说明）
├── tool.js           # 工作流执行器（内嵌步骤 JSON）
├── workflow.json     # 原始工作流数据
└── package.json      # 依赖声明（仅 playwright skill 路径引用）
```

**SKILL.md 示例**：

```markdown

description: |
  自动化部署到 Staging 环境。通过浏览器登录 CI/CD 平台，
  选择分支，触发部署流水线。

| 参数 | 说明 | 必填 |
|------|------|------|
| branch | 部署分支 | 是 |
| version | 版本号 | 否 |

## 使用

\`\`\`bash
node ~/.cursor/skills/deploy-staging/tool.js run '{"branch":"release/2.3","version":"2.3.1"}'
\`\`\`

- 发布到测试环境
```

### 2.2 已知限制和约束

| 限制 | 影响 | 缓解方案 |
|------|------|----------|
| **需要 Playwright 浏览器** | 必须通过 Playwright Skill 启动的浏览器才能录制 | 文档说明，启动时自动检查 |
| **SPA 页面路由** | 单页应用的路由变化不触发 `navigate` 事件 | 监听 `popstate` 和 `hashchange` |
| **iframe / Shadow DOM** | 跨 iframe 和 Shadow DOM 的操作录制较复杂 | Phase 1 不支持，后续迭代 |
| **验证码 / 二次认证** | 无法自动化 | 录制时标记为"手动步骤"，重放时暂停等待用户操作 |
| **动态生成的选择器** | React/Vue 等框架生成的 class 名不稳定 | 多策略定位 + 文本优先 |
| **跨标签页操作** | 多标签页工作流录制复杂 | Phase 1 仅支持单标签页 |

### 2.3 原型验证

核心可行性已由现有 Skill 验证：

1. **CDP 连接**：api-tracer 已证明可通过 CDP 连接 Playwright 浏览器实例
2. **DOM 事件注入**：Playwright 的 `evaluate` 命令可在页面中注入任意 JS
3. **操作重放**：Playwright 的 `click`、`type`、`fillForm`、`navigate` 等命令覆盖了所有常见操作
4. **数据持久化**：api-tracer 的 session store 和 behavior-prediction 的 pattern store 提供了成熟的存储模式

## 3. 产物形态对比：录制后生成什么

录制完成后，工作流数据需要以某种形态持久化并供后续使用。以下对比 4 种产物形态：

### 形态 A：独立 Skill（完整目录结构）

生成一个符合 skillix-hub 规范的独立 Skill 目录，包含 `SKILL.md`、`tool.js`、`workflow.json`、`package.json`。

```
~/.cursor/skills/deploy-staging/
├── SKILL.md          # 描述、触发词、参数说明
├── tool.js           # 工作流执行器
├── workflow.json     # 步骤数据
└── package.json
```

**使用方式**：
- Cursor 自动识别为 Skill，出现在 Agent Skills 列表中
- 用户通过自然语言触发：「部署到 staging」
- Agent 读取 SKILL.md 后自动调用 `tool.js run '{"branch":"release/2.3"}'`

### 形态 B：Cursor Rule 文件（.mdc）

生成一个 `.cursor/rules/` 下的 Rule 文件，包含工作流描述和执行指令。

```
.cursor/rules/deploy-staging.mdc
```

**使用方式**：
- Cursor 自动加载 Rule，Agent 在相关上下文中自动遵循
- Rule 中描述操作步骤，Agent 按步骤调用 Playwright 命令

### 形态 C：纯 JSON 工作流文件

仅保存工作流 JSON 到数据目录，由 browser-workflow-learner 统一管理和重放。

```
~/.cursor/skills/browser-workflow-learner/workflows/deploy-staging.json
```

**使用方式**：
- 通过 browser-workflow-learner 的 `replay` 命令执行
- `node skills/browser-workflow-learner/tool.js replay '{"id":"deploy-staging","params":{...}}'`

### 形态 D：Playwright 测试脚本（.js）

生成一个标准的 Playwright 测试/自动化脚本。

```
~/.cursor/skills/browser-workflow-learner/scripts/deploy-staging.js
```

**使用方式**：
- 直接 `node deploy-staging.js --branch release/2.3` 执行
- 或通过 Playwright 的 `runCode` 命令执行

### 对比表

| 维度 | A. 独立 Skill | B. Cursor Rule | C. 纯 JSON | D. Playwright 脚本 |
|------|--------------|----------------|------------|-------------------|
| **Cursor 集成度** | 最高（自动识别为 Skill） | 高（自动加载 Rule） | 低（需通过 CLI） | 低（需手动执行） |
| **自然语言触发** | 是（SKILL.md 描述触发词） | 是（Rule 描述触发条件） | 否 | 否 |
| **参数化** | 好（CLI 参数 + SKILL.md 文档） | 中（Rule 中描述，Agent 推断） | 好（JSON 模板） | 好（命令行参数） |
| **可移植性** | 高（独立目录，可分享） | 低（绑定 Cursor） | 中（依赖本 Skill） | 最高（标准 JS 文件） |
| **维护性** | 好（结构清晰） | 差（Rule 文本难以维护复杂逻辑） | 好 | 好 |
| **生成复杂度** | 中（需生成多文件 + LLM 写文档） | 低（单文件） | 最低（直接保存） | 中（需代码生成） |
| **复用性** | 最高（可安装到任何项目） | 低（项目级） | 中（依赖本 Skill） | 高（标准脚本） |
| **Agent 自主调用** | 最好（SKILL.md 指导 Agent） | 好（Rule 指导 Agent） | 需要 Agent 知道 CLI | 需要 Agent 知道脚本路径 |
| **适合场景** | 高频复用的标准化流程 | 项目级的操作规范 | 临时或低频工作流 | 需要脱离 Cursor 使用 |

### 推荐策略：分层产物

不同场景使用不同产物形态，由用户或 LLM 自动判断：

| 场景 | 推荐产物 | 理由 |
|------|----------|------|
| 高频复用、跨项目（如部署、环境配置） | **A. 独立 Skill** | 最高复用性和集成度 |
| 项目级操作规范（如代码审查流程） | **B. Cursor Rule** | 自动加载，无需手动触发 |
| 一次性或低频操作 | **C. 纯 JSON** | 最轻量，无需生成额外文件 |
| 需要脱离 Cursor 使用 | **D. Playwright 脚本** | 最高可移植性 |

**默认行为**：录制完成后先保存为 JSON（形态 C），用户可随时通过 `generate` 命令升级为独立 Skill（形态 A）或导出为 Playwright 脚本（形态 D）。

## 4. 录制方案对比

### 方案 A：纯 CDP 录制 + Playwright 重放

**架构**：通过 CDP 连接浏览器，注入 JS 监听 DOM 事件录制操作序列，重放时调用 Playwright 命令。

```
录制：CDP → 注入 JS → 监听 DOM 事件 → 操作序列 → JSON
重放：JSON → 解析步骤 → Playwright CLI 命令 → 执行
```

| 优点 | 缺点 |
|------|------|
| 录制用户真实操作，最自然 | 需要维护 JS 注入脚本 |
| 不依赖用户使用 Playwright CLI | 元素定位需要额外处理 |
| 可录制任意页面操作 | 录制的操作可能包含噪音（无关点击等） |

### 方案 B：Playwright CLI 包装层录制 + 直接重放

**架构**：在 Playwright Skill 的 tool.js 调用层加一个录制中间件，记录每次 CLI 调用。重放时直接回放 CLI 命令。

```
录制：用户通过 Playwright CLI 操作 → 中间件记录命令 → JSON
重放：JSON → 依次执行 Playwright CLI 命令
```

| 优点 | 缺点 |
|------|------|
| 实现最简单，无需 JS 注入 | 只能录制通过 CLI 发起的操作 |
| 录制数据干净，无噪音 | 用户必须通过 Playwright CLI 操作（不自然） |
| 元素定位直接复用 ref/selector | 无法录制用户手动在浏览器中的操作 |
| 重放时命令格式完全一致 | 依赖 Playwright Skill 的 snapshot ref 机制 |

### 方案 C：混合模式（推荐）

**架构**：支持两种录制模式，用户可选择：

1. **观察模式**（方案 A）：注入 JS 监听用户手动操作
2. **命令模式**（方案 B）：记录 Playwright CLI 调用

两种模式生成统一的工作流 JSON 格式，重放逻辑相同。

| 优点 | 缺点 |
|------|------|
| 灵活，覆盖两种使用场景 | 实现复杂度较高 |
| 观察模式适合学习用户行为 | 需要维护两套录制逻辑 |
| 命令模式适合 Agent 自动化 | — |

### 对比表

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 实现复杂度 | 中 | 低 | 高 |
| 录制自然度 | 高（用户正常操作） | 低（必须用 CLI） | 高 |
| 数据质量 | 中（有噪音） | 高（干净） | 高 |
| Agent 集成 | 中 | 高 | 高 |
| 维护成本 | 中 | 低 | 高 |
| 扩展性 | 高 | 中 | 高 |

## 5. 推荐方案

**推荐方案 B（Playwright CLI 包装层录制），Phase 1 先实现，后续迭代加入方案 A 的观察模式。**

理由：

1. **使用场景匹配**：这个 Skill 的核心用户是 AI Agent，Agent 本身就是通过 Playwright CLI 操作浏览器的。录制 Agent 的操作序列比录制用户手动操作更实用。
2. **实现成本低**：只需在 Playwright Skill 的调用链上加一层记录，无需 JS 注入。
3. **数据质量高**：CLI 命令天然结构化，无需清洗噪音。
4. **重放可靠性高**：录制和重放使用相同的命令格式，减少转换误差。
5. **渐进式扩展**：Phase 2 可以加入观察模式，两种模式共享同一工作流格式和重放引擎。

### CLI 命令设计

```bash
# 开始录制
node skills/browser-workflow-learner/tool.js record '{"name":"登录后台"}'

# 停止录制（自动保存工作流）
node skills/browser-workflow-learner/tool.js stop '{}'

# 列出已保存的工作流
node skills/browser-workflow-learner/tool.js list '{}'

# 查看工作流详情
node skills/browser-workflow-learner/tool.js show '{"id":"wf-001"}'

# 重放工作流（带参数）
node skills/browser-workflow-learner/tool.js replay '{"id":"wf-001","params":{"username":"admin","password":"123456"}}'

# 删除工作流
node skills/browser-workflow-learner/tool.js delete '{"id":"wf-001"}'

# 参数化分析（LLM 分析操作序列，提取参数）
node skills/browser-workflow-learner/tool.js analyze '{"id":"wf-001"}'

# 生成独立 Skill（核心命令）
node skills/browser-workflow-learner/tool.js generate '{"id":"wf-001","skillName":"deploy-staging","target":"~/.cursor/skills/deploy-staging"}'
```

`generate` 命令的执行流程：

1. 读取工作流 JSON
2. 如果尚未参数化，先执行 `analyze` 提取参数
3. 生成 `SKILL.md`（LLM 根据步骤生成描述、触发词、参数说明）
4. 生成 `tool.js`（模板化的工作流执行器，内嵌重放逻辑）
5. 复制 `workflow.json`（原始步骤数据）
6. 生成 `package.json`
7. 复制到 `target` 目录（默认 `~/.cursor/skills/<skillName>/`）

### 工作流 JSON 格式

```json
{
  "id": "wf-001",
  "name": "登录后台系统",
  "description": "自动登录 admin 后台",
  "params": [
    { "id": "username", "label": "用户名", "type": "text", "required": true },
    { "id": "password", "label": "密码", "type": "password", "required": true }
  ],
  "steps": [
    {
```

## 6. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| **元素定位失效** | 高 | 页面更新后 selector 失效，重放失败 | 多策略定位 + 重放时 snapshot 重新匹配 |
| **页面结构变化** | 中 | 目标网站改版导致工作流失效 | 重放失败时提示用户重新录制 |
| **异步等待不足** | 中 | 页面未加载完就执行下一步 | 每步操作前自动 waitFor |
| **敏感数据泄露** | 高 | 密码等敏感信息存储在工作流 JSON 中 | 参数化后敏感值不存储，运行时注入 |
| **Playwright 依赖** | 低 | 必须先安装 Playwright Skill | 启动时检查，给出安装指引 |
| **跨域限制** | 低 | 某些页面限制 JS 注入（仅影响未来的观察模式） | Phase 1 不涉及 |

## 7. 实现计划

| Phase | 内容 | 预估工作量 |
|-------|------|-----------|
| Phase 1 | CLI 包装层录制 + JSON 存储 + 基础重放 | 2-3 天 |

| Phase 3 | **Skill 生成器**（generate 命令，生成独立可复用 Skill） | 2-3 天 |

## 8. 结论

**可行。**

核心依赖（Playwright Skill、CDP 连接、JSON 存储、skill-builder 脚手架）均已在项目中验证。推荐从方案 B（CLI 包装层录制）开始，Phase 1-3 可在 5-8 天内完成包含 Skill 生成能力的 MVP。

关键优势：
- 复用 Playwright Skill 的 48 个浏览器操作命令
- 复用 api-tracer 的 CDP 连接和 browser-state 共享机制
- 参数化由 LLM 推断，无需用户手动标记
- **录制完成后自动生成独立 Skill**，可安装到全局或项目级使用
- 工作流 JSON 格式清晰，易于理解和编辑
- 生成的 Skill 遵循 skillix-hub 标准结构，可被 Cursor 自动识别

**端到端流程**：
```
录制操作 → 保存工作流 JSON → LLM 参数推断 → 生成独立 Skill → 安装到 ~/.cursor/skills/
                                                                    ↓
                                                    下次用户说"部署到 staging"
                                                    Agent 自动调用生成的 Skill
```

建议先确认命名和技术方案后进入 Phase 3 设计文档阶段。
