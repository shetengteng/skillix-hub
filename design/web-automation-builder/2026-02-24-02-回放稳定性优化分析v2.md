# 回放稳定性优化分析 v2

> 日期：2026-02-24
> 前版：`2026-02-24-01-回放稳定性优化分析.md`（基于具体失败案例的底层修补）
> 本版：从架构视角重新梳理，以 LLM-Native 自愈为核心方向，工具层强化为补充

---

## 1. 执行摘要

### 问题

`web-automation-builder` 录制生成的 Skill，其 `tool.js` 是一个**确定性哑脚本**——单一 locator、失败即终止、无恢复接口。但这个脚本的执行者是 **LLM**，它具备视觉理解和推理能力，却在工具失败时得不到任何有效上下文，只能从零开始手动补救。

这导致一个 10 步的自动化工作流，在第 2 步失败后需要约 15 次工具调用才能完成。

### 根本矛盾

```
工具设计哲学：确定性脚本（假设执行者是哑程序）
实际执行者：  LLM（具备视觉理解 + 推理能力）
```

### 解决思路

**不是**：给工具加更多规则（fallback 链、特定 CSS 类名、特殊字符处理……）

**而是**：重新定义工具与 LLM 执行者之间的**协作接口**，让工具专注于快速路径，失败时提供结构化上下文，LLM 负责处理边界情况后再恢复自动化执行。

---

## 2. 架构现状与目标

### 2.1 现状

```
录制 → workflow.json ─→ tool.js（哑脚本）─→ 执行失败
                                                    ↓
                                          LLM 拿到裸 error message
                                          从零开始手动逐步操作
```

**断层点**：工具失败时，LLM 不知道：
- 这步的**操作意图**是什么
- 页面**当前状态**如何
- 从哪步可以**安全恢复执行**

### 2.2 目标架构

```
录制 → workflow.json（意图 + locator）─→ tool.js（快速路径）
                                                ↓ 成功
                                          任务完成 ✓
                                                ↓ 失败
                                    返回结构化失败上下文：
                                    { intent, allLocators, error, recovery }
                                                ↓
                                    LLM：snapshot → 视觉定位 → 手动执行
                                                ↓
                                    tool.js startFrom N+1（恢复自动化）
```

**工具职责**：快速路径 + 结构化失败上下文提供者
**LLM 职责**：视觉推理 + 边界情况处理 + 恢复指令

### 3.1 改动 A：workflow JSON — 增加 `intent` 字段

**涉及文件**：`skills/web-automation-builder/SKILL.md`（LLM 分析提示词）

**变更内容**：在「LLM 分析 rawEvents 的要求」第 3 条强化意图捕获：

```javascript

// 修复后：读取 step.waitAfter
if (step.waitAfter) {
  const w = step.waitAfter;
  if (w.type === 'selector')   execPW(pw, 'waitFor', { selector: w.value, timeout: w.timeout || 10000 });
  if (w.type === 'url')        execPW(pw, 'waitFor', { url: w.value, timeout: w.timeout || 15000 });
  if (w.type === 'time')       execPW(pw, 'waitFor', { time: w.value });
  if (w.type === 'networkIdle') execPW(pw, 'waitFor', { networkIdle: true, timeout: w.timeout || 30000 });
```

#### 失败时返回结构化上下文

```javascript
// 当前
results.push({ ..., success: false, error: e.message });
return success({ completed: false, failedAt: i + 1, results });

// 修改后
const failedStep = wf.steps[i];
results.push({ step: i + 1, command: cmd, description: failedStep.description, success: false, error: e.message });
return success({
  completed: false,
  failedAt: i + 1,
  failedStep: {
    intent: failedStep.intent || failedStep.description,
    allLocators: failedStep.locators,
    error: e.message,
  },
  completedSteps: results.filter(r => r.success),
  recovery: {
    hint: `使用 playwright snapshot 查看当前页面，根据 intent 手动定位元素并执行，完成后调用 run startFrom: ${i + 2} 继续`,
    nextStartFrom: i + 2,
  },
});
```

### 3.3 改动 C：`SKILL.md.tpl` — 增加 LLM 执行协议

**涉及文件**：`skills/web-automation-builder/templates/SKILL.md.tpl`

在生成的 Skill SKILL.md 中增加：

```markdown
## LLM 执行协议

### 正常路径
调用 `node tool.js run '{...}'`，`completed: true` 时任务完成。

### 自愈路径（run 返回 completed: false 时）

1. 读取 `failedStep.intent`，理解这步要实现什么目标
2. 调用 playwright snapshot 获取当前页面状态：
   `node ~/.cursor/skills/playwright/tool.js snapshot '{}'`
3. 在 snapshot 的可访问性树中，根据 intent 识别目标元素
4. 使用 playwright 手动执行该步（click/fill/select 等）
5. 确认步骤效果（页面状态符合预期）
6. 调用 `run '{"startFrom": <recovery.nextStartFrom>, ...}'` 从下一步恢复自动化

### 通用等待原则
任何步骤前如果页面有加载指示（spinner、骨架屏、进度条），
先用 `waitFor` 等待加载完成再操作，不要硬等固定时间。
```

### 3.4 改动 D：`SKILL.md` — 更新录制分析要求

**涉及文件**：`skills/web-automation-builder/SKILL.md`

在现有 6 条分析要求后，修改第 3 条并增加第 7、8 条：

```
3. **捕获操作意图**（关键，影响 LLM 自愈能力）：
   intent = [在什么上下文中] + 操作目标 + [期望产生的页面变化]

7. **识别 UI 容器模式**：
   - 点击按钮 → 弹出 modal：后续步骤 locator 限定在 modal 内
   - 下拉菜单展开：选项 locator 限定在 dropdown 内
   - 表格行操作：记录行锚定文本（行内唯一标识列的文本）

8. **生成等待条件**（waitAfter 字段，优先于固定时间等待）：
   - 导航后：等待目标页面关键元素出现（type: selector）
   - 点击触发 modal：等待 modal 容器出现
   - 表单提交后：等待加载状态结束（type: selectorGone 或 networkIdle）
```

## 4. 工具层强化（补充，可按需实施）

> 以下改动在「快速路径」上增加更多容错规则，作为第 3 节的补充。优先级低于第 3 节。

### P0：ID 特殊字符处理

UUID 类 ID 直接用 `#id` 会导致 CSS 选择器语法错误，改用属性选择器：

```javascript
// tool.js.tpl 中
if (loc.id) {
  args.selector = /^[a-zA-Z_][\w-]*$/.test(loc.id)
    ? `#${loc.id}`
    : `[id="${loc.id}"]`;
}
```

### P1：Locator 按优先级多路尝试

原设计文档（3.4 节）定义了优先级 `text > role > css`，但 `tool.js.tpl` 未实现。按设计落地：

```javascript
function buildLocatorChain(loc) {
  const chain = [];
  // 按设计文档 3.4 的稳定性顺序
  if (loc.testId)                chain.push({ testId: loc.testId });
  if (loc.ariaLabel || loc.label) chain.push({ label: loc.ariaLabel || loc.label });
  if (loc.placeholder)           chain.push({ placeholder: loc.placeholder });
  if (loc.text)                  chain.push({ text: loc.text });
  if (loc.role)                  chain.push({ role: loc.role, ...(loc.name ? { name: loc.name } : {}) });
  if (loc.id)                    chain.push({ selector: /^[a-zA-Z_][\w-]*$/.test(loc.id) ? `#${loc.id}` : `[id="${loc.id}"]` });
  if (loc.css)                   chain.push({ selector: loc.css }); // 最后尝试 CSS
  return chain;
}

// 执行时遍历 chain，每个失败后尝试下一个
for (const args of chain) {
  try {
    execPW(pw, cmd, args, stepTimeout);
    break; // 成功即退出
  } catch (e) {
    lastError = e;
  }
}
```

### P1：fill 后值验证

SPA 受控组件对合成输入事件处理不稳定，fill 后验证并在必要时降级：

```javascript
// fill 命令执行后
if (cmd === 'type') {
```

### P2：录制时捕获容器上下文（inject.js.tpl）

在 `locators()` 函数中增加容器上下文，帮助 LLM 生成更准确的 intent：

```javascript
function getContainerContext(el) {
  let cur = el.parentElement, depth = 0;
  while (cur && depth < 5) {
    const role = cur.getAttribute('role'), cls = cur.className || '';
    if (role === 'dialog' || cls.includes('modal'))    return { type: 'modal' };
    if (role === 'menu'   || cls.includes('dropdown')) return { type: 'dropdown' };
    if (cur.tagName === 'TR')                          return { type: 'table-row', rowText: cur.textContent?.trim().slice(0, 80) };
    if (cur.tagName === 'FORM')                        return { type: 'form' };
    cur = cur.parentElement; depth++;
  }
  return null;
}
```

## 6. 第一轮测试复盘（2026-02-24）

> 基于 zcp-deploy Skill 的实际 replay 测试，验证了 v2 架构的自愈协议有效性，同时暴露了 3 个工具层缺陷。

### 6.1 测试结果

| 步骤 | 描述 | 结果 | 自愈方式 |
|------|------|------|---------|
| 1 | navigate 到 ZCP 部署页 | 成功 | — |
| 2 | 点击 Operations 菜单 | 超时 | 手动 navigate 到目标 URL 后 startFrom:2 |
| 3 | 点击 Deploy History | 成功 | — |
| 4 | 点击 Redeploy 按钮 | 成功 | — |
| 5 | 填写 Artifact Version | 失败 | 手动用 `[id$="_artifactVersion"]` 定位后 startFrom:6 |
| 6 | 选择 Branch | 失败 | 跳过（弹窗已预填 branch），startFrom:7 |
| 7 | 点击 Deploy 确认 | 成功 | — |

**自愈协议验证**：结构化失败上下文（intent + allLocators + recovery.nextStartFrom）让 LLM 能快速理解问题并手动补救，`startFrom` 参数实现了无缝恢复。全程仅 2 次手动介入。

### 6.2 暴露的问题与根因

#### 问题 A：navigate 后 waitAfter 超时（Step 2）

**现象**：`navigate` 到 `https://zcp-eng.corp.zoom.us/...` 后，`waitAfter: { type: "selector", value: "#zcp-app-center" }` 超时（15s），导致 Step 2 的 `execSync` 超时抛出 `ETIMEDOUT`。

**根因**：
- 页面首次加载可能触发 SSO 跳转（Okta），此时 `#zcp-app-center` 不存在
- `navigate` 的 `waitAfter` 超时被设为 15s，但 SSO + 页面渲染可能需要 30s+
- 更根本的问题：`navigate` 步骤的 `execSync` timeout 和 `waitAfter.timeout` 是两个独立的超时，前者（`stepTimeout`）取的是 `waitAfter.timeout || 60000`，但 `waitAfter` 的执行是在 `navigate` 成功之后的独立调用

**修复方向**：
1. `navigate` 步骤的 `waitAfter.timeout` 应设为更大值（30s+），或在 SKILL.md 分析要求中指导 LLM 对需要认证的页面设置更长超时
2. `tool.js.tpl` 中 `navigate` 后的 `waitAfter` 失败不应直接终止——导航本身已成功，`waitAfter` 失败只是页面还在加载，应降级为 warning 继续执行

#### 问题 B：locator 链中 `role` 无 `name` 导致匹配不精确（Step 5）

**现象**：Step 5 的 locators 有 `{ id: "95dd597c-..._artifactVersion", role: "combobox" }`。`buildLocatorChain` 按优先级先尝试 `role: combobox`，但页面上有多个 combobox，Playwright strict mode 拒绝。然后尝试 `[id="95dd597c-..."]`，Playwright 的 `type` 命令对属性选择器处理异常。

**根因**：
- `role` locator 缺少 `name` 属性来精确匹配（如 `role: combobox, name: "Artifact Version"`）
- 录制时 `inject.js.tpl` 的 `locators()` 函数没有为 `role` 提取关联的 `name`（accessible name）
- `buildLocatorChain` 中 `role` 排在 `id` 前面，但没有 `name` 的 `role` 反而更不精确

**修复方向**：
1. `inject.js.tpl` 录制时为 `role` 提取 `aria-label` 或关联 `<label>` 的文本作为 `name`
2. `buildLocatorChain` 中，无 `name` 的 `role` 应降低优先级，排在 `id` 之后
3. 对 `id` 类型的 locator，直接使用 `[id$="..."]`（后缀匹配）替代完整 ID 匹配，因为 UUID 前缀可能是动态生成的

#### 问题 C：text locator 在页面多处匹配（Step 6）

**现象**：Step 6 的 `text: "dev-2026-feb"` 在页面上出现 14 次（Deploy History 表格每行的 branch 列都有），Playwright strict mode 拒绝。

**根因**：
- `text` locator 是全页面搜索，不限定作用域
- 虽然 workflow 中有 `context: { type: "modal" }`，但 `buildLocatorChain` 完全忽略了 `context` 信息
- 弹窗（modal）内的元素应该限定在 `[role=dialog]` 或 `.modal` 容器内

**修复方向**：
1. `buildLocatorChain` 应读取 `context` 字段，当 `context.type === "modal"` 时，为所有 locator 添加 `[role=dialog] ` 前缀作用域限定
2. 当 `context.type === "table-row"` 时，用 `rowAnchor` 文本定位到具体行
3. 这是最关键的改动——它能同时解决 modal 内元素定位和表格行操作的精确性问题

### 6.3 综合优化方案

基于以上 3 个问题，需要在 `tool.js.tpl` 和 `replayer.js` 中实施以下改动：

#### 改动 1：context-aware locator 作用域限定

```javascript
function buildLocatorChain(loc) {
  const chain = [];
  if (!loc) return chain;

  // 根据 context 构建作用域前缀
  let scopePrefix = '';
  if (loc.context) {
    if (loc.context.type === 'modal')    scopePrefix = '[role=dialog] ';
    if (loc.context.type === 'dropdown') scopePrefix = '[role=menu] ';
  }

  if (loc.testId)                    chain.push({ testId: loc.testId });
  if (loc.ariaLabel || loc.label)    chain.push({ label: loc.ariaLabel || loc.label });
  if (loc.placeholder)               chain.push({ placeholder: loc.placeholder });
  if (loc.text)                      chain.push({ selector: `${scopePrefix}:text("${loc.text}")` });
  if (loc.role && loc.name)          chain.push({ role: loc.role, name: loc.name });
  if (loc.role && !loc.name)         chain.push({ selector: `${scopePrefix}[role="${loc.role}"]` });
  if (loc.id) {
    // 使用后缀匹配，避免动态 UUID 前缀
    const idSuffix = loc.id.includes('_') ? loc.id.split('_').pop() : loc.id;
    chain.push({ selector: `${scopePrefix}[id$="${idSuffix}"]` });
  }
  if (loc.css)                       chain.push({ selector: `${scopePrefix}${loc.css}` });
  return chain;
}
```

#### 改动 2：navigate 后 waitAfter 失败降级为 warning

```javascript
if (stepSuccess) {
  if (step.waitAfter) {
    try {
      execWait(pw, step.waitAfter);
    } catch (waitErr) {
      if (cmd === 'navigate') {
        // navigate 后 waitAfter 失败不终止，降级为 warning
        results.push({ step: i+1, command: cmd, description: step.description,
                       success: true, warning: 'waitAfter timeout: ' + waitErr.message });
        continue;
      }
      // 非 navigate 步骤的 waitAfter 失败也不终止（best-effort）
    }
  }
}
```

#### 改动 3：录制时为 role 提取 accessible name

在 `inject.js.tpl` 的 `locators()` 函数中：

```javascript
// 提取 role 的 accessible name
if (role) {
  const ariaLabel = el.getAttribute('aria-label');
  const labelledBy = el.getAttribute('aria-labelledby');
  const labelEl = labelledBy ? document.getElementById(labelledBy) : null;
  const associatedLabel = el.id ? document.querySelector(`label[for="${el.id}"]`) : null;
  const name = ariaLabel || labelEl?.textContent?.trim() || associatedLabel?.textContent?.trim();
  if (name) loc.roleName = name;
}
```

### 6.5 修复总结

| 问题 | 根因 | 修复 |
