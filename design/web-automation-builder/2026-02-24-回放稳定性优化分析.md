# 回放稳定性优化分析

> 基于 ZCP 部署 Skill 实际回放失败案例的根因分析与优化方案

## 1. 问题现象

使用 `web-automation-builder` 录制生成的 `zcp-deploy` Skill 执行回放时，10 步工作流在第 2 步即失败。Agent 被迫手动逐步操作，过程中反复遇到元素定位失败、多元素歧义、ID 特殊字符、对话框遮挡等问题，共耗费约 15 次工具调用才完成本应自动化的部署流程。

### 失败步骤复盘

| 步骤 | 预期操作 | 失败原因 |
|------|----------|----------|
| Step 2 | 点击部署按钮 | CSS `table tbody tr:nth-of-type(1) td:nth-of-type(15) button span` 失效（表格列数/结构变化） |
| 手动补救 | `getByText('Deploy')` | strict mode violation：匹配到 6 个元素 |
| 手动补救 | `getByRole('button', {name:'Deploy'})` | strict mode violation：匹配到 31 个元素（每行都有 Deploy 相关按钮） |
| 手动补救 | 填写 Job Name 输入框 | ID 含 UUID（`01d1cfad-..._job_name`），`#id` 选择器语法错误 |
| 手动补救 | 点击对话框内 Deploy 按钮 | 顶部 Deploy 按钮被 `ant-modal-content` 遮挡 |

## 2. 根因分析

问题分布在三个层级：**录制层 → 生成层 → 回放层**。

### 2.1 录制层（inject.js.tpl）

**问题 A：CSS 选择器天然脆弱**

`cssSelector()` 函数生成的是基于 DOM 树结构路径的选择器：

```javascript
// 当前实现：逐层拼接 tag:nth-of-type(n)
tag += ':nth-of-type(' + idx + ')';
parts.unshift(tag);
```

这种选择器依赖 DOM 结构的精确位置。页面新增/删除一列、调整布局、甚至 React 重新渲染顺序变化都会导致失效。在 ZCP 案例中，`td:nth-of-type(15)` 说明表格有 15+ 列，任何列的增删都会打破定位。

**问题 B：locator 采集全面但缺少质量评估**

`locators()` 函数采集了 css/text/role/ariaLabel/placeholder/testId/name/id 共 8 种定位信息，但：
- 没有标注每种 locator 的**稳定性评分**
- 没有记录元素在页面中的**唯一性**（是否有多个匹配）
- text 内容截取 100 字符，对于短文本如 "Deploy" 容易产生歧义

**问题 C：未采集上下文信息**

缺少对元素所在容器的上下文记录（如 modal 内、table 内、header 内），导致回放时无法区分同名元素。

### 2.2 生成层（LLM 分析 → workflow JSON）

**问题 D：workflow 步骤只保留单一 locator**

LLM 分析 rawEvents 后，每步只选择了一种定位方式。例如 Step 2 只保留了 `css`，Step 10 只保留了 `text: "Deploy"`。没有生成 fallback 链。

**问题 E：缺少等待条件**

workflow 步骤之间没有 `wait` 字段。只有 replayer 在 navigate 后硬编码 `waitFor({time: 2})`。对于 SPA 页面（如 ZCP 使用 React），DOM 异步渲染完成的时机不确定，固定等待不可靠。

**问题 F：未识别 UI 模式**

LLM 没有识别出 "点击按钮 → 弹出 modal → 在 modal 内操作" 这种常见 UI 模式，导致回放时在 modal 外点击被拦截。

### 2.3 回放层（replayer.js / tool.js.tpl）

**问题 G：locator 选择是单路径 if-else**

```javascript
if (loc.text)             args.text = loc.text;
else if (loc.css)         args.selector = loc.css;
else if (loc.role)        { args.role = loc.role; ... }
// ... 只用第一个匹配的，无 fallback
```

如果首选 locator 失败，整步直接报错，不会尝试其他 locator。

**问题 H：失败即终止，无重试**

```javascript
} catch (e) {
  results.push({ ..., success: false, error: e.message });
  return success({ completed: false, failedAt: i + 1, results });
}
```

一步失败就终止整个工作流。没有重试机制，没有降级策略。

**问题 I：不处理 Playwright strict mode violation**

当 `getByText('Deploy')` 匹配到多个元素时，Playwright 抛出 strict mode violation。回放层没有解析这个错误并尝试更精确的定位。

## 3. 优化方案

### 3.1 录制层优化

#### 3.1.1 增强 locator 采集质量

在 `inject.js.tpl` 的 `locators()` 函数中增加：

```javascript
function locators(el) {
  var base = { /* 现有字段 */ };

  // 新增：唯一性检测
  base.uniqueness = {
    byId: el.id ? document.querySelectorAll('#' + CSS.escape(el.id)).length : 0,
    byText: base.text ? document.querySelectorAll('*').length : -1, // 简化
    byAriaLabel: base.ariaLabel
      ? document.querySelectorAll('[aria-label="' + base.ariaLabel + '"]').length : 0,
    byPlaceholder: base.placeholder
      ? document.querySelectorAll('[placeholder="' + base.placeholder + '"]').length : 0,
    byTestId: base.testId
      ? document.querySelectorAll('[data-testid="' + base.testId + '"]').length : 0,
  };

  // 新增：容器上下文
  base.context = getContainerContext(el);

  return base;
}

function getContainerContext(el) {
  var cur = el.parentElement;
  var depth = 0;
  while (cur && depth < 5) {
    var role = cur.getAttribute('role');
    var cls = cur.className || '';
    // 识别常见容器模式
    if (role === 'dialog' || cls.includes('modal') || cls.includes('Modal'))
      return { type: 'modal', selector: buildContainerSelector(cur) };
    if (role === 'menu' || cls.includes('dropdown') || cls.includes('Dropdown'))
      return { type: 'dropdown', selector: buildContainerSelector(cur) };
    if (cur.tagName === 'FORM')
      return { type: 'form', selector: buildContainerSelector(cur) };
    if (cur.tagName === 'TABLE')
      return { type: 'table', selector: buildContainerSelector(cur) };
    if (cur.tagName === 'HEADER' || role === 'banner')
      return { type: 'header', selector: buildContainerSelector(cur) };
    cur = cur.parentElement;
    depth++;
  }
  return null;
}
```

#### 3.1.2 生成复合选择器

为每个元素生成一个**稳定性排序的 locator 优先级列表**：

```
稳定性排序：data-testid > aria-label > id(无UUID) > placeholder > role+name > text(短且唯一) > css
```

### 3.2 生成层优化

#### 3.2.1 workflow 步骤包含 fallback locator 链

当前 workflow step 的 locators 是扁平对象。改为支持有序 fallback：

```json
{
  "command": "click",
  "description": "点击 Deploy 按钮",
  "locators": {
    "primary": { "selector": ".ant-modal-content button.ant-btn-primary" },
    "fallback": [
      { "role": "button", "name": "Deploy", "within": ".ant-modal-content" },
      { "ariaLabel": "Deploy" },
      { "text": "Deploy" }
    ]
  },
  "context": { "type": "modal", "hint": "ant-modal-content" }
}
```

#### 3.2.2 添加智能等待条件

LLM 分析时应为关键步骤生成 wait 条件：

```json
{
  "command": "click",
  "wait": {
    "type": "selector",
    "value": ".ant-modal-content",
    "timeout": 10000,
    "description": "等待 modal 对话框出现"
  }
}
```

等待条件类型：
- `selector`：等待元素出现
- `text`：等待文本出现
- `url`：等待 URL 变化
- `networkIdle`：等待网络空闲
- `time`：固定等待（最后手段）

#### 3.2.3 LLM 分析提示词增强

在 SKILL.md 的 "LLM 分析 rawEvents 的要求" 中增加：

```
7. **识别 UI 容器模式**：
   - 点击按钮后出现 modal → 后续步骤的 locator 应限定在 modal 内
   - 下拉菜单展开 → 选项的 locator 应限定在 dropdown 内
   - 表格行操作 → locator 应包含行上下文

8. **生成 fallback locator 链**：
   - 每步至少提供 2 种定位方式
   - 优先使用 aria-label、data-testid 等语义化属性
   - CSS 路径选择器作为最后 fallback

9. **生成等待条件**：
   - 导航后：等待目标页面关键元素出现
   - 点击触发 modal 后：等待 modal 容器出现
   - 填写表单后提交：等待加载状态结束
```

### 3.3 回放层优化

#### 3.3.1 Fallback locator 链执行

改造 `buildPlaywrightArgs` 为 `buildLocatorChain`，返回有序的定位尝试列表：

```javascript
function buildLocatorChain(step, params) {
  const cmd = CMD_MAP[step.command] || step.command;
  const chain = [];

  const loc = renderValue(step.locators, params);

  // primary locator
  if (loc.primary) {
    chain.push(buildArgsFromLocator(loc.primary));
  }

  // fallback locators
  if (loc.fallback && Array.isArray(loc.fallback)) {
    for (const fb of loc.fallback) {
      chain.push(buildArgsFromLocator(fb));
    }
  }

  // 兼容旧格式：从扁平 locators 生成多种尝试
  if (chain.length === 0) {
    if (loc.ariaLabel) chain.push({ label: loc.ariaLabel });
    if (loc.testId) chain.push({ testId: loc.testId });
    if (loc.placeholder) chain.push({ placeholder: loc.placeholder });
    if (loc.text) chain.push({ text: loc.text });
    if (loc.css) chain.push({ selector: loc.css });
    if (loc.id) chain.push({ selector: `[id$="${loc.id.split('_').pop()}"]` });
  }

  // 如果有 context，为每个 locator 添加容器限定
  if (step.context?.hint) {
    chain.forEach(args => {
      if (!args.selector?.includes(step.context.hint)) {
        args._withinSelector = step.context.hint;
      }
    });
  }

  return { cmd, chain };
}
```

#### 3.3.2 重试与降级机制

```javascript
function executeStep(step, index, params, playwrightTool, prevCmd) {
  const { cmd, chain } = buildLocatorChain(step, params);

  // 执行等待条件
  if (step.wait) {
    executeWait(step.wait, playwrightTool);
  }

  // 尝试 locator 链
  let lastError = null;
  for (let attempt = 0; attempt < chain.length; attempt++) {
    const args = chain[attempt];
    try {
      const result = execPlaywright(playwrightTool, cmd, args, stepTimeout);
      return { success: true, step: index + 1, command: cmd, usedLocator: attempt };
    } catch (e) {
      lastError = e;
      // 解析 strict mode violation，尝试 .first()
      if (e.message.includes('strict mode violation')) {
        try {
          args._first = true;
          const result = execPlaywright(playwrightTool, cmd, args, stepTimeout);
          return { success: true, step: index + 1, command: cmd, usedLocator: attempt, note: 'used .first()' };
        } catch { /* continue to next locator */ }
      }
    }
  }

  return { success: false, step: index + 1, error: lastError?.message };
}
```

#### 3.3.3 处理特殊字符 ID

当 locator 中有 `id` 字段时，不要直接拼 `#id`（UUID 中的特殊字符会导致 CSS 选择器语法错误），改用属性选择器：

```javascript
// 旧：
if (loc.id) args.selector = `#${loc.id}`;

// 新：
if (loc.id) {
  if (/^[a-zA-Z_][\w-]*$/.test(loc.id)) {
    args.selector = `#${loc.id}`;
  } else {
    args.selector = `[id="${loc.id}"]`;
  }
}
```

## 4. 优先级排序

| 优先级 | 改动 | 影响范围 | 预期效果 |
|--------|------|----------|----------|
| P0 | 回放层 fallback locator 链 | replayer.js, tool.js.tpl | 单一 locator 失败不再终止，尝试备选 |
| P0 | 回放层 ID 特殊字符处理 | replayer.js, tool.js.tpl | 修复 UUID 类 ID 的选择器语法错误 |
| P1 | 录制层 locator 唯一性检测 | inject.js.tpl | 录制时即知道哪些 locator 可靠 |
| P1 | 录制层容器上下文采集 | inject.js.tpl | 区分 modal 内外的同名元素 |
| P1 | 生成层 LLM 提示词增强 | SKILL.md | 生成更健壮的 workflow JSON |
| P2 | 回放层智能等待 | replayer.js | 替代固定 2 秒等待 |
| P2 | 回放层 strict mode 处理 | replayer.js | 多元素匹配时自动取第一个 |
| P3 | 生成层 workflow schema 升级 | generator.js, SKILL.md.tpl | 支持 fallback 和 context 字段 |

## 5. 兼容性

所有改动需**向后兼容**现有 workflow JSON 格式：
- 回放层检测到旧格式（扁平 locators）时，自动从中提取多种 locator 作为 fallback 链
- 新格式（primary + fallback）是可选增强，不是必须
- 已生成的 Skill（如 zcp-deploy）无需重新录制，回放层升级后自动受益

## 6. 补充优化项（2026-02-24 评审后追加）

### 6.1 元素可见性与滚动进视区（P0）

**问题**：Playwright 虽然在交互前会自动 scroll into view，但当元素被 sticky header 或 `overflow:hidden` 容器遮挡时，click 命令会击中遮挡物而非目标元素，且不会抛出明显错误，导致静默失败。

**方案**：回放层在每次 `click` / `fill` 执行前，先调用 `scrollIntoViewIfNeeded`：

```javascript
// 在 execPlaywright 调用前插入
await locator.scrollIntoViewIfNeeded();
await locator.click(/* ... */);
```

### 6.2 加载遮罩检测（selectorGone 等待类型）（P0）

**问题**：Ant Design 的 `ant-spin-spinning`、通用的全屏 Loading Overlay 在覆盖页面时会拦截所有点击。当前等待条件只有 `selector`（等待元素出现），缺少**等待元素消失**的类型。

**方案**：在 workflow 的 `wait` 字段和回放层同时增加 `selectorGone` 类型：

```json
{
  "command": "click",
  "wait": {
    "type": "selectorGone",
    "value": ".ant-spin-spinning",
    "timeout": 15000,
    "description": "等待加载遮罩消失"
  }
}
```

等待条件类型完整列表（补充到 3.2.2）：

| 类型 | 描述 |
|------|------|
| `selector` | 等待元素出现（已有） |
| `selectorGone` | **新增**：等待元素消失（用于加载遮罩、Toast） |
| `text` | 等待文本出现（已有） |
| `url` | 等待 URL 变化（已有） |
| `networkIdle` | 等待网络空闲（已有） |
| `time` | 固定等待，最后手段（已有） |

### 6.3 fill 后值验证与 pressSequentially 降级（P1）

**问题**：React/Vue 受控组件对 Playwright `fill()` 产生的合成 InputEvent 处理不稳定，部分情况下值写入了 DOM 但没有触发组件状态更新，导致后续提交时字段为空。

**方案**：

1. fill 后验证 `input.value`：

```javascript
await page.fill(selector, value);
const actual = await page.$eval(selector, el => el.value);
if (actual !== value) {
  // 降级：清空后逐字符输入
  await page.click(selector, { clickCount: 3 });
  await page.keyboard.type(value, { delay: 30 });
}
```

2. workflow step 增加可选的 `fillStrategy` 字段，LLM 在识别到受控输入框（含 React state 特征）时设为 `pressSequentially`：

```json
{
  "command": "fill",
  "value": "my-job-name",
  "fillStrategy": "pressSequentially"
}
```

### 6.4 失败时自动截图（P1）

**问题**：当前失败结果只记录 `e.message` 字符串。对于 Agent 来说，仅凭错误文本很难判断页面实际状态（是否有遮挡、按钮是否渲染完成）。

**方案**：在 `executeStep` 捕获到最终失败（所有 locator 都尝试完毕）时，自动截图并将路径附加到结果：

```javascript
// 所有 locator 均失败后
const screenshotPath = `./replay-fail-step${index + 1}-${Date.now()}.png`;
await page.screenshot({ path: screenshotPath, fullPage: false });
return { success: false, step: index + 1, error: lastError?.message, screenshot: screenshotPath };
```

### 6.5 表格行级锚定策略（P1）

**问题**：文档 3.1.1 提到了 `table` 类型容器上下文，但核心问题是**如何定位到正确的行**，而非只是知道在 table 内。`td:nth-of-type(15)` 崩溃是因为列变了；同理，行顺序也可能随排序、分页等操作变化，导致 `tr:nth-of-type(n)` 同样不稳定。

**方案**：录制层在采集 table 内按钮的上下文时，额外记录**同行锚定文本**（当前行某列的唯一标识文字）：

```javascript
function getTableRowAnchor(el) {
  const row = el.closest('tr');
  if (!row) return null;
  // 取行内第一个有唯一文本的 td（通常是 name 列）
  const cells = Array.from(row.querySelectorAll('td'));
  for (const cell of cells) {
    const text = cell.textContent.trim();
    if (text && text.length < 60 && text.length > 2) {
      return text;
    }
  }
  return null;
}
```

生成层 LLM 利用此信息生成行级定位：

```json
{
  "context": {
    "type": "table",
    "rowAnchor": "my-service-name",
    "hint": "tr:has(td:text(\"my-service-name\"))"
  }
}
```

回放层使用 `locator.filter` + row anchor 实现行级精确定位，而非依赖位置索引。

### 6.6 录制事件去重与合并（P2）

**问题**：React double-render、快速连击、输入防抖等场景下，`inject.js.tpl` 会记录多余的重复事件（如同一元素的两次连续 click，或 input 事件高频触发导致记录数十条 change 事件）。这会导致 LLM 生成冗余步骤，回放时执行多余操作。

**方案**：在 LLM 分析 rawEvents 的提示词中增加去重规则（补充到 3.2.3 第 10 条）：

```
10. **事件去重合并**：
    - 同一元素的连续多次 click，合并为一次
    - 同一 input 元素的连续 change/input 事件，只保留最后一次（最终值）
    - navigate 后立即发生的 click（< 500ms）视为误触，丢弃
```

### 6.7 步骤后置断言 assert（P2）

**问题**：当前 workflow 步骤之间没有检查点。步骤执行后无法确认是否产生了预期的 DOM 变化。若某步骤操作成功但后置状态未达预期（如 modal 没有弹出），回放层会继续执行下一步，直到后续步骤失败才报错，错误定位困难。

**方案**：workflow step 增加可选 `assert` 字段，支持在步骤执行后验证预期状态：

```json
{
  "command": "click",
  "description": "点击 Deploy 按钮打开对话框",
  "locators": { "primary": { "selector": ".ant-btn-primary span:text('Deploy')" } },
  "assert": {
    "type": "selectorVisible",
    "value": ".ant-modal-content",
    "timeout": 8000,
    "description": "验证 Deploy 对话框已出现"
  }
}
```

断言类型：

| 类型 | 描述 |
|------|------|
| `selectorVisible` | 元素可见 |
| `selectorGone` | 元素消失（用于确认遮罩/Toast 已消失） |
| `urlContains` | URL 包含指定字符串（用于确认导航成功） |
| `inputValue` | 输入框值等于预期 |

断言失败时回放层将该步骤标记为 `assertFailed`（区别于 `locatorFailed`），并触发截图，便于问题分类。

## 8. LLM-Native 回放架构重设计（架构方向）

> 第 3-6 节的优化方案本质上是「用更多规则对抗不确定性」。这是一场持续投入且难以穷举的军备竞赛。
> 本节提出一个根本性的方向调整：**让 LLM 承担回放时的容错工作，而不是把所有容错逻辑编码到工具里**。

### 8.1 当前架构的根本问题

```
录制 → workflow.json（选择器为核心） → tool.js.tpl（哑脚本，确定性执行） → 失败返回 error
                                                                                    ↓
                                                                            LLM 从零开始手动补救
```

**问题：tool.js 和 LLM 之间没有协作接口。** 失败时 LLM 只拿到一行 error message，完全不知道：
- 这一步的**操作意图**是什么
- 页面**当前实际状态**是什么
- 工作流从哪一步可以**安全恢复**

### 8.2 目标架构

```
录制 → workflow.json（意图为核心 + 选择器为提示） → tool.js.tpl（快速路径）
                                                            ↓ 失败时
                                                    返回结构化失败上下文
                                                            ↓
                                                    LLM：snapshot → 定位 → 手动执行
                                                            ↓
                                                    tool.js.tpl（从失败步骤恢复）
```

工具的职责从「负责所有执行」变为「**快速路径 + 失败上下文提供者**」。LLM 利用视觉理解能力处理工具无法处理的边界情况，再通过 `startFrom` 衔接回自动化轨道。

### 8.3 需要改动的四个位置

#### 改动 A：workflow JSON 格式 — 每步增加 `intent` 字段

这是最小代价的改动，只需在 LLM 分析 rawEvents 时额外生成一个字段：

```json
{
  "step": 2,
  "intent": "点击列表第一个部署任务的 Deploy 按钮",
  "command": "click",
  "locators": { "css": "table tbody tr:nth-of-type(1) td:nth-of-type(15) button", "text": "Deploy" }
}
```

`intent` 是**自然语言目标描述**，与 `description`（操作描述）的区别：
- `description`：「点击 Deploy 按钮」（记录了做什么）
- `intent`：「点击列表第一个部署任务的 Deploy 按钮，以打开部署配置对话框」（记录了为什么、在哪个上下文、期望结果）

#### 改动 B：`tool.js.tpl` — 支持 `startFrom` 参数 + 结构化失败上下文

**新增 `startFrom` 参数**：

```bash
# 从第 3 步开始执行（前 2 步已由 LLM 手动完成）
node tool.js run '{"startFrom": 3, "params": {...}}'
```

**失败时返回结构化上下文**（而非裸 error message）：

```json
{
  "completed": false,
  "failedAt": 2,
  "failedStep": {
    "intent": "点击列表第一个部署任务的 Deploy 按钮，以打开部署配置对话框",
    "allLocators": {
      "css": "table tbody tr:nth-of-type(1) td:nth-of-type(15) button",
      "text": "Deploy",
      "role": "button"
    },
    "error": "strict mode violation: locator resolved to 6 elements"
  },
  "completedSteps": [
    { "step": 1, "description": "导航到部署页面", "success": true }
  ],
  "recovery": {
    "hint": "使用 snapshot 工具查看当前页面状态，根据 intent 定位目标元素，手动执行后使用 startFrom: 3 继续"
  }
}
```

LLM 拿到这个响应后的行动路径：
1. 读取 `failedStep.intent` — 了解目标
2. 调用 `snapshot` 工具 — 看清当前页面状态
3. 根据 intent 在 snapshot 中定位元素 — 用 playwright 手动执行这步
4. 调用 `run '{"startFrom": 3}'` — 恢复自动化执行

#### 改动 C：`SKILL.md.tpl` — 增加 LLM 执行协议

生成的 Skill 的 SKILL.md 目前没有任何 LLM 行为指导。需要增加：

```markdown
## LLM 执行协议

### 正常路径
直接调用 `node tool.js run '{...}'`，成功时任务完成。

### 自愈路径（run 返回 completed: false 时）

1. 读取返回的 `failedStep.intent`，理解这步的目标
2. 调用 playwright snapshot 工具查看当前页面状态：
   `node ~/.cursor/skills/playwright/tool.js snapshot '{}'`
3. 根据 intent 在 snapshot 的可访问性树中定位目标元素
4. 使用 playwright 手动执行该步骤（click/fill 等）
5. 验证步骤结果符合预期（页面状态变化）
6. 调用 `node tool.js run '{"startFrom": <failedAt + 1>, ...}'` 从下一步恢复

### 通用等待策略
任何步骤执行前，如果页面有加载指示（spinner、进度条、骨架屏），先等待其消失再操作。
使用 playwright waitFor 或 snapshot 判断页面是否已稳定。
```

#### 改动 D：`SKILL.md`（web-automation-builder）— 更新 LLM 分析要求

在「LLM 分析 rawEvents 的要求」第 3 条「识别意图」中强化：

```
3. **捕获操作意图**（关键）：
   - 每步的 intent 必须包含：目标元素的语义描述 + 所在上下文 + 期望产生的页面变化
   - 格式：「[在什么上下文中] 点击/填写 [目标的语义描述]，以 [期望结果]」
   - 示例：「在任务列表第一行，点击 Deploy 按钮，以打开部署配置对话框」
   - 避免：「点击 Deploy」（无上下文、无期望结果）
```

### 8.5 与原设计文档（v2）的对应关系

原设计文档（`2026-02-23-02-设计文档.md`）已有正确方向，但实现时未完全落地：

| 原设计文档描述 | 实际实现状态 | 第 8 节改动如何落地 |

### 8.6 与第 3-6 节方案的关系

| 维度 | 第 3-6 节方案 | 第 8 节方案 |
|------|------------|-----------|
| 本质 | 工具层面的规则补丁 | 架构层面的职责重分工 |
| 适用范围 | 针对特定 UI 框架（ant-design 等） | 对任何页面通用 |

| 执行速度 | 快（无 LLM 介入） | 失败时略慢（需 snapshot + 推理） |
| 建议 | 可作为快速修复 | 应作为核心方向 |

**建议实施顺序**：先实施第 8 节（低代码量、解决根本问题），再按需补充第 3-6 节的具体规则作为锦上添花。
