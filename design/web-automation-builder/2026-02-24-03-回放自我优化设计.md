# 回放自我优化设计

## 1. 问题背景

### 1.1 现状

当前 web-automation-builder 生成的 Skill 在回放时，即使最终完成了任务，过程往往"磕磕绊绊"：

- 录制时的 locator 在回放时失效（动态 ID、DOM 结构变化）
- LLM 需要多次 snapshot + evaluate 才能找到正确元素
- waitAfter 超时但实际页面已加载完成
- 存在多个同名按钮，需要通过坐标或 JS 精确定位

**关键问题**：这些"曲折经验"在回放结束后完全丢失，下次执行同样的 Skill 时，LLM 会重复踩同样的坑。

### 1.2 目标

建立**回放 → 总结 → 优化 → 更新**的闭环，让每次执行都比上一次更顺畅。

## 2. 核心设计

### 2.1 架构概览

```
┌─────────────────────────────────────────────────────────┐
│                    LLM 执行 Skill                        │
│                                                          │
│  steps → step 1 → step 2 → ... → step N                │
│     ↓        ↓        ↓              ↓                  │
│  [执行日志: 每步的实际操作方式、耗时、是否回退]            │
└──────────────────────┬──────────────────────────────────┘
                       │ 执行完成
                       ▼
┌──────────────────────────────────────────────────────────┐
│              Phase 1: 执行报告生成                        │
│                                                          │
│  对比 workflow.json 的预期路径 vs 实际执行路径             │
│  识别: 哪些步骤顺利、哪些步骤绕路、用了什么替代方案        │
│  生成结构化的 replay-report                               │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│              Phase 2: 优化建议生成                        │
│                                                          │
│  基于 replay-report，LLM 分析可优化点:                    │
│  - locator 更新（用实际成功的 locator 替换失败的）        │
│  - waitAfter 调整（超时的改策略）                         │
│  - 新增步骤提示（如"需要先滚动到弹窗底部"）              │
│  生成 workflow.json 的 patch                              │
└──────────────────────┬──────────────────────────────────┘

│  - 执行摘要（成功/失败步骤数、总耗时）                    │
│  - 差异对比（哪些步骤有优化建议）                         │
│  - 用户选择: 应用优化 / 跳过 / 查看详情                   │
└──────────────────────┬──────────────────────────────────┘
                       │ 用户确认
                       ▼

│                                                          │
│  应用 patch 到 workflow.json                              │
│  更新 SKILL.md（如果步骤描述变化）                        │
│  记录优化历史到 optimization-log.jsonl                    │
└──────────────────────────────────────────────────────────┘
```

### 2.2 执行日志收集（Replay Trace）

在 LLM-First 模式下，LLM 是执行者。需要在 SKILL.md 中指导 LLM 在执行过程中收集 trace 数据。

  "totalSteps": 5,

#### 2.2.2 收集方式

LLM 在执行完所有步骤后，根据 SKILL.md 中的指引，调用 `node tool.js report-replay '{trace}'` 命令提交 trace 数据。

### 2.3 优化建议生成

#### 2.3.1 优化类型

| 类型 | 触发条件 | 优化动作 |
|------|---------|---------|
| **locator-update** | 步骤使用了 LLM 手动操作且找到了更好的 locator | 更新 workflow.json 中该步骤的 locators |
| **locator-reorder** | 工具执行成功但不是第一个 locator 命中 | 调整 locator 优先级顺序 |
| **waitAfter-fix** | waitAfter 超时但步骤实际成功 | 调整 waitAfter 策略或超时时间 |
| **step-hint** | LLM 发现需要额外操作（如滚动、等待） | 在步骤 intent 中添加提示 |
| **context-update** | 元素的上下文信息变化（如不在 modal 内） | 更新 locator 的 context 字段 |

#### 2.3.2 Patch 格式

```json
{
  "optimizations": [
    {
      "type": "locator-update",
      "step": 2,
      "reason": "录制的 CSS locator 失效，ariaLabel 更稳定",
      "before": {
        "css": "#zcp-app-center > div > main > ... > button"
      },
      "after": {
        "ariaLabel": "Redeploy with LATEST CODE on branch {{branch}}",
        "css": "button[aria-label^='Redeploy with LATEST CODE']"
      }
    },
    {
      "type": "waitAfter-fix",
      "step": 1,
      "reason": "waitAfter text 'Redeploy' 超时，改用 time 等待",
      "before": { "type": "text", "value": "Redeploy", "timeout": 60000 },
      "after": { "type": "time", "value": 5 }
    },
    {
      "type": "locator-update",
      "step": 5,
      "reason": "弹窗内有多个 Deploy 按钮，需要更精确的 selector",
      "before": {
        "text": "Deploy",
        "css": "div:nth-of-type(2) > ... > button"
      },
      "after": {
        "css": ".ant-modal-content .ant-btn-primary:last-child",
        "text": "Deploy",
        "context": { "type": "modal" }
      }
    }
  ]
}
```

### 2.5 SKILL.md 模板更新

在生成的 SKILL.md 中添加**自我优化协议**章节，指导 LLM 在执行完成后进行自我优化。

#### 新增章节（追加到 SKILL.md.tpl 末尾）

```markdown
## 自我优化协议

本 Skill 支持执行后自我优化。每次执行完成后，LLM 应执行以下流程：

### 回放后总结

执行完所有步骤后（无论成功或失败），回顾整个执行过程：

1. **识别差异**：哪些步骤直接用 `step` 命令成功？哪些需要切换到手动模式？
2. **记录发现**：
   - 失效的 locator 和实际有效的替代方案
   - waitAfter 超时但页面已就绪的情况
   - 需要额外操作（滚动、等待、处理弹窗）的步骤
3. **提交报告**：调用 `node tool.js report-replay '{trace_json}'` 提交执行 trace

### Trace 格式

```json
{
  "overallSuccess": true,
  "steps": [
    {
      "step": 1,
      "method": "tool|llm-manual",
      "issues": ["问题描述"],
      "successLocator": { "type": "locator类型", "value": "值" }
    }
  ]
}
```

### 优化触发

提交 trace 后，工具会生成优化建议。如果有可优化项，通过弹窗询问用户是否应用。
```

### 3.1 涉及文件

| 文件 | 变更类型 | 说明 |

### 3.2 tool.js.tpl 新增命令

#### `report-replay` — 接收 LLM 提交的执行 trace

```javascript
async 'report-replay'(params) {
  // 1. 验证 trace 格式
  // 2. 保存到 replay-history.jsonl
  // 3. 对比 workflow.json，生成优化建议
  // 4. 如果有优化建议，通过 agent-interact 弹窗展示
  // 5. 返回优化建议摘要
}
```

#### `apply-optimization` — 应用优化 patch

```javascript
async 'apply-optimization'(params) {
  // 1. 读取 workflow.json
  // 2. 按 patch 更新 locators/waitAfter/intent
  // 3. 写回 workflow.json
  // 4. 记录到 optimization-log.jsonl
  // 5. 返回更新摘要
}
```

### 3.3 优化建议生成逻辑（optimizer.js）

```javascript
function generateOptimizations(trace, workflow) {
  const opts = [];

  for (const stepTrace of trace.steps) {
    const wfStep = workflow.steps[stepTrace.step - 1];

    // Case 1: LLM 手动操作且找到了更好的 locator
    if (stepTrace.method === 'llm-manual' && stepTrace.successLocator) {
      opts.push({
        type: 'locator-update',
        step: stepTrace.step,
        reason: stepTrace.issues?.join('; '),
        after: buildNewLocators(wfStep.locators, stepTrace.successLocator),
      });
    }

    // Case 2: waitAfter 超时
    if (stepTrace.issues?.some(i => i.includes('waitAfter'))) {
      opts.push({
        type: 'waitAfter-fix',
        step: stepTrace.step,
        reason: 'waitAfter 超时',
        after: { type: 'time', value: 3 },
      });
    }
  }

  return opts;
}
```

### 3.4 数据存储

每个生成的 Skill 目录下新增：

```
<skill>/
├── tool.js
├── workflow.json
├── SKILL.md
├── package.json
├── replay-history.jsonl    ← 每次执行的 trace 记录
└── optimization-log.jsonl  ← 每次优化的变更记录
```

#### replay-history.jsonl 格式

每行一条 JSON，记录每次执行的 trace：

```json
{"timestamp":"2026-02-24T06:00:00Z","overallSuccess":true,"steps":[...],"optimizationsApplied":3}
```

#### optimization-log.jsonl 格式

```json
{"timestamp":"2026-02-24T06:15:00Z","type":"locator-update","step":2,"before":{...},"after":{...}}
```

## 5. 执行流程（端到端）

```
用户: "使用 ZCP 部署"
  │
  ├─ LLM 读取 SKILL.md，获取步骤列表
  ├─ LLM 逐步执行，记录每步的执行方式和问题
  ├─ 所有步骤完成
  │
  ├─ LLM 按 SKILL.md 中的"自我优化协议"：
  │   ├─ 回顾执行过程，构建 trace JSON
  │   └─ 调用 node tool.js report-replay '{trace}'
  │
  ├─ report-replay 内部：
  │   ├─ 保存 trace 到 replay-history.jsonl
  │   ├─ 对比 workflow.json 生成优化建议
  │   └─ 如果有优化建议：
  │       ├─ 通过 agent-interact 弹窗展示报告
  │       └─ 返回优化建议给 LLM
  │
  ├─ 用户在弹窗中选择"应用全部优化"
  │
  └─ LLM 调用 node tool.js apply-optimization '{patch}'
      ├─ 更新 workflow.json
      ├─ 记录到 optimization-log.jsonl
      └─ 返回"优化完成"
```

## 6. 渐进式优化效果

```
第 1 次执行:
  Step 1: ✅ tool (warning: waitAfter timeout)
  Step 2: ⚠️ llm-manual (locator 失效，用 ariaLabel 替代)
  Step 3: ✅ tool (预填值正确)
  Step 4: ✅ tool (预填值正确)
  Step 5: ⚠️ llm-manual (多按钮歧义，用坐标点击)
  → 优化: 更新 Step 2 locator, 修复 Step 1 waitAfter, 精确化 Step 5 locator

第 2 次执行:
  Step 1: ✅ tool (waitAfter 已修复)
  Step 2: ✅ tool (ariaLabel locator 直接命中)
  Step 3: ✅ tool
  Step 4: ✅ tool
  Step 5: ✅ tool (精确 locator 直接命中)
  → 无优化建议，全部步骤一次通过
```

## 7. 实现优先级

| 阶段 | 内容 | 复杂度 |
|------|------|--------|
| P0 | SKILL.md.tpl 添加自我优化协议章节 | 低 |
| P0 | tool.js.tpl 添加 report-replay 命令 | 中 |
| P1 | optimizer.js 优化建议生成 | 中 |
| P1 | tool.js.tpl 添加 apply-optimization 命令 | 中 |
| P2 | agent-interact 弹窗展示优化报告 | 中 |
| P2 | replay-history / optimization-log 持久化 | 低 |

## 8. 风险与约束

1. **LLM 执行不确定性**：LLM 可能不严格遵循 SKILL.md 中的自我优化协议。通过明确的指令和结构化格式降低风险。
2. **优化过度**：频繁更新 locator 可能导致"震荡"。通过 optimization-log 记录历史，避免反复修改。
3. **trace 数据质量**：依赖 LLM 准确回忆执行过程。可以在 tool.js 的 step 命令中自动记录部分数据（如 locator 命中索引），减少对 LLM 回忆的依赖。
