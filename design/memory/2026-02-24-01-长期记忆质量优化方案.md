# 长期记忆（MEMORY.md）质量优化方案

## 1. 问题分析

### 1.1 当前架构

```
Agent 调用 save_fact.py → daily/YYYY-MM-DD.jsonl → distill_to_memory.py → MEMORY.md
                                                         ↑
                                                    纯规则驱动
```

当前 `distill_to_memory.py` 的提炼流程：
1. 从 daily jsonl 筛选 `confidence >= 0.85`、`age >= 1天` 的事实
2. 按 `memory_type` + 关键词匹配分类到 MEMORY.md 章节
3. **原文复制**到对应章节，每次最多 5 条

### 1.2 质量问题

| 问题 | 根因 | 影响 |
|------|------|------|
| 事实内容冗长、含实现细节 | save_fact 提示词未约束内容格式 | MEMORY.md 条目过长，AI 加载时浪费 token |
| 相似事实重复堆积 | 去重仅靠子串匹配，无语义判断 | 同一决策的多次表述都会写入 |
| 分类不准确 | 仅靠 memory_type 和中文关键词 | 英文内容全部落入"项目背景" |
| 无信息精炼 | distill 直接复制原文 | 临时性细节（Bug 修复步骤、文件路径）混入核心记忆 |
| 章节结构固定 | 硬编码 4 个章节 | 无法适应不同项目的记忆需求 |
| memory_type 单一化 | Agent 几乎只使用 W 类型 | 分类第一道防线失效，全部走关键词兜底 |
| save_fact 调用污染 | Agent 将命令本身混入 content | 产生包含 python3 命令的脏数据 |

### 1.3 实际数据分析（基于 semantic_metadata 项目）

> 以下分析基于 semantic_metadata 项目的真实 memory-data（28 条 facts，17 条 sessions，MEMORY.md 44 行）。

| memory_type 分布 | W: 28, O: 0, B: 0 | **100% 为 W 类型**，分类第一道防线完全失效 |
| 内容平均长度 | 253 字 | 最短 96 字，最长 508 字 |
| 超过 80 字 | 28/28 (100%) | 无一条符合"一句话原则" |

| confidence 分布 | 全部 >= 0.85 | 无区分度，筛选形同虚设 |
| entities 为空 | 2/28 (7%) | 两条最长的 fact 反而没有 entities |

|------|------|---------|

#### 1.3.3 典型低质量条目

**问题 1：冗长的过程描述（508 字）**
```
"修复SemanticDimensionManager NoClassDefFoundError: 将populateStatistics方法中的
responses.forEach(lambda)改为标准for-each循环，避免匿名内部类python3

--content \"修复SemanticDimensionManager...\" --type W --entities \"...\" --session \"...\"

```
**问题**：(1) 包含 Bug 修复的具体步骤 (2) save_fact.py 命令本身被混入 content (3) 508 字远超合理长度。应精炼为"Java 禁止 forEach+lambda，改用 for-each 循环避免 NoClassDefFoundError"。

**问题 2：临时性数量统计**
```
"Dimension模块回归测试完成：18/28通过(64.3%)，后端NoClassDefFoundError已修复
(SemanticDimensionManager lambda改for-each)。整体回归测试81/139通过(58.3%)"
```
**问题**：测试通过率是快照数据，下次回归后就过时。应只保留规范性结论。

**问题 3：重复的数据回填记录（4 条描述同一件事）**
```
fact[15]: "BDP业务数据回填全部完成(107条)..."
fact[18]: "BDP数据回填补充完成: METRIC_PROCESS 31条..."
fact[20]: "BDP补充回填完成(113条新增)..."
fact[16]: "BDP业务数据回填缺少两类关联关系..."
```
**问题**：4 条 fact 描述同一个数据回填任务的不同阶段，全部进入 MEMORY.md 造成冗余。应合并为一条"BDP 业务数据回填共 210 条操作已全部完成"。

**问题 4：MEMORY.md 中的信息重复**

MEMORY.md 中"常用 Skills"章节和"重要决策"章节包含完全相同的 Skills 列表，只是格式不同。一个是列表格式，一个是编号段落格式。这是因为同一信息被两次 save_fact 写入不同 memory_type，而去重机制（子串匹配）无法识别语义重复。

#### 1.3.4 MEMORY.md 当前状态评估

| 章节 | 条目数 | 质量评估 |
|------|--------|---------|
| 用户偏好 | 0 | 空（Agent 从未使用 O 类型） |
| 项目背景 | 18 | **严重膨胀**：含大量实现细节、API 路径、级联删除步骤 |
| 常用 Skills | 8 | 合理，但与"重要决策"章节重复 |
| 重要决策 | 3 | 第一条与"常用 Skills"完全重复 |
| 项目规范 | 1 | 过少，实际有多条规范散落在"项目背景"中 |

**核心问题**：MEMORY.md 从"空"变成了"信息垃圾场"——distill 机制开始工作后，由于无精炼能力，原文照搬导致 3160 字符的 MEMORY.md 中有效信息密度极低。

---

## 2. 优化目标

1. **精炼**：将冗长的事实浓缩为简洁、可操作的核心记忆（平均 253 字 → ≤80 字）
2. **去重合并**：语义相似的事实合并为一条（如 4 条数据回填记录 → 1 条）
3. **过滤**：排除临时性、已过时的信息（预计过滤 40%+ 的 facts）
4. **分类**：修复 memory_type 单一化问题，准确分类到合适的章节
5. **可读性**：生成的 MEMORY.md 对人和 AI 都易于理解
6. **防污染**：防止 save_fact 命令本身混入 content

---

### 3.2 Stage 0：入口防污染（新增）

**发现的真实问题**：semantic_metadata 项目中 `log-132231339-jh3b` 这条 fact 的 content 中混入了完整的 `python3 save_fact.py --content "..." --type W --entities "..."` 命令，导致 508 字的脏数据进入 daily jsonl。

**根因**：Agent 在构造 save_fact.py 的 `--content` 参数时，将自身的工具调用命令也拼接进了内容。

**修复方案**：在 `save_fact.py` 中增加输入清洗：

```python
def sanitize_content(content: str) -> str:
    """清洗 fact content，移除混入的命令片段"""
    # 移除混入的 save_fact.py 命令
    content = re.sub(
        r'python3\s+\S*save_fact\.py\s+--content\s+"[^"]*".*?--session\s+"[^"]*"',
        '', content
    )
    # 移除混入的 save_summary.py 命令
    content = re.sub(
        r'python3\s+\S*save_summary\.py\s+--topic\s+"[^"]*".*?--session\s+"[^"]*"',
        '', content
    )
    content = re.sub(r'\s{2,}', ' ', content).strip()
    
    if len(content) > 800:
        log.warning("fact content 超长 (%d 字)，截断至 800", len(content))
        content = content[:800]
    
    return content
```

**memory_type 自动推断**：当 Agent 传入的 type 为 W 时，根据内容关键词二次推断：

```python
def infer_memory_type(content: str, declared_type: str) -> str:
    """当 declared_type 为 W 时，尝试根据内容推断更准确的类型"""
    if declared_type != "W":
        return declared_type
    
    # O 类型关键词
    if re.search(r'偏好|习惯|中文|英文|风格|不要|必须|禁止|规范|约定', content):
        return "O"
    # B 类型关键词
    if re.search(r'项目|架构|技术栈|模块|子项目|工作区|仓库', content):
        return "B"
    return "W"
```

### 3.3 Stage 1：入口质量提升

#### 3.3.1 增强 FLUSH_TEMPLATE（flush_memory.py）

**当前问题**：提取规则过于宽泛，没有格式和质量约束。实际数据显示 100% 的 facts 超过 80 字，平均 253 字。

**优化方向**：

```diff
  ## 提取规则
  
- 1. **技术决策**：架构选择、配置变更、Bug 修复方案
- 2. **用户偏好**：编码风格、工具选择、沟通方式、命名习惯
- 3. **项目经历**：里程碑、重要变更、团队信息
+ 1. **技术决策**：架构选择、技术选型（不含实现细节） → type W
+ 2. **用户偏好**：编码风格、工具选择、沟通方式 → type O
+ 3. **项目规范**：必须/禁止的规则、约定 → type O
+ 4. **项目背景**：项目结构、技术栈、模块关系 → type B
+
+ ## 质量要求
+
+ - 每条事实必须是**一句话**，不超过 80 字
+ - 只保存**长期有效**的信息，排除：
+   · 已修复的 Bug 的具体修复步骤
+   · 临时性的数量统计（如"测试从 N 增至 M"、"共 107 条"）
+   · 文件路径、行号等易变信息
+   · 正在进行中的任务状态和进度
+   · 一次性操作的详细记录（如数据回填的具体条数）
+ - 用**结论性表述**而非过程描述
+   · ✗ "修复SemanticDimensionManager NoClassDefFoundError: 将forEach(lambda)改为for-each循环"
+   · ✓ "Java 禁止 forEach+lambda，改用 for-each 避免 NoClassDefFoundError"
+   · ✗ "BDP数据回填补充完成: METRIC_PROCESS 31条, METRIC_DIMENSION从3条补充到22条"
+   · ✓ "BDP 业务数据回填已全部完成（含所有关联关系）"
+ - 如果一个事实需要超过 80 字才能表达，拆分为多条独立事实
+ - **不要将同一件事的多个阶段分别保存**，只保存最终结论
```

#### 3.3.2 增强 SAVE_TEMPLATE（prompt_session_save.py）

**当前问题**：Agent 几乎只使用 type W，导致 memory_type 分类完全失效。需要在提示词中明确各类型的使用场景和示例。

**优化方向**：

```diff
  ## 逐条提取关键事实（必须执行）
  
  回顾本次会话，将以下类型的信息**逐条**保存为独立事实：
  
- - 新确立的项目原则或规范 → type O
- - 重要的架构决策或技术选型 → type W
- - 用户明确表达的偏好 → type O
- - 关键的 Bug 修复方案 → type W
+ - 用户偏好、项目规范 → **type O**（如"中文回答"、"禁止 forEach+lambda"）
+ - 技术决策、架构选型 → **type W**（如"日志路径用环境变量传递"）
+ - 项目结构、技术栈 → **type B**（如"后端 Spring Boot + MyBatis-Plus"）
+
+ **重要**：请根据内容性质选择正确的 type，不要全部使用 W。
+
+ ## 事实质量要求
+
+ - **一句话原则**：每条事实不超过 80 字，用结论性表述
+ - **长期有效性**：只保存 3 个月后仍然有价值的信息
+ - **不要保存**：
+   · Bug 修复的具体步骤和过程（只保存结论性规范）
+   · 临时性的数量统计（测试通过率、数据条数、文件数）
+   · 已完成任务的进度描述（"已完成"、"全部通过"）
+   · 与特定会话绑定的上下文信息
+   · 一次性操作的详细记录
+ - **合并同类**：如果多个事实描述同一件事的不同阶段，只保存最终结论
+ - **不要重复**：如果某信息已在之前的 save_fact 中保存过，不要再次保存
```

### 3.4 Stage 2：LLM 精炼（核心改动）

#### 3.4.1 流程设计

```
distill_to_memory.py（重构）
│
├── 1. 规则预筛选（保留现有逻辑，调整参数）
│   ├── confidence >= 0.85
│   ├── age >= 1 天
│   ├── 排除 session_start/session_end/audit/session_metrics
│   └── 子串去重
│
├── 2. LLM 精炼（新增）
│   ├── 输入：全部候选事实 + 现有 MEMORY.md 全文
│   ├── 处理：
│   │   ├── 过滤临时性/已过时信息（预计过滤 40%+）
│   │   ├── 合并语义重复的事实（如 4 条回填记录 → 1 条）
│   │   ├── 精炼为简洁表述（253字均值 → ≤80字）
│   │   ├── 准确分类（不再依赖 memory_type=W 的单一值）
│   │   └── 对现有 MEMORY.md 条目也进行精炼和去重
│   └── 输出：完整的 MEMORY.md 内容（全量重写，非追加）
│
└── 3. 全量重写 MEMORY.md（改为全量重写模式）
│   ├── 备份旧文件到 MEMORY.md.bak
│   └── 写入 LLM 精炼后的完整内容
│
└── 4. 标记已提炼的 facts（新增）
    └── 在 daily jsonl 中标记 distilled=true，避免重复处理
```

**关键变更：从"追加"改为"全量重写"**

当前 distill 是追加模式，这导致 MEMORY.md 只增不减，历史冗余无法清理。改为全量重写模式后，LLM 每次都能看到完整上下文，对现有条目和新候选一起精炼，确保 MEMORY.md 始终保持最优状态。

#### 3.4.2 LLM 精炼 Prompt

```python
REFINE_PROMPT = """你是一个记忆管理专家。你的任务是将一组原始事实和现有记忆精炼为高质量的长期记忆。

## 输入

### 新候选事实（待精炼）
{candidates_json}

### 现有核心记忆（MEMORY.md 当前内容）
{existing_memory}

## 任务

综合新候选事实和现有记忆，生成一份**完整的**精炼后记忆列表。

### 1. 过滤（丢弃以下类型）
- 已修复 Bug 的具体修复步骤和过程描述（只保留结论性规范）
- 临时性数量统计（如"测试从 N 增至 M"、"共 107 条"、"通过率 58.3%"）
- 一次性操作的详细记录（如数据回填的具体条数和步骤）
- 与特定会话/时间点绑定的状态信息
- 纯实现细节（具体文件路径、行号、变量名、方法签名）
- 重复信息（同一内容在不同章节出现、同一事件的多个阶段描述）
- 已过时的信息（被后续事实更新或否定的旧信息）

### 2. 精炼
- 每条记忆不超过 80 字
- 使用结论性表述，不描述过程
- 保留关键技术名词和决策理由
- 如果多条事实描述同一主题，合并为一条
- 现有 MEMORY.md 中过长的条目也要精炼

### 3. 分类
将每条精炼后的记忆分类到以下章节之一：
- **用户偏好**：用户的个人偏好、习惯、沟通方式
- **项目规范**：必须遵守的规则、约定、禁止事项
- **重要决策**：技术选型、架构决策及其理由
- **项目背景**：项目结构、技术栈、模块关系（简洁概述，不含实现细节）
- **常用工具**：经常使用的 Skills、工具、命令

### 4. 去重
- 如果新事实与现有记忆语义相同，保留更精炼的版本
- 如果同一信息出现在多个章节，只保留最合适的章节中的版本
- 合并描述同一主题不同方面的条目

## 输出格式

返回 JSON 对象：
```json
{
  "memories": [
    {
      "section": "章节名",
      "content": "精炼后的记忆内容（≤80字）",
      "source": "new|existing|merged",
      "source_ids": ["原始事实ID列表（如适用）"],
      "action": "keep|merge|refine|discard",
      "reason": "简短说明处理理由"
    }
  ],
  "stats": {
    "new_kept": 3,
    "existing_kept": 8,
    "merged": 2,
    "discarded": 5,
    "refined": 4
  }
}
```

- source="new": 来自新候选事实
- source="existing": 保留自现有 MEMORY.md
- source="merged": 新旧合并
- action="discard" 的条目也要列出（用于日志审计）

只返回 JSON 对象，不要其他内容。"""
```

#### 3.4.3 LLM 调用方式

有三种可选方案：

##### 方案 A：利用 Cursor Agent 自身模型能力（推荐）

**核心思路**：在 `stop` hook 中通过 `followup_message` 注入精炼指令，让 Cursor Agent 自己完成精炼并写回 MEMORY.md。

**工作原理**：
```
stop hook 触发（status=completed）
  ├── prompt_session_save.py 已通过 followup_message 让 Agent 保存摘要和事实
  └── 新增：在同一个 followup_message 中追加精炼指令
       ├── Agent 读取 daily/*.jsonl 中的候选事实
       ├── Agent 读取当前 MEMORY.md
       ├── Agent 用自身模型能力进行精炼（合并、过滤、分类）
       └── Agent 调用 distill_refined.py 写入精炼结果
```

**优势**：
- **零额外成本**：使用 Cursor 已有的模型配额，不需要 OpenAI API key
- **模型能力更强**：Cursor 使用的是 Claude/GPT-4 级别模型，精炼质量远超 gpt-4o-mini
- **无需配置**：不需要用户配置 API key、base_url 等
- **已有先例**：`stop` hook 的 `followup_message` + `loop_limit` 机制已被 `prompt_session_save.py` 验证可行

**劣势**：
- 增加 Agent 的 stop 阶段工作量（多一轮精炼操作）
- 依赖 Agent 正确执行指令（可能被忽略或执行不完整）
- `loop_limit: 1` 限制了只能追加一轮，需要和现有的 session save 合并在同一个 followup_message 中

**实现方式**：修改 `prompt_session_save.py` 的 `SAVE_TEMPLATE`，在末尾追加精炼指令段：

```python
DISTILL_SECTION = """

```bash

```

##### 方案 B：外部 LLM API 调用（备选）

在 `sessionEnd` hook 中使用 `urllib` 调用 OpenAI 兼容 API。

**适用场景**：方案 A 不可行时的降级方案（如 Agent 未正确执行精炼指令）。

```python
import urllib.request
import json

def call_llm(prompt, config):
    """调用 LLM API 进行事实精炼"""
    api_key = config.get("api_key") or os.environ.get("OPENAI_API_KEY", "")
    base_url = config.get("base_url", "https://api.openai.com/v1")
    model = config.get("model", "gpt-4o-mini")

    if not api_key:
        return None

    payload = {
        "model": model,
        "messages": [{"role": "user", "content": prompt}],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }

    req = urllib.request.Request(
        f"{base_url}/chat/completions",
        data=json.dumps(payload).encode(),
        headers={
            "Authorization": f"Bearer {api_key}",
            "Content-Type": "application/json",
        },
    )

    with urllib.request.urlopen(req, timeout=30) as resp:
        result = json.loads(resp.read())
        return json.loads(result["choices"][0]["message"]["content"])
```
