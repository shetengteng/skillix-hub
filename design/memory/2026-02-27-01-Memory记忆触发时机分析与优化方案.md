# Memory 记忆触发时机分析与优化方案

## 1. 当前架构概览

Memory Skill 采用 **多层级兜底** 架构，通过 Cursor Hooks + Rules 协作完成记忆的存储与召回。

### 1.1 组件清单

| 组件 | 文件 | 触发方式 |
|------|------|----------|
| 会话加载 | `load_memory.py` | sessionStart Hook |
| 上下文刷写 | `flush_memory.py` | preCompact Hook |
| 会话保存提示 | `prompt_session_save.py` | stop Hook |
| 会话清理 | `sync_and_cleanup.py` | sessionEnd Hook |
| 审计-回复 | `audit_response.py` | afterAgentResponse Hook |
| 审计-思考 | `audit_thought.py` | afterAgentThought Hook |
| 审计-工具 | `audit_tool_use.py` | postToolUse Hook (matcher=Shell) |
| Agent 规则 | `memory-rules.mdc` | alwaysApply Rule |

### 1.2 数据存储结构

```
.cursor/skills/memory-data/
├── config.json           # 项目配置
├── MEMORY.md             # 核心记忆（精炼后的长期记忆）
├── NOTES.md              # 用户笔记
├── sessions.jsonl        # 会话摘要
├── daily/                # 每日事实
│   └── YYYY-MM-DD.jsonl  # 包含 fact、audit、session_start/end、warning 等
├── session_state/        # 会话状态追踪
│   └── {conv_id}.json    # 记录 summary_saved、fact_count 等
└── logs/                 # 日志
```

## 2. 记忆存储的完整触发流程

### 2.1 四层架构

```
┌──────────────────────────────────────────────────────────────────┐
│ Layer 1: Rules 主动存储（memory-rules.mdc）                       │
│ 触发者：Agent 自身                                                │
│ 时机：任务执行中 / 任务完成后                                      │
│ 动作：save_fact.py（逐条保存关键事实）                              │
│ 不触发 save_summary                                               │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│ Layer 2: preCompact 刷写（flush_memory.py）                       │
│ 触发者：Cursor 引擎（上下文接近阈值时）                             │
│ 时机：context_usage_percent 达到阈值                               │
│ 动作：注入 [Memory Flush] → Agent 执行 save_fact.py + 阶段摘要     │
│ 输出字段：user_message                                            │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│ Layer 3: sessionEnd 自动生成（sync_and_cleanup.py）               │
│ 触发者：Cursor 引擎（会话彻底关闭后）                               │
│ 时机：sessionEnd Hook                                             │
│ 动作：auto_generate_summary（从 facts + 阶段摘要聚合）              │
│ 条件：仅在 Layer 1/4 均未保存摘要时执行                              │
│ source: layer3_auto                                               │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────────┐
│ Layer 4: stop Hook 兜底（prompt_session_save.py）                 │
│ 触发者：Cursor 引擎（status=completed/aborted）                    │
│ 时机：Agent 完成任务 或 用户关闭/切换会话                           │
│ 动作：注入 [Session Save] → Agent 执行 save_summary + save_fact    │
│       + distill_refined（精炼 MEMORY.md）                         │
│ 条件：has_session_data() 为 False 时才注入                         │
│ 输出字段：followup_message                                        │
│ source: layer4_stop                                               │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 审计层（辅助，不直接触发存储）

| Hook | 脚本 | 作用 |
|------|------|------|
| afterAgentResponse | `audit_response.py` | 检测 Agent 回复中的信号词（"记下来了"、"决定使用"等），写入 audit 日志 |
| afterAgentThought | `audit_thought.py` | 检测 Agent 思考中的决策意图，写入 audit 日志 |
| postToolUse (Shell) | `audit_tool_use.py` | 捕获 git commit/push 等操作，写入 audit 日志 |

审计层仅记录日志，不注入消息、不触发 Agent 行为。

### 2.3 信号与处理

| 信号 | 发送者 | 接收者 | 输出字段 | 处理规则 |
|------|--------|--------|----------|----------|
| `[Memory Flush]` | flush_memory.py (preCompact) | Agent | user_message | save_fact + 阶段摘要 |
| `[Session Save]` | prompt_session_save.py (stop) | Agent | followup_message | save_summary + save_fact + distill |
| `[Session Save]` | 用户手动发送 | Agent | — | 同上（memory-rules.mdc 匹配） |

## 3. 会话生命周期完整流程

```
[会话开始]
    │
    ▼
sessionStart Hook → load_memory.py
    │  加载 MEMORY.md、NOTES.md、近期事实、上次会话摘要
    │  输出 {"additional_context": "..."}
    │
    ▼
[对话进行中]
    │
    ├─ Agent 任务执行中 → Layer 1: 主动 save_fact（rules 驱动）
    │
    ├─ 上下文接近阈值 → Layer 2: preCompact → [Memory Flush]
    │                              → Agent save_fact + 阶段摘要
    │
    ├─ afterAgentResponse → audit_response.py → 审计日志
    ├─ afterAgentThought  → audit_thought.py  → 审计日志
    ├─ postToolUse(Shell) → audit_tool_use.py → 审计日志
    │
    ▼
[任务完成 / 会话关闭]
    │
    ▼
stop Hook (status=completed/aborted)
    │  → prompt_session_save.py
    │  → 检查 has_session_data()
    │     ├─ True  → 跳过（已有数据）
    │     └─ False → 注入 [Session Save] followup_message
    │                → Agent 执行 save_summary + save_fact + distill
    │
    ▼
[会话彻底关闭]
    │
    ▼
sessionEnd Hook → sync_and_cleanup.py
    ├─ sync_index（JSONL → SQLite）
    ├─ truncate_sessions（保留最近 500 条）
    ├─ auto_generate_summary（Layer 3 兜底）
    ├─ check_summary_saved（写 warning 如未保存）
    ├─ log_session_metrics（记录会话指标）
    ├─ log_session_end（记录结束元数据）
    ├─ clean_old_logs（清理过期日志）
    └─ clean_old_session_states（清理过期状态文件）
```

## 4. 当前问题分析

### 4.1 [Session Save] 触发时机不合理

**现状**：`[Session Save]` 由 stop Hook 在 `status=completed/aborted` 时自动触发。

**问题**：

1. **每次任务完成都会触发**：用户可能只是完成了一个小任务就触发 Session Save，浪费 Agent 的一个 turn 来执行记忆操作
2. **用户无法控制时机**：用户说"结束"时想触发 Session Save，但当前机制是 Cursor 引擎自动判断 status
3. **重复触发风险**：如果用户手动说了"结束"，Agent 执行了 Session Save，随后 stop Hook 又触发一次（虽然 `has_session_data` 可以拦截，但多了一次不必要的 Hook 调用）

### 4.2 memory-rules.mdc 的矛盾

规则中写着：
> 会话摘要只在收到 `[Session Save]` 信号时保存，不在每次任务完成时触发。

但 stop Hook 在每次 `status=completed` 时都会发送 `[Session Save]`，相当于每次任务完成都触发。

---

## 5. 期望行为

| 场景 | 期望行为 |
|------|----------|
| 用户说"结束" | Agent 检测到结束意图 → 执行 Session Save → stop Hook 检测到已保存，跳过 |
| 用户没说"结束"，直接关闭会话 | stop Hook 作为兜底 → 发送 [Session Save] |
| 用户没说"结束"，切换到新会话 | stop Hook 作为兜底 → 发送 [Session Save] |
| 短对话（一问一答） | 不触发 Session Save（无实质内容） |

## 6. 优化方案

> 排除方案：userinput.py 不做修改，保持只读 prompts.txt 的简单逻辑。

### 6.1 核心思路：Rules 驱动 + stop Hook 纯兜底

```
用户说"结束"
    │
    ▼
memory-rules.mdc 匹配 → Agent 主动执行 Session Save
    │  save_summary (--source layer1_rules)
    │  save_fact（关键事实）
    │  distill_refined（精炼 MEMORY.md）
    │  → session_state 中 summary_saved = True
    │
    ▼
后续 stop Hook 触发
    │  → prompt_session_save.py
    │  → has_session_data() 检测到 summary_saved = True
    │  → 跳过，不注入 [Session Save]
    │
    ▼
结束 ✅（不重复触发）
```

```
用户没说"结束"，直接关闭会话
    │
    ▼
stop Hook 触发 (status=completed/aborted)
    │  → prompt_session_save.py
    │  → has_session_data() 检测到 summary_saved = False
    │  → 注入 [Session Save] followup_message（兜底）
    │  → Agent 执行 save_summary + save_fact + distill
    │
    ▼
结束 ✅（兜底生效）
```

### 6.2 需要修改的组件

#### 修改 1：memory-rules.mdc — 增加"结束"检测规则

在 `memory-rules.mdc` 中增加「会话结束处理」章节：

```markdown
## 会话结束处理

当用户明确表示结束会话（如"结束"、"结束会话"、"结束了"、"今天就到这"等）时：
1. 执行 Session Save（同 [Session Save] 信号的处理逻辑）
2. save_summary 使用 --source layer1_rules
3. 逐条提取关键事实
4. 精炼 MEMORY.md
5. 完成后停止，不再执行新任务
```

同时更新模板文件 `skills/memory/templates/memory-rules.mdc`（安装源）和项目级 `.cursor/rules/memory-rules.mdc`。

#### 修改 2：prompt_session_save.py — has_session_data 只检查 summary_saved

当前 `has_session_data()` 的逻辑：
- `summary_saved = True` → 跳过 ✅
- `fact_count > 0` → 跳过 ❌（有 facts 不代表会话被正确结束）
- daily 中有匹配 session 的 fact → 跳过 ❌

问题：Agent 在任务中主动保存了几条 fact（Layer 1），`has_session_data()` 就返回 True，导致 stop Hook 跳过。但此时没有 summary，只有 Layer 3 auto_generate 兜底（质量低）。

优化后的判断逻辑：

```python
def has_session_data(memory_dir: str, conv_id: str) -> bool:
    """只检查 summary 是否已保存，fact 不作为跳过条件"""
    state = read_session_state(memory_dir, conv_id)
    if state and state.get("summary_saved"):
        return True
    return False
```

这样：
- 用户说"结束" → Agent 保存 summary → stop Hook 检测到 summary_saved=True → 跳过 ✅
- 用户没说"结束"，但 Agent 保存了 facts → stop Hook 检测到 summary_saved=False → 触发 [Session Save] ✅
- 短对话无任何记忆数据 → stop Hook 触发，但 session_save_template 中有"如果会话内容太少不做操作"的指引 ✅

### 6.3 不需要修改的组件

| 组件 | 原因 |
|------|------|
| flush_memory.py | preCompact 流程不变 |
| sync_and_cleanup.py | sessionEnd 清理流程不变，Layer 3 auto_generate 保持作为最后兜底 |
| save_summary.py | 已支持 `--source layer1_rules` |
| save_fact.py | 无变化 |
| session_state.py | 无变化 |
| 审计脚本 | 无变化 |

### 6.4 优化后的四层架构

```
Layer 1: Rules 主动存储
  ├─ 任务中/完成后 → save_fact（不变）
  └─ 用户说"结束" → save_summary + save_fact + distill（新增）★
      └─ source: layer1_rules

Layer 2: preCompact → [Memory Flush]（不变）

Layer 3: sessionEnd → auto_generate_summary（不变，最终兜底）

Layer 4: stop → [Session Save]
  ├─ 检测 summary_saved = True → 跳过（用户已手动结束）
  └─ 检测 summary_saved = False → 注入提示词（兜底）★ 判断条件修改
      └─ source: layer4_stop
```

### 6.5 影响评估

| 变更 | 风险 | 影响 |
|------|------|------|
| memory-rules.mdc 增加"结束"规则 | 低 | Agent 行为变更，用户说"结束"时自动执行 Session Save |
| has_session_data 改为只检查 summary | 中 | 会话中保存了 facts 但没说"结束"时，stop Hook 会触发（增加 1 个 Agent turn） |
