# Agent Workflow — Claude 模型兼容性评估

> 日期：2026-06-01
> 背景：评估 agent-workflow 作为 caller skill 时，配合 OpenCode 使用各 Claude 模型的可行性
> 结论：**Sonnet 4.6 是最佳性价比选择；Haiku 4.5 可用于简单直线 workflow，复杂场景风险高**

---

## 1. Agent-Workflow 对 Caller 模型的核心要求

从 SKILL.md 主循环协议中提炼出 8 项关键能力需求：

| # | 能力需求 | 难度 | 说明 |
|---|---|---|---|
| R1 | JSON 输出解析 | 低 | 解析 CLI 返回的 `{"result":{...}}` 结构 |
| R2 | 4-action 路由分发 | 中 | 根据 `action` 字段选择 execute_agent / wait_user / continue / done |
| R3 | `continue` 正确处理 ★ | **高** | 收到 continue 后**立即** advance（不推理、不展示、不 sleep） |
| R4 | Shell 命令构造 | 中 | 构造含嵌套 JSON 的 shell 命令，引号转义正确 |
| R5 | 多轮状态跟踪 | 中高 | 跨 turn 维护 run_id，loop 中追踪迭代 |
| R6 | 7 条强制约束遵守 | 高 | DON'T 列表：不改 state、不暴露 continue、不并发等 |
| R7 | 上下文窗口 | 可变 | 长链 workflow 累积 vars/history 可达数万 token |
| R8 | Authoring Protocol | 高 | 8 步创作协议，多轮结构化对话 |

**其中 R3（continue 处理）是最关键的差异化指标**——它要求模型在收到一个看起来"有内容"的响应时，**违反直觉地**不做任何推理直接转发，弱模型极易在此出错。

## 3. 逐模型评估

### Opus 4.7 — 完全兼容 ✅✅✅

| 需求 | 评分 | 备注 |
|---|---|---|
| R1 JSON 解析 | ✅ | 无压力 |
| R2 Action 路由 | ✅ | 无压力 |
| R3 Continue 处理 | ✅ | 指令遵循最强，能严格按 DON'T 执行 |
| R4 Shell 构造 | ✅ | 复杂 JSON 嵌套无问题 |
| R5 多轮状态 | ✅ | 1M 上下文 + 最强推理 |
| R6 约束遵守 | ✅ | 无压力 |
| R7 上下文 | ✅ | 1M 窗口充裕 |
| R8 Authoring | ✅ | 多步对话最强 |

**适用场景**：所有 workflow 场景，特别是 >5 节点的复杂长链、跨 LLM 协作、需要高质量推理输出的节点。

**代价**：贵（$5/$25），延迟稍高。对于简单 workflow 性价比低。

### Sonnet 4.6 — 完全兼容 ✅✅（推荐默认）

| 需求 | 评分 | 备注 |
|---|---|---|
| R1 JSON 解析 | ✅ | 无压力 |
| R2 Action 路由 | ✅ | 无压力 |
| R3 Continue 处理 | ✅ | 指令遵循强，SKILL.md 写清楚后能稳定执行 |
| R4 Shell 构造 | ✅ | Tool use 与 Opus 持平 |
| R5 多轮状态 | ✅ | 1M 上下文充裕 |
| R6 约束遵守 | ✅ | 偶尔需 SKILL.md 强调，整体可靠 |
| R7 上下文 | ✅ | 1M 窗口 |
| R8 Authoring | ✅ | 多步对话可靠 |

**适用场景**：所有 workflow 场景的默认选择。质量接近 Opus，成本仅 40%。

**推荐配置**（OpenCode）：

```json
{
  "agent": {
    "build": {
      "mode": "primary",
      "model": "anthropic/claude-sonnet-4-6",
      "permission": { "bash": "allow", "edit": "allow" }

  }
```

### Haiku 4.5 — 有条件兼容 ⚠️（限简单场景）

| 需求 | 评分 | 风险等级 | 备注 |
|---|---|---|---|
| R1 JSON 解析 | ✅ | 低 | 简单 JSON 解析无问题 |
| R2 Action 路由 | ✅ | 低 | 4 分支 if-else 能处理 |
| R3 Continue 处理 | ⚠️ | **高** | 可能尝试"解释"或"推理" continue 内容 |
| R4 Shell 构造 | ⚠️ | 中 | 深层嵌套 JSON + 引号转义偶发错误 |
| R5 多轮状态 | ⚠️ | 中高 | 200K 窗口在长 loop 中可能溢出 |
| R6 约束遵守 | ⚠️ | **高** | 多条 DON'T 约束同时生效时容易遗漏 |
| R7 上下文 | ❌ | **高** | 200K << 1M，复杂 workflow 累积 context 快速耗尽 |
| R8 Authoring | ❌ | **高** | 8 步创作协议对 Haiku 过于复杂 |

**Haiku 能胜任的场景**：
- ✅ 2-3 节点直线 workflow（无 loop、无 wait_user 嵌套）
- ✅ 纯 mock 测试运行（结果确定、无需推理）
- ✅ 简单 status / list / view 查询
- ✅ validate 调用（输入输出都是确定的）

**Haiku 大概率失败的场景**：
- ❌ 含 loop + wait_user 的复杂交互 workflow
- ❌ 需要 Authoring Protocol 创作新 workflow
- ❌ 长链 workflow（>5 节点）的 continue 保活
- ❌ 跨 LLM 协作（需要理解 executor 差异和 spawn 行为）
- ❌ 断点续跑（需要组合 list → status → resume 的多步决策）

## 5. 具体失败模式预测（Haiku）

### 5.1 Continue 误处理（最危险）

```
CLI 返回: {"result": {"action": "continue", "run_id": "wf-..."}}

Haiku 可能做的:
  ❌ "我注意到 workflow 返回了 continue 状态，让我检查一下..."（推理了）
  ❌ "workflow 正在继续执行..."（展示给用户了）
  ❌ advance 时带了 result（不该带）

正确行为:
  ✅ 立即 advance '{"run_id":"wf-..."}'（不带 result，不输出任何文字）
```

### 5.2 Loop 迭代中状态丢失

```
Loop 第 3 轮，用户输入 approved=false：

Haiku 可能做的:
  ❌ 忘记之前的 run_id，重新 start 一个 workflow
  ❌ 把 resume 的 JSON 格式搞错（字段名拼错）
  ❌ 在 200K 上下文快满时，丢失早期 loop 历史

正确行为:
  ✅ 用原 run_id resume，传入正确结构的 user_input
```

### 5.3 嵌套 JSON Shell 命令格式错误

```
正确:
  python3 tool.py resume '{"run_id":"wf-xxx","user_input":{"approved":true,"feedback":"ok"}}'

Haiku 可能产出:
  ❌ python3 tool.py resume {"run_id":"wf-xxx",...}（缺外层引号）
  ❌ python3 tool.py resume '{"run_id":"wf-xxx","user_input":{"approved":"true"}}'（boolean 写成 string）
```

## 6. 推荐策略

### 方案 A：统一用 Sonnet（推荐）

最简单、最稳妥。所有 agent-workflow 操作都用 Sonnet 4.6。

```json
{
  "agent": {
    "build": {
      "model": "anthropic/claude-sonnet-4-6",
      "permission": { "bash": "allow", "edit": "allow" }
    }
  }
}
```

**成本预估**：单次完整 workflow 执行（5 节点 + 2 轮 loop）≈ 30K 输入 + 5K 输出 ≈ $0.17

### 方案 B：分层策略（Haiku 做简单事 + Sonnet 做复杂事）

```json
{
  "agent": {
    "build": {
      "model": "anthropic/claude-sonnet-4-6",
      "permission": { "bash": "allow", "edit": "allow" }
    },
    "plan": {
      "model": "anthropic/claude-haiku-4-5",
      "permission": { "bash": "deny", "edit": "deny" }
    }
  }
}
```

让 Haiku 处理只读操作（plan/status 查看），Sonnet 处理实际 workflow 执行。

### 方案 C：Opus 兜底（高可靠场景）

当 workflow 是生产级 SOP（不允许出错）时，用 Opus：

```json
{
  "agent": {
    "build": {
      "model": "anthropic/claude-opus-4-7",
      "permission": { "bash": "allow", "edit": "allow" }
    }
  }
}
```

## 7. 结论

| 模型 | 推荐度 | 适用范围 | 成本/次(5节点) |
|---|---|---|---|
| **Sonnet 4.6** | ⭐⭐⭐⭐⭐ | 所有场景 | ~$0.17 |
| **Opus 4.7** | ⭐⭐⭐⭐ | 复杂/高可靠 | ~$0.43 |
| **Haiku 4.5** | ⭐⭐ | 仅简单直线 workflow | ~$0.03 |

**最终建议**：用 **Sonnet 4.6** 作为 OpenCode 中 agent-workflow 的默认 caller 模型。Haiku 单独用不建议——如果实在想省钱，用分层策略（方案 B）让 Haiku 只做 plan/查询。

---

## 8. 验证建议

如果想亲自验证 Haiku 的表现，建议用以下测试序列：

1. **能过**：测试 1（executors）、测试 2（templates）、测试 3（validate）— 确定性操作
2. **边界**：测试 4（mock 端到端）— 简单 chain，Haiku 大概率能过
3. **关键测试**：测试 8（loop 3 轮）— Haiku 在第 2-3 轮容易出错
4. **必挂**：手动构造一个 10 节点 workflow 带 continue 返回 — Haiku 几乎必然误处理

在 OpenCode 中切模型非常方便（Tab 键切 agent），可以同一个 workflow 分别用 Haiku 和 Sonnet 跑一遍对比。
