## Q1 — 作用对象：caller-only / +spawn 文本前缀 / +spawn 原生 API

### 选项 A：仅 caller 节点 ★

CLI 把 `agent` 字段原样塞到 caller 收到的 payload；spawn executor（claude/codex）忽略该字段。

**caller 收到**：
```json
{
  "action": "execute_agent",
  "payload": {
    "prompt": "请设计 ...",
    "agent": {
      "role": "You are a senior backend architect.",
      "skills": ["knowledge-compiler"]
    }
  }
}
```

| 指标 | 值 |
|---|---|
| 代码改动 | ~50 行 |
| spawn 行为 | 不变 |
| 测试成本 | 低（caller payload 单测） |
| 适合阶段 | **v1.5.4 推荐** |

**优点**：零侵入；保持 spawn 协议简洁
**缺点**：spawn executor 拿不到 agent 上下文

---

### 选项 B：caller + spawn 文本前缀

选项 A 全部 + 对 spawn executor，CLI 自动把 `role` 拼到 stdin prompt 前。

**spawn 实际收到的 stdin**：
```
=== Role ===
You are a senior backend architect.

=== Available skills (advisory) ===
- knowledge-compiler

=== Task ===
请设计 ...
```

| 指标 | 值 |
|---|---|
| 代码改动 | ~80 行 |
| spawn 行为 | stdin 前缀拼接 |
| 测试成本 | 中（需 spawn 集成测试） |
| 适合阶段 | v1.5.5 |

**优点**：spawn 端也能用 role 影响输出风格
**缺点**：
- prompt 总长度增加（可能溢 context window）
- 文本拼接破坏 prompt 完整性（LLM 可能把 `=== Role ===` 当业务输入）
- skills 列表对 spawn executor 无实际意义

---

### 选项 C：caller + spawn 原生 API

选项 B 全部 + spawn executor 改用原生 system flag（如 `claude --system "<role>"`），而非拼接 stdin。

**spawn 命令变化**：
```bash
# 之前
echo "请设计..." | claude

# 改为
echo "请设计..." | claude --system "You are a senior backend architect."
```

| 指标 | 值 |
|---|---|
| 代码改动 | ~150 行 |
| spawn 行为 | 原生 system flag |
| 测试成本 | 高（多 LLM 真测） |
| 适合阶段 | v1.6 大版本 |

**优点**：行为最"原生"，最像真实 system prompt
**缺点**：
- 需逐个 LLM CLI 验证 flag 支持（codex CLI 据查未必有 `--system`）
- 老 workflow cmd 模板需要兼容性处理

---

## Q2 — skills 语义：建议 / 白名单 / 推迟

### 选项 A：仅建议 ★

`agent.skills` 是"参考列表"，CLI 不做运行时强制。caller 自己决定是否真去加载。

**示例**：
```yaml
agent:
  skills:
    - "knowledge-compiler"
    - "不存在的-skill"   # CLI 不报错；caller 自己处理
```

**校验**：类型 array of string，每项 1-64 字符，列表 0-20 项。**不**校验 skill 真实性。

**优点**：实现极简；caller 可自由扩展含义
**缺点**：用户拼错 skill 名时无提示
**适用**：MVP，先把字段定下来

---

### 选项 B：硬白名单

`agent.skills` 是 caller 在此节点**唯一可用**的 skill 列表。CLI 在 start 时校验是否注册。

**校验**：必须存在于全局 skill 注册表（如 `~/.agent-workflow/registered-skills.yaml`），失败抛 `SKILL_NOT_REGISTERED`。

**优点**：能在 start 时拦截拼写错误
**缺点**：
- 需要建立 skill 注册表（agent-workflow 不应负责管理 skill 元数据）
- caller 千差万别，CLI 强制白名单意义有限（caller 不一定听）
- 与 Cursor / Claude Code 的 skill 体系耦合，迁移性差

**适用**：v2 与 skill-store 协同时

---

### 选项 C：v1.5.4 不上 skills，仅 role

只加 `agent.role`，不引入 `agent.skills`。等 v1.5.5 真实场景反馈后再决定。

**优点**：避免过早设计；focus 最高价值的 role
**缺点**：用户已提 skills 需求，推迟可能让他失望

---

## Q3 — 字段命名：agent.xxx / context.xxx / 平面

### 选项 A：`agent: {role, skills}` 嵌套 ★

```yaml
- alias: design
  type: agent_call
  executor: caller
  agent:
    role: "You are an architect."
    skills: ["knowledge-compiler"]
  prompt: "..."
  output: doc
```

**优点**：分组清晰；未来加 `agent.tools` / `agent.persona` / `agent.memory` 易扩展
**缺点**：嵌套深一层

---

### 选项 B：`context: {role, skills}` 嵌套

```yaml
context:
  role: "You are an architect."
  skills: ["knowledge-compiler"]
```

**优点**：泛意命名，未来非 agent 场景（如 wait_user 上下文）也可复用
**缺点**：语义不够具体，易与"用户上下文 / 业务上下文"混淆

---

### 选项 C：平面字段 `role:` + `skills:`

```yaml
- alias: design
  type: agent_call
  executor: caller
  role: "You are an architect."
  skills: ["knowledge-compiler"]
  prompt: "..."
  output: doc
```

**优点**：YAML 最简洁，少一层缩进
**缺点**：未来再加新字段（tools/memory/persona）会让节点字段越来越平铺，与 `description` / `timeout_ms` 等混在一起、无分组

---

## Q4 — 交付范围：MVP / 完整 caller / 完整两端

### 选项 A：只改 schema + parser + SKILL.md

**做**：schema 加 `agent` 字段，parser L2 自动通过，SKILL.md 增加描述

**不做**：caller payload 透传 / example yaml / 单测

**优点**：30 分钟交付，让 yaml 能合法写 `agent:`
**缺点**：caller 收不到这些字段，**用户写了等于白写**（不闭环）

**适用**：纯文档承诺，不真交付

---

### 选项 B：完整 caller 闭环 ★

选项 A 全部 +
- `lib/executors/caller.py`：把 node 的 `agent` 塞进 payload
- `lib/nodes/agent_call.py`：把 `agent` 从 node 提取归一化
- 新增 `examples/agent-with-role.yaml`
- 新增 `tests/agent-workflow/unit/test_agent_context.py`

**优点**：caller 侧端到端可用；用户写了立即生效
**缺点**：~1 小时交付

**适用**：v1.5.4 推荐

---

### 选项 C：完整 caller + spawn 两端

选项 B 全部 +
- `lib/executors/base.py` SpawnExecutor：role 拼到 stdin 前缀（或注入 system flag，依 Q1）
- 真实 claude 集成测试
- examples 增加 claude executor + role 的 yaml

**优点**：spawn executor 也能用 role
**缺点**：~2 小时；需真实 claude 集成测试

**适用**：v1.5.5 或一起做（依用户优先级）

## 推荐组合的最终效果（用 Q1A + Q2A + Q3A + Q4B 演示）

**YAML 写法**：
```yaml
nodes:
  - alias: design
    type: agent_call
    executor: caller
    agent:
      role: |
        You are a senior backend architect.
        Focus on scalability, security, and observability.
      skills:
        - "knowledge-compiler"
        - "swagger-api-reader"
    prompt: |
      请设计 {{topic}} 的 API 架构。
    output: design_doc
```

**caller 收到的 payload**：
```json
{
  "result": {
    "action": "execute_agent",
    "payload": {
      "prompt": "请设计 OAuth 登录的 API 架构。",
      "node": { ... },
      "agent": {
        "role": "You are a senior backend architect.\nFocus on scalability, security, and observability.",
        "skills": ["knowledge-compiler", "swagger-api-reader"]
      }
    }
  }
}
```

caller agent 收到 `payload.agent.role` 后，可把它设为本次推理的 system prompt；收到 `payload.agent.skills` 后，可决定是否真的加载这些 skill。**CLI 不强制**，给 caller 最大灵活性。
