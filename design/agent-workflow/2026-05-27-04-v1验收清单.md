## 2. 节点类型验收

| 节点 | 必做 | 验收标准 |
|------|:---:|---------|
| `agent_call` (executor=caller) | ✅ | start/advance 流程通；不存渲染后 prompt 到 history |
| `agent_call` (executor=mock) | ✅ | mock 立即返回预设 output；不真实 spawn |
| `agent_call` (executor=claude，stdin) | ✅ | spawn `claude` 命令，prompt 从 stdin 传入；exit_code != 0 → EXECUTOR_NONZERO_EXIT |
| `agent_call` (executor=codex，stdin) | ✅ | 同上，命令 `codex exec` |
| `agent_call` (executor=custom) | ✅ | 通过 ~/.agent-workflow/config.yaml 配置任意 cmd/args/input_mode；input_mode 支持 stdin/args |
| `agent_call` output 必填 | ✅ | YAML 缺 output → L2 失败 |
| `agent_call` context_files | ✅ | 外部 executor spawn 时，文件内容拼到 prompt 前；caller 不注入 |
| `agent_call` timeout | ✅ | 超时 → SIGTERM 子进程 → code=NODE_TIMEOUT |
| `wait_user` | ✅ | 返回 action=wait_user；state=waiting_user；payload 含 message+schema+node 元信息 |
| `wait_user` schema 校验 | ✅ | 字段 string/boolean/number；顶层 object；required 列表；default 值 |
| `wait_user` 在 loop.body 内 | ✅ | 每轮 body 执行到 wait_user → 阻塞 → resume 后继续 body；body 完重新评估 condition |
| `loop` 正常退出 | ✅ | condition 转 false → 退出，节点 completed |
| `loop` max 退出 | ✅ | condition 仍 true 且 iteration_count==max → code=LOOP_EXCEEDED |
| `loop` max=0 | ✅ | body 不执行直接进入下一节点（合法语义） |
| `loop` 嵌套 loop | ✅（最少 2 层） | 内外 loop_counters 用 internal_id 区分 |
| `sleep` 基本 | ✅ | seconds=2，advance/resume 调用阻塞约 2 秒后返回；events.ndjson 见 sleep_start/sleep_end |
| `sleep` 字段校验 | ✅ | seconds 缺失 → L2 失败；seconds<0 或 >300 → L2 失败；seconds 非数值 → L2 失败 |
| `sleep` 含 `{{var}}` | ✅ | start vars 传 `{wait:3}`，节点 `seconds: "{{wait}}"` 正确生效 |
| `sleep` 在 loop.body 内 | ✅ | 轮询场景 TC：每轮 check + sleep；max_iterations 兜底 |
| `sleep` 不写 vars | ✅ | 节点执行完 vars 中无新字段；history 记录 status=completed |

**v1.5 推迟**：`condition`（if-else 节点）/ `parallel` / `sub_workflow` / 节点级 `retry` / `sleep` 异步版（可 abort）

## 4. 持久化验收

## 5. 校验验收

| 能力 | 必做 | 验收标准 |
|------|:---:|---------|
| L1 YAML 语法 | ✅ | PyYAML 错误 → 输出 line/col |
| L2 JSON Schema | ✅ | Ajv 校验；errors 含 location（json pointer） |
| L3 alias 同层唯一 | ✅ | 跨层（loop.body）允许重名 |

| 错误对象包含 location | ✅ | line / col / json pointer / suggestion 至少有一项 |

## 12. v1 上线后回看清单

收到至少 3 个真实使用反馈后，回看以下问题再启动 v1.5：

1. caller 节点的 prompt 渲染是否够用（要不要加多模板系统）
2. claude/codex 真实 spawn 速度（1-3 秒开销可接受吗）
3. wait_user terminal 模式 vs agent-interact 弹窗，用户偏好
4. 用户写 YAML 的痛点（要不要自然语言转 YAML 工具）
5. 错误信息是否足够帮 AI 自我修正（连续修 3 轮内成功率）
6. events.ndjson 的实际用途（要不要做查询工具）

如这些问题中超过 2 个有明显改进空间 → 启动 v1.5，否则继续打磨稳定性。
