# Agent Workflow 设计审核

> 日期：2026-05-27
> 审核对象：`2026-05-27-01-设计文档.md`、`2026-05-27-02-流程介绍.html`、`skills/agent-workflow/` 当前骨架
> 结论：设计方向成立，但不建议直接进入 Phase 5 实现。需要先收敛 v1 语义边界、错误协议、循环模型和 executor 安全策略，否则实现阶段会出现契约反复和不可兼容的状态文件。

## 总体判断

当前方案的核心方向是合理的：用 CLI + YAML + 落盘状态机实现跨 IDE workflow，用 `caller` executor 保留当前 IDE agent 的上下文能力，用外部 CLI executor 做跨 LLM 调度。这几个关键选择是可落地的。

主要问题不在方向，而在 v1 范围内同时塞入了过多能力，并且部分章节之间存在语义冲突：文档说 v1 不做 DAG，但 schema/校验出现 `edges`；文档说 loop 支持用户反馈迭代，但又限制 loop body 不嵌套 `wait_user`；错误输出同时有字符串、`action=error` 和结构化 error object 三套形态。这些需要在实现前定稿。

## P0 必须先修

### 1. v1 流程模型混入 DAG 概念，执行顺序不明确

设计在非目标中明确 v1 不做 DAG，只支持串行 + 串行内 loop/wait_user（`2026-05-27-01-设计文档.md:88`），但校验机制和 schema 又引入了 `edges`（`:882`、`:988`），失败示例也使用 `edges[3].to`。同时 YAML 示例只有 `nodes` 顺序，没有 `edges` 或 `next`。

影响：parser、engine、validate 会不知道到底按数组顺序执行，还是按 edges 执行。AI 生成 YAML 时也会混用两种模型，导致同一个 workflow 在不同版本实现中行为不同。

建议：v1 删除 `edges`，明确 `nodes` 数组即执行顺序；`loop.body` 内部同理按数组顺序。若必须保留未来扩展，只在“v2 预留”描述，不进入 v1 schema、validate 和错误示例。

### 2. loop + wait_user 的设计互相冲突，用户反馈迭代示例无法正确结束

示例中 `refine_loop.condition` 使用 `{{review.approved}} == false`，body 只做一次 refine，没有再次询问用户（`:381-400`）。如果用户第一次拒绝，`review.approved` 不会变化，循环会一直执行到 `max_iterations` 后失败。另一方面，L3 校验又写了 `loop.body 不嵌套 wait_user（v1 限制）`（`:882`），这和“多轮用户反馈优化”的核心场景冲突。

影响：`iterative-refine`、跨 LLM 设计评审等主打场景会天然失败；用户无法在每轮 refine 后重新 approve。

建议二选一：

- v1 允许 `loop.body` 包含 `wait_user`，并规定每次 body 后重新评估 condition。
- 或者 v1 不支持“用户确认式 loop”，只支持自动质量分循环；把用户反馈迭代放到 v1.5，并从 v1 模板中移除。

更推荐第一种，因为 `wait_user + loop` 是本设计最核心的差异化能力。

### 3. 错误协议不统一，调用方 agent 无法稳定分支处理

输出格式章节里错误是字符串（`:591-597`）；成功结果里 `action` 又允许 `"error"`（`:560-564`）；错误处理章节又定义了顶层结构化 error object（`:1108-1125`）。F1-F6 表格中的 CLI 返回还混用了 `{action: "error", error}`（`:1095-1100`）。

影响：SKILL.md 无法给调用方 agent 写出可靠的判断逻辑。到底看 `error != null`，还是看 `result.action == "error"`，会直接影响 `advance/resume` 的控制流。

建议：统一为一套协议：

```json
{
  "result": null,
  "error": {
    "code": "NODE_TIMEOUT",
    "message": "...",
    "node_id": "design",
    "retryable": false,
    "suggestion": "..."
  }
}
```

`result.action` 只保留 `execute_agent`、`wait_user`、`done`。用户输入校验失败这类可重试错误也走顶层 `error`，用 `retryable: true` 表达。

### 4. `agent_call.output` 在 schema 中非必填，但成功路径强依赖它

schema 只要求 `id/type/prompt`（`:991-1000`），但成功路径无条件写 `vars[node.output]`（`:1084-1085`）。如果 AI 生成了没有 `output` 的 `agent_call`，运行时会出现 undefined key 或直接失败。

影响：validate 通过的 workflow 可能 start 后崩溃，破坏“AI 生成后 validate 即可信”的目标。

建议：v1 将 `agent_call.output` 设为必填。若未来需要 side-effect-only 节点，显式增加 `output: null` 或 `save_output: false`，不要让缺省行为隐含。

### 5. prompt 通过命令参数和 state/history 记录存在安全与隐私风险

executor 模板使用 `claude -p "{{prompt}}"`、`codex exec --prompt "{{prompt}}"`，history 示例记录完整 `cmd`（`:503`、`:860`、`:1120`）。虽然风险表提到用 spawn 数组避免 shell 注入，但完整 prompt 仍可能出现在进程列表、shell/系统审计、state.json、错误详情中。prompt 往往包含源码、需求、用户反馈甚至密钥片段。

影响：本地工具也会造成敏感信息落盘和进程侧泄漏；一旦 workflow 目录被同步或提交，风险扩大。

建议：v1 默认使用 `stdin` 或临时文件传 prompt；state/history 只记录 `cmd` + 参数模板，不记录渲染后的 prompt。需要 debug 时用显式参数开启 `--show-prompt` 或写入独立受控文件，并默认脱敏。

## P1 建议实现前收敛

### 6. `running` 状态同时表示“等待 caller advance”和“外部 executor 正在执行”

Run 状态表把 `running` 定义为“caller 节点等待 advance（或非 caller 节点正在 spawn）”（`:813-816`）。这两个状态的可操作性不同：前者允许 `advance`，后者不应该允许外部 `advance`，只能 `status/abort`。

影响：并发控制和错误提示会复杂化；agent 看到 `running` 不知道下一步该执行节点还是等待 CLI 内部 chain。

建议：增加 `pending_action` 或拆分状态，例如 `awaiting_agent`、`executing_external`、`waiting_user`。至少在 state 中记录 `current_action` 和 `expected_command`，让 `status` 可明确提示下一步只能做什么。

### 7. L4 executor 缺失在 `start` 时仅 warning，容易制造半途失败的 run

设计规定 `start` 对 L4 只 warning 仍启动（`:887-890`）。如果 workflow 前几个节点是 caller，后面才到 `codex`，用户会在做完人工步骤后才失败。

影响：长流程里浪费用户时间，且失败后 v1 不支持 recover，只能新 run。

建议：默认 `start` 对当前 workflow 用到的非 caller executor 做强校验，缺失则拒绝启动。若确实要允许延迟失败，增加 `allow_missing_executors: true` 或 `--force`，并要求返回 warnings 让 caller 明确征询用户。

### 8. 变量引用校验需要做数据流分析，不能只看初始 `vars`

L3 写“template `{{var}}` 引用变量已定义”（`:882`），但 workflow 变量会由前序 `agent_call.output`、`wait_user` 和 loop body 动态产生。示例里的 `research_result`、`design_doc` 都不是初始 vars，而是前序节点输出。

影响：如果只按顶层 `vars` 校验，会误报；如果完全不校验，又无法发现引用未来节点、拼写错误和 loop 作用域问题。

建议：定义 v1 数据流规则：

- 初始可用变量 = 顶层 `vars` + start 入参 vars。
- 每个 `agent_call.output` 在该节点完成后加入可用变量集合。
- 每个 `wait_user` 将 `vars[node.id]` 加入可用变量集合。
- `loop.body` 可读 loop 前变量，body 输出在 loop 后仍可见。
- 引用当前节点之后才产生的变量报错。

### 9. 外部 executor 自动 chain 会让长流程缺少可观测进度

设计选择 v1 非 caller 节点全部在 CLI 内部连续执行，直到遇到 caller/wait_user/done 才返回（`:465-466`、`:1320-1323`）。如果连续多个外部 LLM 节点耗时数分钟，调用方只看到一个阻塞 shell 命令。

影响：IDE agent 和用户不知道卡在哪个节点；中途 abort/status 的体验不清晰。

建议：v1 至少落地 `events.ndjson` 或在 stdout 支持 progress JSON lines。若坚持单 JSON 输出，必须保证另一个进程可 `status` 查到当前 `executing_external` 节点、开始时间和最近事件。

### 10. v1/v1.5 范围有多处矛盾

文档前面把 `abort`、`prune` 放在管理命令表中（`:100-104`、`:783-790`），实现计划又把 `abort`、`prune` 放到 Phase 2（`:1198-1206`）。`audit.log` 在持久化章节列为三件落盘文件之一（`:743-749`），但又标注 v1 可选/v1.5。`retry` 在节点 schema 示例中出现（`:450-459`），但决策 Q15 推荐 v1.5。

影响：验收口径不清，测试计划无法判断 v1 到底必须实现哪些命令和文件。

建议：做一张“v1 必须 / v1 可选 / v1.5 / v2”能力矩阵，并让 CLI 命令、schema、目录结构、测试计划严格对齐。v1 不实现的字段不要进入 v1 schema。

## P2 可后续优化

### 11. wait_user schema 过于简化

示例 schema 使用 `approved: boolean`、`feedback: string`（`:381-387`），但没有定义 required、默认值、数组/object、枚举、空字符串策略，也没有说明 schema 校验器是自研还是 JSON Schema 子集。

建议：直接复用 JSON Schema 子集，或者明确只支持 `{ field: "string|boolean|number" }` 且所有字段默认必填。不要介于两者之间。

### 12. 状态目录默认全局，项目隔离和清理策略需要提前想清楚

推荐 `~/.agent-workflow/runs` 并允许环境变量覆盖（Q2），但 workflow YAML 属于项目资产。全局 runs 会混合多个项目，`list` 输出可能噪声较大，也存在跨项目 prompt/history 混放问题。

建议：state 中强制记录 `project_root`，`list` 默认只显示当前项目相关 runs；全局列表需要显式参数。

### 13. 当前实现骨架与设计差距很大，SKILL.md 尚不能指导 agent

当前 `skills/agent-workflow/tool.js` 只有 `example` 命令（`skills/agent-workflow/tool.js:12-17`），`SKILL.md` 仍是 TODO（`skills/agent-workflow/SKILL.md:8`、`:26-34`、`:47-51`），测试也是占位（`tests/agent-workflow/src/unit/test_example.js:15-21`）。

建议：在 Phase 5 之前先把 SKILL.md 的调用协议写到可执行程度，哪怕 CLI 未实现，也要先固定命令名称、输入输出和 agent 分支逻辑。否则实现过程中会反复改协议。

## 建议的下一步

1. 先开一次 Phase 4 决策，只讨论 P0/P1 中的协议问题，不进入代码实现。
2. 修改设计文档：删除 v1 `edges`；修正 loop 示例；统一错误协议；明确 `output` 必填；补充 prompt 安全策略。
3. 产出 v1 能力矩阵和验收清单，再开始 Phase 5。
4. 第一批实现只做最小闭环：`validate`、`start`、`advance`、`resume`、`status`、`caller executor`、mock external executor。真实 `claude/codex` 适配放在核心状态机稳定之后。
