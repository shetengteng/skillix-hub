跨 IDE 通用

CLI 形态发布，Claude Code / Cursor / Codex / OpenCode / GitHub Copilot 的 agent 都能 shell spawn 调用，不绑定任何 IDE 私有抽象。

跨 LLM 协作

每个节点声明 executor：设计交给 Claude、实现交给 Codex、评审回到当前 agent。CLI 不内嵌 SDK，spawn 你已登录的 CLI。

状态机驱动

每个 Run 一份 state.json，崩溃可恢复、wait_user 可暂停几天、多 Run 并发互不干扰，文件锁串行推进。

💡

一句话设计哲学

CLI 是纯状态机，自身不调用任何 LLM API。 节点执行由两类角色完成：caller（当前 IDE 的 agent，利用 IDE 内置 LLM 与工具能力）和 外部 LLM CLI（spawn 子进程）。 workflow 用 YAML 描述，状态全部落盘可恢复。

§4.4 节点

四类节点（v1.5.1）

三类核心 + 一个原子 sleep。每个节点共享 alias / type / description 三个通用字段（CLI 自动生成 UUID 主键）。

A

agent_call

让某个 agent 或 LLM CLI 执行一段 prompt，输出写回 vars。

- alias: design
  type: agent_call
  description: 由 Claude 产出技术设计
  executor: claude
  prompt: |
    基于调研结果设计：
    {{research_result}}
  output: design_doc        # v1.5 必填
  timeout: 600
  context_files: ["./README.md"]
executor:caller / claude / codex / opencode / gemini / 自定义
prompt:支持 {{var}} 插值
output:结果写回 vars 的字段名
W

wait_user

阻塞节点。CLI 把 status 设为 waiting，落盘后退出，等用户调 resume 才继续。

- alias: review
  type: wait_user
  description: 用户审阅设计文档
  message: 请审阅设计文档，是否通过？
  schema:                  # v1.5 JSON Schema 子集
    type: object
    required: [approved]
    properties:
      approved: { type: boolean }
      feedback: { type: string }
message:展示给终端用户的提示语
schema:约束 user_input 结构（可选）
超时:默认 7 天未 resume 自动 abort
L

loop

条件循环。v1.5：body 内允许嵌套 wait_user / sleep，每轮完重新评估 condition。

- alias: refine_loop
  type: loop
  description: 用户反馈式循环
  condition: "{{review.approved}} == false"
  max_iterations: 3
  body:
    - alias: review        # ✅ v1.5 允许
      type: wait_user
      message: 是否通过？
    - alias: refine
      type: agent_call
      executor: claude
      output: design_doc
condition:白名单运算符 == != < > && ||
上限:max_iterations 最大 100
嵌套:body 含 wait_user / sleep / agent_call
S

sleep

v1.5.1 新增

原子节点。同步等待 N 秒（0-300），任何位置都能插入做 rate limit / 轮询间隔。

- alias: cool_down
  type: sleep
  description: 防 LLM rate limit
  seconds: 5             # 必填，可 {{var}}

# 典型：loop.body 内做轮询
- alias: poll_loop
  type: loop
  condition: "{{job_status}} != 'done'"
  body:
    - alias: check
      type: agent_call
      ...
    - alias: gap
      type: sleep
      seconds: 5
seconds:0-300（5 分钟上限）
实现:v1 同步 time.sleep()；不持久化、不可 abort（v1.5 异步）
无 output:不写 vars，纯时间流逝

名称 默认命令 是否 v1 内置 行为
caller — 不 spawn 子进程 ✅ v1 返回 execute_agent 给调用方 agent

§4.10 重点回答

执行记录如何处理？

是的：每启动一次 workflow = 创建一个独立 Run = 一份 state.json。多次启动互不干扰。

1️⃣

一次启动 = 一个 run_id

格式 wf-YYYY-MM-DD-xxxxxx，例：wf-2026-05-27-a3f9c1

2️⃣

独立目录

每个 Run 一个文件夹：~/.agent-workflow/runs/<run_id>/

3️⃣

3 个文件

state.json（运行状态）+ workflow.yaml（启动时快照）+ audit.log（append-only）

4️⃣

并发安全

不同 run_id 完全独立；同 run_id filelock 文件锁串行

目录结构

~/.agent-workflow/
├── config.yaml             # 全局：executor 配置 + defaults
└── runs/
    ├── wf-2026-05-27-a3f9c1/  # 第 1 次启动（completed）
    │   ├── state.json
    │   ├── workflow.yaml
    │   └── audit.log
    ├── wf-2026-05-27-b8d2e3/  # 第 2 次启动（waiting）
    │   ├── state.json
    │   └── workflow.yaml
    └── wf-2026-05-27-c4f1a2/  # 第 3 次启动（running）
        ├── state.json
        └── workflow.yaml

3 个文件的职责

state.json
运行时状态：current_node / vars / history / status。每次 advance/resume 更新。
workflow.yaml
启动时的工作流定义快照。只读——避免源 YAML 改了导致状态对不上。
audit.log
append-only 审计日志。每次 advance/resume/spawn 追加一行。v1.5 引入，v1 可选。
💡
为什么 workflow.yaml 要快照？
避免你在 run 进行到一半时修改源 YAML，导致 advance 解析时 schema 对不上 vars/history。快照让 Run 完全可复现。

保留策略（可在 config.yaml 改）

策略默认
Run 完成后永久保留
wait_user 超时7 天
history 长度最近 100 条
磁盘上限不限（v1.5）

并发场景

✓
同一 workflow YAML 启动多个 Run（不同 run_id）→ 完全独立
✓
同一 run_id 串行 advance/resume → 文件锁保护
✗
同一 run_id 并行 advance → 后者返回 error: run busy
✓
不同 IDE/进程操作同 run_id → 跨进程文件锁排队

§4.11 校验 (v1.4)

Workflow 校验机制

AI 生成的 YAML 可能出错，本 skill 内置 4 级校验体系，错误信息带 code / location / line / suggestion，AI 可自动修正。

最严 L1

YAML 语法

PyYAML 能否成功解析

YAML parse error at line X col Y

严 L2

Schema 结构

JSON Schema 校验字段类型、必填、枚举值

nodes[2].type expected enum

中 L3

引用一致性

edges 引用、template 变量、loop 嵌套

unknown node 'finalreview'

宽 L4

环境就绪

executor 在 PATH 中、timeout 合理

'codex' not found in PATH

校验时机

命令 强制级别 失败行为
start L1+L2+L3 拒绝启动，不创建 Run（L4 仅 warning）
validate L1+L2+L3+L4 不实际执行，返回完整报告
advance L1+L2（重读快照） 节点 status=failed

AI 自我修正循环

1. AI 生成 workflow.yaml
2. AI 调 validate
3. if errors:
     ├─ 读 errors[].code / location / line
     ├─ 自动修复 YAML
     └─ 回到第 2 步（最多 3 轮）
4. if warnings (L4):
     └─ 提示用户安装 CLI 或改用 caller
5. if valid:
     └─ 调 start 启动

§4.13 错误处理 (v1.4)

节点结果与错误处理

每个节点的成功 / 失败路径都有明确定义，错误对象规范化便于 AI 自动决策。

✅ 成功路径

节点类型 成功判定 vars 写入 状态变化
agent_call
caller		 advance 时 result.error == null 且 output 非空 vars[node.output] = result.output completed → 下一节点
agent_call
外部 CLI		 spawn exit_code == 0 且 stdout 非空 vars[node.output] = parseOutput(stdout) completed → 下一节点
wait_user resume 时 user_input 通过 schema 校验 vars[node.id] = user_input completed → 下一节点
loop iteration_count < max_iterations 且 condition true —— 进入 body

❌ 失败路径（6 种情况）

# 失败原因 节点 status run status 关键错误码
F1 caller advance 传 result.error failed failed CALLER_ERROR
F2 非 caller spawn 非零退出 failed failed EXECUTOR_NONZERO_EXIT
F3 非 caller spawn 超时 failed failed NODE_TIMEOUT
F4 非 caller stdout 为空或解析失败 failed failed EMPTY_OUTPUT
F5 wait_user user_input 不符 schema waiting（不变） waiting SCHEMA_VIOLATION (retryable)
F6 loop 超过 max_iterations 仍 true failed failed LOOP_EXCEEDED
💡 F5 特殊："用户输错"不是节点逻辑错。节点状态不变，CLI 返回 retryable: true，提示 caller 重新索取输入再 resume，不让整个 run 进入 failed。

📐 统一错误对象规范（v1.5）

v1.5 关键变化：错误协议统一为顶层 error 对象。删除原 action="error" 形态。result.action 仅保留 execute_agent / wait_user / done。 
{
  "result": null,
  "error": {
    "code": "NODE_TIMEOUT",
    "message": "node 'design' (7f3a9c2e) timed out after 600s",
    "node": {
      "internal_id": "7f3a9c2e",             // UUID 主键
      "alias": "design",                     // 用户起的语义化名
      "description": "由 Claude 基于调研结果产出技术设计"
    },
    "executor": "claude",
    "details": {
      "exit_code": 124,
      "stderr_tail": "...",
      "cmd_template": "claude (stdin)"      // v1.5 不存渲染后 prompt
    },
    "retryable": false,
    "suggestion": "Increase timeout or split prompt"
  }
}
code：机器可读错误码，AI 据此分支处理
retryable：是否可直接重试（F5/SCHEMA_VIOLATION 为 true）
suggestion：人类可读修复建议，AI 透传给用户
§4.3 示例

完整跨 LLM Workflow YAML

调研 → Claude 设计 → 用户审阅 → 循环优化 → Codex 实现 → 当前 agent 评审

# v1.5 修订：删 edges / alias 主键 / loop.body 含 wait_user（用户反馈式循环）
name: cross-llm-pipeline
version: 1.0

vars:
  topic: ""
  max_refine: 3

nodes:
  - alias: research                       # v1.5 alias 替代 id；CLI 自动生成 UUID 主键
    type: agent_call
    description: 调研主题，给出核心要点列表
    executor: caller
    prompt: |
      调研主题：{{topic}}
    output: research_result               # v1.5 必填

  - alias: design
    type: agent_call
    description: 由 Claude 基于调研结果产出技术设计
    executor: claude
    prompt: |
      基于调研结果设计：{{research_result}}
    output: design_doc
    timeout: 600
    context_files:                     # v1.5：注入项目文件给外部 LLM
      - "./README.md"

  - alias: refine_loop
    type: loop
    description: 用户反馈式循环（v1.5：body 内允许 wait_user）
    condition: "{{review.approved}} == false"
    max_iterations: "{{max_refine}}"
    body:
      - alias: review                    # v1.5 关键：每轮先问用户
        type: wait_user
        description: 用户审阅本轮设计
        message: 是否通过？
        schema:                         # v1.5 JSON Schema 子集
          type: object
          required: [approved]
          properties:
            approved: { type: boolean }
            feedback: { type: string }
      - alias: refine
        type: agent_call
        description: 根据反馈优化设计
        executor: claude
        prompt: "反馈：{{review.feedback}}\n原设计：{{design_doc}}"
        output: design_doc

  - alias: implement
    type: agent_call
    description: 由 Codex 实现代码
    executor: codex
    prompt: "基于设计实现：{{design_doc}}"
    output: code

  - alias: final_review
    type: agent_call
    description: 回到当前 agent 做最终评审
    executor: caller
    prompt: "评审代码风格：{{code}}"
    output: final_report
# 注意：v1.5 移除 edges，仅按 nodes 数组顺序执行
