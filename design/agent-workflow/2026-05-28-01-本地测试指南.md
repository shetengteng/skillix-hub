## 5. TC-D：Mock 2 节点端到端 ★

**目的**：用 mock executor 跑通 `start → 内部 chain → completed` 最小状态机闭环，**无需 caller agent 介入**。

**步骤 1：写一个 2 节点 mock workflow**

```bash
cat > /tmp/aw-smoke/mock-smoke.yaml <<'EOF'
name: tc-d-mock-smoke
version: "1.0"
description: 最小 2 节点 mock workflow，验证状态机闭环
nodes:
  - alias: ping
    type: agent_call
    executor: mock
    prompt: "ping"
    output: pong
  - alias: echo
    type: agent_call
    executor: mock
    prompt: "echo {{pong}}"
    output: final
EOF
```

**步骤 2：用环境变量启用 mock + 注入输出，然后 start**

```bash
AGENT_WORKFLOW_ENABLE_MOCK=1 \
AGENT_WORKFLOW_MOCK_PING="hello world" \
AGENT_WORKFLOW_MOCK_ECHO="echoed: hello world" \
python3 skills/agent-workflow/tool.py start '{
  "workflow":"/tmp/aw-smoke/mock-smoke.yaml",
  "caller":"manual-test"
}'
```

**预期输出**：

```json
{
  "result": {
    "action": "completed",
    "run_id": "wf-20260528-090030-ec63e7",
    "status": "completed",
    "vars": {
      "pong": "hello world",
      "final": "echoed: hello world"
    }
  }
}
```

**通过判定**：
- `action: "completed"`（一次 start 内部自动 chain 完两个 mock 节点，因为 mock 不需要 caller 推理）
- `vars.pong` = 你注入的字符串
- `vars.final` 包含上一节点 output（验证 `{{pong}}` 模板渲染生效）
- `.agent-workflow/runs/` 下能看到新增的 `wf-YYYYMMDD-HHMMSS-XXXXXX/` 目录，包含 `state.json` / `workflow.yaml` / `events.ndjson` / `audit.log`

**关键 gotcha**（实测踩坑）：
- ✅ mock 启用方式 = **环境变量** `AGENT_WORKFLOW_ENABLE_MOCK=1`
- ❌ **不要**在 YAML 写 `executors: { mock: {} }`——会被 `_normalise_spec` 强制要求 `cmd` 字段，报错 `PARAMS_INVALID: executor 'mock' requires 'cmd'`（这是个边界 case，已记入待修清单）
- 注入输出的环境变量命名规则：`AGENT_WORKFLOW_MOCK_<ALIAS 大写>=<text>`，alias 内部大小写不敏感
- 不注入时默认输出 `mock-<alias>` 字符串

## 6. TC-E：view 浏览器可视化 ★

**目的**：验证 v1.5.3 view 子系统 SPA 改造（Python 端只输出 JSON 数据 + 浏览器侧 JS 渲染）的回归正确性。

```bash
python3 skills/agent-workflow/tool.py view '{}'
```

**预期**：浏览器自动打开 `.agent-workflow/views/index.html`。

页面构成：

- 表格列：`run_id` 短形式（wf-xxxxxx）/ `name` / status badge / 节点进度（完成/总数）/ `updated`

**通过判定**：
- 浏览器自动打开
- 表格至少 1 行（TC-D 刚跑过的那个 run）
- 顶部 status filter 点 `completed` 只剩跑通的 run
- search 框输入 `mock` 能过滤出 TC-D 的 run
- 右上角 theme 切换（light / dark）正常

### 步骤 2：单 run 详情页

把 TC-D 返回的 `run_id` 代入：

```bash
python3 skills/agent-workflow/tool.py view '{"run_id":"wf-20260528-090030-ec63e7"}'
```

**预期**：浏览器打开 `workflow.html#<run_id>`，页面有 5 个分区：

1. **Header**：workflow name + run_id + status badge + caller + created/updated
2. **Nodes**：节点列表（mock executor、duration、cursor 当前位置高亮）
3. **History**：表格展示每个节点执行记录（status / output_head / duration_ms）
4. **Events**：时间流形式展示 `events.ndjson` 的事件
5. **vars**：JSON 形式展示 workflow 变量

**通过判定**：
- 5 个分区全部渲染出来
- 节点列表里有 `ping` 和 `echo`，status 都是 completed
- Events 流里能看到 `run_start` / `node_start` / `node_end` / `run_end`
- vars 里看到 `pong: "hello world"` 和 `final: "echoed: hello world"`
- F5 刷新不报错（说明 SPA 重新加载 `data.js` 正常）

## 附录 A：故障排查

| 现象 | 原因 | 解决 |
|---|---|---|
| `ModuleNotFoundError: yaml` 等 | 缺依赖 | `pip3 install -r skills/agent-workflow/requirements.txt` |
| validate 报 `executor 'mock' requires 'cmd'` | YAML 写了 `executors: { mock: {} }` | 删掉这段，改用环境变量 `AGENT_WORKFLOW_ENABLE_MOCK=1` |
| validate 报 `VAR_NOT_IN_SCOPE` | 节点 prompt 引用了未定义的 `{{var}}` | 检查 `vars:` 定义或前置节点的 `output:` 字段 |
| `EXECUTOR_NOT_FOUND: claude` | `claude` CLI 不在 PATH | `which claude` 检查；可在 workflow 里 override `executors.claude.cmd` 写绝对路径 |
| view 命令后浏览器没弹出 | 系统未配 default browser，或 SSH 远程 | 加 `"open": false`，拿到 url 后手动打开 |
| view 页面空白 | `.agent-workflow/views/_assets/data.js` 没被加载 | 直接打开 url 走 `file://` 协议，无需起 server；Chrome 严格模式下可能拦 file 协议，换 Safari 或 `python3 -m http.server` |
