# Agent Workflow — 本地测试指南（自然语言版）

> 日期：2026-06-01
> 适用版本：v1.5.3
> 适用对象：首次接触 agent-workflow 的开发者，希望快速验证功能是否正常

本指南用**场景故事**的方式引导你完成 5 个测试，每个测试都是一个"你想做什么 → 系统应该怎么反应"的对话。无需事先了解内部实现。

---

## 你需要先准备什么？

确认 Python 版本 ≥ 3.10，然后在项目根目录执行一次安装：

```bash
pip3 install -e skills/agent-workflow
```

安装完成后你就有了 `python3 skills/agent-workflow/tool.py` 这个入口命令。后面的每个测试都是对它发一条指令。

---

## 测试 1：我想知道系统支持哪些执行器

**你的意图**：了解当前环境能用哪些 AI Agent 来执行 workflow 节点。

**怎么做**：运行 `executors` 命令，不需要任何参数。

**你应该看到**：系统返回一个列表，包含 4 种执行器：
- **caller** — 表示"由当前 AI Agent（也就是你）自己来推理"
- **claude** — 表示可以 spawn Claude CLI 来做推理
- **codex** — 表示可以 spawn Codex CLI 来做推理
- **mock** — 测试用的假执行器（默认禁用）

如果你的机器装了 `claude` 或 `codex` 命令，对应的 `available_in_path` 会是 `true`；没装也没关系，不影响后续测试。

---

## 测试 2：我想看看有哪些现成的 workflow 模板

**你的意图**：不想从零写 YAML，想看系统自带了哪些可以直接用的模板。

**怎么做**：运行 `create` 命令，参数指定 `action` 为 `list_templates`。

**你应该看到**：返回 4 个模板，每个都有名字、描述和节点数量：
- **code-review** — 代码审查流程
- **cross-llm-pipeline** — 多个 LLM 接力协作
- **iterative-refine** — 迭代优化（循环直到满意）
- **research-and-implement** — 先调研再实现

---

## 测试 3：我想用模板创建一个 workflow 并验证它是否合法

**你的意图**：从模板生成一份 workflow 文件，然后让系统做全面校验，确保语法、结构、数据流、执行器都没问题。

**怎么做**：分两步——

1. **从模板生成**：用 `create` 命令的 `from_template` 动作，选 `iterative-refine` 模板，输出到一个临时文件。
2. **四级校验**：用 `validate` 命令指向刚生成的文件。

**你应该看到**：
- 第一步返回确认文件已写入
- 第二步返回校验结果，`violations` 为空（没有错误），`levels_checked` 包含 L1~L4 四级全部通过：
  - L1 = YAML 语法正确
  - L2 = 结构符合 JSON Schema
  - L3 = 数据流合法（变量引用都能找到来源）
  - L4 = 执行器在 PATH 中可用

## 测试 4：我想跑通一个完整的 workflow（端到端闭环）

**你的意图**：验证 workflow 从启动到完成的完整生命周期，包括节点间数据传递。用 mock 执行器避免真正调用 LLM。

**怎么做**：

1. **写一个最小 workflow**：2 个节点——`ping` 节点产出一段文字，`echo` 节点引用上一个节点的结果再产出最终结果。
2. **通过环境变量启用 mock 并注入输出**：
   - `AGENT_WORKFLOW_ENABLE_MOCK=1` 启用 mock 执行器
   - `AGENT_WORKFLOW_MOCK_PING="hello world"` 让 ping 节点返回这段话
   - `AGENT_WORKFLOW_MOCK_ECHO="echoed: hello world"` 让 echo 节点返回这段话
3. **启动 workflow**：用 `start` 命令。

**你应该看到**：
- 状态直接变为 `completed`（因为 mock 不需要外部推理，内部自动 chain 完所有节点）
- `vars` 中包含两个变量：`pong = "hello world"` 和 `final = "echoed: hello world"`
- 项目目录下 `.agent-workflow/runs/` 里出现一个新的 run 目录，包含执行记录

**关键知识**：
- mock 是通过环境变量启用的（不是在 YAML 中配置）
- 环境变量命名规则是 `AGENT_WORKFLOW_MOCK_<节点ALIAS大写>=<输出内容>`

## 测试 5：我想在浏览器里看 workflow 的执行状态

**你的意图**：用可视化界面查看所有 workflow run 的列表和详情。

**怎么做**：分三步体验——

1. **看总览**：运行 `view` 命令不带参数，浏览器自动打开一个表格页面。
2. **看单个 run 详情**：运行 `view` 命令并指定上一步测试产生的 `run_id`，浏览器打开详情页。
3. **CI 模式（不开浏览器）**：运行 `view` 命令加 `"open": false`，只返回文件路径不弹浏览器。

**总览页应该有**：
- 状态筛选（all / running / completed / failed …）
- 搜索框
- 表格：每行一个 run，显示名称、状态、节点进度、更新时间
- 点击某行可以跳转到详情

**详情页应该有 5 个区域**：
- 顶部信息（workflow 名、run_id、状态、创建时间）
- 节点列表（每个节点的执行器、耗时、当前进度）
- 执行历史表格
- 事件时间流
- 变量 JSON 面板

## 测试完毕后的清理

测试过程中会产生以下副作用文件，可按需删除：
- `/tmp/aw-smoke/` — 模板生成的临时文件
- `.agent-workflow/runs/` 下的 mock 测试 run
- `.agent-workflow/views/` — view 命令生成的 HTML

## 小结

| 测试 | 验证的能力 | 是否需要联网 |
|---|---|---|
| 1. 列执行器 | CLI 入口 + 注册表 | 否 |
| 2. 列模板 | 模板系统 | 否 |
| 3. 模板生成 + 校验 | Builder + 4 级 Validator | 否 |
| 4. Mock 端到端 | 状态机完整闭环 + 变量传递 | 否 |
| 5. 浏览器可视化 | View 子系统 SPA | 否 |

全部通过说明本地环境一切正常，可以进入更复杂的集成场景（真实 LLM 协作、长链超时、用户阻塞等）。
