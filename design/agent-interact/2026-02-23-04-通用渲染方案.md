# Agent Interact — 通用渲染方案

> 日期：2026-02-23

## 1. 问题

当前 agent-interact 有 7 种预定义交互类型（confirm、wait、chart、form、approval、progress、notification），每种类型有固定的 UI 模板和数据结构。

**局限性**：

| 问题 | 说明 |
|------|------|
| 模板固定 | LLM 只能在 7 种预设模板中选择，无法自由组合 |
| 扩展成本高 | 每新增一种交互类型，需要前后端同时开发新组件 |
| 无法混合内容 | 不能在一个弹框中同时展示表格 + 图表 + 按钮 |
| LLM 创造力受限 | LLM 可能想展示一个自定义的对比表格，但没有对应模板 |

**期望**：LLM 通过一个 JSON 描述，自由编排弹框内容（标题、文本、表格、图表、按钮、输入框等），前端根据 JSON 动态渲染。

## 2. 方案优化：Schema + Registry 双层架构

### 2.1 核心思路

在 `custom` 的基础上补齐两层能力：

1. **协议层（Schema）**：用严格 JSON Schema 定义可用字段和结构，服务端先校验再广播。  
2. **渲染层（Registry）**：前端通过 `kind -> renderer` 注册表渲染，新增组件只需“加组件 + 注册”，不改大段分支逻辑。

```json
{
  "type": "custom",
  "schemaVersion": "1.0",
  "title": "API 分析报告",
  "content": [
    { "kind": "text", "value": "以下是过去 7 天的 API 性能分析：" },
    { "kind": "table", "columns": ["端点", "P50", "P99", "错误率"], "rows": [["/api/users", "45", "320", "0.1%"]] },
    { "kind": "chart", "chartType": "line", "data": { "labels": ["Mon"], "datasets": [{ "label": "P99", "data": [320] }] } },
    { "kind": "divider" },
    { "kind": "text", "value": "建议：优先优化 /api/orders 的慢查询" }
  ],
  "actions": [
    { "id": "optimize", "label": "开始优化", "variant": "default", "submit": true, "requireValid": true },
    { "id": "ignore", "label": "暂不处理", "variant": "outline", "submit": false }
  ],
  "meta": {
    "maxWidth": "xl",
    "scroll": "auto"
  }
}
```

### 2.2 可用组件类型（按职责分层）

| 分类 | kind | 关键属性 |
|------|------|----------|
| 展示类 | `text`, `heading`, `divider`, `badge`, `alert`, `code`, `image`, `kv`, `progress` | `value`, `level`, `variant`, `language`, `src`, `items` |
| 数据类 | `table`, `chart` | `columns/rows`, `chartType/data` |
| 输入类 | `input`, `textarea`, `select`, `checkbox` | `id`, `label`, `default`, `required`, `options`, `placeholder` |
| 布局类 | `row`, `column`, `grid`, `group`, `section` | `children`, `columns`, `gap`, `title` |

> `group` 保留兼容，推荐新场景优先用 `row/column/grid/section`，避免布局表达力不足。

### 2.3 actions 协议增强

所有 `custom` 弹框包含 `actions`，除现有字段外新增语义字段：

| 字段 | 含义 |
|------|------|
| `submit` | 是否提交表单数据（默认 `true`） |
| `requireValid` | 是否要求输入校验通过（默认 `false`） |
| `closeOnSubmit` | 提交后是否自动关闭（默认 `true`） |

### 2.4 响应协议（可诊断）

用户点击按钮后统一返回：

```json
{
  "dialogId": "d-xxx",
  "action": "optimize",
  "data": { "reason": "慢查询告警" },
  "valid": true,
  "errors": []
}
```

校验失败时返回（或前端阻止提交并展示）：

```json
{
  "dialogId": "d-xxx",
  "action": "optimize",
  "valid": false,
  "errors": [
    { "fieldId": "reason", "message": "部署原因必填" }
  ]
}
```

### 2.5 容错与错误反馈策略

- **默认严格模式**：服务端 Schema 校验失败直接 400，返回 `path + message`，不进入渲染阶段。  
- **前端降级模式**：仅在兼容老数据时启用，对未知 `kind` 显示 “Unsupported 卡片”，不静默吞掉。  
- **版本兼容**：通过 `schemaVersion` 做向前兼容，不同版本走不同校验器。

## 3. 与现有类型的关系

| 维度 | 预定义类型（confirm 等） | custom 类型 |
|------|------------------------|-------------|
| 适用场景 | 标准交互模式 | 自由编排 |
| 开发成本 | 已完成 | 需要新增通用渲染器 |
| LLM 使用难度 | 低（参数少） | 中（需要构造 content 数组） |
| UI 一致性 | 高（固定模板） | 中（取决于 LLM 的编排能力） |
| 推荐使用 | 简单交互优先用预定义类型 | 复杂展示或自定义需求用 custom |

**策略**：预定义类型和 custom 类型共存。LLM 优先使用预定义类型（更简单、UI 更一致），当预定义类型无法满足需求时使用 custom。

## 4. 技术实现（针对当前代码结构）

### 4.1 后端：请求校验与归一化

`server.js` 不能只加 `validTypes`，还需要增加 `custom` 专属校验流程：

1. `validTypes` 新增 `'custom'`  
2. 对 `request.type === 'custom'` 执行 Schema 校验（建议 Ajv）  
3. 校验通过后做 `normalize`（填默认值、裁剪超限内容）  
4. 再进入 `dm.create()` 阻塞链路

推荐错误返回：

```json
{
  "error": "Invalid custom schema",
  "details": [
    { "path": "/content/2/rows/301", "message": "rows exceeds limit 200" }
  ]
}
```

### 4.2 前端：注册表式递归渲染

新增 `CustomDialog.vue` + `ContentRenderer.vue`，并使用渲染注册表替代硬编码分支：

```
CustomDialog.vue
  └─ ContentRenderer.vue（递归）
       └─ rendererRegistry[kind](node, ctx)
```

注册表示例（概念）：

```ts
const rendererRegistry = {
  text: TextRenderer,
  heading: HeadingRenderer,
  table: TableRenderer,
  chart: ChartRenderer,
  input: InputRenderer,
  select: SelectRenderer,
  row: RowRenderer,
  column: ColumnRenderer,
  grid: GridRenderer,
}
```

### 4.3 类型定义（严格联合）

```typescript
type DisplayNode =
  | { kind: 'text'; value: string }
  | { kind: 'heading'; value: string; level?: 1 | 2 | 3 | 4 }
  | { kind: 'divider' }
  | { kind: 'alert'; value: string; level?: 'info' | 'warning' | 'error' | 'success' }
  | { kind: 'badge'; value: string; variant?: 'default' | 'success' | 'warning' | 'error' }

type InputNode =
  | { kind: 'input'; id: string; label: string; required?: boolean; placeholder?: string; default?: string }
  | { kind: 'textarea'; id: string; label: string; required?: boolean; placeholder?: string; default?: string }
  | { kind: 'select'; id: string; label: string; options: string[]; required?: boolean; default?: string }
  | { kind: 'checkbox'; id: string; label: string; default?: boolean }

type LayoutNode =
  | { kind: 'row'; children: CustomNode[]; gap?: 'sm' | 'md' | 'lg' }
  | { kind: 'column'; children: CustomNode[]; gap?: 'sm' | 'md' | 'lg' }
  | { kind: 'grid'; children: CustomNode[]; columns?: 2 | 3 | 4; gap?: 'sm' | 'md' | 'lg' }
  | { kind: 'section'; title?: string; children: CustomNode[] }

type DataNode =
  | { kind: 'table'; columns: string[]; rows: string[][]; sortable?: boolean }
  | { kind: 'chart'; chartType: 'line' | 'bar' | 'pie' | 'doughnut' | 'radar'; data: ChartData }

type CustomNode = DisplayNode | InputNode | LayoutNode | DataNode

interface CustomAction extends DialogAction {
  submit?: boolean
  requireValid?: boolean
  closeOnSubmit?: boolean
}

interface CustomDialog extends BaseDialog {
  type: 'custom'
  schemaVersion: '1.0'
  content: CustomNode[]
  actions?: CustomAction[]
}
```

### 4.4 安全策略（必须落地）

- Markdown 渲染必须走 sanitize（禁用脚本和危险属性）  
- `image.src` 仅允许 `https`（或受控白名单）  
- `code` 仅文本展示，不执行  
- 输入字段 `id` 仅允许字母数字下划线，防止原型污染键名

### 4.5 性能边界（服务端兜底）

- `content` 最大长度：40  
- 最大嵌套深度：4  
- 表格最大行数：200  
- 图表总点数：1000  
- 超限直接返回 400，避免前端卡顿

## 5. 实施计划（按优先级）

### P0（先解决核心问题）

- 后端增加 `custom` 严格校验（Schema + 错误路径返回）
- 前端新增 `CustomDialog.vue` + `ContentRenderer.vue`（注册表机制）
- 首批支持：`text/heading/table/divider/alert/input/select/checkbox/row/column`
- 响应体支持 `valid/errors`

### P1（补齐复杂展示）

- 支持 `chart/code/image/kv/progress/grid/section`
- Markdown sanitize 与图片 URL 限制
- 完善窗口尺寸策略（custom 默认更高、内容区滚动）

### P2（体验优化）

- Schema 版本升级机制（`1.0 -> 1.1`）
- 渲染性能监控（渲染耗时、节点数量统计）
- LLM 侧最佳实践模板（提示词示例 + 反例）

## 6. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| LLM 生成非法 JSON | 弹框失败、任务中断 | 服务端严格校验并返回 `path + message` |
| 未知组件被静默忽略 | 用户看到不完整界面 | 默认严格拒绝；兼容模式展示 Unsupported 卡片 |
| 富文本/图片引入安全风险 | XSS、恶意资源加载 | Markdown sanitize + URL 白名单 |
| 内容过载导致窗口卡顿 | 交互延迟、体验差 | 服务端限制节点/深度/数据量，前端滚动容器 |
| 自由布局导致 UI 不一致 | 可读性下降 | 提供布局规范与示例模板，限制布局节点组合复杂度 |

## 7. 缓解方案落地细则

### 7.1 LLM 生成非法 JSON

- 在 `server.js` 增加 `validateCustomDialog(request)`，对 `custom` 请求做 Schema 校验。  
- 校验失败直接返回 400，并带 `details: [{ path, message }]`，Agent 可据此自动重试。  
- 在 `SKILL.md` 增加“失败重试模板”：收到 400 时仅修正报错路径字段，不重写整份 JSON。

### 7.2 未知组件被静默忽略

- 默认开启严格模式：存在未知 `kind` 直接拒绝请求，不进入前端。  
- 兼容模式仅用于历史数据回放：前端渲染 `Unsupported` 占位卡片，显示 `kind` 与错误原因。  
- 记录未知 `kind` 统计（次数、来源 Agent、schemaVersion），用于决定是否正式支持该组件。

### 7.3 富文本 / 图片安全风险

- Markdown 渲染前统一 sanitize，禁用 HTML 标签中的脚本相关属性。  
- `image.src` 限制为 `https` 或配置白名单域名，拒绝 `javascript:`、`file:`、`data:`。  
- 长文本与代码块做长度限制（如 20KB），防止渲染阻塞和内存抖动。

### 7.4 内容过载导致窗口卡顿

- 服务端增加硬限制：`content <= 40`、`depth <= 4`、`tableRows <= 200`、`chartPoints <= 1000`。  
- 前端 `CustomDialog` 内容区固定最大高度并启用滚动，不随内容无限增高。  
- 渲染器按节点数量输出耗时日志（仅开发模式），用于识别性能瓶颈组件。

### 7.5 自由布局导致 UI 不一致

- 定义布局白名单：`row/column/grid/section`，限制嵌套层级和组合复杂度。  
- 提供 3 套推荐模板（分析报告、部署确认、对比结果），LLM 优先套模板再填数据。  
- PR 审核新增检查项：`custom` JSON 必须通过 Schema 校验且符合布局规范。
