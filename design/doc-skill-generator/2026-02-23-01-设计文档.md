# Doc Skill Generator 设计文档

> 日期：2026-02-23
> 状态：设计阶段

## 1. 需求

构建一个 Skill 生成器，专门从**文档**（网页、PDF、本地文件）中提取内容，基于提取的内容自动生成可用的 Cursor Skill。

### 核心场景

1. 用户给出一个 API 文档 URL → 生成一个调用该 API 的 Skill
2. 用户给出一个产品使用手册 PDF → 生成一个操作该产品的 Skill
3. 用户给出多个文档源 → 合并理解后生成一个综合 Skill
4. 用户给出一个 CLI 工具文档 → 生成一个封装该 CLI 的 Skill

### 关键需求

- 支持多种文档源：网页（含 SPA）、PDF、本地 Markdown/文本文件
- 提取内容尽可能全面（不丢失关键信息）
- 自动识别文档类型和结构（API 文档、教程、参考手册等）
- 生成的 Skill 符合 skill-builder 规范（SKILL.md + tool.js + package.json）
- 支持 install/update 命令

## 2. 整体流程

**两阶段确认流程**：先提取到临时文件 → 用户/LLM 检查确认 → 再生成 Skill。

```
用户输入文档源（URL / PDF / 文件路径）
        ↓
   ┌────────────────────┐
   │   文档采集层        │  ← Playwright BFS + PDF 解析
   │  (Document Fetcher) │
   └────────┬───────────┘
            ↓
   ┌────────────────────┐
   │   内容提取层        │  ← 结构化提取
   │  (Content Extractor)│
   └────────┬───────────┘
            ↓
   ┌────────────────────────────┐
   │   保存到临时文件            │  ← data/extracts/<id>.json
   │  (Intermediate Storage)     │
   └────────┬───────────────────┘
            ↓
   ┌────────────────────────────┐
   │   检查确认                  │  ← 用户/LLM 审查提取内容
   │  (Review & Confirm)         │     是否完整？是否有遗漏？
   └────────┬───────────────────┘
            ↓ 确认 OK
   ┌────────────────────┐
   │   文档分析层        │  ← LLM 理解文档意图
   │  (Doc Analyzer)     │
   └────────┬───────────┘
            ↓
   ┌────────────────────┐
   │   Skill 生成层      │  ← 生成代码 + 文档
   │  (Skill Generator)  │
   └────────┬───────────┘
            ↓
   生成的 Skill 目录
```

### 两阶段设计原因

1. **提取阶段耗时长**（Playwright 多页面 BFS 可能需要几分钟），不应该在提取失败时浪费 LLM 生成的时间
2. **提取结果可复用**：同一份文档可以生成多个不同的 Skill
3. **可人工干预**：用户可以在提取后修改/补充内容，再生成 Skill
4. **增量采集**：如果发现遗漏，可以追加采集而不是从头开始

## 3. 技术方案

### 3.1 文档采集层（Document Fetcher）

根据输入类型自动选择采集策略：

| 输入类型 | 策略 | 依赖 |
|---------|------|------|
| HTTP/HTTPS URL | **Playwright BFS 深度采集**（渲染 SPA + 展开 Tab/折叠 + 多页面遍历） | playwright skill |
| Swagger/OpenAPI URL | swagger-api-reader（解析 spec，生成 Markdown）— 可选 | swagger-api-reader skill |
| PDF 文件（本地路径） | pdf-parse 库解析 | pdf-parse npm 包 |
| PDF URL | 先下载到临时目录，再用 pdf-parse 解析 | node-fetch + pdf-parse |
| 本地 .md / .txt 文件 | 直接读取 fs.readFileSync | 无 |

> **统一使用 Playwright 采集网页**。Playwright 本身就能渲染 Vue/React/Angular 等 SPA 页面，
> 无需额外的 SPA 检测逻辑。所有网页（静态/SPA/多页面）统一走 Playwright BFS 流程。

**采集策略选择**：

```
输入 URL
  ↓
检测是否为 Swagger/OpenAPI → 是 → swagger-api-reader（可选，也可走 Playwright）
  ↓ 否
Playwright navigate → snapshot → LLM 分析页面结构
  ↓
LLM 返回：导航链接 / Tab / 折叠区域
  ↓
有导航链接 → BFS 遍历子页面
无导航链接 → 单页面深度采集（Tab + 折叠 + 懒加载）
```

#### 网页采集（Playwright 统一引擎）

统一使用 Playwright 作为网页采集引擎，不依赖 web-content-reader：

- **SPA 渲染**：Playwright 直接渲染 Vue/React/Angular 页面，无需 SPA 检测
- **内容提取**：通过 Playwright snapshot（无障碍树）+ evaluate（DOM 操作）提取结构化内容
- **多页面 BFS**：LLM 分析 snapshot 识别导航链接，BFS 遍历
- **页面内深度采集**：LLM 识别 Tab/折叠元素，逐个展开采集

#### PDF 解析

使用 `pdf-parse` npm 包：

```javascript
const pdfParse = require('pdf-parse');
const fs = require('fs');

async function readPdf(filePath) {
  const buffer = fs.readFileSync(filePath);
  const data = await pdfParse(buffer);
  return {
    text: data.text,
    pages: data.numpages,
    info: data.info,
    metadata: data.metadata,
  };
}
```

对于 PDF URL，先下载：

### 3.2 内容提取层（Content Extractor）

将原始内容转为结构化格式：

```json
{
  "source": "https://docs.example.com/api",
  "type": "webpage",
  "title": "Example API Documentation",
  "sections": [
    {
      "heading": "Authentication",
      "level": 2,
      "content": "All API requests require...",
      "codeBlocks": [
        { "language": "bash", "code": "curl -H 'Authorization: Bearer TOKEN' ..." }
      ],
      "tables": [],
      "links": []
    }
  ],
  "metadata": {
    "language": "en",
    "docType": "api-reference",
    "estimatedTokens": 5200
  }
}
```

#### 文档类型识别

通过内容特征自动识别：

| 文档类型 | 特征 |
|---------|------|
| API Reference | endpoints、HTTP methods、request/response、status codes |
| CLI Reference | commands、flags、options、usage |
| Tutorial | step-by-step、"first"/"then"/"next"、代码示例 |
| Configuration | settings、parameters、key-value pairs |
| SDK/Library | classes、methods、imports、installation |

#### Token 预算管理

文档可能很长，需要控制传给 LLM 的内容量：

- 计算每个 section 的 token 数
- 优先保留：标题、代码块、表格、API 端点
- 次要保留：说明文本
- 可裁剪：示例输出、changelog、FAQ
- 总预算：根据模型上下文窗口动态调整

### 3.3 文档分析层（Doc Analyzer）

由 LLM 完成，输入结构化文档，输出 Skill 规格：

```json
{
  "skillName": "example-api-client",
  "description": "调用 Example API 的客户端 Skill",
  "commands": [
    {
      "name": "listUsers",
      "description": "获取用户列表",
      "params": [
        { "name": "page", "type": "number", "required": false, "default": 1 },
        { "name": "limit", "type": "number", "required": false, "default": 20 }
      ],
      "implementation": "GET /api/users?page={{page}}&limit={{limit}}"
    }
  ],
  "dependencies": ["node-fetch"],
  "authType": "bearer",
  "baseUrl": "https://api.example.com"
}
```

**分析策略**：

1. **API 文档** → 提取 endpoints、参数、认证方式 → 生成 HTTP 客户端 Skill
2. **CLI 文档** → 提取命令、参数、选项 → 生成 CLI wrapper Skill
3. **SDK 文档** → 提取类、方法、初始化 → 生成 SDK adapter Skill
4. **教程** → 提取步骤、代码示例 → 生成 workflow Skill
5. **配置文档** → 提取配置项、默认值 → 生成 config manager Skill

## 8. 文档采集策略详解

### 核心思路：Playwright 驱动的深度采集

网页文档不是简单的静态 HTML。现代文档站点有：
- **多 Tab 内容**：同一页面内有多个 Tab 面板（如 API 文档的 Request / Response / Example）
- **折叠区域**：Accordion、Collapse、Details 等可展开组件
- **懒加载**：滚动到底部才加载更多内容
- **分页**：内容分布在多个页面
- **嵌套链接**：侧边栏导航 → 子页面 → 子子页面

**必须使用 Playwright 在真实浏览器中操作**，才能获取完整内容。

### 采集算法：BFS（广度优先）

使用 BFS 而非 DFS，原因：
1. 优先采集同层级的所有页面，保证核心内容先获取
2. 避免深度过深导致采集偏离主题
3. 更容易控制采集范围和超时

```
BFS 采集流程:

Queue = [入口 URL]
Visited = {}
Results = []

while Queue 非空 && Results.length < maxPages:
    url = Queue.dequeue()
    if url in Visited: continue
    Visited[url] = true

    // Step 1: 导航到页面
    playwright.navigate(url)
    snapshot = playwright.snapshot()

    // Step 2: LLM 分析页面结构
    analysis = LLM.analyze(snapshot)
    // analysis = { navLinks, tabs, collapsible, hasMoreContent, contentArea }

    // Step 3: 页面内深度采集（根据 LLM 指令）
    for tab in analysis.tabs:
        playwright.click(tab)       ← 点击 Tab
        tabContent = extractContent(page)
        Results.push(tabContent)

    for item in analysis.collapsible:
        playwright.click(item)      ← 展开折叠区域

    if analysis.hasMoreContent:
        scrollToBottom(page)        ← 触发懒加载

    content = extractContent(page)
    Results.push(content)

    // Step 4: 发现子页面链接（LLM 已从 snapshot 中提取）
    for link in analysis.navLinks:
        if 同域 && depth < maxDepth && link not in Visited:
            Queue.enqueue(link)
```

### 页面内深度采集（关键）

在每个页面内，需要展开所有隐藏内容：

#### Tab 面板采集（LLM 驱动）

LLM 从 snapshot 中识别 Tab 元素并返回 ref 列表，crawler 逐个点击：

```javascript
async function expandTabs(page, tabRefs) {
  const results = [];
  for (const ref of tabRefs) {
    await execPlaywright('click', { ref });
    await page.waitForTimeout(300);
    const snapshot = await execPlaywright('snapshot', {});
    results.push(snapshot);
  }
  return results;
}
```

#### 折叠区域展开（LLM 驱动）

LLM 从 snapshot 中识别折叠元素并返回 ref 列表：

```javascript
async function expandCollapsible(page, collapsibleRefs) {
  for (const ref of collapsibleRefs) {
    await execPlaywright('click', { ref });
    await page.waitForTimeout(200);
  }
}
```

**关键设计**：不硬编码任何 CSS 选择器。所有元素识别由 LLM 通过 Playwright snapshot（无障碍树）完成，适配任意文档框架。

#### 懒加载触发

```javascript
async function scrollToBottom(page) {
  let previousHeight = 0;
  let retries = 0;

  while (retries < 5) {
    const currentHeight = await page.evaluate(() => document.body.scrollHeight);
    if (currentHeight === previousHeight) {
      retries++;
    } else {
      retries = 0;
    }
    previousHeight = currentHeight;
    await page.evaluate(() => window.scrollTo(0, document.body.scrollHeight));
    await page.waitForTimeout(500);
  }
}
```

### 导航链接提取（LLM 驱动）

**不硬编码 CSS 选择器**。页面结构千变万化，应该让 LLM 通过 Playwright snapshot 自己判断：

```
采集流程:

1. Playwright navigate → 目标 URL
2. Playwright snapshot → 获取页面无障碍树（accessibility tree）
3. 将 snapshot 交给 LLM 分析：
   - 识别导航区域（侧边栏、目录、菜单）
   - 提取导航链接列表
   - 识别 Tab 面板和折叠区域
   - 判断哪些元素需要点击展开
4. LLM 返回结构化指令：
   - 需要访问的子页面 URL 列表
   - 需要点击的 Tab 元素 ref
   - 需要展开的折叠元素 ref
5. 执行 LLM 的指令，采集内容
```

**LLM 分析 Prompt**：

```
你是一个文档采集助手。以下是一个文档页面的无障碍树快照。

请分析页面结构，返回 JSON：
{
  "navLinks": ["url1", "url2", ...],     // 导航中的子页面链接
  "tabs": ["ref1", "ref2", ...],          // 需要点击的 Tab 元素
  "collapsible": ["ref3", "ref4", ...],   // 需要展开的折叠区域
  "hasMoreContent": true/false,           // 是否有分页/加载更多
  "contentArea": "ref5"                   // 主内容区域的 ref
}

页面快照：
{{snapshot}}
```

**优势**：
- 不依赖特定框架的 CSS 类名
- 能处理任意文档站点（Docusaurus、GitBook、VuePress、自定义等）
- LLM 能理解语义，区分导航链接和正文链接
- 能识别非标准的 Tab/折叠实现

## 9. 与 LLM 的交互设计

doc-skill-generator 的核心是 LLM 分析和代码生成。设计原则：

1. **结构化输入** → 给 LLM 的不是原始文本，而是结构化的 sections + codeBlocks + tables
2. **分步生成** → 先生成 Skill 规格（JSON），确认后再生成代码
3. **模板辅助** → 提供 tool.js / SKILL.md 模板，LLM 只需填充业务逻辑
4. **Token 控制** → 长文档分批处理，每批控制在 token 预算内

### LLM Prompt 设计

```
你是一个 Skill 生成器。根据以下文档内容，生成一个 Cursor Skill 的规格。

文档类型：{{docType}}
文档内容：
{{structuredContent}}

请输出 JSON 格式的 Skill 规格，包含：
- skillName: 建议的 Skill 名称
- description: 一句话描述
- commands: 命令列表（name, description, params, implementation）
- dependencies: npm 依赖
- authType: 认证方式（如有）
```

## 11. 风险与注意事项

| 风险 | 影响 | 缓解 |
|------|------|------|
| 文档内容过长超出 LLM 上下文 | 生成质量下降 | Token 预算管理 + 分批处理 |
| PDF 格式复杂（扫描件、多栏） | 提取质量差 | 降级为 OCR 或提示用户提供文本版 |
| 网页反爬（CAPTCHA、登录墙） | 无法采集 | Playwright 回退 + 提示用户手动提供 |
| 生成的代码有 bug | 用户体验差 | preview 预览 + 人工确认 |
| 多文档源内容冲突 | 生成混乱 | 标记来源，冲突时优先 API 文档 |
| 依赖 LLM 质量 | 生成结果不稳定 | 结构化 prompt + 模板约束 |
