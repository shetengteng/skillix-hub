# project-knowledge-base-nex (PKB) Skill 设计分析

---

## 1. PKB Skill 概述

### 1.1 定位

PKB 是一个**项目级文档生成与维护 Skill**，为软件项目创建 AI 可读的结构化文档体系。核心理念："人机共维"——AI 维护机械性事实（文件路径、依赖版本、API 签名），人类维护判断性内容（架构决策、业务约束、权衡取舍）。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| C4 架构模型 | 按 Context → Container → Component → Code 四层组织架构文档 |
| Diátaxis 分类 | 文档分为 Tutorial / How-to / Reference / Explanation 四象限 |
| Sphinx 发布 | 支持 MyST + sphinx-rtd-theme 生成 HTML/PDF |
| 双语支持 | 英文为源，通过 sphinx-intl + gettext 翻译到中文 |
| AI 可读性 | llms.txt 入口、YAML frontmatter、严格 heading 层级 |
| 质量审计 | 5 维度评分（Accuracy 30%, Completeness 25%, Findability 20%, Clarity 15%, Consistency 10%） |
| 过期检测 | 基于 git commit 对比和时间窗口的文档陈旧检测 |

## 2. 架构设计分析

### 2.1 文件结构

```
project-knowledge-base-nex/
├── SKILL.md                          # Skill 主入口（600 行，包含完整规范）
├── prompts.md                        # 预设提示词集合（AI Onboarding 用）
├── references/
│   ├── commands.md                   # 命令详细参考
│   ├── quality-rubric.md             # 质量评分标准
│   ├── drift-detection.md            # CI 漂移检测
│   ├── bilingual-sphinx.md           # 双语 Sphinx 配置
│   ├── ai-readability.md             # AI 可读性标准
│   ├── diagram-examples.md           # Mermaid 图表示例
│   └── changelog.md                  # 版本历史
└── templates/
    ├── docs/                         # 文档模板（00-overview ~ 03-workflows）
    ├── adr/template.md               # ADR 模板
    ├── changes/_template/            # 变更提案模板
    └── sphinx/                       # Sphinx 配置模板
```

**设计洞察**：PKB 是一个"纯 Markdown 规范"型 Skill——没有代码，全部逻辑通过 SKILL.md 中的自然语言指令让 AI Agent 解释执行。这与 llm-wiki-compiler 的设计哲学一致。

### 2.2 命令体系

#### 4 个 Workflow（编排型命令）

| Workflow | 触发 | 编排模式 | 步骤 |
|----------|------|---------|------|
| WF-1: Full Onboarding | `/PKB-onboard` | 流水线 + 并行 | init → discovery → core docs(串行) → extended(并行) → ops(并行) → verify |
| WF-2: Maintenance Loop | `/PKB-maintain` | 条件分流 | scan → triage → update → re-verify → report |
| WF-3: AI Onboarding | `/PKB-ai-onboard` | 迭代循环 | R1 基础 → R2 架构 → R3 模块 → 验证 |
| WF-4: Change Documentation | `/PKB-document-change` | 流水线 | ADR → proposal → impact scan → selective update → verify |

#### Single Command Engine（3 步引擎）

所有单命令（`/PKB-overview`、`/PKB-architecture` 等）走统一的三步流程：

1. **Discover** — 分析项目标记、扫描目录、读取关键源文件，展示 summary 等确认
2. **Generate** — 加载命令规范 + 模板 → 填充 → 验证引用路径
3. **Refine** — 去冗余 → 统一术语 → 写入 metadata footer → 预览 → 写入

### 2.3 质量保障体系

#### 验证门控

| 类型 | 数量 | 典型规则 |
|------|------|---------|
| Hard Gates | 10 项 | 非空输出、模板合规、有效引用、无 TODO 占位、Mermaid 语法、metadata footer... |
| Soft Gates | 8 项 | 图表存在、命名一致性、合理长度、所有权标记、Diátaxis 覆盖... |

#### 5 维质量评分

```
总分 = Accuracy × 0.30 + Completeness × 0.25 + Findability × 0.20 + Clarity × 0.15 + Consistency × 0.10
```

满分 5.0，每维度 1-5 分。

#### 覆盖率指标

| 指标 | 目标 |
|------|------|
| API 文档覆盖率 | ≥ 80% |
| ADR 覆盖率 | ≥ 70% |
| 模块文档覆盖率 | ≥ 90% |
| 文档新鲜度 | ≥ 70% |

### 2.4 过期检测机制

```
每个文档末尾的 <!-- PKB-metadata --> footer:
  last_updated: 2026-03-30
  commit: a1b2c3d
  updated_by: human+ai
  quality_score: 4.2
  doc_type: reference

检测逻辑:
  1. commit 与 HEAD 对比 → git log 计算 commits-behind
  2. 10+ commits → 严重过期（建议 regenerate）
  3. 30-60 天 → 轻度过期（建议 incremental update）
  4. 无 footer → 未追踪
```

## 3. 设计思路总结

### 3.1 核心设计原则

| 原则 | 体现 |
|------|------|
| **人机共维** | 每个文档标记 `maintained-by: human+ai`，明确区分 AI 可维护部分和需人工判断部分 |
| **声明式驱动** | 全部逻辑用自然语言规范描述，由 AI Agent 解释执行，零代码 |
| **标准优先** | C4 架构模型、Diátaxis 文档分类、llms.txt 规范——不造轮子，对接行业标准 |
| **渐进披露** | AI Onboarding 分三轮渐进加载知识，而非一次性 dump |
| **可验证性** | Hard Gates 保底、5 维度量化评分、覆盖率指标，文档质量可度量 |
| **防腐化** | metadata footer + git commit 对比实现自动过期检测 |

### 3.2 与 llm-wiki-compiler 的关键差异

| 维度 | llm-wiki-compiler | project-knowledge-base-nex (PKB) |
|------|-------------------|----------------------------------|
| **文档化对象** | 个人知识材料（文章、论文、笔记） | 软件项目（代码、架构、API） |
| **编译方向** | raw/ → wiki/（知识合成） | 项目源码 → man/（文档生成） |
| **编译输入** | Markdown 原始文件 | 代码文件、配置、目录结构 |
| **核心价值** | 多源信息合成 + 覆盖度追踪 | 架构分析 + 文档标准化 + 质量保障 |
| **Schema** | 由编译器和用户共维 | 由 C4 + Diátaxis 标准驱动 |
| **索引模型** | INDEX.md（主题索引） | index.md + llms.txt（层级索引 + AI 入口） |
| **过期机制** | mtime 对比 | git commit 对比 + 时间窗口 |
| **质量保障** | 覆盖度三级（high/medium/low） | 5 维度评分 + 10 Hard Gates + 8 Soft Gates |
| **发布能力** | 无（纯 Markdown） | Sphinx HTML/PDF + 双语 |

### 3.3 互补关系

两个 Skill 解决的是知识管理的不同面向：

```
                        ┌─────────────────────────┐
                        │   知识管理全景           │
                        └─────────────────────────┘
                               ┌───┴───┐
                        ┌──────┘       └──────┐
                  ┌─────┴──────┐       ┌──────┴─────┐
                  │ 个人知识库  │       │ 项目文档化  │
                  │ (llm-wiki) │       │ (PKB)      │
                  └─────┬──────┘       └──────┬─────┘
                        │                     │
              raw/ → wiki/           源码 → man/
              多源合成 + 覆盖度       架构分析 + 标准化
              个人积累               团队协作
```

## 4. 可复用的设计模式

以下是 PKB 中值得复刻到 skillix-hub 的通用设计模式：

### 4.1 Discover → Generate → Refine 三步引擎

通用的文档/内容生成管道：
1. Discover：分析上下文，展示 summary 等确认
2. Generate：加载规范 + 模板 → 填充
3. Refine：验证 → 去冗余 → 写入 metadata → 预览 → 写入

### 4.2 Metadata Footer 过期检测

基于 git commit 的文档陈旧检测：
```markdown
<!-- PKB-metadata
last_updated: YYYY-MM-DD
commit: short-hash
updated_by: human+ai
-->
```

### 4.3 Hard Gate / Soft Gate 验证框架

分两级验证：
- Hard Gate：必须通过，不通过则失败/重试
- Soft Gate：建议性，警告但不阻断

### 4.4 渐进式 AI Onboarding

分轮加载知识，每轮有验证：
- R1：宏观认知
- R2：架构深入
- R3：模块级细节

### 4.5 Intent Router 意图路由

用户自然语言输入 → 意图匹配 → 路由到对应 Workflow 或 Single Command。

## 5. 对 llm-wiki Skill 复刻的启示

| 从 PKB 借鉴的设计 | 应用到 llm-wiki 的方式 |
|-------------------|----------------------|
| Metadata footer 过期检测 | 为 wiki/concepts/ 文章添加 footer，支持基于 git 的陈旧检测 |
| 5 维质量评分 | 可简化为 3 维（Accuracy + Coverage + Consistency）用于 wiki-lint |
| Discover → Generate → Refine | 作为 wiki-compile 的通用编译管道模式 |
| Hard/Soft Gate | 用于 wiki-lint 的检查分级 |
| AI 可读性标准 | wiki/ 输出遵循 heading 层级、code block 标签等规范 |
| Intent Router | wiki 命令的自然语言路由 |

## 6. 风险与注意事项

1. **SKILL.md 过长（600 行）**：复刻时应考虑拆分为 SKILL.md（入口） + references/（详细规范），避免 Token 浪费
2. **纯声明式的局限**：复杂编译逻辑（如增量 mtime 对比）用自然语言描述可能导致 AI 执行不一致，关键路径应有 Python 兜底
3. **C4 模型不适用于个人知识库**：llm-wiki 的主题分类需要自己的 taxonomy，不能直接套用 C4
4. **Sphinx 栈过重**：个人知识库不需要 HTML 发布能力，保留纯 Markdown 即可
