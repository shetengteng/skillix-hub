# Knowledge Compiler — AI 检索层优化方案

> 日期：2026-04-11
> 目标：将 `Knowledge Compiler` 从“可阅读 wiki 生成器”进一步收敛为“AI 优先的知识检索层”
> 结论：后续优化应优先增强可检索性、可路由性、可验证性，而不是继续增加展示型文档特性

---

## 1. 背景

当前 `Knowledge Compiler` 已具备以下能力：

- 从源材料编译出结构化 wiki
- 对主题进行增量编译
- 为章节打覆盖度标签
- 基于 wiki 回答问题
- 提供 lint、browse、status 等辅助命令

但从当前设计演进来看，系统已经不再只是“写出一批 wiki 文档”，而是在承担一个更关键的角色：

> 为 AI 提供稳定、低歧义、低成本的知识检索入口。

因此，优化方向应从“文档展示效果”切换为“检索层契约设计”。

---

## 2. 优化目标

### 核心目标

把 Knowledge Compiler 收敛成一个面向 AI 的检索层，具备以下特征：

1. **可检索**
   - AI 能快速定位相关 topic
   - 不需要逐篇读全文做召回

2. **可路由**
   - AI 能根据问题，把请求路由到少量最相关的主题
   - 路由行为依赖明确字段，而不是隐式猜测

3. **可验证**
   - 检索字段和来源字段都有结构约束
   - lint 可以检查“检索质量”，而不只是“文档完整性”

4. **低歧义**
   - topic 的边界、摘要、典型问答场景要清晰
   - 同类 topic 之间要能区分职责

### 非目标

以下方向不应成为当前版本的主要优化重点：

- 继续增强“给人读的排版装饰”
- 强制每篇文章都追求强视觉效果
- 在没有明确检索收益时继续堆叠 Mermaid 或复杂模板元素

---

## 3. 当前问题

## 问题 1：`INDEX.md` 更像展示页，不像检索注册表

当前 `INDEX.md` 的主要结构是分类表格，偏向人工浏览。

问题在于：

- 对 AI 来说，分类表格不是最稳定的召回结构
- `summary / tags / answers` 这些关键信息只存在于 concept frontmatter
- AI 为了路由问题，不得不再去打开多篇 concept 的 frontmatter

这使得：

- 首轮召回成本偏高
- 路由逻辑依赖模型临场发挥
- 一致性和性能都不够稳

---

## 问题 2：检索字段已经很关键，但仍缺少硬约束

当前设计里，以下字段已经承担检索职责：

- `summary`
- `tags`
- `answers`

但这些字段虽然被强调，却还没有形成完整的数据契约：

- 长度没有严格边界
- 内容质量没有门控
- 主题间去重与边界控制不足

这会导致：

- 摘要过泛
- tags 太少或太噪
- answers 写得像介绍而不是问题
- 多个 topic 对同一个问题都“看起来能答”

---

## 问题 3：命令层尚未完全按“检索层模式”工作

当前虽然在 `SKILL.md` 中已经提出“渐进式披露协议”，但命令层仍然偏文档工作流：

- `kc query`：仍主要描述“读 INDEX → 读文章”
- `kc browse`：偏展示视图
- `kc status`：偏统计视图

如果目标是 AI 检索层，那么命令层应该更明确地体现：

- 先召回
- 再筛选
- 再展开正文
- 最后必要时回源验证

---

## 问题 4：质量门控主要检查文档完整性，缺少检索质量门控

现有 Hard/Soft Gate 更多关注：

- frontmatter 是否完整
- coverage 是否齐全
- source 是否有效
- schema 是否一致

但真正影响 AI 检索效果的问题还没有被检查，例如：

- 摘要是否过于空泛
- answers 是否能形成精准路由
- tags 是否足够区分主题
- topic 之间是否存在问答边界重叠

---

## 问题 5：Mermaid 被强制要求，但对检索层收益有限

Mermaid 对“人理解架构”有帮助，但对“AI 首轮检索”帮助很小。

如果每篇文章都强制包含 1 个 Mermaid 图，会带来几个问题：

- 增加生成负担
- 增加文档噪音
- 对纯概念型 topic 收益不高
- 会把系统继续拉向“展示型 wiki”而不是“检索层”

---

## 问题 6：多语言策略会显著增加检索复杂度

当前已经支持 `language=zh/en`。

但对于 AI 检索层，语言切换不是简单的“标题翻译”问题，而会直接影响：

- summary 的召回质量
- tags 的语义一致性
- answers 的问题匹配能力
- 多语言 topic 之间是否需要双份索引

如果不控制，会出现：

- 中文问题召回英文 answers 不稳定
- tags 跨语言不一致
- 相同 topic 在不同语言下检索结果偏差大

---

## 4. 推荐优化方向

## 优化 1：把 `INDEX.md` 升级为 Topic Registry

### 目标

让 `INDEX.md` 成为 AI 的稳定入口，而不仅是给人看的分类页。

### 做法

在 `INDEX.md` 中新增一个机器优先的注册表区块，例如：

```md
## Topic Registry

| slug | title | summary | tags | answers | coverage | updated | sources_count |
|------|-------|---------|------|---------|----------|---------|---------------|
| api-gateway-design | API 网关设计 | 网关的职责边界、认证策略与流量入口设计 | api-gateway, auth, traffic | API 网关做了哪些设计取舍？; 认证在什么层处理？ | medium | 2026-04-11 | 3 |
```

### 收益

- `kc query` 可直接在 registry 层做首轮召回
- AI 不需要为每个问题先打开多篇文章 frontmatter
- topic 元数据集中、可测试、可比较

### 建议

保留“给人看的分类视图”，但把 `Topic Registry` 作为 Layer 1 的核心结构。

---

## 优化 2：把 frontmatter schema 升级成硬性检索契约

### 建议字段

每篇 concept frontmatter 至少应包含：

```yaml
id:
title:
summary:
tags:
answers:
sources:
relations:
created:
updated:
compile_count:
```

### 建议约束

- `summary`
  - 1-3 句
  - 必须包含主题专有名词
  - 不允许只写泛化描述

- `tags`
  - 3-12 个
  - 至少包含：
    - 技术术语
    - 组件名 / 模块名
    - 业务域关键词

- `answers`
  - 3-5 条
  - 必须是自然语言问题句
  - 应直接反映该 topic 的回答边界

### 收益

- 路由行为结构化
- lint 可对检索字段做校验
- topic 间的边界更容易控制

---

## 优化 3：把渐进式披露协议下沉到命令层

### 目标

让“先索引、后正文、再回源”的行为变成命令契约，而不是写在总说明里。

### 建议调整

#### `kc query`

改成 4 步：

1. 读 `INDEX.md` 的 Topic Registry
2. 基于 `summary / tags / answers` 做 topic 召回
3. 仅读被召回主题的 frontmatter 和相关章节
4. 仅在 low 覆盖或高风险问题时回源

#### `kc browse`

改成 3 层：

1. 浏览 registry 概览
2. 浏览 topic 元数据
3. 按需展开正文

#### `kc status`

优先从：

- `.compile-state.json`
- `INDEX.md` registry
- `wiki/log.md`

汇总状态，而不是默认全文扫描所有 topic 文件。

### 收益

- 命令层和检索层目标一致
- AI 行为更稳定
- 执行成本更低

---

## 优化 4：新增“检索质量门控”

### 建议新增 Hard Gate

1. `summary` 缺失
2. `answers` 缺失
3. `tags` 缺失
4. `answers` 不是问题句
5. `summary` 与标题重复、没有新增信息

### 建议新增 Soft Gate

1. `summary` 过泛
   - 如只写“这是一个系统设计文档”

2. `tags` 区分度过低
   - 多篇文章 tags 基本一致

3. `answers` 边界重叠
   - 多个 topic 能回答几乎同样的问题

4. topic 召回歧义
   - 某个问题可能同时命中多个近似 topic

5. registry 信息漂移
   - `INDEX.md` 中 registry 与 concept frontmatter 不一致

### 收益

- 从“文档质量”升级到“检索质量”
- 让 AI 检索层可持续维护

---

## 优化 5：将 Mermaid 从“全量强制”调整为“场景化推荐”

### 建议策略

不要要求“每篇文章至少 1 个 Mermaid 图”。

改为：

- 架构类 topic：推荐
- 流程类 topic：推荐
- 生命周期 / 状态机类 topic：推荐
- 概念解释类 / FAQ 类 topic：不强制

### 收益

- 降低编译噪音
- 减少无效内容
- 让输出更贴近检索层本质

### 建议

把 Mermaid 从 Hard 要求降级为 Soft 推荐项，并可在 lint 中输出建议而非失败。

---

## 优化 6：收缩多语言策略，优先保证检索一致性

### 建议方案 A：V1 只支持 `zh`

最稳的策略是：

- 当前版本只支持中文检索字段
- 暂不开放 `language=en`

适合场景：

- 当前目标用户主要是中文环境
- 优先验证检索契约而不是国际化

### 建议方案 B：检索字段单语、正文可多语

如果必须保留多语言能力，建议采用：

- `summary / tags / answers` 固定一种主语言
- 正文允许按 `language` 切换

例如：

- frontmatter 检索字段固定中文
- 正文可中文或英文

### 原因

检索字段如果跟着语言漂移，会直接影响召回稳定性。

---

## 5. 建议的数据契约

## 5.1 Topic Registry 契约

建议在 `INDEX.md` 中维护统一 registry：

```yaml
topic_registry_fields:
  - slug
  - title
  - summary
  - tags
  - answers
  - coverage
  - updated
  - sources_count
```

### 规则

- 每个 topic 在 registry 中必须且仅有一条记录
- registry 与 concept frontmatter 必须一致
- query / browse / status 优先依赖 registry，而不是全文扫描

---

## 5.2 Concept Frontmatter 契约

```yaml
id: api-gateway-design
title: API 网关设计
summary: API 网关的职责边界、认证策略与入口流量治理方案。
tags:
  - api-gateway
  - auth
  - traffic
answers:
  - API 网关的设计取舍是什么？
  - 认证策略在哪一层执行？
  - 网关负责哪些入口流量能力？
sources:
  - design/api-gateway-design.md
relations:
  related:
    - auth-flow
  depends_on:
    - identity-service
created: 2026-04-11
updated: 2026-04-11
compile_count: 1
```

## 6. 推荐实施顺序

## P0

1. 新增 Topic Registry 设计并落到 `INDEX.md`
2. 将 `summary / tags / answers` 正式提升为硬性 frontmatter 字段
3. 为 lint 增加检索质量门控

## P1

1. 重写 `kc query`，改为基于 registry 的两阶段召回
2. 重写 `kc browse`，让浏览逻辑优先依赖 registry
3. 重写 `kc status`，更多依赖 registry 与 state 文件

## P2

1. 将 Mermaid 从强制要求改为场景化推荐
2. 收缩或重构 `language` 能力
3. 增加 topic 边界冲突检测

## 7. 推荐修改项清单

建议后续优先修改这些文件：

  - 明确 frontmatter 质量要求

## 8. 最终建议

如果后续明确选择“AI 检索层优先”，建议把对外表述调整为：

> Knowledge Compiler 不是单纯的 wiki 生成器，而是一个把原始知识材料编译成“AI 可检索知识层”的 Skill。
> 它的核心价值不在于产出多少文档，而在于让 AI 能稳定、低成本、可验证地找到正确知识并回答问题。

这会让整个 Skill 的优化方向更统一，也能避免继续在“展示型文档”上投入过多设计成本。
