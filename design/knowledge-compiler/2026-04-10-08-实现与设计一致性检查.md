## 1. 结论

当前实现已经不再只是早期骨架，和设计文档主干基本一致：

- 目录结构、Markdown-first 形态、frontmatter、coverage 标记已对齐
- Scan / Classify / Schema / Index / Verify / State / Query / Lint / Status / Browse / Session Injector 都已有实现
- 测试已从早期的 31 个增加到 68 个，覆盖了 scanner/classifier/compiler/schema/indexer/state/verifier

但还不能判定为“完全与设计一致”。目前仍有几处关键行为没有落到设计要求的强度，尤其是：

- `kc compile` 仍然不是“真实编译并安全合并”的实现
- 用户手工编辑保留策略还没有真正落地
- classifier 还没有做到设计里的歧义确认和 schema 约束分类
- verifier 与 query 仍是简化版，少了设计里的一部分闭环行为

当前更准确的定义是：

**设计主干已落地，但仍未达到设计文档要求的完整 MVP 行为。**

## 3. 仍未对齐的关键项

以下是当前实现和设计文档仍存在的主要差异。

### 3.1 `kc compile` 仍然不是“真实编译”

设计要求：

- 读取概念全部来源全文
- 生成/更新概念文章各 section
- 标记 coverage
- 标注来源引用
- 输出可直接使用的概念文章

当前实现：

- `generate_compile_prompt()` 会生成用于 AI 编译的 prompt
- `compile_cmd()` 实际落盘仍主要依赖 `create_stub_article()`
- 输出更接近“待 AI 补全草稿”，而不是“已完成编译的概念文章”

影响：

- 核心能力“把原始材料编译为结构化知识”尚未真正闭环

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`
- `skills/knowledge-compiler/src/compiler.py`

### 3.2 用户手工编辑保留策略未真正实现

设计要求：

- 识别编译器生成内容与用户手工内容
- 重编译时只更新编译器生成部分
- 用户修改后的内容优先保留

当前实现：

- prompt 中写了“保留用户手动编辑”
- 但代码没有真正的 merge 逻辑
- 重新编译时仍可能整体重写文章

影响：

- 设计中的“人机共维”核心约束未真正成立

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`
- `skills/knowledge-compiler/src/compiler.py`

### 3.3 classifier 仍是简化版

设计要求：

- 读取标题 + 前 500 字
- 匹配已有概念，优先合并
- 歧义时交互确认
- 受 schema 的命名/分类约束

当前实现：

- 主要依据标题生成 slug
- `_extract_preview()` 没有成为核心分类输入
- 没有歧义确认流程
- 还没有真正按 schema 做分类决策

影响：

- 分类准确性和可控性低于设计要求

涉及代码：

- `skills/knowledge-compiler/src/classifier.py`

### 3.4 verifier 还没覆盖设计中的全部规则

设计要求中的 Soft Gate 包含：

- 孤立概念
- 低覆盖集群
- 过期检测
- 内容矛盾
- 断裂链接
- 缺失交叉引用

当前实现：

- 已有 orphan / low coverage / broken links / aging
- 但没有真正实现“内容矛盾”
- 没有实现“缺失交叉引用”
- 设计里强调的 stale source 关联校验也未在 verifier 中完整体现

影响：

- 质量治理还不是设计要求的完整集合

涉及代码：

- `skills/knowledge-compiler/src/verifier.py`

### 3.5 frontmatter 生命周期还不完全符合设计

设计要求：

- `created` 保持首次创建时间
- `updated` 反映最新编译或编辑时间
- `compile_count` 在更新时递增

当前实现：

- stub 生成路径会重写这些字段
- 没有稳定地读取旧值并做增量维护

影响：

- frontmatter 元数据的可追踪性还不够可靠

涉及代码：

- `skills/knowledge-compiler/src/compiler.py`

### 3.6 关系与交叉引用还没有自动建立

设计要求：

- frontmatter `relations`
- 正文 `## Related`
- schema 的 cross-reference rules 真正参与结构维护

当前实现：

- schema 可以读取 cross-ref 规则
- 但文章内容不会自动维护 `related/depends_on`
- `## Related` 也没有自动填充逻辑

影响：

- 知识图谱结构仍然偏弱

涉及代码：

- `skills/knowledge-compiler/src/compiler.py`
- `skills/knowledge-compiler/src/schema.py`

### 3.7 `kc query --save` 还没有形成完整保存闭环

设计要求：

- 基于 Wiki 回答问题
- low 覆盖时自动回查 raw
- `--save` 可保存为新概念文章

当前实现：

- 已能筛选相关概念、生成 query prompt、补充 low 覆盖来源
- 但 `--save` 还只是提示用户后续用 AI Agent 保存
- 没有真正自动落盘新概念文章

影响：

- query 闭环仍差最后一步

涉及代码：

- `skills/knowledge-compiler/src/commands/query.py`

### 3.8 compile 阶段还未真正接入 Hard Gate 阻断

设计要求：

- Hard Gate 失败时编译失败

当前实现：

- `verifier.py` 已存在
- `lint` 可输出质量报告
- 但 `compile` 主流程还没有将 Hard Gate 结果作为阻断条件严密接入

影响：

- 编译和质量门禁仍是分离的，未完全符合设计

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`
- `skills/knowledge-compiler/src/verifier.py`

## 4. 测试覆盖判断

新增单元测试已经明显优于早期版本，但仍不能称为“覆盖完整”。

已覆盖较好的部分：

- scanner
- classifier 的基础 slug 行为
- compiler 的 frontmatter / prompt / stub
- schema 的解析与增量更新
- indexer 的分类输出
- state 的基础状态计算
- verifier 的部分 Hard/Soft Gate

仍明显不足的部分：

- `commands/*` 的 CLI 层行为几乎没有单测
- `session/injector.py` 没有单测
- `compile` 的“保留用户编辑”回归场景没有测试
- `compile_count / created / updated` 生命周期没有强测试
- verifier 缺失规则没有测试，也没有暴露实现缺口
- `query --save` 没有测试

结论：

**当前测试覆盖达到“核心库模块中等覆盖”，未达到“完整覆盖”。**
