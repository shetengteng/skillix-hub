## 1. 结论

当前代码实现与设计文档在整体方向上是一致的：

- 已落地 `kc init` / `kc add` / `kc compile` 三个核心入口
- 已有 Phase 1/2/3/4/5 的最小骨架
- 目录结构、frontmatter 基本形态、coverage 标记、`wiki/concepts/` 输出路径与设计一致

但当前实现仍然是 **MVP 骨架版**，距离设计文档定义的完整行为还有明显缺口。主要问题不是“方向跑偏”，而是：

- 设计里写了很多约束，代码里只做了静态模板或提示语，没有真正执行
- 若干关键能力在文档中是“必须行为”，但实现里仍是 placeholder 或未实现
- 测试 31/31 通过，只说明当前骨架自洽，不代表已覆盖设计要求

## 3. 明确缺失项

以下项在设计文档中是明确要求，但当前代码未实现，或只做了不满足设计要求的简化版本。

### 3.1 `kc compile` 还没有真正“编译”文章

设计要求：

- 读取所有关联来源全文
- 生成/更新概念文章各 Section
- 标记覆盖度
- 标注来源引用
- 保留用户手动编辑内容

当前实现：

- `generate_compile_prompt()` 只生成 prompt
- `compile_cmd()` 实际落盘的是 `create_stub_article()` 生成的骨架文件
- 这意味着真实输出并不是“根据来源编译后的概念文章”，而是 fallback stub

影响：

- 设计中的核心价值“把原始材料编译成结构化知识”尚未真正落地
- 现在的 `kc compile` 更接近“生成待 AI 补全的草稿”

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`
- `skills/knowledge-compiler/src/compiler.py`

### 3.2 “保留用户手动编辑”只存在于 prompt 文案，没有实际合并逻辑

设计要求：

1. 识别编译器生成内容
2. 识别用户手动添加内容
3. 只更新编译器生成部分
4. 用户修改后的内容优先

当前实现：

- 已有文章内容会被读入 prompt
- 但最终仍直接 `article_path.write_text(stub, ...)`
- 没有 merge 算法，没有 section 级保留，没有“用户版本优先”

影响：

- 用户对 `wiki/concepts/*.md` 的编辑在再次编译时会被覆盖
- 这与设计中的“人机共维”核心原则直接不一致

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`
- `skills/knowledge-compiler/src/compiler.py`

### 3.3 `schema.md` 还没有进入实际编译闭环

设计要求：

- Phase 3.5 自动生成和增量更新 `wiki/schema.md`
- schema 定义 taxonomy、命名规则、交叉引用规则、废弃概念
- 编译器读取 schema 作为输入约束
- INDEX 依据 schema 分类组织

当前实现：

- `schema.md` 只在 `init` 时按模板创建
- 没有 `schema.py`
- classifier 不读取 schema
- compile 不读取 schema
- index 不按 schema 分类聚合

影响：

- `schema.md` 目前只是静态说明文件，不是系统行为约束

涉及代码：

- `skills/knowledge-compiler/src/commands/init.py`
- `skills/knowledge-compiler/src/classifier.py`
- `skills/knowledge-compiler/src/commands/compile.py`

### 3.4 Hard Gates / Soft Gates 未实现

设计要求：

- Hard Gates：frontmatter 完整、coverage 存在、来源有效、schema 一致、section 非空
- Soft Gates：孤立概念、低覆盖集群、过期检测、内容矛盾、断裂链接、缺失交叉引用

当前实现：

- 没有 `verifier.py`
- 没有 `lint.py`
- `compile` 不会因 Hard Gate 失败而阻断
- 没有任何质量报告产出

影响：

- 当前知识库无法进行设计文档要求的“可信度治理”

### 3.5 过期检测状态未实现

设计要求：

- 来源 stale
- 概念 needs_recompile
- aging / weak / orphan 等状态

当前实现：

- scanner 只做 mtime 对比，产出文件变更
- 没有概念级状态计算
- `status` / `lint` / `browse` 均未实现

影响：

- “知识腐化可观测”这个目标未落地

### 3.6 `kc add` 未覆盖设计中的外部来源注册能力

设计要求：

- 支持 `kc add <url> --title "..."`
- 支持注册 `raw/` 外的文件作为补充来源

当前实现：

- `source` 参数类型是 `Path`
- 仅支持本地文件或目录复制到 `raw/`
- 不支持 URL
- 不支持“注册外部来源但不复制入 raw/”的模式

影响：

- 输入源能力比设计收缩了一档

涉及代码：

- `skills/knowledge-compiler/src/commands/add.py`

### 3.7 分类策略未实现“歧义确认”和 schema 约束

设计要求：

- 读取标题 + 前 500 字
- 匹配已有概念，优先合并
- 歧义时交互确认
- 受 schema 的命名和分类规则约束

当前实现：

- 只用了标题生成 slug
- `_extract_preview()` 存在但未用于分类判断
- 没有歧义检测与确认流程
- 没有 schema 约束输入

影响：

- 分类准确度和可控性与设计存在明显差距

涉及代码：

- `skills/knowledge-compiler/src/classifier.py`

### 3.8 INDEX 生成逻辑未按设计分类聚合

设计要求：

- INDEX 按 schema 分类聚合
- 包含覆盖度摘要
- 体现结构化知识地图

当前实现：

- `_update_index()` 只按文件名平铺输出概念表
- 不区分 taxonomy
- 不读取 schema
- `Updated` 列直接写当天日期，不是真实文章更新时间

影响：

- INDEX 目前更像文件列表，而不是知识索引

涉及代码：

- `skills/knowledge-compiler/src/commands/compile.py`

### 3.9 frontmatter 生命周期字段未按设计更新

设计要求：

- `created` 保持首次创建时间
- `updated` 反映最近编译或编辑时间
- `compile_count` 更新时递增

当前实现：

- stub 每次都重写 `created/updated/compile_count`
- 不读取旧值做增量更新

影响：

- frontmatter 元数据不可信，无法支持设计中的状态治理

涉及代码：

- `skills/knowledge-compiler/src/compiler.py`

### 3.10 关系与交叉引用没有自动建立

设计要求：

- frontmatter `relations`
- 正文 `## Related`
- schema 中的 cross-reference rules

当前实现：

- 模板中有 `relations`
- 但不会从来源中抽取、推断或维护 `related/depends_on`
- `## Related` 只是空段落

影响：

- 知识图谱结构没有真正形成

### 3.11 会话注入只在配置层存在，未实现运行时集成

设计要求：

- 自动向上查找 `.kc-config.json`
- 根据 `session_mode` 注入上下文
- 支持 `staging/recommended/primary`

当前实现：

- `config.json` 中有 `session_mode`
- `find_project_root()` 可向上查找
- 但没有 `session/injector.py`，也没有 hook 集成

影响：

- AI 会话集成仍停留在设计层

### 3.12 设计中的查询/健康检查/状态/浏览命令都未实现

缺失命令：

- `kc query`
- `kc lint`
- `kc status`
- `kc browse`

影响：

- 当前系统还没有形成设计里的使用闭环

### 3.13 测试覆盖集中在骨架行为，未覆盖设计级行为

当前测试已覆盖：

- scan 基本变更检测
- classify 基本 slug 推断
- compiler 的 frontmatter / stub / prompt 生成

未覆盖的关键设计行为：

- 用户手动编辑保留
- schema 约束输入
- Hard Gate / Soft Gate
- INDEX 分类聚合
- frontmatter 生命周期更新
- 关系链接抽取
- query / lint / status / browse

## 5. 优先级建议

如果按“先让产品成立，再补增强功能”的顺序推进，建议优先级如下：

1. 先修正 `compile` 的真实落盘行为，避免继续覆盖用户编辑。
2. 让 schema 进入编译闭环，否则 taxonomy 和命名规则无法真正生效。
3. 补 Hard Gate / Soft Gate，否则“可信知识库”目标不成立。
4. 再补 query / lint / status / browse 等使用层命令。
