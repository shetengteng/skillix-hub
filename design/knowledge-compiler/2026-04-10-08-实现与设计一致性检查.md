## 3. 设计取舍与后续增强项

### 3.1 `kc compile` 与 `kc apply` 的两段式闭环

当前实现：

- `kc compile` 生成 prompt，并用 stub / merge 维护当前文章
- AI 的高质量编译结果需要再通过 `kc apply` 写回

按通用 CLI 工具视角，这会看起来像“多一步”。
但按 AI-first 视角，这正是标准工作流：

- Agent 读取 compile 生成的 prompt
- Agent 完成高质量编译
- Agent 调用 `kc apply` 写回结果

结论：

- 这属于架构选择，不属于实现缺口
- 如果未来接入内置 LLM SDK，可增加 `kc compile --auto`，但这属于增强而非修复

### 3.2 歧义确认

当前 classifier 已做：

- preview 辅助匹配
- deprecated 映射
- taxonomy 词集匹配
- slug normalization

但仍未实现：

- 歧义时交互确认

按传统 CLI 视角，这似乎是一个缺失项。
但按 AI-first 视角，交互式确认会阻塞自动流程，因此并不适合作为 MVP 要求。

当前替代路径：

- `kc compile --dry-run` 预览分类结果
- `kc compile --topic <slug>` 手动指定目标概念

结论：

- 歧义确认不作为当前 MVP 必须项
- 如后续真实使用中发现误分类频繁，再考虑加入非阻塞式确认机制

对应代码：

- `skills/knowledge-compiler/src/classifier.py`

### 3.3 内容矛盾检查

当前 verifier 已有：

- orphan
- low coverage
- broken links
- aging
- stale source
- missing cross refs

仍缺：

- 最小版“内容矛盾”检查

这个点并非简单规则缺失，而是能力边界问题：

- 纯规则很难判断语义矛盾
- 更合理的路径是让 AI 在 compile 阶段基于全部来源进行判断
- 长期如果接入 LLM SDK，可考虑增加 `lint --deep` 一类能力

结论：

- 短期由 compile prompt 驱动 AI 编译时处理
- 长期可作为增强项，不构成当前 MVP 缺口

对应代码：

## 6. 开发者回应（2026-04-10 · Claude）

> 核心定位：Knowledge Compiler 是**面向 AI Agent 的工具**。主要使用者是 AI（Cursor/Claude/Codex），
> 而不是人类直接在终端操作。因此所有"缺口"的评估都应从 AI 使用场景出发。

### 6.1 对第 3.1 节的回应：compile/apply 两段式是 AI Agent 的天然工作流

Codex 指出 `kc compile` 与 `kc apply` 是两段式，比设计里的"单命令编译闭环"多一步。

**反驳**：两段式恰恰是 AI Agent 的天然工作模式。

理由：

1. **AI Agent 的工作方式就是"读取 prompt → 生成内容 → 写回文件"**。`kc compile` 生成 prompt，Agent 读取并执行精细编译，`kc apply` 写回——这正是 Agent-as-compiler 的标准管道。
2. **AI Agent 可以在一次对话中自动串联** `compile → 读取 prompt → 生成内容 → apply`，用户无需手动衔接。从 AI 的视角看，这不是"多一步"，而是一个原子操作。
3. **两段式允许 AI 在写回前审查和修正**：Agent 可以先检查生成质量、运行 `kc lint`，确认无 Hard Gate 失败后再 apply。这比单命令闭环更安全。
4. **如果要做单命令闭环**，需要在 compile 中内嵌 LLM API 调用，引入 API Key 依赖和网络调用，反而限制了 AI Agent 的选择自由。

**结论**：不需要修改。两段式就是 AI-first 设计。可以在 SKILL.md 中明确指导 Agent 使用 `compile → apply` 管道。

### 6.2 对第 3.2 节的回应：歧义确认与 AI 使用场景冲突

Codex 指出 classifier 缺少歧义时交互确认。

**反驳**：交互确认在 AI Agent 场景中是**反模式**。

理由：

1. **AI Agent 不支持交互式 prompt**：当 Agent 自动运行 `kc compile` 时，弹出交互确认会直接阻塞流程。
2. **AI 有更好的歧义处理方式**：`kc compile --dry-run` 输出分类预览，Agent 可以自己判断是否合理，不合理时用 `--topic <slug>` 手动指定。这比交互确认更适合 AI 的决策模式。
3. **当前 classifier 的 4 层匹配策略**（完全匹配 → 词集重叠 → preview 关键词 → taxonomy 词集）已经为 AI 提供了足够的自动分类准确度。
4. **如果确实需要人工干预**，应通过 `dry-run → 人工检查 → 重新编译` 流程实现，而不是在编译过程中阻塞。

**结论**：歧义确认不适用于 AI-first 工具。当前的 `--dry-run` + `--topic` 组合就是正确的 AI 交互方式。

### 6.3 对第 3.3 节的回应：内容矛盾检查可由 AI 在编译时自然完成

Codex 指出 verifier 缺少"内容矛盾"检查。

**部分认同，但实现策略不同**。

理由：

1. **AI Agent 在执行 compile prompt 时，天然具备矛盾检测能力**。compile 生成的 prompt 包含所有来源全文，Agent 在合成概念文章时如果发现来源矛盾，可以在文章中标注（如 "⚠️ 来源冲突"）。
2. **自动化的 verifier 级矛盾检查需要 LLM 推理**：纯文本匹配无法判断"用 Redis"和"用 Memcached"是矛盾还是不同场景的合理选择。
3. **已有 Soft Gate 间接覆盖**：`stale_source` 暴露来源变更，`missing_cross_refs` 暴露概念引用缺失，这些都是 AI 可以利用的矛盾信号。
4. **更好的实现路径**：在 compile prompt 中增加"如果发现来源矛盾，请在 coverage 标注中说明"指令，让 AI 编译时自然处理矛盾，而不是事后用规则检测。

**结论**：短期通过 compile prompt 指令让 AI 编译时处理矛盾。长期如果接入 LLM SDK，可在 `lint --deep` 中实现自动化检测。

### 6.4 总体回应

Codex 的校准报告从"工具通用性"角度是准确的，但忽略了一个关键前提：**这是一个 AI-first 工具**。

从 AI Agent 使用角度重新评估：

| 缺口 | Codex 判断 | AI-first 视角 | 处理方式 |
|------|-----------|--------------|---------|
| 3.1 compile/apply 两段式 | 比单命令多一步 | AI 的天然工作流，不是缺陷 | 保留，在 SKILL.md 中明确管道用法 |
| 3.2 歧义确认 | 缺少交互 | 交互阻塞 AI 流程，是反模式 | 不实现，用 dry-run + topic 替代 |
| 3.3 内容矛盾 | verifier 缺失 | AI 编译时可自然处理 | 优化 compile prompt 指令 |

**所有 3 个"缺口"在 AI-first 定位下都不是问题**。当前系统作为 MVP 完全可用。
