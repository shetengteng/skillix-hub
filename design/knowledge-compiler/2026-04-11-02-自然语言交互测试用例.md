# Knowledge Compiler — 自然语言交互测试用例

> 日期：2026-04-11
> 目标：验证 `Knowledge Compiler` 的自然语言交互设计与 Markdown Skill 实现一致
> 范围：`kc init` / `kc add` / `kc compile` / `kc query` / `kc lint` / `kc browse` / `kc status` / `language`

---

## 1. 测试目标

本测试文档用于验证以下内容：

- 自然语言是否能正确路由到对应命令
- 默认 `raw/` 场景与自定义 source 场景是否都能成立
- `high / medium / low` 覆盖度驱动的回答策略是否一致
- `kc lint` 与 `kc lint --fix` 的交互边界是否清晰
- `language=zh` 与 `language=en` 的输出约定是否一致
- 不确定时是否会按设计要求向用户确认，而不是擅自决定

---

## 2. 测试前置条件

### 通用前置条件

- `skills/knowledge-compiler/` 目录完整存在
- `SKILL.md`、`commands/*.md`、`skills/wiki-compiler.md`、`templates/*.md`、`references/*.md` 可读取
- 测试执行者可以创建临时目录和 Markdown 文件

### 测试数据约定

建议准备两组测试工程：

1. 默认 raw/ 场景工程
2. 自定义 source 场景工程（如 `doc/` + `design/`）

建议准备以下示例文件：

- 设计文档：`api-gateway-design.md`
- 调研文档：`grpc-evaluation.md`
- ADR：`auth-strategy-adr.md`
- 笔记：`meeting-notes.md`

## 3. 测试矩阵

| 编号 | 模块 | 场景 | 重点验证 |
|------|------|------|---------|
| TC-01 | init | 默认 raw/ 初始化 | 默认目录、默认语言、默认模式 |
| TC-02 | init | 自定义 source 初始化 | `doc/`、`design/` 等已有目录 |
| TC-03 | add | 明确意图添加设计文档 | 无需确认，直达正确目录 |

| TC-06 | compile | dry-run 预览 | 不写文件，只输出影响范围 |
| TC-07 | query | high/medium/low 覆盖度回答 | 直接引用、加限定、回查原文 |
| TC-08 | lint | 只读检查 | 不修改文件，不提示交互修复 |

## 4. 详细测试用例

## TC-01：默认 raw/ 初始化

### 前置条件

- 当前目录无 `.kc-config.json`
- 当前目录无现成 `doc/`、`design/` 等知识源目录

### 用户输入

```text
帮我初始化一个知识库
```

### 预期行为

1. 路由到 `kc-init.md`
2. 扫描 Markdown 文件
3. 若无合适已有目录，则采用默认 `raw/` 结构
4. 询问语言，默认 `zh`
5. 询问会话模式，默认 `recommended`
6. 创建：
   - `raw/designs/`
   - `raw/decisions/`
   - `raw/research/`
   - `raw/notes/`
   - `wiki/concepts/`
   - `wiki/analyses/`
7. 写入 `.kc-config.json`
8. 生成 `wiki/INDEX.md` 与 `wiki/log.md`

### 预期结果

`.kc-config.json` 应包含：

```json
{
  "session_mode": "recommended",
  "language": "zh",
  "sources": ["raw/designs", "raw/decisions", "raw/research", "raw/notes"],
  "output_dir": "wiki"
}
```

输出应包含：

```text
源目录:  raw/designs/, raw/decisions/, raw/research/, raw/notes/
输出:    wiki/
语言:    zh
模式:    recommended
```

---

## TC-02：自定义 source 初始化

### 前置条件

- 当前目录存在：
  - `doc/`
  - `design/`
- 目录下有若干 `.md` 文件
- 当前目录无 `.kc-config.json`

### 用户输入

```text
帮我初始化一个知识库
```

### 预期行为

1. 扫描到 `doc/`、`design/`
2. 询问用户选择 source
3. 用户选择 `doc/`, `design/`
4. 不创建 `raw/` 子目录
5. 仅创建：
   - `wiki/concepts/`
   - `wiki/analyses/`
6. 默认语言 `zh`
7. 默认模式 `recommended`

### 预期结果

`.kc-config.json` 应包含：

```json
{
  "session_mode": "recommended",
  "language": "zh",
  "sources": ["doc", "design"],
  "output_dir": "wiki"
}
```

输出应反映实际 source：

```text
源目录:  doc/, design/
输出:    wiki/
语言:    zh
模式:    recommended
```

---

## TC-03：明确意图添加设计文档

### 前置条件

- 已存在 `.kc-config.json`
- `sources = ["doc", "design"]`
- 文件 `api-gateway-design.md` 已存在于项目内但不在 source 目录中，或用户明确指向 `design/`

### 用户输入

```text
把 api-gateway-design.md 加到知识库里
```

### 预期行为

1. 路由到 `kc-add.md`
2. 读取 `.kc-config.json.sources`
3. 判断内容属于设计文档
4. 选择 `design/` 作为目标目录
5. 若文件已在 `design/` 中，则跳过复制
6. 不额外询问确认

### 预期结果

输出应类似：

```text
已添加: design/api-gateway-design.md
运行 `kc compile` 将其编入 wiki。
```

---

## TC-04：模糊意图添加文件

### 前置条件

- 已存在 `.kc-config.json`
- `sources = ["doc", "design"]`
- 文件内容类型不能从用户话术中明确判断

### 用户输入

```text
加个文件到知识库
```

### 预期行为

1. 路由到 `kc-add.md`
2. 基于文件内容推荐目录
3. 向用户发起确认，而不是直接决定

### 预期结果

输出应类似：

```text
放到 `design/`？(y/n/自定义路径)
```

---

## TC-05：增量编译

### 前置条件

- 已初始化知识库
- 已存在 source 文件
- 存在或不存在 `.compile-state.json`

### 用户输入

```text
编译一下
```

### 预期行为

1. 路由到 `kc-compile.md`
2. 调用 `wiki-compiler.md`
3. 执行：
   - Phase 1 Scan
   - Phase 2 Classify
   - Phase 3 Compile
   - Phase 3.5 Schema
   - Phase 4 Verify
   - Phase 5 State + Log

### 预期结果

输出应包含：

```text
编译完成: X 新建, Y 更新, Z 未变
Hard Gates: 全部通过
Soft Gates: N 个警告
Wiki: wiki/INDEX.md
```

并更新：

- `wiki/concepts/*.md`
- `wiki/schema.md`
- `wiki/INDEX.md`
- `wiki/log.md`
- `.compile-state.json`

---

## TC-06：dry-run 预览编译

### 用户输入

```text
先看看编译会改什么，别直接写
```

### 预期行为

1. 识别 `--dry-run`
2. 只执行 Phase 1 和 Phase 2
3. 不写入任何文件

### 预期结果

输出应包含：

```text
预览模式 — 仅预览，不写入任何文件。
变更的源文件:
受影响主题:
运行 `kc compile` 执行。
```

且以下文件不应发生变化：

- `wiki/concepts/*.md`
- `wiki/schema.md`
- `wiki/INDEX.md`
- `wiki/log.md`
- `.compile-state.json`

---

## TC-07：query 的覆盖度分级回答

### 前置条件

- 已有编译完成的 wiki
- 某主题存在：
  - `high` 覆盖章节
  - `medium` 覆盖章节
  - `low` 覆盖章节

### 用户输入

```text
API 网关的设计取舍是什么？
```

### 预期行为

1. 读取 `wiki/INDEX.md`
2. 读取相关主题文章
3. 对不同覆盖度采用不同策略：
   - `high`：直接引用
   - `medium`：引用但标注“基于单一来源，可能需补充”
   - `low`：回查 source 原文

### 预期结果

回答中应同时体现：

- wiki 引用
- raw/source 引用
- `medium` 的不确定性说明
- `low` 的补充验证说明

---

## TC-08：kc lint 只读模式

### 用户输入

```text
检查一下知识库的质量
```

### 预期行为

1. 路由到 `kc-lint.md`
2. 运行 Hard Gate / Soft Gate
3. 只报告，不修改文件
4. 不出现交互式 `Fix? (y/n)`

### 预期结果

输出应类似：

```text
kc lint 报告 — YYYY-MM-DD
Hard Gates: ...
Soft Gates: ...
1 个可自动修复的问题。运行 `kc lint --fix` 修复。
```

---

## TC-09：kc lint --fix 自动修复

### 用户输入

```text
运行 kc lint --fix
```

### 预期行为

1. 自动修复安全可修复的问题
2. 不逐条交互确认
3. 记录修复结果到 `wiki/log.md`

### 预期结果

应修复的典型问题包括：

- 缺失覆盖度注释
- 缺失 schema 条目
- 缺失 frontmatter 基础字段

输出中应明确列出修复项。

### 用户输入

```text
知识库状态怎么样
```

1. 路由到 `kc-status.md`
2. 统计 source 文件数
3. 统计 topics 数

## TC-12：language=zh

### 前置条件

- `.kc-config.json.language = "zh"`

### 用户输入

```text
编译一下
```

### 预期结果

生成内容应为中文：

- 文章章节标题为中文
- INDEX 为中文
- schema 正文为中文
- YAML frontmatter key 保持英文

---

## TC-13：language=en

### 前置条件

- `.kc-config.json.language = "en"`

### 用户输入

```text
compile the knowledge base
```

### 预期行为

编译器按 `wiki-compiler.md` 中的映射规则将章节标题切换为英文，正文使用英文生成。

### 预期结果

生成内容应满足：

- `Summary`
- `Key Decisions`
- `Current State`
- `Gotchas`
- `Open Questions`
- `Related`
- `Sources`

同时：

- YAML frontmatter key 仍为英文
- 不应混入中文章节标题

---

## TC-14：歧义场景必须确认

### 前置条件

- 一个文件可能同时属于两个 topic
- 或新 topic 与已有 topic 高度相似

### 用户输入

```text
编译一下
```

### 预期行为

编译流程暂停并发起确认，而不是擅自决定。

### 预期结果

输出应类似：

```text
该文件涉及 `topic-a` 和 `topic-b`，应归入哪个主题？还是拆分为两条？
```

## 5. 验收标准

通过标准：

- 所有测试用例的“预期行为”和“预期结果”都能成立
- 默认 raw/ 场景与自定义 source 场景都自洽
- `kc lint` / `kc lint --fix` 的行为边界明确
- `medium` / `low` 覆盖度规则在问答中可观测
- `language=zh` / `language=en` 输出规则可观测
- 歧义场景下会暂停确认

失败标准：

- 同一文档中混用互相冲突的 source 路径
- `language=en` 仍生成中文章节
- `kc lint` 在无 `--fix` 时修改文件
- 歧义场景下未询问直接决策
