# llm-wiki Skill 复刻设计文档


---

## 1. 原始 Skill 概述

### 1.1 是什么

llm-wiki-compiler 是一个 Claude Code 插件，将用户积累的 Markdown 知识原料（文章、论文、笔记、书摘）编译成结构化的、AI 可查询的 Wiki。核心特性：

- **自动主题发现**：扫描原始文件，推断主题分类
- **覆盖度追踪**：每个章节标记 high/medium/low 三级置信度
- **会话感知注入**：SessionStart hook 自动检测 Wiki 并注入上下文
- **增量编译**：基于 mtime 对比实现增量更新

### 1.2 架构组成

```
llm-wiki-compiler/
├── .claude-plugin/plugin.json   # 插件清单（5 个命令 + 1 个 skill + 1 个 hook）
├── hooks/
│   ├── hooks.json               # SessionStart hook 声明
│   └── session-start            # Bash 脚本，自动检测 .wiki-compiler.json
├── skills/
│   └── wiki-compiler.md         # 5 阶段编译算法（核心逻辑）
├── commands/
│   ├── wiki-init.md             # 初始化：personal / project 两种模式
│   ├── wiki-compile.md          # 触发编译（增量/全量/单主题/dry-run）
│   ├── wiki-query.md            # 基于 Wiki 回答问题，可保存到 analyses/
│   ├── wiki-lint.md             # 健康检查（8 类问题检测）
│   └── wiki-upgrade.md          # 自更新（git pull）
└── templates/
    ├── article-template.md      # 主题文章模板（含覆盖度标记）
    ├── schema-template.md       # Wiki 结构契约模板
    └── index-template.md        # 索引模板
```

**关键设计约束**：纯 Markdown + JSON，零代码、零基础设施。所有逻辑由 AI Agent 解释和执行。

### 1.3 数据结构

```
{username}-llm-wiki/
├── .wiki-compiler.json          # 项目配置（mode, sources, compile_options）
├── .compile-state.json          # 编译状态（文件路径 + mtime 快照）
├── raw/                         # 原始知识材料（只读，不可变）
│   ├── articles/
│   ├── papers/
│   ├── books/
│   ├── notes/
│   └── assets/                  # 被排除
└── wiki/                        # 编译产物
    ├── INDEX.md                 # 主题索引（自动维护）
    ├── schema.md                # 结构契约（人机共维）
    ├── log.md                   # 编译日志
    ├── concepts/                # 主题文章
    │   ├── {topic-slug}.md
    │   └── ...
    └── analyses/                # 查询结果保存
        └── {date}-{slug}.md
```

## 2. 核心设计思路分析

### 2.1 编译管道（5 阶段）

这是整个系统最精妙的设计：

| 阶段 | 名称 | 输入 | 输出 | 核心逻辑 |
|------|------|------|------|---------|
| Phase 1 | Scan Sources | .compile-state.json + 源文件 | new/changed/deleted 三个列表 | mtime 对比，--full 时全部视为 changed |
| Phase 2 | Classify & Discover Topics | new+changed 文件的标题和摘要 | topic_map (slug → [files]) | 推断主题 slug，优先合并已有主题 |
| Phase 3 | Compile Topic Articles | 每个主题的全部源文件 | wiki/concepts/{slug}.md | 读全文 → 模板生成 → 覆盖度标记 → 引用标注 |
| Phase 3.5 | Schema | 已编译的文章 | wiki/schema.md | 首次生成，后续增量更新，不覆盖手动编辑 |
| Phase 4 | Update INDEX | 所有概念文章 | wiki/INDEX.md | 分类聚合 + 覆盖度摘要 + 更新标记 |
| Phase 5 | State & Log | 编译结果 | .compile-state.json + wiki/log.md | 快照更新 + 日志追加 |

**设计洞察**：
- Phase 2 的"优先合并已有主题"避免了主题碎片化
- Phase 3.5 的 schema 设计为"人机共维契约"——编译器自动生成但尊重手动编辑
- 增量编译基于 mtime 而非内容哈希，简单高效

### 2.2 三级覆盖度模型

| 级别 | 含义 | 行为建议 |
|------|------|---------|
| high | 多个一致来源 | 可直接信赖 |
| medium | 单一来源或部分覆盖 | 酌情使用 |
| low | 推断或稀疏 | 需要回查 raw/ 原始文件验证 |

这个设计让 AI Agent 能自主决定何时需要深入查看原始材料。

### 2.3 三种会话模式

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| staging | Wiki 是补充参考，需要时查阅 | 项目文档（默认） |
| recommended | 先读 Wiki 再读原始文件 | 个人知识库（默认） |
| primary | Wiki 为主，仅 low 覆盖才看 raw/ | 成熟度高的知识库 |

### 2.4 SessionStart Hook 自动注入

```bash
# 从当前目录向上查找 .wiki-compiler.json
# 找到 → 根据 mode 注入上下文提示
# 未找到 → 静默退出，不干扰其他项目
```

这是一个低侵入设计：插件只在检测到 Wiki 项目时才介入。

## 3. 与现有 knowledge-base Skill 的对比

skillix-hub 中已存在 `skills/knowledge-base/`，功能有显著重叠：

| 维度 | llm-wiki-compiler | knowledge-base (现有) |
|------|-------------------|----------------------|
| **实现方式** | 纯 Markdown + JSON（Claude Code 插件规范） | Python CLI (typer) |
| **平台** | Claude Code 专属 | Cursor / Claude Code / 通用 |
| **编译执行** | AI Agent 解释 Markdown 指令后执行 | Python 脚本直接执行 |
| **索引管理** | 自动扫描 raw/ 目录 | 手动 `add` / `import-project` |
| **主题发现** | 自动推断 topic slug | 自动提取 concept |
| **覆盖度** | high/medium/low 三级 | 无 |
| **知识图谱** | 无（仅交叉引用） | graph.json + Mermaid |
| **Schema 契约** | schema.md（人机共维） | categories/ 分类索引 |
| **会话注入** | SessionStart hook | 无自动注入 |
| **支持文件类型** | 仅 Markdown | 多类型（md, code, config, dataset, image, binary, repo, link） |
| **资料管理** | raw/ 只读，编译器不修改 | CRUD 操作（add/remove/edit） |

### 关键差异总结

1. **llm-wiki 的优势**：覆盖度追踪、Schema 契约、会话自动注入、纯声明式（无代码依赖）
2. **knowledge-base 的优势**：多文件类型支持、知识图谱、Python 可编程、更丰富的 CRUD
3. **设计哲学差异**：llm-wiki 偏"编译器"思维（raw→wiki 单向流），knowledge-base 偏"数据库"思维（CRUD + 编译）

## 4. 复刻方案

### 4.1 复刻策略

**不是直接复制代码**，而是将 llm-wiki 的设计精华融入 skillix-hub 的架构中，作为独立 Skill 存在。

核心取舍：
- **保留**：5 阶段编译管道、覆盖度模型、Schema 契约、会话模式
- **适配**：从 Claude Code 插件规范 → skillix-hub 的 Skill 规范（Python + SKILL.md）
- **增强**：结合 knowledge-base 的多类型支持和知识图谱能力
- **去除**：git clone 安装流程（改为 Skill 安装）、Claude Code 专属 hook 机制

### 4.2 目标结构

```
skills/llm-wiki/
├── SKILL.md                     # Skill 入口（Cursor/Claude Code 兼容）
├── main.py                      # 命令入口（typer CLI）
├── config.json                  # 默认配置
├── src/
│   ├── __init__.py
│   ├── scanner.py               # Phase 1: 源文件扫描 + mtime 对比
│   ├── classifier.py            # Phase 2: 主题发现与分类
│   ├── compiler.py              # Phase 3: 主题文章编译
│   ├── schema.py                # Phase 3.5: Schema 管理
│   ├── indexer.py               # Phase 4: INDEX.md 生成
│   ├── state.py                 # Phase 5: 状态和日志管理
│   └── commands/
│       ├── __init__.py
│       ├── init.py              # wiki-init（personal/project）
│       ├── compile.py           # wiki-compile（增量/全量/dry-run）
│       ├── query.py             # wiki-query（Wiki 检索问答）
│       ├── lint.py              # wiki-lint（8 类健康检查）
│       └── status.py            # 状态查看
├── templates/
│   ├── article.md               # 主题文章模板（含覆盖度标记）
│   ├── schema.md                # Wiki 结构契约模板
│   └── index.md                 # 索引模板
└── prompts/
    ├── classify_topics.md       # 主题分类 prompt
    └── compile_article.md       # 文章编译 prompt
```

### 4.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现语言 | Python | 与 skillix-hub 现有 skill 一致 |
| 命令框架 | typer | 与 knowledge-base 一致，统一风格 |
| 编译执行 | Python 脚本生成 prompt → Agent 执行 | 保留"AI 驱动编译"的设计本质 |
| 数据存储 | {project}/llm-wiki-data/ | 遵循 skillix-hub 的 data 目录规范 |
| 覆盖度追踪 | 保留 high/medium/low 三级 | llm-wiki 的核心价值 |
| Schema 契约 | 保留人机共维设计 | 避免编译器覆盖用户编辑 |
| 会话注入 | Cursor hook 适配 | 从 Claude Code hook → Cursor rules |

### 4.4 与 knowledge-base 的关系

两个 Skill 独立共存，不合并：
- `knowledge-base`：通用知识索引 + CRUD + 图谱，适合项目级资料管理
- `llm-wiki`：专注 raw→wiki 编译流水线 + 覆盖度追踪，适合个人知识积累

未来可考虑让 `llm-wiki` 使用 `knowledge-base` 的索引作为数据源。

## 6. 风险与注意事项

1. **原始设计是"AI 解释执行"**：llm-wiki 的编译逻辑写在 Markdown 中让 AI 执行，复刻到 Python 后需要明确 AI 参与的边界（哪些是确定性 Python 逻辑，哪些仍需 AI 推理）
2. **覆盖度标记的准确性**：取决于 AI 对多源信息的理解能力，需要设计合理的 prompt
3. **Schema 契约的冲突处理**：需要实现 diff/merge 逻辑，避免覆盖用户手动编辑
4. **与 knowledge-base 的边界**：需清晰定义两个 Skill 的使用场景，避免用户困惑
