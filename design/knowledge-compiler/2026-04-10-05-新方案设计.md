# Knowledge Compiler — 新方案设计 (v2)

> 日期：2026-04-10
> 设计参考：llm-wiki-compiler、project-knowledge-base-nex
> 审核修订：基于苏格拉底审核报告（2026-04-10-06）

---

## 0. 设计决策前提

| 决策 | 选择 | 理由 |
|------|------|------|
| V1 目标用户 | 工程团队 | 先专注一个用户群，做好再扩展 |
| 数据格式 | Markdown-first | 用户可直接阅读和编辑，无工具依赖 |
| 输入约定 | `raw/` 目录 | 清晰心智模型："所有原始材料在这里" |
| 输出格式 | 单一 wiki 输出 | 不做视图系统，避免过度设计 |
| 人机共维 | 直接编辑 Markdown | 用户修改概念文章，编译器尊重手动编辑 |

---

## 1. 核心问题

工程团队积累了大量知识材料（设计文档、架构决策、技术调研、会议纪要、代码注释），但这些知识：
- 散落在各处，缺少结构化组织
- 没有可信度标记，AI Agent 无法判断哪些信息需要进一步验证
- 无法自动检测过期，知识随时间腐化
- 新人或新 Agent 无法快速获得项目全貌

**Knowledge Compiler 将团队知识材料编译为结构化、有覆盖度标记的 Wiki，让人和 AI 都能高效使用。**

---

## 2. Markdown-first 架构

### 2.1 核心原则

```
输入是 Markdown → 编译产物是 Markdown → 用户直接编辑 Markdown
```

没有 JSON 中间层。概念文章的元数据通过 YAML frontmatter 存储，正文就是概念内容。

### 2.2 概念文章格式

```markdown
---
id: transformer-architecture
title: Transformer 架构
tags: [deep-learning, nlp, architecture]
sources:
  - raw/papers/attention-is-all-you-need.md
  - raw/articles/transformer-guide.md
relations:
  related: [attention-mechanism, positional-encoding]
  depends_on: [matrix-multiplication]
created: 2026-03-15
updated: 2026-04-08
compile_count: 3
---

# Transformer 架构

> 基于自注意力机制的序列到序列模型架构。

## Summary
<!-- coverage: high -->

Transformer 是 Google 在 2017 年提出的...
[source: raw/papers/attention-is-all-you-need.md]

## Key Decisions
<!-- coverage: medium -->

选择多头注意力而非单头...
[source: raw/articles/transformer-guide.md]

## Open Questions
<!-- coverage: low -->

关于稀疏注意力的效率优化仍在研究中...

## Related
- [[attention-mechanism]]
- [[positional-encoding]]
```

**关键设计**：
- frontmatter 存所有元数据（来源、关系、编译信息）
- 正文中每个章节有 `<!-- coverage: X -->` 标记
- 来源引用用 `[source: path]` 内联标注
- 用户可以直接打开文件编辑任何部分
- 编译器更新时会保留用户的手动编辑

## 3. 目录结构

```
{project}/
├── .kc-config.json                  # 配置
├── .compile-state.json              # 编译状态（mtime 快照）
│
├── raw/                             # 原始材料（只读约定）
│   ├── designs/                     # 设计文档
│   ├── decisions/                   # 架构决策
│   ├── research/                    # 技术调研
│   ├── notes/                       # 会议纪要、笔记
│   └── assets/                      # 图片等（排除编译）
│
└── wiki/                            # 编译产物
    ├── INDEX.md                     # 概念索引
    ├── schema.md                    # 结构契约（分类 + 命名规则 + 交叉引用）
    ├── log.md                       # 编译日志
    └── concepts/                    # 概念文章（Markdown + frontmatter）
        ├── {topic-slug}.md
        └── ...
```

### 3.1 raw/ 目录约定

- `raw/` 是只读约定：编译器永远不修改 `raw/` 下的文件
- 子目录可以自定义（designs、decisions、research、notes 只是建议）
- 也支持注册 `raw/` 外的文件作为补充来源（但 `raw/` 是主要入口）
- `assets/` 默认排除，不参与编译

### 3.2 wiki/ 目录

- 编译产物全部在 `wiki/`
- 用户可以直接编辑 `wiki/concepts/` 下的文件
- 编译器再次运行时，会**合并**而非覆盖用户编辑
- INDEX.md 和 schema.md 由编译器维护，但 schema.md 中的手动编辑会被保留

## 4. 编译管道

### 4.1 流程

```
Phase 1: Scan
  读取 .compile-state.json → 对比 raw/ 文件 mtime
  → 输出: new_files, changed_files, deleted_files
  → --full 时全部视为 changed

Phase 2: Classify
  读取变更文件的标题 + 前 500 字
  → 推断 topic slug，匹配已有概念（优先合并）
  → 歧义时交互确认
  → 输出: topic_map {slug: [file_paths]}

Phase 3: Compile
  对每个有变更来源的概念:
    → 读取所有关联来源全文
    → 生成/更新概念文章各 Section
    → 标记覆盖度 (high/medium/low)
    → 标注来源引用
    → 保留用户手动编辑的内容
  支持并行（多概念同时编译）

Phase 3.5: Schema
  首次: 生成 wiki/schema.md（分类 + 命名规则 + 交叉引用）
  后续: 增量更新，不覆盖手动编辑部分

Phase 4: Index + Verify
  更新 wiki/INDEX.md（分类聚合 + 覆盖度摘要）
  Hard Gate 检查:
    - 每篇文章有 frontmatter
    - 每个 Section 有 coverage 标记
    - 所有来源引用路径有效
    - 新概念已录入 schema.md

Phase 5: State + Log
  更新 .compile-state.json（mtime 快照）
  追加 wiki/log.md（编译摘要）
```

### 4.2 保留用户编辑的策略

编译器更新概念文章时：
1. 读取现有文章
2. 识别哪些内容是编译器生成的（有 `[source: ...]` 标注）
3. 识别哪些内容是用户手动添加的（无来源标注）
4. 只更新编译器生成的部分，保留用户添加的段落
5. 如果用户修改了编译器生成的内容（手动调整覆盖度、补充信息），以用户版本为准

## 5. 覆盖度标记

### 5.1 三级覆盖度

| 级别 | 含义 | 行为建议 |
|------|------|---------|
| high | 多个一致来源支撑 | 可直接信赖 |
| medium | 单一来源或部分覆盖 | 酌情使用，可能需要补充 |
| low | 推断或稀疏，信息不足 | 需要回查 raw/ 原始文件验证 |

### 5.2 标记方式

```markdown
## Summary
<!-- coverage: high -->
```

- 每个 Section 顶部一行 HTML 注释
- 编译器自动标记，用户也可手动调整
- AI Agent 读取时据此决定是否需要深挖原始来源

### 5.3 覆盖度在会话中的作用

当 AI Agent 使用 Wiki 回答问题时：
- high 章节 → 直接引用
- medium 章节 → 引用但标注"可能需补充"
- low 章节 → 自动回查 raw/ 原始文件后再回答

## 6. 质量保障

### 6.1 Hard Gates（必须通过，否则编译失败）

| Gate | 检查内容 |
|------|---------|
| frontmatter 完整 | 每篇概念文章有 id, title, sources, created, updated |
| coverage 标记 | 每个 Section 有 coverage 注释 |
| 来源引用有效 | 所有 `[source: path]` 指向存在的文件 |
| schema 一致 | 新概念已出现在 schema.md 中 |
| 非空内容 | 每个 Section 有实质内容，不只是标题 |

### 6.2 Soft Gates（警告但不阻断）

| Gate | 检查内容 |
|------|---------|
| 孤立概念 | 无入链（其他概念未引用） |
| 低覆盖集群 | 所有 Section 均为 low |
| 过期检测 | 来源文件 mtime 变化但概念未重编译 |
| 内容矛盾 | 跨文章的同一主题有冲突描述 |
| 断裂链接 | `[[slug]]` 引用指向不存在的概念 |
| 缺失交叉引用 | 文章提到的概念未建立链接 |

### 6.3 过期检测

```
来源级别:
  raw/ 文件 mtime > .compile-state.json 记录的 mtime
  → 标记来源 stale → 关联概念标记 needs_recompile

概念级别:
  30+ 天未更新 → 标记 aging
  所有 Section 覆盖度 low → 标记 weak
  无入链 → 标记 orphan
```

## 7. Schema 人机共维

### 7.1 schema.md 的角色

schema.md 是知识库的结构契约：
- 定义概念分类（Topic Taxonomy）
- 定义命名规则（slug 格式）
- 定义交叉引用规则（哪些概念应该互相链接）
- 记录已废弃的概念（避免编译器重建）

### 7.2 共维规则

- 编译器自动生成和更新 schema.md
- 用户可以手动修改分类、添加规则
- 编译器读取 schema.md 作为编译的输入约束
- 冲突时，用户手动内容优先
- schema.md 中的分类决定 INDEX.md 的组织方式

## 8. 命令体系

### 8.1 核心命令

```bash
# 初始化
kc init                              # 创建 raw/ + wiki/ + .kc-config.json

# 添加来源
kc add <path> [--tags "t1,t2"]       # 添加文件/目录到 raw/
kc add <url> --title "..."           # 添加外部链接

# 编译
kc compile                           # 增量编译
kc compile --full                    # 全量重编译
kc compile --dry-run                 # 预览变更，不写入
kc compile --topic <slug>            # 编译单个概念

# 查询
kc query "问题"                      # 基于 Wiki 回答问题
kc query "问题" --save               # 回答并保存为新概念

# 健康检查
kc lint                              # 报告问题
kc lint --fix                        # 报告 + 自动修复安全问题

# 状态
kc status                            # 概念统计 + 覆盖度分布 + 过期概念列表

# 浏览
kc browse                            # 知识地图概览
kc browse <category>                 # 查看分类下的概念
```

### 8.2 会话集成

进入含 `.kc-config.json` 的目录时，自动注入上下文：

| 模式 | 行为 |
|------|------|
| staging | "Wiki 可用，需要深入上下文时查阅" |
| recommended | "先读 Wiki 概念文章，再读 raw/ 原始文件" |
| primary | "Wiki 为主要知识源，仅 low 覆盖时回查 raw/" |

未找到 `.kc-config.json` 时静默退出，不干扰无关项目。

## 12. 功能出处追溯

### 编译管道

| 功能 | 出处 | 说明 |
|------|------|------|
| Phase 1: Scan（mtime 对比增量检测） | **llm-wiki** | `.compile-state.json` + mtime 快照，`--full` 全量模式 |
| Phase 2: Classify（自动主题发现 + slug 推断） | **llm-wiki** | 读标题 + 前 500 字推断 topic slug，优先合并已有主题 |
| Phase 3: Compile（多源合成 + 覆盖度标记 + 来源引用） | **llm-wiki** | 读全文 → 生成文章 → `<!-- coverage: X -->` + `[source: path]` |
| Phase 3.5: Schema 人机共维 | **llm-wiki** | 编译器自动生成，不覆盖手动编辑 |
| Phase 4: Index 生成 | **llm-wiki** | INDEX.md 按分类列出主题 + 覆盖度摘要 |
| Phase 4: Verify（Hard/Soft Gate 分级验证） | **PKB** | Hard Gate 阻断 vs Soft Gate 警告的分级模式 |
| Phase 5: State + Log | **llm-wiki** | `.compile-state.json` + `wiki/log.md` |
| 并行编译（多概念同时编译） | **llm-wiki** | `parallel_topics` + `max_parallel` 配置 |

### 覆盖度与质量

| 功能 | 出处 | 说明 |
|------|------|------|
| 三级覆盖度（high/medium/low） | **llm-wiki** | 每个 Section 独立标记可信度 |
| 覆盖度指导 Agent 回查策略 | **llm-wiki** | low → 自动回查 raw/，high → 直接信赖 |
| Hard Gates（frontmatter、coverage、引用有效等） | **PKB** | 必须通过否则编译失败 |
| Soft Gates（孤立概念、低覆盖、断裂链接等） | **llm-wiki** + **PKB** | llm-wiki 的 8 类检查 + PKB 的分级模式 |
| 过期检测（mtime 对比） | **llm-wiki** | 来源 mtime 变化 → 概念标记 needs_recompile |
| 过期检测（时间窗口 30 天） | **PKB** | `last_updated` 超 30 天标记 aging |

## 13. 实施优先级

| 阶段 | 内容 | 产出 | 验证标准 |
|------|------|------|---------|
| **P0** | Skill 骨架 + init + add | 能创建知识库 + 添加文件到 raw/ | `kc init && kc add doc.md` 成功 |
| **P1** | scanner + classifier + compiler (Phase 1-3) | 能编译出概念文章 | 10 篇文档编译出 3-5 个概念，覆盖度标记合理 |
| **P2** | schema + indexer + state (Phase 3.5-5) | 完整编译闭环 | INDEX.md 正确反映所有概念 |
| **P3** | query | 基于 Wiki 的问答 | 问 5 个问题，回答引用正确的概念和来源 |
| **P4** | lint (Hard/Soft Gate) | 质量检查 | 能检出孤立概念、断裂链接、过期文档 |
| **P5** | 会话注入 | 自动上下文 | 进入项目时自动识别 Wiki |
| **P6** | status + browse | 知识库概览 | 能看到覆盖度分布和过期列表 |

**最小闭环验证**：P0 + P1 完成后，用团队真实的 10-20 篇设计文档测试 `init → add → compile`，验证：
1. 概念发现是否合理
2. 覆盖度标记是否准确
3. 来源引用是否正确
4. 用户手动编辑后再编译，编辑是否保留
