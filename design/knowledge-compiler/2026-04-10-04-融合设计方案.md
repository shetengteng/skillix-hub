# Knowledge Compiler — llm-wiki × PKB 融合设计方案


---

## 1. 设计目标

构建一个**通用知识编译器**，同时支持两种场景：

| 场景 | 输入 | 输出 | 来源 |
|------|------|------|------|
| 个人知识库 | 文章、论文、笔记、书摘 | 结构化 Wiki（主题文章 + 覆盖度） | llm-wiki |
| 项目文档化 | 源码、配置、目录结构 | 标准工程文档（架构 + API + 运维） | PKB |

**一个编译器，两种模式，统一的质量保障框架。**

---

## 2. 核心设计原则

| 原则 | 来源 | 说明 |
|------|------|------|
| 人机共维 | PKB | AI 维护机械事实，人维护判断内容，文档标记 `maintained-by` |
| Raw 不可变 | llm-wiki | 原始材料只读，编译器从不修改输入源 |
| 覆盖度透明 | llm-wiki | 每个章节标记可信度，让 Agent 自主决定是否需要回查 |
| 渐进披露 | PKB | 知识分轮加载，不一次性 dump |
| 可验证性 | PKB | Hard/Soft Gate 分级验证，质量可量化 |
| 低侵入 | llm-wiki | 非目标项目静默退出，不干扰无关工作流 |
| 声明式优先 | 两者 | 核心逻辑用自然语言规范，关键路径 Python 兜底 |

## 4. 融合编译管道

核心创新：**5 阶段管道 + 3 步引擎的融合**

### 4.1 统一管道流程

```
┌──────────────────────────────────────────────────────┐
│ Phase 0: Discover (from PKB)                         │
│ • 检测 .kc-config.json → 确定模式                      │
│ • personal: 扫描 raw/ 目录                            │
│ • project: 分析源码 + 配置 + 目录结构                   │
│ • 展示 discovery summary → 等待确认（可选）             │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 1: Scan (from llm-wiki)                        │
│ • 读取 .compile-state.json（mtime 快照）               │
│ • 对比当前文件 → new / changed / deleted               │
│ • 可选: git commit 对比辅助判断                         │
│ Output: scan_result                                   │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 2: Classify & Discover (from llm-wiki)         │
│ • personal: 自动推断 topic slug，优先合并已有主题       │
│ • project: 按 C4 层级 + 固定分类映射                   │
│ • 歧义时交互确认                                       │
│ Output: topic_map                                     │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 3: Compile / Generate (融合)                    │
│ • 加载 prompts/ 和 templates/ 对应模板                 │
│ • personal: 读全文 → 合成主题文章 → 覆盖度标记          │
│ • project: 源码分析 → 标准文档生成 → 引用验证           │
│ • 支持并行（多主题/多文档同时编译）                      │
│ Output: 编译/生成的 Markdown 文件                      │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 3.5: Schema (from llm-wiki)                    │
│ • 首次运行: 生成 schema.md                             │
│ • 后续: 增量更新，不覆盖手动编辑                        │
│ • project 模式: schema 包含 C4 层级和模块关系          │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 4: Refine + Index (融合)                        │
│ • Refine (from PKB): 路径验证 → 去冗余 → 统一术语      │
│ • Index (from llm-wiki): 更新 INDEX.md                │
│ • 写入 metadata footer（git commit + 时间戳）          │
│ • Hard Gate 检查: 非空、有效引用、无 TODO、元数据完整    │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│ Phase 5: State & Log (from llm-wiki)                 │
│ • 更新 .compile-state.json                            │
│ • 追加 log.md                                         │
│ • 更新覆盖度统计                                       │
└──────────────────────────────────────────────────────┘
```

### 4.2 模式差异表

| 阶段 | personal 模式 | project 模式 |
|------|-------------|-------------|
| Phase 0 Discover | 扫描 raw/ 子目录 | 分析源码 + 配置 + 入口 |
| Phase 1 Scan | mtime 对比 | mtime + git commit 双轨 |
| Phase 2 Classify | 自动推断 topic slug | C4 层级 + 固定分类 |
| Phase 3 Compile | 多源合成 → 主题文章 | 源码分析 → 标准文档 |
| Phase 3.5 Schema | Wiki taxonomy 契约 | 模块关系 + 架构约束 |
| Phase 4 Refine | 覆盖度标记 + INDEX | 路径验证 + 术语统一 + INDEX |
| Phase 5 State | compile-state + log | compile-state + log + metadata |

## 5. 质量保障框架

### 5.1 覆盖度模型（来自 llm-wiki，增强版）

```
每个章节标记：
  <!-- coverage: high -->    多个一致来源，可直接信赖
  <!-- coverage: medium -->  单一来源或部分覆盖，酌情使用
  <!-- coverage: low -->     推断或稀疏，需验证原始文件
  <!-- coverage: none -->    [新增] 无来源支撑，需要补充
```

**personal 和 project 模式都使用**。project 模式中，覆盖度反映文档与源码的匹配程度。

### 5.2 质量评分（来自 PKB，简化版）

| 维度 | 权重 | 说明 |
|------|------|------|
| Accuracy | 40% | 内容与源材料/源码一致 |
| Coverage | 35% | 主题/模块的文档化覆盖率 |
| Consistency | 25% | 术语、格式、交叉引用统一性 |

总分 1.0-5.0，相比 PKB 的 5 维度简化为 3 维度（去掉 Findability 和 Clarity，因为这些由模板保证）。

### 5.3 Hard / Soft Gate 验证

#### Hard Gates（必须通过）

| Gate | 检查内容 |
|------|---------|
| 非空输出 | 生成内容超过标题+模板框架 |
| 有效引用 | 所有文件路径引用指向存在的文件 |
| 无原始占位 | 不存在 `[TODO]`，只允许 `[NEEDS INPUT: reason]` |
| 元数据完整 | 每篇文章包含 frontmatter + metadata footer |
| 覆盖度标记 | 每个章节有 coverage 标记 |
| Schema 一致 | 新主题已录入 schema.md |

#### Soft Gates（警告但不阻断）

| Gate | 检查内容 |
|------|---------|
| 孤立页面 | Wiki 页面无入链（from llm-wiki） |
| 低覆盖集群 | 所有章节均为 low（from llm-wiki） |
| 过期文档 | git commit 偏移 10+ 或 30+ 天未更新（from PKB） |
| 内容矛盾 | 跨文章的同一主题有冲突描述（from llm-wiki） |
| 术语不一致 | 同一概念用不同名称（from PKB） |

### 5.4 过期检测（双轨）

```python
def detect_staleness(entry):
    # Track 1: mtime (from llm-wiki) — 快速，文件级
    if entry.current_mtime > entry.compiled_mtime:
        return StaleReason.FILE_CHANGED

    # Track 2: git commit (from PKB) — 精确，变更级
    if entry.metadata_commit != current_head:
        commits_behind = count_commits(entry.metadata_commit, current_head)
        if commits_behind >= 10:
            return StaleReason.MAJOR_DRIFT
        
    # Track 3: time window — 兜底
    if days_since(entry.last_updated) > 30:
        return StaleReason.TIME_EXPIRED
    
    return None
```

## 7. 会话集成

### 7.1 自动上下文注入（来自 llm-wiki）

```
进入项目 → 向上查找 .kc-config.json
  ├── 找到 → 读取 mode → 注入上下文
  │    ├── staging:     Wiki 可用，按需查阅
  │    ├── recommended: 先读 Wiki 再读原始文件
  │    └── primary:     Wiki 为主，low 覆盖才看原始
  └── 未找到 → 静默退出
```

### 7.2 AI Onboarding（来自 PKB，适配）

```
Round 1 — 宏观认知:
  personal: 喂 INDEX.md + schema.md → AI 总结知识版图
  project:  喂 overview + repo-map → AI 总结项目结构

Round 2 — 深入理解:
  personal: 喂 2-3 个 high 覆盖主题 → AI 总结关键知识
  project:  喂 architecture + workflows → AI 总结架构

Round 3 — 验证:
  AI 生成 3 个问题并回答 → 用户判断正确性
```

## 10. 设计取舍决策

| 决策 | 选择 | 舍弃 | 理由 |
|------|------|------|------|
| 实现方式 | Python CLI + Markdown 规范混合 | 纯 Markdown | 关键路径（扫描、mtime 对比、状态管理）需要确定性逻辑 |
| 质量维度 | 3 维（Accuracy + Coverage + Consistency） | 5 维 | Findability 和 Clarity 由模板保证，不需独立评分 |
| 覆盖度粒度 | 章节级（每个 section 独立标记） | 文档级 | 章节级更精确，让 Agent 知道具体哪段需要回查 |
| 过期检测 | mtime + git 双轨 | 仅 mtime 或仅 git | mtime 快但不精确，git 精确但慢，双轨互补 |
| 发布能力 | 不内置 | Sphinx + 双语 | 个人知识库不需要，project 模式可后续按需集成 |
| SKILL.md 长度 | 精简入口 + references/ 详细文档 | 600 行单文件 | 减少 Token 消耗，Agent 按需加载 |
| 命令前缀 | `kc`（knowledge-compiler） | `wiki-` 或 `/PKB-` | 短、统一、不绑定特定概念 |
| Schema 策略 | 人机共维，编译器不覆盖手动编辑 | 完全自动 | 尊重用户的领域知识 |
| Intent Router | 可选启用 | 必须 | 低门槛入口，但不强制 |
| AI Onboarding | 3 轮渐进 | 一次性加载 | 验证理解正确性比速度更重要 |

## 11. 融合价值总结

```
llm-wiki 贡献:
  ✓ 5 阶段编译管道（增量编译的核心骨架）
  ✓ 覆盖度三级标记（每个章节的可信度）
  ✓ Schema 人机共维契约
  ✓ 会话自动注入（SessionStart hook）
  ✓ wiki-query 知识问答
  ✓ Raw 不可变原则

PKB 贡献:
  ✓ Discover → Generate → Refine 三步引擎（Phase 0 + Phase 4）
  ✓ Hard/Soft Gate 分级验证
  ✓ Git commit 过期检测
  ✓ Metadata footer 标准
  ✓ AI Onboarding 三轮渐进
  ✓ Intent Router 自然语言路由
  ✓ 维护循环 Workflow
  ✓ C4 架构模型（project 模式）

融合创新:
  ★ 双模式统一管道（personal + project 共享基础设施）
  ★ 双轨过期检测（mtime + git 互补）
  ★ 3 维质量评分（简化但实用）
  ★ 精简入口 + 按需加载（避免 Token 浪费）
```
