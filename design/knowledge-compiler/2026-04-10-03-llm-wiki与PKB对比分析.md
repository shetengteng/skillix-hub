# llm-wiki-compiler vs project-knowledge-base-nex (PKB) 对比分析

> 日期：2026-04-10

---

## 1. 定位对比

| 维度 | llm-wiki-compiler | PKB |
|------|-------------------|-----|
| **一句话定位** | 个人 Markdown 知识库 → 结构化 Wiki 编译器 | 软件项目源码 → AI 可读工程文档生成器 |
| **目标用户** | 个人知识工作者 | 软件工程团队 |
| **文档化对象** | 文章、论文、笔记、书摘 | 代码仓库、架构、API、流程 |
| **输入源** | raw/ 目录下的 Markdown 文件 | 项目源码、配置文件、目录结构 |
| **输出物** | wiki/ 目录（主题文章 + 索引 + Schema） | man/ 目录（7 类标准文档 + ADR + 变更提案） |
| **核心价值** | 多源信息合成、知识覆盖度量化 | 架构分析标准化、文档质量保障 |

## 2. 架构对比

### 2.1 技术实现

| 维度 | llm-wiki-compiler | PKB |
|------|-------------------|-----|
| **实现方式** | 纯 Markdown + JSON + Bash | 纯 Markdown（零代码） |
| **插件规范** | Anthropic Claude Code Plugin | Cursor/Claude Skill 规范 |
| **代码量** | ~300 行 Markdown + 60 行 Bash | ~600 行 Markdown |
| **安装方式** | git clone + hook 注册 | Skill Market symlink |
| **配置** | `.wiki-compiler.json` | 无持久配置（每次 discovery） |
| **状态管理** | `.compile-state.json`（mtime 快照） | `<!-- PKB-metadata -->` footer（git commit） |

## 3. 编译/生成管道对比

### 3.1 llm-wiki 编译管道（5 阶段）

```
Phase 1: Scan Sources
  .compile-state.json + 源文件 mtime → new/changed/deleted
      ↓
Phase 2: Classify & Discover Topics
  文件标题 + 摘要 → topic_map {slug: [files]}
      ↓
Phase 3: Compile Topic Articles
  全文阅读 → 模板填充 → 覆盖度标记 → 引用标注
      ↓
Phase 3.5: Schema
  首次生成 / 增量更新（不覆盖手动编辑）
      ↓
Phase 4: Update INDEX
  分类聚合 → 覆盖度摘要 → 更新标记
      ↓
Phase 5: State & Log
  .compile-state.json 更新 + wiki/log.md 追加
```

### 3.2 PKB 生成管道（3 步引擎）

```
Discover:
  项目标记验证 → config 读取 → 目录扫描 → 入口识别 → 源文件阅读
  → 展示 summary → 等待用户确认
      ↓
Generate:
  加载 commands.md 步骤 + templates/ 模板
  → 填充内容 → 验证引用路径
  → Mermaid 图表生成 → 双语处理
      ↓
Refine:
  路径验证 → 去冗余 → 统一术语
  → 写入 PKB-metadata footer → 预览
  → 用户确认 → 写入 man/
```

### 3.3 管道差异

| 方面 | llm-wiki | PKB |
|------|----------|-----|
| **增量策略** | mtime 对比（文件级） | git commit 对比（文档级） |
| **主题发现** | 自动推断 topic slug | 固定 7 类文档编号体系 |
| **并行能力** | 支持多主题并行编译（subagents） | WF-1 中 extended/ops docs 并行 |
| **用户确认** | Phase 2 主题歧义时才问 | Discover 和 Refine 都等确认 |
| **状态持久化** | .compile-state.json | 文档内 metadata footer |
| **Dry Run** | Phase 1-2 only，预览不写入 | 无显式 dry-run |

## 4. 质量保障对比

### 4.1 覆盖度 / 质量模型

| 维度 | llm-wiki | PKB |
|------|----------|-----|
| **模型** | 3 级覆盖度（high / medium / low） | 5 维质量评分 + 4 项覆盖率指标 |
| **粒度** | 每个章节独立标记 | 每个文档整体评分 |
| **标记位置** | 文章内 `<!-- coverage: high -->` | 文档末尾 `<!-- PKB-metadata -->` |
| **可信度语义** | high=可信任, medium=酌情, low=需验证 | 1-5 分量化 |

### 4.2 健康检查

| 检查项 | llm-wiki (`wiki-lint`) | PKB (`/PKB-verify`) |
|--------|----------------------|---------------------|
| 过期检测 | 基于 mtime | 基于 git commit + 时间窗口 |
| 孤立页面 | ✅ | — |
| 缺失交叉引用 | ✅ | — |
| 低覆盖集群 | ✅ | — |
| 内容矛盾 | ✅ | — |
| Schema 漂移 | ✅ | — |
| 断裂链接 | ✅ | ✅（Reference Validation） |
| 缺失元数据 | ✅ | ✅（Hard Gate） |
| 质量评分 | — | ✅（5 维度） |
| 覆盖率统计 | — | ✅（4 项指标） |
| 一致性检查 | — | ✅（术语 + 格式） |
| Mermaid 语法 | — | ✅（Hard Gate） |

### 4.3 验证策略

| 策略 | llm-wiki | PKB |
|------|----------|-----|
| 门控模型 | 无分级，统一报告 | Hard Gate（阻断） + Soft Gate（警告） |
| 自动修复 | `--fix` 参数，用户确认后修复 | Hard Gate 自动重试/修复 |
| 审计模式 | 单一模式 | Quick（CI 用）/ Full（审查用） |

## 9. 融合复刻建议

在 skillix-hub 中复刻 llm-wiki 时，可从 PKB 借鉴以下设计：

| 借鉴点 | llm-wiki 现状 | PKB 做法 | 融合建议 |
|--------|-------------|---------|---------|
| 过期检测 | mtime 对比 | git commit + 时间窗口 | 保留 mtime 为主，可选 git commit 辅助 |
| 质量评分 | 3 级覆盖度（章节级） | 5 维度评分（文档级） | 保留章节级覆盖度，增加文档级综合评分 |
| 验证门控 | 统一报告 | Hard/Soft 分级 | 采用分级门控模型 |
| 用户确认 | 仅歧义时问 | Discover 和 Refine 都确认 | 保持 llm-wiki 的低交互风格，仅关键决策点确认 |
| 元数据 | schema.md（全局） | 每文档 footer + frontmatter | 文章级 frontmatter + 全局 schema |
| Intent Router | 无 | 自然语言 → 命令路由 | 可选引入，降低命令记忆门槛 |

### 不应借鉴的部分

| 设计 | 原因 |
|------|------|
| C4 架构模型 | 个人知识库不需要 Context/Container/Component/Code 分层 |
| Sphinx 发布栈 | 个人知识库不需要 HTML/PDF 生成 |
| 7 类固定编号文档 | llm-wiki 的主题应该自动发现，不应预定义编号体系 |
| 600 行 SKILL.md | 应拆分为入口 + 参考文档，避免 Token 浪费 |
| 严格的双语支持 | 个人知识库通常单语言 |
