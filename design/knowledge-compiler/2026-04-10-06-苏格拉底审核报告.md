# 🏗️ Socratic Design Review — Knowledge Compiler 新方案

> 日期：2026-04-10
> 审核对象：`design/llm-wiki/2026-04-10-05-新方案设计.md`
> 审核方法：苏格拉底式 Review Mode
> 检测域：🏗️ 系统设计（架构提案审查）
> 思维锚点：好的设计能清楚说出自己放弃了什么；没有最好的设计，只有最合适的取舍

---

## 核心目标是否清晰：部分

文档陈述了痛点："人和 AI Agent 都需要结构化、可信赖的知识，但知识散落在各处，没有统一的编译和信任体系"。

但**谁在痛**不够清晰——个人知识工作者和工程团队是两类截然不同的用户，他们的工作流、频率、协作模式完全不同。试图用一个工具同时服务两类用户，是力量分散还是真正的协同？

---

## Dimension A — 问题定义

### 发现

1. **缺少成功标准**：文档定义了"以 Concept 为中心"的新范式，但没有定义做出来之后，怎么判断它比 llm-wiki + PKB 各自独立使用更好

2. **"双模式"的统一性存疑**：personal 模式和 project 模式看似共享管道，但两种模式的编译逻辑差异极大：
   - personal：多源信息合成（需要 AI 推理能力）
   - project：源码分析生成（需要代码理解能力）
   - 共享管道的 Phase 2（Classify）和 Phase 3（Compile）在两种模式下几乎完全不同

### 建议

- 明确回答：**这个工具首先服务谁？** 如果是个人知识工作者，project 模式可以作为未来扩展而非 V1 必须。如果是工程团队，personal 模式可以后置
- 定义 3 个可验证的成功标准，例如：
  - "编译后的知识库让新 AI Agent 回答问题的准确率 > X%"
  - "编译耗时 < Y 分钟/百篇文档"
  - "覆盖度标记的准确率 > Z%"

---

## Dimension B — 假设挑战

### 发现 1："概念图谱优于文件编译"

这是整个方案的**基石假设**，但未经验证。

- 大多数知识工作者的心智模型是**文件和文件夹**，不是抽象的概念图
- 引入 `.kc/concepts/*.json` 中间层意味着用户无法直接编辑概念数据——必须通过工具
- llm-wiki 的 Markdown-in-Markdown-out 之所以简洁，正因为用户可以直接打开 `wiki/` 里的文件阅读和编辑
- 如果这个前提是错的：应退回 Markdown-first 架构，概念元数据用 frontmatter 表示，去掉 JSON 中间层

### 发现 2："来源不限，任何位置"

听起来自由，但失去约束就失去了可预测性。

- `raw/` 目录虽然"限制"了输入位置，却带来了清晰的心智模型：**所有原始材料在这里**
- 散布在任意位置的来源意味着：
  - 文件移动/重命名/删除时，`sources.jsonl` 中的路径失效
  - 用户需要记住注册了哪些文件
  - 跨项目的来源管理变得复杂
- 文件系统是易变的，用户重命名一个目录，所有该目录下的来源就失效——这不是边缘情况，而是日常操作

### 发现 3："视图系统让输出灵活"

视图系统增加了一层抽象，但问题是：**用户真的需要多种视图吗？**

- 如果 90% 的使用场景只用一种视图（wiki 或 project docs），那视图系统就是过度设计
- 每新增一种视图就是一个新的维护面——模板 + 渲染逻辑 + 测试
- 与其一开始就设计视图系统，不如先做好一种输出，需要时再抽象

### 建议

- 对假设 1：考虑 **Markdown-first** 架构——概念文章本身是 Markdown 文件（带 frontmatter），既是内部表示也是外部呈现
- 对假设 2：提供"约定优于配置"的默认目录结构（如 `raw/`），但允许用户注册外部来源作为补充
- 对假设 3：V1 先实现一种输出格式（wiki），视图系统作为架构预留而非 V1 功能

## Dimension C — 视角盲点

### 发现

1. **缺少迁移路径**：skillix-hub 已有 `knowledge-base` skill，数据格式为 JSONL + wiki/concepts/。新方案没有说明如何迁移现有数据

2. **协作场景空白**：没有讨论多人/多 Agent 同时操作一个知识库的情况。概念级别的冲突如何解决？编译是否幂等？

3. **性能未考虑**：扫描散布在任意位置的来源文件，比扫描单一 `raw/` 目录慢得多。当来源数量达到 500+ 文件时，Scan 阶段的性能如何？

4. **冷启动体验**：新用户面对空的概念图谱，第一次体验是什么？从"安装完毕"到"得到第一个有用的编译结果"需要几步？

5. **人机共维的边界模糊**：文档说"Schema 人机共维"，但 `.kc/concepts/*.json` 是机器格式——用户如何参与共维？如果用户想手动调整某个概念的覆盖度标记或关系，怎么操作？

6. **与已有 memory 系统的关系**：skillix-hub 有完整的 memory skill（save_fact, save_summary, distill_refined），新方案和 memory 系统的知识是否重叠？两者的边界在哪？

### 建议

- 增加"从 knowledge-base / llm-wiki 迁移"章节
- 如果采用 Markdown-first，人机共维自然解决（用户直接编辑 Markdown）
- 定义与 memory 系统的关系：memory = 短期工作记忆，knowledge compiler = 长期结构化知识

## Dimension D — 风险压测

### 发现

1. **复杂度是最大风险**：方案包含 6 阶段管道 + 视图系统 + 双轨过期检测 + 自然语言路由 + AI Onboarding。实施优先级列了 P0-P7 共 8 级。实施周期长意味着长时间没有可用产物

2. **概念图谱的可逆性低**：JSON 概念图谱一旦构建，回退到纯 Markdown 方案的成本较高。而如果以 Markdown 为主数据格式，回退成本几乎为零

3. **"任意位置来源"的脆弱性**：不是理论风险，而是确定性问题。文件移动、项目重组、磁盘迁移都会导致 sources.jsonl 中的路径失效。需要设计路径修复机制

4. **如果流量增长 10x**：假设从 20 篇文档增长到 200 篇，概念数从 10 增长到 100——编译时间是否线性增长？概念间关系的复杂度是否爆炸？

### 建议

- **最小可行验证（MVP）**：先实现 `add → compile → query` 最小闭环，用真实的 10-20 篇文档跑一遍，验证"概念为中心"是否真的优于"文件为中心"
- 选择 **Markdown-first** 可以显著降低可逆性风险——即使方案失败，产出的 Markdown 文件仍然有价值
- 默认目录结构 + 可选外部注册，平衡自由度和可预测性

## 最高优先级改进项 (Top 3)

| 优先级 | 改进项 | 影响 |
|--------|--------|------|
| **#1** | **明确首要用户和 V1 范围** | 选择 personal 或 project 作为 V1 主场景。"两种都做"≈"两种都做不好" |
| **#2** | **重新评估 JSON 概念图谱 vs Markdown-first** | 如果概念文章就是 Markdown（带 frontmatter），可大幅简化架构，保留覆盖度和来源追踪的核心价值 |
| **#3** | **定义最小闭环验证** | 用真实文档跑 `add → compile → query`，验证核心假设后再展开视图系统/多模式/AI Onboarding |

## Recommended Next Step

先回答一个问题：

> **这个工具 V1 首先服务谁——积累个人知识的个体，还是文档化项目的团队？**

答案决定了：
- 编译管道的核心逻辑（多源合成 vs 源码分析）
- 默认目录结构（raw/ vs src/）
- 视图系统是否 V1 必须

建议回答后，用 first-principles 从确认的基石假设出发，重构实施方案。
