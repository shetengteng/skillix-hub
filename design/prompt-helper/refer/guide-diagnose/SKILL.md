# 行为诊断指南

## 核心问题

**模型做错了，问题出在 Prompt 的哪里？**

这个 Skill 从**观察到的行为问题**出发，定位到需要修改的 Facet 和 Section。它不检查 Prompt 的静态质量（那是 `workflow-audit-prompt` 的事），也不评估修改是否合理（那是 `workflow-review-edit` 的事）——它只负责**从症状找到病因**。

**与其他 Skill 的分工**：

- **guide-diagnose**（本 Skill）：有问题时用。从行为症状 → 定位根因 → 指导修复方向
- **workflow-audit-prompt**：日常检查。从 Prompt 文本 → 逐项检查质量
- **workflow-review-edit**：改完后用。从具体修改 → 评估修改是否合理

典型使用链路：guide-diagnose 定位根因 → 输出诊断报告 + 创建 Trial 任务列表 → 逐 Trial 使用 workflow-edit-prompt 执行修复。

---

## 前置知识

本 Skill 全程使用 Facet 概念（六种功能类型：定义/知识/场景/策略/边界/规范）进行错误分类和定位。Facet 类型的完整定义见 `@guide-placement/references/facet-types.md`。

---

## 诊断输入

诊断需要以下信息：

- **错误行为**：模型实际做了什么（越具体越好）
- **期望行为**：模型应该做什么
- **触发输入**：导致错误的输入内容（如果有的话）
- **Prompt 文本**：当前的 Prompt 内容

---

## Step 1: 收集 + 快筛

先做「低挂果实」检查——很多问题不需要深度诊断，简单的结构性缺陷就是根因。

### 快筛清单

- **缺少步骤编号 / 执行顺序不明确**：工作流程没有清晰的步骤标号，模型容易跳步或乱序
- **执行区在解释"是什么"而非"怎么做"**：工作流程里混入了定义性内容，模型分不清"理解"和"执行"
- **同一规则在多处描述且措辞矛盾**：模型无法判断该遵循哪个版本
- **完全缺失某个必需 Facet**：如无输出格式（规范型）、无场景定义（场景型）、无工作流程（策略型）
- **无兜底场景**：Super Agent 缺少处理"不属于任何已知场景"的分支

**如果快筛命中** → 直接给出修复建议，引用对应的 guide skill。不需要进入深度诊断。

**如果快筛未命中** → 进入 Step 2。

---

## Step 2: 错误分类 + Facet 定位

这一步由**错误→Facet 映射表**驱动。但在开始分类之前，先做**诊断切分**。

映射表的完整内容在 `references/error-facet-map.md` 中。

### 前置：诊断切分（Trial Segmentation）

不要试图一次性诊断整个 Prompt 的所有问题。先把问题空间切成独立的诊断单元（Trial），形成一个 **Trial 队列**，然后按优先级逐个处理——每个 Trial 独立走完 Step 2→3→4 的完整流程。

**切分规则**：

- **多个症状** → 每个症状是一个独立 Trial，从影响最大的开始排入队列
- **多个场景的 Prompt** → 先隔离出问题场景，只诊断那个场景对应的 Section
- **长 Prompt（超过 200 行）** → 按 Section 分段，逐段进入阶段 B，不要一次性分析整个 Prompt
- **Super Agent + Sub Agent** → 先诊断 Super Agent 本身（一个 Trial），如果问题不在协调层，再诊断相关 Sub Agent（另一个 Trial）

**Trial 队列是动态的**：诊断过程中发现的新症状（如修复一个问题后暴露出另一个）作为新 Trial 追加到队列末尾。

**为什么切分**：TRAIL 论文发现诊断性能与轨迹长度反相关——分析的内容越多，准确率越低。切分后每个 Trial 的分析范围缩小，定位精度更高。

### 阶段 A：识别错误类型（全局视角）

从观察到的症状，对照映射表，识别属于哪种错误类型。映射表覆盖两大来源：

- **TRAIL 错误分类学**（20 种具体错误类型）：覆盖 Agent 执行层面的问题——推理错误、系统执行错误、规划与协调错误
- **PRISM 特有缺陷**（7 种）：覆盖 Prompt 设计层面的问题——场景互斥性破坏、信息传递断链等

**注意区分 Facet 问题和非 Facet 问题**：系统执行错误（API 报错、超时、配置错误等）不是 Prompt 的问题，不要去改 Prompt。映射表中这些类型标注为「非 Facet — 检查工具/环境配置」。

给出 1-2 个最可能的错误类型。不需要 100% 确定——后面用干预验证来确认。

### 阶段 B：从错误类型定位 Facet（局部检查）

映射表中每种错误类型指向 1-2 个最可能的 Facet 类型。定位到 Facet 类型后，在当前 Trial 对应的 Section 中逐段检查（切分范围见前置步骤）。

---

## Step 3: 假设 + 最小干预

### 形成假设

用一句话描述根因假设：

> 因为 **[Facet 类型]** 在 **[Section]** 中 **[具体问题]**，导致 Agent **[错误行为]**。

如果发现多个问题，按影响优先级排序。**先处理「决定性错误」**——修正它对行为改善影响最大的那个问题。不要试图一次修所有问题。

### 最小干预

对最高优先级假设做**最小化修改**来验证。这不是正式修复，只是测试假设是否正确。

**干预模板**：

- 怀疑**定义型** Facet → 临时加一句角色澄清或职责边界
- 怀疑**知识型** Facet → 临时补一条领域规则或术语定义
- 怀疑**场景型** Facet → 临时加一句场景边界或意图识别条件
- 怀疑**策略型** Facet → 临时加一个步骤或调整步骤顺序
- 怀疑**边界型** Facet → 临时加一条禁止规则或特殊处理
- 怀疑**规范型** Facet → 临时加一个输出格式说明或模板

**原则**：改动越小越好。只加 1-2 句话，不做结构性调整。

---

## Step 4: 验证 + Trial 流转

### 验证

用原始的问题触发输入测试干预后的行为。根据结果走三条路径：

**路径 A：假设成立（行为改善）** → 当前 Trial 标记为「已确认」，记录修复方案（见下），进入下一个 Trial。

**路径 B：假设不成立（行为未改善），还有候选 Facet** → 回到 Step 2 阶段 A，尝试下一个候选错误类型/Facet（仍在同一 Trial 内）。

**路径 C：假设不成立，已耗尽候选** → 当前 Trial 标记为「未确认——建议人工分析」，进入下一个 Trial。

**部分改善**：属于路径 A 的变体——最小干预改善了当前症状但暴露了新症状。当前 Trial 按路径 A 处理（标记为「已确认」），新症状作为新 Trial 追加到队列末尾。

### 停止条件

- **单 Trial**：同一 Trial 内最多尝试 3 个候选 Facet 类型。第 3 次仍不成立 → 走路径 C。
- **全局**：所有 Trial 处理完毕（无论已确认/未确认）→ 输出诊断报告。

### 记录修复方案（路径 A）

假设确认后，**撤销最小干预**，记录诊断结论和修复方向建议：

- 明确定位到的 Facet 类型和 Section
- 给出修复方向（应该怎么改，而非直接改）

### 行为测试

为修复的 Facet 类型设计 2-3 个测试输入：

1. **正常输入**：应该正确处理的典型输入
2. **边界输入**：之前出问题的那个输入
3. **混淆输入**：与其他场景/行为容易混淆的输入

### Trial 流转

当前 Trial 完成（已确认或未确认）后，检查队列中是否还有待处理的 Trial。如果有，从队列中取出下一个 Trial，回到 Step 2 开始新一轮诊断。如果没有，输出诊断报告。

---

## 输出格式

诊断完成后，先输出诊断报告，再创建 Trial 任务列表。

```
## 行为诊断报告

**总览**：共 [N] 个 Trial，[X] 个已确认，[Y] 个未确认

### 快筛结果
- [快筛发现 / 无明显结构问题]

---

### Trial 1: [症状简述]
**症状**：[Agent 的错误行为]
**期望**：[应有的正确行为]

**错误类型**：[从映射表识别的错误类型，如「推理错误 > 任务误解」]
**来源**：[TRAIL 分类 / PRISM 特有缺陷]

**Facet 类型**：[定位到的 Facet 类型]
**位置**：[具体 Section / 内容]
**根因假设**：因为 [X]，导致 Agent [Y]
**验证结果**：已确认 / 未确认（已尝试 N 个候选）

**修复方案**：
- **使用 Skill**：`workflow-edit-prompt`
- **修复方向**：[具体建议]
- **行为测试**：
  1. [正常输入] → 期望 [X]
  2. [边界输入] → 期望 [Y]
  3. [混淆输入] → 期望 [Z]

### Trial 2: [症状简述]
...（同上结构）
```

**情境变体**：

- 如果只有 1 个 Trial，省略总览行，直接输出该 Trial 的内容
- 如果快筛阶段已全部命中（未进入 Step 2），不需要 Trial 结构——直接列出快筛发现和修复建议
- 如果所有 Trial 均未确认，仍输出完整报告，总览行标注全部未确认，并在末尾说明建议人工分析的原因

### Trial 任务列表

诊断报告输出后，使用 TodoWrite 将已确认的 Trial 转化为修复任务列表，驱动逐 Trial 修复。

**创建规则**：

- 每个已确认 Trial → 一个任务：`Trial N: [症状简述] — [修复方向摘要]`
- 第一个任务标记为 `in_progress`，其余标记为 `pending`
- 未确认的 Trial 不纳入任务列表（报告中已标注"建议人工分析"）
- 快筛直接命中的问题也作为独立任务纳入

**执行方式**：

创建任务列表后，按顺序逐个 Trial 使用 `workflow-edit-prompt` 执行修复。完成一个 Trial 后标记为 `completed`，启动下一个。

**为什么用任务列表**：诊断可能产生多个 Trial，一次性全部修复容易遗漏或混乱。任务列表让模型聚焦当前 Trial，修完一个再修下一个，与 Trial 队列的"逐个处理"设计一致。
