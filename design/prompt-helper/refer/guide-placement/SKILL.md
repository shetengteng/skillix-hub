# Prompt 信息布局原则

## 适用范围

本 Skill 指导 prompt 中**信息的布局设计**，帮助决定：
- 一块信息属于什么功能类型（Facet 分类）
- 哪些信息应该打包在一起、哪些应该分开（布局决策）
- 打包在一起的信息如何组织内部顺序（Section 内部组织）
- 如何设计 section 使其可独立增删改（模块化）

整体结构设计（标准章节组成、信息归属）参考 `@guide-skeleton`。
示例值的写法（占位符、情境模板）参考 `@guide-example`。
场景型 Facet 的深度写作指南参考 `@guide-scenario`。
规范型 Facet 的深度写作指南参考 `@guide-output`。

## Part 1-2 - Facet 核心概念与功能型分类

Facet（任务侧面）是 prompt 设计的基本单位——模型需要理解的任务的一个独立方面。框架定义了六种功能类型：定义型、知识型、场景型、策略型、边界型、规范型。每种类型的完整定义、归属维度、识别方法见 `references/facet-types.md`。

以下 Part 3-7 基于 Facet 概念，指导布局决策和 Section 组织。

---

## Part 3 - Facet 到 Section 的布局决策

**核心问题**：哪些 facet 打包进同一个 section，哪些分开？

### 判据 1：facet 间耦合度

- 紧耦合（改一个 facet 必须改另一个）→ 打包进同一个 section
- 松耦合（可以独立变化）→ 分开放

**内聚性参考**（从合理到不合理）：

- **功能内聚**（一组 facet 为同一个功能构成 What → When → How 链条）→ 打包合理
  - What（定义型 + 知识型）：这个功能是什么、规则是什么
  - When（场景型 + 边界型）：什么时候适用、什么时候不适用
  - How（策略型）：具体怎么执行
  - 不要求三段都齐全（简单功能可能只有 What + How），但 facet 之间必须形成链条关系
- **顺序内聚**（一个 facet 的输出是另一个的输入）→ 打包合理
- **通信内聚**（操作同一组数据）→ 打包合理
- **逻辑内聚**（只是归类相似但本质不同）→ 不应打包，应分开

**内聚性区分方法**：

功能内聚 vs 逻辑内聚是最易混淆的一对。用以下问题区分：

- 功能内聚检验：
  - 依赖测试："去掉 facet A，facet B 还能独立完成它服务的功能吗？" 如果不能 → 功能内聚
  - 链条测试："这些 facet 能否为同一个功能构成 What → When → How 链条？" 如果能 → 功能内聚
- 逻辑内聚检验："去掉'相关/相似'这个标签，A 和 B 之间还有实际的数据流或功能依赖吗？" 如果没有 → 逻辑内聚
- 链条测试的典型反例：把所有知识型 facet 归在一个"规则汇总" section → 它们服务不同功能，无法构成链条 → 逻辑内聚

**关键澄清**：facet 功能类型（知识/策略/边界...）和耦合度是**正交维度**。不同类型的 facet 完全可以是功能内聚——判断依据是"能否为同一功能构成 What → When → How 链条"，而非"是否属于同一类型"。

### 判据 2：facet 作用域

- 专属 facet（只有 1 处使用）→ 嵌入使用位置
- 共享 facet（多处引用）→ 提取到集中位置

### 决策路径

```
这个 facet 是专属的还是共享的？
│
├── 共享 → 提取到集中位置（独立 section 或前置步骤）
│
└── 专属 → 和消费者的耦合度？
    │
    ├── 紧耦合（功能/顺序/通信内聚）
    │   → 打包进消费者的 section
    │
    └── 松耦合（逻辑内聚或无内聚）
        → 独立 section，用名称引用
```

### 常见误判模式

- **"facet 类型不同 → 松耦合"**：知识型规则 + 策略型流程，如果能为同一功能构成 What → How 链条，就是功能内聚。类型不同不等于松耦合
- **"section 太大 → 应该拆"**：大小不是拆分依据，耦合度和作用域才是。大而内聚的 section 比小而碎片化的多个 section 更健康
- **"知识型 facet 应该集中管理"**：只有**共享**知识型 facet 才需要集中。专属知识型 facet 应留在使用位置，保持功能自包含
- **"规则应从流程中分离"**：如果规则（What）和执行步骤（How）为同一个功能构成链条，分离就是打破功能内聚。例如把"术语匹配特殊规则"从"术语匹配流程"中拆出来单独成 section → 链条断裂，模型可能忽略优先级关系
- **"逐 section 检查通过 → 整体布局合理"**：每个 section 内部可以各自内聚良好，但不同 section 中服务同一功能的 facet 被按逻辑分类拆开仍是布局问题。需要跨 section 执行链条测试：不同 section 中的 facet 能否为同一功能构成 What → How 链条？若能 → 应合并

### 应用示例

以下示例均来自真实 prompt：

- **场景定义 + 执行流程**（Super Agent V6 场景 A）：场景型 + 策略型，功能内聚且专属 → 打包在同一个 section
- **Agent 接口规范**（知识型 + 规范型）：改 Agent 能力必须改调用模板 → 紧耦合打包
- **suggestionActions 处理**（混合归属）：解析逻辑（多处使用）→ 共享 → 提取到集中步骤；跳过规则（只服务 Classifier）→ 专属 → 留在 Classifier 接口。即使在同一个 section 中，也要检查是否混入了应提取的共享 facet
- **语言传递规则**（边界型）：所有调用都遵循 → 共享 → 集中在通用规则
- **"核心原则"章节**（多种边界型）：通常是多种**共享**边界型 facet 的集合 → 按共享 facet 规则集中放置（不是逻辑内聚，而是作用域驱动的集中）

## Part 4 - Section 内部组织

**当多个 facet 打包在同一个 section 时**，用以下顺序组织内部内容。

### 内部顺序原则

```
1. 名称/标识 — 给 section 一个可引用的名字
2. What（定义型 + 知识型 facet）— 先说清楚"是什么"
3. When（场景型 + 边界型 facet）— 再说"什么时候适用"
4. How（策略型 facet）— 最后说"怎么做"
5. 输出/约束（规范型 + 边界型 facet）— 补充"输出什么、不做什么"
```

**顺序原则**：先理解 → 再判断 → 再行动。

**与 Part 3 的关系**：What → When → How 既是功能内聚的判定标准（能构成这条链 → 应打包），也是打包后的内部排列顺序。两者是同一个底层逻辑的两面。

### 示例

以 Super Agent V6 场景 A 为例，它在一个 section 中打包了多种 facet：

```
**场景 A：资产信息填写引导**        ← 名称
用户希望 AI 帮助检查或补充...       ← What（定义型）

**可协作的 Agent**：                ← What（知识型）
- Quality_Checker / Guide_Agent

**适用条件**：                      ← When（场景型）
- Request 状态为 DRAFT 或 REJECTED
- 用户请求引导、信息检查...

**执行流程**：                      ← How（策略型）
1. 先调用 Quality_Checker
2. 提取 JSON
3. 再调用 Guide_Agent
```

## Part 5 - Section 间关系与模块化

### Section 间三种关系

- **前向引用**：定义 section 提供信息 → 执行 section 用名称引用（最健康）
- **查找引用**：执行 section 指向格式定义 section（松耦合时使用）
- **共置**：多个 facet 在同一个 section，无跨 section 引用（紧耦合时使用）

### 健康判断

- 单向引用 → 健康
- 无引用（共置）→ 紧耦合时健康
- 双向引用 → 不健康（facet 归属有问题）
- 内容重复（同一个 facet 出现在多个 section）→ 不健康（应提取为共享 facet）

### 模块化设计原则

使 section 可独立增删改：

1. **自包含**：紧耦合的 facet 打包在同一个 section，增加同类 section 时复制模式即可
2. **名称接口**：section 之间只通过名称引用，不引用具体 facet 内容
3. **模式化工作流程**：工作流程用通用模式描述，不枚举所有情况
4. **可预测影响**：增删改一个 section 时，能清楚知道还需要改哪里

### 操作指南

- **添加 section**：复制同类 section 的 facet 模式，填入新内容
- **添加 facet**：识别 facet 的功能类型和归属 → 按布局决策放到正确的 section
- **编辑**：自包含 section 只改该 section；检查名称引用是否仍然有效
- **删除**：删掉 section，检查是否有其他 section 通过名称引用了它

## Part 6 - 自然语言输出描述维度

当 Prompt 需要输出自然语言时，规范型 facet 需要覆盖以下四个维度。缺少这些描述，模型会按默认方式输出——通常过于冗长和正式。

- **语气** — 告诉 agent 用什么口吻说话
- **内容边界** — 告诉 agent 说什么、不说什么
- **词汇约束** — 告诉 agent 哪些词可以用、哪些不能用
- **输出时机** — 告诉 agent 什么时候输出

每个维度的详细写法、好/差示例、以及混合输出说明参考 `@guide-output`。

## Part 7 - 检查清单

### Facet 识别

- [ ] 能否列出 prompt 包含的所有 facet？
- [ ] 每个 facet 的功能类型是否明确（定义/知识/场景/策略/边界/规范）？
- [ ] 每个 facet 的归属是否明确（专属/共享）？
- [ ] 去掉任一 facet，模型会犯什么类型的错？（验证必要性和分类）

### 布局决策

- [ ] 共享 facet 是否已提取到集中位置？（核心原则章节有意重复关键约束不算）
- [ ] 专属 facet 的打包是否基于功能内聚（而非逻辑内聚）？
- [ ] 打包的 facet 能否构成 What → When → How 链条？
- [ ] 通用 section 中是否混入了专属 facet？（应移到专属 section）
- [ ] 专属 section 中是否混入了共享 facet？（应提取到集中位置）
- [ ] 不同 section 中的 facet 能否为同一功能构成链条？（能 → 应合并）
- [ ] 同一 facet 是否只在匹配其作用域的 section 保留？
- [ ] 修改一个功能是否只需改一处？（需改多处 → facet 分散）

### Section 组织

- [ ] 多 facet 共置时，内部顺序是否遵循 What → When → How？
- [ ] Section 间引用方向是否健康（单向）？
- [ ] 有无同一 facet 重复出现在多个 section？

### 模块化

- [ ] 新增同类 section 是否只需复制模式？
- [ ] 工作流程是否模式化（非枚举）？
- [ ] 增删改一个 section 的影响是否可预测？
