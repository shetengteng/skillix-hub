# 规范型 Facet 写作指南

## Facet 定位

**规范型（Specification）facet** 告诉模型输出长什么样。

缺失时的典型症状：
- 输出格式错误，JSON 缺少必需字段
- 前端无法解析输出
- 输出过于冗长或格式不统一
- 示例值被 AI 照搬

**与其他 Facet 的关系**：
- 规范型定义"输出什么格式"→ 策略型定义"怎么执行"——两者可以紧耦合（嵌入工作流程）或松耦合（独立章节）
- 规范型和知识型都涉及"数据格式"，但规范型描述"输出格式"，知识型描述"输入格式 / 领域数据"

整体布局原则参考 `@guide-placement`。
示例值写法参考 `@guide-example`。

## 1. 结构化输出设计（JSON）

### JSON 模板设计

**必需字段 vs 可选字段**：

在 prompt 中明确区分：
```
**必需字段**：
- `id` — 字段唯一标识
- `logicalTableId` — 所属表标识
- `result` — 处理结果

**可选字段**：
- `confidence` — 置信度（有时不适用）
- `reason` — 判断理由（调试用）
```

**字段命名原则**：
- 使用 camelCase（与前端一致）
- 名称自解释（`fieldName` 而非 `fn`）
- 与输入数据保持一致（如果输入用 `logicalTableId`，输出也用）

**空值处理**：

在 prompt 中明确空值策略，不要让模型自行决定：
- 字段值为空时用 `null` 还是省略字段？
- 无结果时输出空数组 `[]` 还是不输出？
- 与前端约定一致

```
**空值处理**：
- 可选字段无值时省略（不输出 `null`）
- 无结果时输出空数组 `[]`
```

### 输出设计决策

设计输出格式时需要做两个关键决策：放在哪里（位置）和什么时候输出（时序）。

#### 位置：嵌入 vs 独立

**嵌入工作流程**：当输出格式和执行步骤紧密绑定时

```
### 步骤 3：逐字段输出结果

处理完一个字段，立即输出：

```json async_cui_callback
{"id": "字段ID", "logicalTableId": "表ID", "result": "分类结果"}
```（反引号闭合）

然后继续处理下一个字段。
```

**独立章节**：当输出格式是全局规范，多个步骤都要遵循时

```
## 输出格式

所有结果使用以下格式输出：

```json async_cui_callback
{"id": "字段ID", "logicalTableId": "表ID", "result": "结果值"}
```（反引号闭合）
```

**判断方法**：
- 输出只在一个步骤产生 → 嵌入（专属，紧耦合）
- 输出是全局规范 → 独立（共享或松耦合）
- 不确定时 → 独立章节更安全（易于查找和修改）

#### 时序：流式 vs 批量

**流式（边处理边输出）**：

适用场景：逐个处理对象（如逐字段分类、逐条检查）

```
处理完一个字段，立即输出这个字段的结果，
然后继续处理下一个。用户能看到实时进度。
```

优势：
- 用户体验好（实时看到进度）
- 即使中途出错，已输出的结果不丢失
- 前端可以实时处理每个 callback

**批量**：

适用场景：需要汇总分析后再输出（如统计所有字段后给出总结）

```
先处理完所有数据，然后一次性输出汇总结果。
```

适用条件：
- 结果之间有依赖关系（如需要排序、统计比例）
- 用户期望看到完整的分析报告
- 输出量很小（只有一个 JSON 对象）

**在 Prompt 中说明**：

```
**输出方式**：边处理边输出。
处理完一个字段就立即输出该字段的 JSON callback，
不要等全部处理完再批量输出。
```

## 2. 自然语言输出设计

当 Prompt 需要输出面向用户的自然语言时，规范型 facet 需要覆盖四个描述维度。缺少这些描述，模型会按默认方式输出——通常过于冗长和正式。

> `@guide-placement` Part 6 概述了这四个维度，本节提供深度写法指导和示例。

### 语气

告诉 agent 用什么口吻说话。

好的描述：
```
- 口语化，像同事聊天
- 直接说结果，不用"经过分析，我认为..."这种开场白
```

差的描述：
```
- 请使用专业且友好的语气
```
（太模糊，模型不知道具体怎么做）

### 内容边界

告诉 agent 说什么、不说什么。

好的描述：
```
- 直接说结果，不描述检测过程
- 只输出有问题的字段，没问题的不提
- 空值不是问题，不要提及
```

差的描述：
```
- 输出相关信息
```
（没有说清楚"相关"的边界）

### 词汇约束

告诉 agent 哪些词可以用、哪些不能用。

好的描述：
```
- 不暴露技术术语（suggestionActions、incompleteFieldsInfo 等）
- 用"描述"代替"description"，用"示例值"代替"sampleValue"
```

差的描述：
```
- 使用通俗易懂的语言
```
（没有给出具体的替换规则）

### 输出时机

告诉 agent 什么时候输出。

好的描述：
```
发现问题时**立即输出**自然语言 + JSON，不要等到最后。
```

差的描述：
```
- 在适当的时候输出结果
```
（"适当"是什么时候？）

## 3. 混合输出设计

### NL + JSON 交替

当自然语言和 JSON 交替出现时，需要在 prompt 中说明两者的关系：

```
**输出方式**：
- 自然语言部分负责"给人看"——简洁、口语化
- JSON 部分负责"给系统用"——完整、结构化
- 两者可以说同一件事，但用不同的方式

**示例**：
正在分析字段"用户邮箱"...描述填得不太规范，建议改成...

```json async_cui_callback
{"id": "字段ID", "logicalTableId": "表ID", "fieldName": "user_email",
 "result": {"description": "建议的描述"}}
```（反引号闭合）
```

混合输出的流式/批量策略参考 Section 1「输出设计决策 > 时序」。

Sub Agent 被 Super Agent 调用时通常只需要纯 JSON 输出，不需要面向用户的自然语言——详见 Section 5「Sub Agent 输出约束」。

## 4. 前端交互格式

前端无法直接理解自然语言输出，需要特定的代码块标记来识别和解析 Agent 的结构化数据。以下两种格式是前端与 Agent 之间的通信协议。

### async_cui_callback（数据回传）

用于将 Agent 的处理结果回传给前端系统，前端识别到这个标记后会解析 JSON 并更新界面：

```markdown
```json async_cui_callback
{
  "id": "字段ID",
  "logicalTableId": "表ID",
  "fieldName": "字段名称",
  "result": "结果值"
}
```（反引号闭合）
```

**关键要求**：
- `id` 和 `logicalTableId` 是必需字段（前端用来定位）
- 代码块必须以 ` ```json async_cui_callback ` 开头（前端靠这个标记识别）
- 每个对象单独一个代码块（配合流式输出，参考 Section 1「输出设计决策 > 时序」）

### async_cui_suggest（建议按钮）

用于在聊天界面显示可点击的建议按钮，前端将数组中的每个字符串渲染为一个按钮：

```markdown
```json async_cui_suggest
["建议1", "建议2", "建议3"]
```（反引号闭合）
```

**关键要求**：
- 内容是字符串数组
- 每个字符串会变成一个可点击按钮
- 通常放在 Agent 输出的最后

## 5. Sub Agent 输出约束

当 Prompt 作为被调用方（如 Sub Agent 被 Super Agent 调用）时，需要额外的输出约束：

**核心原则**：被调用方不面向用户，只面向调用方。

```
**输出约束**：
- 只输出结构化结果，不输出面向用户的总结
- 不输出统计信息（"共检查了 X 个字段"）
- 不输出建议（"建议您..."）
- 让 Super Agent 决定如何面向用户呈现
```

→ 详细的 Sub Agent 输出约束原则参考 `@guide-skeleton`（Super Agent 结构 > 核心原则）

## 常见误区

- **空值策略缺失**：没有在 prompt 中说明空值怎么处理，导致模型自行决定（有时用 `null`，有时省略字段，有时输出 `""`）→ 在 JSON 模板设计中明确空值策略（Section 1）
- **示例值被照搬**：用了具体值（如 `"user_email"`）而非 high-level 占位符（如 `"字段名"`），模型误以为是实际值 → 参考 `@guide-example`
- **语气描述太模糊**：写"专业且友好"而不是给出具体行为（"直接说结果，不加开场白"）→ 参考 Section 2 语气 / 词汇约束
- **混淆输出格式与输入格式**：规范型描述"输出什么"，输入数据结构属于知识型 facet → 参考「Facet 定位」
- **流式/批量选择不当**：对逐个处理的任务使用批量输出，导致用户等待时间长 → 参考 Section 1「输出设计决策 > 时序」

## 检查清单

### 结构化输出（JSON）→ Section 1
- [ ] 必需字段和可选字段是否明确区分？
- [ ] 字段命名是否自解释？
- [ ] 空值处理策略是否明确？（`null` / 省略字段 / 空字符串）
- [ ] 输出位置是否合理？（嵌入 vs 独立）
- [ ] 流式 vs 批量选择是否合理？（逐个处理 → 流式；需汇总 → 批量）
- [ ] 示例是否使用 high-level 占位符？（参考 `@guide-example`）

### 自然语言输出 → Section 2
- [ ] 是否描述了语气？（口语化/正式/专业简洁）
- [ ] 是否描述了内容边界？（说什么、不说什么）
- [ ] 是否有词汇约束？（避免哪些技术术语）
- [ ] 是否说明了输出时机？（实时/批量/结束时）

### 混合输出 → Section 3
- [ ] NL 和 JSON 的关系是否说明？
- [ ] 两者的职责是否清晰？（NL 给人看，JSON 给系统用）

### 前端交互 → Section 4
- [ ] `async_cui_callback` 格式是否正确？（必需字段 + 标记）
- [ ] `async_cui_suggest` 格式是否正确？（字符串数组）
- [ ] 是否配合流式输出？（每个对象单独一个代码块）

### Sub Agent 约束 → Section 5
- [ ] 是否限制了不该输出的内容？（总结、统计、建议）
- [ ] 是否只面向调用方输出结构化数据？
