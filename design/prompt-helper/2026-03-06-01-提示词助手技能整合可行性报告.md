# Prompt Helper Skill 整合可行性报告

## 1. 背景与目标

当前 `design/prompt-helper/` 目录下存在一套完整的 Prompt 编写辅助体系，由 **7 个 Guide Skill + 3 个 Workflow Skill + 2 个 Command + 4 个 Reference 文件** 组成。这些组件分散在多个目录中，使用时需要手动加载多个 Skill 文件，学习成本高、操作链路长。

**目标**：将上述所有组件整合为一个统一的 `prompt-helper` Skill，用户只需安装一个 Skill 即可获得完整的 Prompt 编写辅助能力。

### 2.3 核心发现

1. **高度互引**：7 个 Guide Skill 之间存在大量交叉引用，`guide-placement` 和 `guide-skeleton` 是几乎所有流程的必读依赖

3. **知识体系完整**：覆盖了 Prompt 设计的完整生命周期——从结构设计、信息布局、场景/输出/示例编写，到质量检测、行为诊断、修改评审
4. **Reference 文件是核心知识**：`facet-types.md` 和 `error-facet-map.md` 是诊断和检测流程的关键参考

## 3. 整合方案分析

### 3.1 方案 A：单文件合并（全量内联）

将所有内容合并到一个 `SKILL.md` 文件中。

**优势**：
- 安装和使用最简单，一个文件搞定
- Agent 一次性加载全部知识，无需运行时读取

**劣势**：
- 文件约 25,000+ tokens，超过大多数 Skill 的合理大小
- 每次触发都加载全部内容，即使只需要其中一个功能
- 维护困难，修改一个维度需要在巨大文件中定位

**结论**：不推荐。Token 浪费严重，违背按需加载原则。

### 3.2 方案 B：主文件 + 子模块（推荐）

一个主 `SKILL.md` 作为入口和路由，子模块按功能分文件存放在 `docs/` 目录下。

**结构设计**：

```
prompt-helper/
├── SKILL.md                          # 主入口（~3,000 tokens）
│   ├── 角色定义 + 核心概念（Facet 类型定义内联）
│   ├── 三大功能入口（编写/检测/诊断）
│   ├── 路由逻辑（根据用户意图分发）
│   ├── 通用原则精华（从 skeleton + placement 提炼）
│   └── 子模块寻径规则（基于 SKILL.md 所在目录的相对路径）
│
└── docs/
    ├── guide-structure.md            # 结构与布局（skeleton + placement 合并精炼）
    ├── guide-scenario.md             # 场景设计
    ├── guide-output.md               # 输出设计
    ├── guide-example.md              # 示例编写
    ├── guide-info-passing.md         # 信息传递
    ├── workflow-edit.md              # 编辑流程 + 修改评审（合并）
    ├── workflow-audit.md             # 质量检测 + 质量等级 + 常见问题（合并）
    ├── workflow-diagnose.md          # 行为诊断 + 错误映射表（合并）
    └── checklists.md                 # 所有检查清单汇总
```

**子模块寻径机制**：

Agent 通过 Read 工具加载子模块时，如果文档中只给出相对路径（如 `docs/xxx.md`），Agent 通常会默认在用户当前工作区根目录下寻找，而非 Skill 的实际安装目录，导致文件读取失败。这是多文件 Skill 的常见失败模式。

解决方案：在主 `SKILL.md` 中内置寻径指令——

```
## 子模块加载规则

本 Skill 的子模块位于与本文件同级的 docs/ 目录下。
加载子模块前，请先确定本 SKILL.md 的所在目录路径（即 ${SKILL_DIR}），
然后使用 ${SKILL_DIR}/docs/<文件名> 作为完整路径进行读取。

示例：如果本文件路径为 .cursor/skills/prompt-helper/SKILL.md，
则 guide-structure.md 的完整路径为 .cursor/skills/prompt-helper/docs/guide-structure.md
```

**优势**：
- 主文件控制在 3,000 tokens 以内，包含核心概念和路由逻辑
- 按需加载：Agent 根据任务类型只读取需要的子模块
- 子模块可以合并同类内容，减少冗余（如 skeleton + placement 合并）
- 维护友好，每个子模块职责清晰
- 内置寻径规则，避免多文件 Skill 的路径解析失败

**劣势**：
- 仍需运行时读取子模块（但次数从 5-10 次降至 1-3 次）
- 需要精心设计主文件的路由逻辑

**结论**：推荐。平衡了易用性和 Token 效率。

### 3.3 方案 C：分层内联（核心内联 + 深度按需）

主文件内联高频使用的核心知识（Facet 定义、结构原则、检查清单），深度内容按需加载。

**结构设计**：

```
prompt-helper/
├── SKILL.md                          # 主入口 + 核心知识内联（~6,000 tokens）
│   ├── 角色定义
│   ├── Facet 类型定义（完整内联）
│   ├── 结构设计精华（从 skeleton 提炼）
│   ├── 布局决策精华（从 placement 提炼）
│   ├── 三大功能入口 + 路由
│   └── 核心检查清单（合并精炼）
│
└── docs/
    ├── deep-scenario.md              # 场景设计深度指南
    ├── deep-output.md                # 输出设计深度指南
    ├── deep-example.md               # 示例编写深度指南
    ├── deep-info-passing.md          # 信息传递深度指南
    ├── deep-diagnose.md              # 行为诊断完整流程 + 错误映射
    ├── deep-edit.md                  # 编辑流程 + 修改评审
    └── deep-audit.md                 # 质量检测完整流程
```

**优势**：
- 主文件包含 80% 场景所需的核心知识，大多数操作无需读取子模块
- 深度内容仅在复杂场景下按需加载
- 用户体验最好——简单任务零额外读取

**劣势**：
- 主文件 6,000 tokens 偏大（但仍在可接受范围）
- 核心知识提炼需要仔细设计，避免丢失关键信息
- 主文件和深度文件之间可能存在内容重叠

**结论**：可选。适合追求极致用户体验的场景。

## 4. 整合关键决策

### 4.1 内容合并策略

以下内容可以安全合并，减少文件数量和冗余：

| 合并前 | 合并后 | 理由 |
|--------|--------|------|
| `guide-skeleton` + `guide-placement` | `guide-structure.md` | 两者高度互补：skeleton 管章节，placement 管 section 内部 |
| `workflow-edit-prompt` + `workflow-review-edit` | `workflow-edit.md` | review-edit 是 edit 流程的阶段 3，天然属于同一流程 |
| `workflow-audit-prompt` + `quality-levels.md` + `common-issues.md` | `workflow-audit.md` | reference 文件只被 audit 使用，合并减少引用跳转 |
| `guide-diagnose` + `error-facet-map.md` | `workflow-diagnose.md` | 映射表只被 diagnose 使用 |
| `facet-types.md` | 内联到主 SKILL.md | 所有流程的基础概念，应直接可用 |
| `edit-prompt` + `audit-prompt` | 保留为轻量 Command 入口 | 改写为空壳路由转发器，指向主 SKILL.md 对应分支（见 4.3） |

### 4.2 精炼策略

整合不是简单拼接，需要精炼：

1. **去重**：skeleton 和 placement 中关于"定义与执行分离"的描述有重叠，合并时保留最清晰的版本
2. **提炼检查清单**：7 个 Skill 各自有检查清单，合并为统一的分类检查清单
3. **简化引用链**：当前 Skill 间大量 `@guide-xxx` 引用，合并后改为 section 内部引用
4. **保留深度**：Guide 的深度内容（如 placement 的 5 种内聚性判断、scenario 的多级场景设计）不应被过度简化

### 4.3 功能入口设计

统一 Skill 需要一个清晰的路由机制：

```
用户意图识别：
├── "编写/创建 Prompt"  → 结构设计 + 布局指南
├── "修改/修复 Prompt"  → 编辑流程（含诊断）
├── "检查 Prompt 质量"  → 质量检测流程
├── "诊断行为问题"      → 行为诊断流程
└── "优化 Prompt"       → 质量检测 → 编辑流程
```

**保留轻量 Command 入口**：整合后不直接删除原有的 `edit-prompt` 和 `audit-prompt`，而是将其改写为"空壳路由转发器"，维持用户已有的触发习惯。Command 内部逻辑精简为一句指令，例如：

```
# 编辑 Prompt（Command 入口）
用户希望执行编辑任务，请立即加载 prompt-helper/SKILL.md，进入编辑路由分支。
```

这样既收拢了核心逻辑到统一 Skill，又避免了用户触发习惯的断裂。

### 4.4 子模块设计约束

整合后的子模块需要遵循以下设计约束，确保 Agent 在长上下文中的执行质量：

**约束 1：核心原则重申（Context Anchor）**

当 Agent 深入加载一个 4,000+ tokens 的子模块时，主文件中的核心概念容易在冗长上下文中被稀释。因此每个 `docs/` 子模块的顶部必须用最简练的语言重申 1-2 条不可妥协的核心原则：

```
> **核心原则提醒**：
> - 六大 Facet 类型：定义型 / 知识型 / 场景型 / 策略型 / 边界型 / 规范型
> - 定义与执行分离：角色/职责在定义区声明，具体步骤在执行区编写
```

**约束 2：强制状态锚点（State Checkpoints）**

`workflow-edit.md` 和 `workflow-audit.md` 等长流程子模块中，Agent 在多次工具调用后容易"遗忘"后续步骤（如改完 Prompt 后跳过自检）。需要在提示词中要求 Agent 在每个阶段结束时显式输出进度标志：

```
[Phase 2 Completed] 修改已完成。开始 Phase 3: 修改后自检（不可跳过）
```

这种显式输出能有效刷新 Agent 的上下文注意力，防止流程中断。

## 5. 风险评估

### 5.1 子模块寻径失败风险（高→已缓解）

Agent 通过 Read 工具加载子模块时，默认在工作区根目录寻找相对路径，导致文件读取失败。

**缓解措施**（已纳入方案 B 设计）：
- 主 SKILL.md 内置寻径指令，要求 Agent 先确定 SKILL.md 所在目录再拼接子模块路径
- 提供明确的路径示例，降低 Agent 的路径推理负担

### 5.2 信息丢失风险（中）

精炼过程中可能丢失细节。缓解措施：
- 保留所有检查清单的完整条目
- 深度内容放在 docs/ 子模块中，不做过度简化
- 整合完成后用原 `workflow-audit-prompt` 的检查维度自检

### 5.3 长流程状态丢失风险（中→已缓解）

合并后的 `workflow-edit.md` 流程跨度变长，Agent 在多次工具调用后容易遗忘后续步骤（如跳过修改后自检）。

**缓解措施**（已纳入 4.4 设计约束）：
- 子模块中嵌入强制状态锚点（State Checkpoints），要求 Agent 在每个阶段结束时显式输出进度标志
- 关键步骤标注"不可跳过"，强化 Agent 的执行意识

### 5.4 主文件过大风险（低）

方案 B 控制在 ~3,000 tokens，方案 C 控制在 ~6,000 tokens，均在合理范围。

### 5.5 按需加载失效风险（低）

如果 Agent 无法正确判断需要加载哪个子模块，可能遗漏关键知识。缓解措施：
- 主文件中提供明确的路由表和触发条件
- 每个子模块的 description 清晰描述适用场景

### 5.6 核心概念上下文稀释风险（低→已缓解）

Agent 深入子模块工作时，主文件中的核心概念可能在长上下文中被稀释。

**缓解措施**（已纳入 4.4 设计约束）：
- 每个子模块顶部重申 1-2 条核心原则（Context Anchor）
- 降低对主文件短期记忆的过度依赖

### 5.7 维护同步风险（低）

整合后只需维护一个 Skill 目录，比当前 16 个分散文件更易维护。

## 7. 能力保留度对比分析

现有体系能良好运转的核心原因：

1. **Guide 的深度内容**：每个 Guide 都有详细的原则、判断方法、红灯信号、检查清单——这些细节是"能力"的真正来源
2. **按需加载机制**：Workflow 在运行时通过 Read 工具加载对应 Guide，确保 Agent 拿到完整的方法论知识
3. **检查清单驱动**：每个维度都有具体的检查项，Agent 逐项执行而非凭感觉

### 三种方案的能力保留度

| 方案 | 能力保留度 | 分析 |
|------|-----------|------|
| **A 单文件合并** | ~95% | 内容 100% 保留，但 25,000+ tokens 的超大文件会导致 Agent 注意力分散，深度检查项容易被忽略，实际执行质量反而可能下降 |
| **B 主文件 + 子模块** | **~98%** | 子模块保留完整原始内容（仅合并同类文件，不删减深度内容），按需加载确保 Agent 聚焦当前任务。唯一损失：跨 Skill 引用链变为 section 内部引用，极少量语境丢失 |
| **C 分层内联** | ~90% | 主文件的精炼过程不可避免地压缩细节（如 placement 的内聚性判断方法、scenario 的退化检测清单），这些细节正是诊断和检测能力的关键 |

### 关键结论

**方案 B 的能力保留度最高**，前提是调整精炼策略：

- **子模块以"合并"为主，不做"精炼"**：`guide-skeleton` + `guide-placement` 合并为一个文件时，保留两者的完整内容，只去除重复段落和简化引用链
- **检查清单 100% 保留**：所有检查项原样保留，仅重新分类组织
- **深度内容不压缩**：placement 的 5 种内聚性判断、scenario 的多级场景设计、diagnose 的 Trial 流程等，全部保留原文
- **主文件只做路由 + 核心概念**：不试图在主文件中"概括"Guide 内容，避免概括导致的信息丢失

这样整合后的 Skill 本质上是"同一套知识体系的重新打包"，而非"精简版"。

## 8. 最终建议

**推荐方案 B（主文件 + 子模块），采用"合并不精简"策略**，理由：

1. **能力保留最高（~98%）**——子模块保留完整原始内容，不压缩深度知识
2. **Token 效率合理**——主文件 ~3,000 tokens 做路由，按需加载 1-3 个子模块（每个 3,000-5,000 tokens）
3. **维护友好**——每个子模块职责清晰，修改不影响其他模块
4. **扩展性好**——未来新增维度只需添加子模块 + 更新路由表
5. **与现有 Skill 体系一致**——skillix-hub 的其他 Skill（如 memory、playwright）也采用类似结构
6. **执行健壮性高**——内置寻径规则、状态锚点、核心原则重申三重保障

| `guide-scenario` | `docs/guide-scenario.md` | 原样保留 |
| `guide-output` | `docs/guide-output.md` | 原样保留 |

### 子模块编写规范

所有 `docs/` 子模块必须遵循以下规范：

1. **顶部核心原则重申**：每个子模块开头用 1-2 句话重申与该模块最相关的核心原则（Context Anchor），防止长上下文稀释
2. **状态锚点**（仅 workflow 类）：长流程子模块在每个阶段边界嵌入显式进度标志，关键步骤标注"不可跳过"
3. **自包含性**：子模块应包含执行所需的全部信息，不依赖 Agent 对主文件内容的记忆

独立不变的 Guide（scenario/output/example/info-passing）占现有体系的 4/7，直接原样保留，最大程度保证能力不丢失。
