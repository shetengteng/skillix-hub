# Prompt Helper 现有 Skill 说明分析报告

## 概述

`design/prompt-helper/` 目录下包含一套完整的 **Prompt 编写辅助体系**，基于自研的 **PRISM 框架**（Facet 驱动的 Prompt 设计方法论）。体系由三层组件构成：

- **Command 层**（2 个）：用户入口，触发对应工作流
- **Workflow 层**（3 个）：编排执行流程，协调多个 Guide 的加载和使用
- **Guide 层**（7 个）：方法论知识库，提供具体的设计原则和检查标准

附带 **4 个 Reference 文件**，作为 Guide/Workflow 的补充参考。

---

## 核心概念：Facet

整个体系围绕 **Facet（任务侧面）** 展开。Facet 是 Prompt 设计的基本单位——模型需要理解的任务的一个独立方面。框架定义了六种功能类型：

| 类型 | 作用 | 缺失后果 |
|------|------|---------|
| **定义型** | 模型是谁、做什么 | 角色偏移、职责越界 |
| **知识型** | 领域背景、数据格式 | 领域错误、参数错误 |
| **场景型** | 区分不同情况 | 走错分支、场景混淆 |
| **策略型** | 具体怎么执行 | 步骤遗漏、顺序错误 |
| **边界型** | 特殊规则、禁止行为 | 边界处理错误 |
| **规范型** | 输出格式 | 格式错误、不可解析 |

## Command 层（入口）

### 1. edit-prompt-command

**功能**：编辑 Prompt 文件的入口命令。

**使用方式**：当用户需要修改、修复或优化一个 Prompt 文件时触发。它会读取 `workflow-edit-prompt/SKILL.md` 并按其流程执行。

**触发场景**：
- "帮我修改这个 Prompt"
- "这个 Agent 的行为不对，修一下 Prompt"
- "优化一下这个 Prompt 的结构"

---

### 2. audit-prompt-command

**功能**：检查 Prompt 质量的入口命令。

**使用方式**：当用户需要审查一个 Prompt 的质量时触发。它会：
1. 读取 `workflow-audit-prompt/SKILL.md`，按 8 个维度检查目标 Prompt
2. 向用户报告检查结果
3. 如有需要修复的项，自动读取 `workflow-edit-prompt/SKILL.md` 执行修复

**触发场景**：
- "检查一下这个 Prompt 的质量"
- "审查一下这个 Agent Prompt"
- "这个 Prompt 写得怎么样？"

## Workflow 层（流程编排）

### 3. workflow-edit-prompt

**功能**：Prompt 编辑的完整工作流，从诊断到修改到自检。

**流程**：

```
阶段 0：确认是否需要先诊断
  ├── 是行为问题 → 加载 guide-diagnose 诊断
  └── 不是行为问题 → 直接进入阶段 1

阶段 1：读取 Skill + 设计方案
  ├── 必读：guide-skeleton + guide-placement
  ├── 按需：guide-scenario / guide-output / guide-example / guide-info-passing
  └── 设计修改方案，向用户确认

阶段 2：执行修改

阶段 3：修改后自检（自动执行，不可跳过）
  └── 加载 workflow-review-edit 逐项检查
```

**核心约束**：
- 每个阶段必须通过 Read 工具加载对应的 Skill 文件，不允许凭记忆操作
- 向用户汇报方案时，必须引用 Skill 中的具体原则

---

### 4. workflow-audit-prompt

**功能**：Prompt 质量检测，基于 8 个维度输出结构化检测报告。

**8 个检查维度**：

1. **Facet 识别** — 能否列出所有 Facet？类型是否明确？
2. **Facet 覆盖度** — 是否有遗漏的必需 Facet？（按 Prompt 类型分别检查）
3. **结构与布局** — 共享 Facet 是否集中？打包是否基于功能内聚？
4. **场景设计质量**（有条件）— 意图识别、场景边界、耦合组织
5. **输出设计** — JSON/NL/混合输出是否规范？
6. **示例规范** — 是否使用 high-level 占位符？
7. **风格一致性** — 术语一致？避免表格？
8. **信息传递**（有条件）— Producer/Coordinator/Consumer 设计是否正确？

**输出**：结构化报告，每项给出 ✅ 通过 / ❌ 需修改 / ⚠️ 可优化。

**适用范围**：所有类型的 Prompt——Agent Prompt、Cursor Skill、Cursor Rule、Cursor Command。

---

### 5. workflow-review-edit

**功能**：评估对 Prompt 的修改是否合理。在修改之后、质量检查之前使用。

**5 个评审维度**：

1. **问题对应性** — 改的是不是该改的地方？修改的 Facet 类型是否与问题匹配？
2. **修改方式** — 是否优先删除/重组而非新增规则？修改是否最小化？
3. **副作用检测** — 有没有引入新的重复定义？有没有破坏引用关系？
4. **Facet 完整性** — 有没有意外合并/拆分 Facet？内部顺序是否合理？
5. **常见坏修改模式** — 加规则代替重构？范围膨胀？信息重复？表述矛盾？

**输出**：每个维度给出"通过 / 有问题 / 有更好的方式"的判断 + 具体建议。

## Guide 层（方法论知识库）

### 6. guide-skeleton

**功能**：指导 Prompt 的整体结构设计。

**核心内容**：
- **Standard Agent 标准结构**：角色 → 核心任务 → 语言支持 → 输入格式 → 工作流程 → 输出格式 → 核心原则
- **Super Agent 标准结构**：核心职责 → Sub Agent 接口 → 核心工作流程 → 核心原则
- **信息归属原则**：每条信息应该放在哪个章节（职责→角色、执行→工作流程、格式→输出格式、约束→核心原则）
- **三个通用原则**：定义与执行分离、一次定义多处引用、耦合程度决定位置
- **结构健康判断**：健康/不健康信号 + 重构方法

**使用场景**：设计新 Prompt 的整体结构、诊断结构性问题、重构混乱的 Prompt。

---

### 7. guide-placement

**功能**：指导 Prompt 的信息布局设计，是整个体系中最核心的方法论。

**核心内容**：
- **Part 1-2**：Facet 核心概念与六种功能型分类
- **Part 3**：Facet 到 Section 的布局决策——基于耦合度（功能/顺序/通信/逻辑内聚）和作用域（专属/共享）
- **Part 4**：Section 内部组织——What → When → How 顺序原则
- **Part 5**：Section 间关系与模块化——前向引用、查找引用、共置
- **Part 6**：自然语言输出描述维度——语气、内容边界、词汇约束、输出时机
- **Part 7**：完整检查清单（Facet 识别 4 项 + 布局决策 8 项 + Section 组织 3 项 + 模块化 3 项）

**使用场景**：设计新 Prompt 布局、诊断信息放置问题、重构混乱的 Section 结构。

**附带 Reference**：`references/facet-types.md` — Facet 类型的权威定义文件，被 placement 和 diagnose 共同引用。

---

### 8. guide-scenario

**功能**：场景型 Facet 的深度写作指南。

**核心内容**：
- **意图识别写法**：从用户目标出发（"用户希望..."），不做关键词匹配。每个场景包含三要素：用户目标 + 适用条件 + 可执行操作
- **场景边界设计**：互斥 vs 可组合、重叠处理（细化条件/设置优先级/合并）、兜底场景
- **场景与策略的耦合**：同一场景的定义和执行流程应打包在同一 Section
- **多级场景**：不超过两层嵌套，同层子场景互斥
- **Standard vs Super Agent 差异**：Standard 的场景是处理逻辑分支，Super 的场景是意图识别和路由

**使用场景**：设计多场景 Prompt、编写意图识别逻辑、诊断场景混淆问题。

---

### 9. guide-output

**功能**：规范型 Facet 的深度写作指南，指导输出格式设计。

**核心内容**：
- **结构化输出（JSON）**：必需/可选字段区分、字段命名原则、空值处理策略、位置决策（嵌入 vs 独立）、时序决策（流式 vs 批量）
- **自然语言输出**：四个描述维度——语气、内容边界、词汇约束、输出时机
- **混合输出**：NL + JSON 交替时的关系说明（NL 给人看，JSON 给系统用）
- **前端交互格式**：`async_cui_callback`（数据回传）和 `async_cui_suggest`（建议按钮）
- **Sub Agent 输出约束**：被调用方只面向调用方输出结构化数据，不输出面向用户的总结/统计/建议

**使用场景**：设计输出格式、诊断输出问题。

---

### 10. guide-example

**功能**：指导编写 Prompt 中的示例值。

**核心内容**：
- **结构化示例（JSON）**：使用 high-level 占位符（`"字段ID"` 而非 `"field_123"`），避免模型照搬具体值。数组只需 1-2 个元素展示结构
- **自然语言示例（情境模板）**：列出每种情况的参考样例（有问题/无问题/边界情况），展示语气和内容范围
- **核心原则**：示例展示模式，不展示具体数据

**使用场景**：编写 JSON 示例或自然语言输出模板。

---

### 11. guide-info-passing

**功能**：指导设计 Prompt 之间传递信息的标准化机制。

**核心内容**：
- **三种角色**：Producer（生产者）、Coordinator（协调者/Super Agent）、Consumer（消费者）
- **三种传递方式**：标记化 JSON（`--- [信息类型] (JSON) ---`）、特殊代码块（`Remote_Agent_Request Inner`）、自然语言提取
- **各角色设计规范**：Producer 的输出格式设计、Coordinator 的提取/组织/转发逻辑、Consumer 的接收/解析/使用
- **标记规范和命名约定**：描述性名称、统一大小写、避免歧义
- **调试指南**：信息丢失、格式不匹配、解析失败的排查方法

**使用场景**：设计 Agent 间信息传递、优化通信流程、调试信息丢失问题。

---

### 12. guide-diagnose

**功能**：行为诊断方法论，从错误行为定位 Prompt 缺陷根因。

**核心流程**：

```
Step 1: 收集 + 快筛
  └── 5 项低挂果实检查（缺步骤编号、执行区解释"是什么"、规则矛盾、缺必需 Facet、无兜底）

Step 2: 错误分类 + Facet 定位
  ├── 诊断切分（Trial Segmentation）— 多症状拆分为独立 Trial
  ├── 阶段 A：识别错误类型（TRAIL 20 种 + PRISM 7 种）
  └── 阶段 B：从错误类型定位 Facet

Step 3: 假设 + 最小干预
  └── 最小化修改验证假设（只加 1-2 句话）

Step 4: 验证 + Trial 流转
  ├── 路径 A：假设成立 → 记录修复方案
  ├── 路径 B：假设不成立，还有候选 → 重试
  └── 路径 C：已耗尽候选 → 标记"建议人工分析"
```

**附带 Reference**：`references/error-facet-map.md` — 27 种错误类型到 Facet 的完整映射表（TRAIL 20 种 + PRISM 7 种）。

**使用场景**：模型行为不符合预期时使用。

## Reference 文件

### 13. facet-types.md（guide-placement 附属）

Facet 类型的权威定义，包含六种功能类型的完整说明、归属维度（专属/共享）、识别方法。被 `guide-placement` 和 `guide-diagnose` 共同引用。

### 14. error-facet-map.md（guide-diagnose 附属）

错误类型到 Facet 的完整映射表。Part 1 覆盖 TRAIL 论文的 20 种错误（推理 8 种 + 系统 8 种 + 规划 4 种），Part 2 覆盖 PRISM 特有的 7 种缺陷，Part 3 提供 Facet 反向索引。

### 15. quality-levels.md（workflow-audit 附属）

质量等级定义（优秀/良好/合格/不合格）+ 检查项分类（必须通过/建议修改/可选优化）+ 评分示例。

### 16. common-issues.md（workflow-audit 附属）

9 种常见问题的诊断和修复方法，覆盖结构问题（缺章节、顺序混乱）、内容问题（执行区重复定义、场景位置不当、缺必需字段）、风格问题（使用表格、具体值示例、语言机械、术语不一致）。

## 使用链路总结

```
场景 1：从零编写 Prompt
  → guide-skeleton（结构设计）
  → guide-placement（布局决策）
  → guide-scenario / guide-output / guide-example（按需）
  → workflow-audit-prompt（质量自检）

场景 2：修改现有 Prompt
  → edit-prompt-command（入口）
  → workflow-edit-prompt（流程编排）
    → guide-diagnose（如有行为问题）
    → guide-skeleton + guide-placement（必读）
    → 其他 Guide（按需）
    → workflow-review-edit（修改后自检）

场景 3：检查 Prompt 质量
  → audit-prompt-command（入口）
  → workflow-audit-prompt（8 维度检查）
  → workflow-edit-prompt（如需修复）

场景 4：诊断行为问题
  → guide-diagnose（诊断流程）
  → error-facet-map.md（错误映射）
  → workflow-edit-prompt（逐 Trial 修复）
```
