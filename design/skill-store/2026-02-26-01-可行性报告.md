# Skill Store 可行性报告

## 一、需求概述

构建一个 Cursor Skill，用于**发现、管理和分发**现有 Skill。核心能力：

1. **配置 Git 仓库 URL**（GitHub 等），仓库中包含 Skill 源码
2. **Clone/同步**仓库到本地 data 目录
3. **会话启动 Hook**：自动拉取远程更新，维护本地索引
4. **自然语言搜索**：用户描述需求，推荐匹配的 Skill
5. **安装 Skill**：支持项目级（`.cursor/skills/`）和全局（`~/.cursor/skills/`）
6. **版本检测与更新提示**：已安装 Skill 有远程更新时，提示用户是否更新

## 二、技术可行性分析

| 功能模块 | 可行性 | 实现方式 | 风险 |
|----------|--------|----------|------|
| Git 仓库配置（自然语言） | **高** | JSON 配置文件 + SKILL.md 指导 Agent 解析自然语言写入 | 低 |
| Clone/同步到本地 | **高** | `git clone --depth 1` / `git pull`，Python 脚本封装 | 低 |
| 会话启动 Hook | **高** | hook.py 脚本 + alwaysApply 规则注册，**异步后台执行不阻塞会话** | 需注册规则 |
| 本地索引构建 | **高** | 扫描仓库中的 SKILL.md，提取 name/description，生成 JSON 索引 | 低 |
| 自然语言搜索推荐 | **中高** | 索引提供候选列表，Agent LLM 做语义理解和推荐 | 依赖 Agent 上下文窗口 |
| 安装到项目/全局 | **高** | 复制 Skill 目录 + 记录安装元信息（来源、commit hash） | 低 |
| 版本检测（Git hash 比对） | **高** | 对比 installed.json 中记录的 commit hash 与远程最新 hash | 低 |
| 更新提示 | **高** | Hook 中检测差异后输出提示文本 | 低 |

## 三、架构设计

### 3.1 目录结构

```
skill-store/
├── SKILL.md                    # 主指令文件
├── scripts/
│   ├── registry.py             # 仓库配置管理（增删改查）
│   ├── sync.py                 # Clone / Pull 同步
│   ├── index.py                # 索引构建与搜索
│   ├── install.py              # Skill 安装 / 卸载 / 更新
│   ├── hook.py                 # 会话启动 Hook（异步调度入口）
│   └── async_worker.py         # 后台异步任务执行器
├── data/                       # 运行时数据（gitignore）
│   ├── config.json             # 仓库配置
│   ├── repos/                  # Clone 的仓库
│   ├── index.json              # Skill 索引
│   ├── installed.json          # 已安装 Skill 的版本记录
│   └── status.json             # 异步任务状态（更新提示缓存）
└── rules/
    └── skill-store-hook.mdc    # alwaysApply 规则（注册 Hook）
```

### 3.2 异步执行机制

**核心原则：所有 Git 操作和索引重建均不阻塞会话流程。**

```
hook.py（会话启动时调用）
  ├── 检查时间阈值 → 未到期则直接返回
  ├── 读取 status.json → 如有待展示的更新提示，输出给 Agent
  └── 启动 async_worker.py 子进程（subprocess.Popen, 立即返回）
        ├── git pull 所有仓库
        ├── 重建 index.json
        ├── 比对已安装 Skill 版本
        └── 将结果写入 status.json（下次 hook 读取）
```

**关键实现**：
- `hook.py` 使用 `subprocess.Popen` 启动 `async_worker.py`，不等待完成
- `async_worker.py` 使用 PID 文件防止重复执行
- 异步结果通过 `status.json` 传递，Agent 下次读取时展示

### 3.3 数据格式

**config.json**（仓库配置）：

```json
{
  "registries": [
    {
      "url": "https://github.com/user/skills-repo",
      "alias": "my-skills",
      "added_at": "2026-02-26T10:00:00Z",
      "last_synced": "2026-02-26T10:05:00Z"
    }
  ]
}
```

**index.json**（Skill 索引）：

```json
{
  "updated_at": "2026-02-26T10:05:00Z",
  "skills": [
    {
      "name": "pdf-processor",
      "description": "Extract text and tables from PDF files...",
      "registry": "my-skills",
      "path": "skills/pdf-processor",
      "commit_hash": "abc123",
      "tags": ["pdf", "document", "extraction"]
    }
  ]
}
```

**installed.json**（安装记录）：

```json
{
  "installations": [
    {
      "name": "pdf-processor",
      "registry": "my-skills",
      "source_commit": "abc123",
      "installed_at": "2026-02-26T11:00:00Z",
      "target": "~/.cursor/skills/pdf-processor",
      "scope": "global"
    }
  ]
}
```

### 3.4 核心流程

```
用户说 "添加 https://github.com/xxx/skills-repo"
  → Agent 调用 registry.py add
  → sync.py clone（浅克隆）
  → index.py rebuild

用户打开新会话
  → hook.py 自动执行（异步后台，不阻塞会话）
  → 后台: git pull 所有仓库 + 重建索引
  → 后台: 检查已安装 Skill 版本
  → 完成后写入状态文件，下次 Agent 读取时展示更新提示

用户说 "我需要一个处理 PDF 的 Skill"
  → Agent 读取 index.json
  → 基于 description 语义匹配推荐
  → 用户选择 → install.py 安装

用户说 "更新所有 Skill"
  → install.py 批量更新
```

## 四、关键设计决策

| 决策点 | 推荐方案 | 理由 |
|--------|----------|------|
| 脚本语言 | **Python** | 项目已有 pyrightconfig.json，memory 等 Skill 均用 Python |
| 索引格式 | **JSON** | 轻量，Agent 可直接读取理解 |
| 仓库约定 | 目录中包含 `SKILL.md` 即为一个 Skill | 兼容现有 Skill 结构 |
| 安装方式 | **复制 + 记录来源** | 符号链接跨平台有问题，复制更稳定 |
| Clone 策略 | `git clone --depth 1` 浅克隆 | 减少磁盘占用和 clone 时间 |
| Hook 触发 | alwaysApply 规则 + 静默执行 | 与现有 memory-rules、userinput-mandatory 模式一致 |
| 异步策略 | **所有 Git 操作（clone/pull）和索引重建均异步后台执行** | 不阻塞会话流程，用户无感知 |
| 更新频率 | Hook 中加入时间阈值（如 1 小时内不重复 pull） | 避免频繁网络请求 |

## 五、与现有 Skill 的关系

| 现有 Skill | 关系 |
|------------|------|
| `skill-builder` | **互补**：skill-builder 负责**开发**新 Skill，skill-store 负责**发现和管理**已有 Skill |
| `doc-skill-generator` | **互补**：doc-skill-generator 从文档**生成** Skill，skill-store 从 Git 仓库**分发** Skill |
| `create-skill`（Cursor 官方） | **不冲突**：create-skill 是创建指南，skill-store 是分发管理 |

## 六、Skill 命名候选

| 候选名称 | 风格 | 说明 |
|----------|------|------|
| **skill-store** | 用户友好 | 类似 App Store，强调发现/安装，**推荐** |
| skill-hub | 简洁直观 | "Skill 中心"，但与项目名 skillix-hub 重叠 |
| skill-registry | 技术精确 | 类似 npm registry，偏技术 |
| skill-sync | 功能导向 | 强调同步更新，但不够全面 |
| skill-forge | 有创意 | "锻造"Skill，偏开发而非管理 |
| skillpkg | 极简 | 类似 pkg 包管理器 |

**最终选择：`skill-store`**

## 七、风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 私有仓库需要认证 | 中 | 依赖用户本地 Git 凭证（SSH key / credential helper），不在 Skill 内存储密码 |
| 大仓库 clone 耗时 | 低 | `--depth 1` 浅克隆 + 异步执行 |
| Hook 影响会话启动速度 | 低（已缓解） | **异步后台进程执行**，hook.py 仅启动子进程后立即返回，不阻塞会话 |
| 索引搜索精度 | 低 | 索引提供候选，最终由 Agent LLM 做语义理解 |
| Skill 目录结构不规范 | 低 | 仅识别包含 SKILL.md 的目录，跳过不规范的 |

## 八、预估工作量

| 模块 | 文件 | 预估 |
|------|------|------|
| SKILL.md | 主指令 | 200-300 行 |
| registry.py | 仓库配置管理 | ~100 行 |
| sync.py | Clone/Pull | ~120 行 |
| index.py | 索引构建与搜索 | ~150 行 |
| install.py | 安装/卸载/更新 | ~180 行 |
| hook.py | 会话 Hook（异步调度） | ~120 行 |
| async_worker.py | 后台异步任务执行器 | ~80 行 |
| skill-store-hook.mdc | 规则文件 | ~30 行 |

**总计**：约 5-6 个 Python 脚本 + 1 个 SKILL.md + 1 个规则文件

## 九、结论

**完全可行**。核心功能（Git 配置、Clone/同步、索引、安装、版本检测）都是成熟技术方案，与现有 Skill 生态（hook 机制、alwaysApply 规则、data 目录模式）完全兼容。建议按 registry → sync → index → install → hook 的顺序逐步实现。
