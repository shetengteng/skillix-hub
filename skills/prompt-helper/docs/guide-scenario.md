> **核心原则提醒**：
> - 六大 Facet 类型：定义型 / 知识型 / 场景型 / 策略型 / 边界型 / 规范型
> - 场景型定义"什么时候" → 策略型定义"怎么做"——两者通常紧耦合

# 场景型 Facet 写作指南

## Facet 定位

**场景型（Scenario）facet** 让模型区分不同情况采取不同行动。

缺失时的典型症状：
- 走错分支，场景混淆
- 只能识别固定关键词，换种说法就失败
- 某些请求无法归类
- 路由到错误的 Sub Agent

**与其他 Facet 的关系**：
- 场景型定义"什么时候"→ 策略型定义"怎么做"——两者通常紧耦合
- 场景型和边界型都是"条件"，但场景型描述正常分支（互斥路径），边界型描述特殊约束（任何路径都可能触发）

整体布局原则参考 `docs/guide-structure.md`。

---

## 1. 意图识别写法

### 从用户目标出发，而非关键词匹配

场景描述应该是**"用户希望..."**，而不是**"用户说了..."**。

✅ **好的写法**（语义理解）：
```
**场景 A：资产信息填写引导**
用户希望 AI 帮助检查或补充资产的分类信息，包括描述、示例值、
标准化建议等。
```

❌ **差的写法**（关键词匹配）：
```
**场景 A：资产信息填写引导**
当用户说"帮我填"、"检查一下"、"补充信息"时触发。
```

**为什么**：
- 用户可能用"看看还缺什么"、"这个字段怎么填"等各种说法表达同一意图
- 关键词匹配无法覆盖所有可能的表达方式
- 语义理解让模型根据意图判断，而非逐词匹配

### 场景描述的三要素

每个场景应包含：

1. **用户目标**：用户想达成什么（一句话）
2. **适用条件**：什么时候属于这个场景（具体条件）
3. **可协调的 Agent / 可执行的操作**：这个场景下做什么

```
**场景 A：资产信息填写引导**

用户希望 AI 帮助检查或补充资产信息。      ← 用户目标

**适用条件**：                              ← 适用条件
- Request 状态为 DRAFT 或 REJECTED
- 用户请求引导、信息检查、补充填写

**可协调的 Agents**：                       ← 可执行操作
- Quality_Checker（检查质量）
- Guide_Agent（引导填写）
```

## 2. 场景边界设计

### 互斥 vs 可组合

**互斥场景**：一次只能匹配一个场景。大多数意图识别属于这类。

```
场景 A：填写引导（用户想填信息）
场景 B：分类请求（用户想分类）
场景 C：状态查询（用户想查状态）
→ 三个场景互斥，用户意图只会落入一个
```

**可组合场景**：一次可能触发多个场景。通常在复杂业务中出现。

```
用户说"帮我填完信息然后提交分类"
→ 先触发场景 A（填写引导），完成后触发场景 B（分类请求）
→ 需要在工作流程中说明组合执行的逻辑
```

**设计原则**：
- 默认设计为互斥场景，除非业务明确需要组合
- 互斥场景之间应有清晰的判断边界
- 可组合场景需要说明执行顺序和数据流转

### 重叠处理

当两个场景的适用条件有重叠时：

**方法 1：细化条件**
```
场景 A：用户想检查已填的信息 → 适用条件：已有数据
场景 B：用户想从零开始填写 → 适用条件：数据为空
```

**方法 2：设置优先级**
```
如果同时满足 A 和 B 的条件，优先匹配 A。
```

**方法 3：合并为一个场景**
```
如果两个场景的执行流程几乎相同，合并为一个场景，
内部用条件分支处理差异。
```

### 兜底场景

**始终设计一个兜底场景**，处理无法归类的请求：

```
**场景 F：其他请求**
用户的请求不属于上述任何场景。

**处理方式**：
- 告知用户当前支持的功能
- 引导用户描述更具体的需求
```

## 3. 场景与策略的耦合

### 打包原则

场景定义（场景型 facet）和该场景的执行流程（策略型 facet）通常是**功能内聚**的——改一个场景的定义，通常也要改它的执行流程。

**因此**：同一场景的定义和执行流程应打包在同一个 section。

```
**场景 A：资产信息填写引导**

用户希望 AI 帮助检查或补充...           ← 场景型 facet（What + When）

**适用条件**：
- Request 状态为 DRAFT 或 REJECTED
- 用户请求引导...

**可协调的 Agents**：                    ← 知识型 facet（What）
- Quality_Checker / Guide_Agent

**执行流程**：                           ← 策略型 facet（How）
1. 先调用 Quality_Checker
2. 提取 JSON
3. 再调用 Guide_Agent
```

### 共享逻辑的提取

如果多个场景有相同的执行步骤（如"解析 suggestionActions"），这是**共享策略型 facet**，应提取到集中位置：

```
## 核心工作流程

### 阶段 2：协调执行
步骤 1：解析 suggestionActions          ← 共享策略型 facet（集中）
步骤 2：按场景执行                      ← 各场景的专属策略型 facet
```

### 内部顺序

场景 section 内部遵循 **What → When → How** 原则：

1. **What**：场景名称 + 用户目标（定义）
2. **When**：适用条件（场景型）
3. **How**：执行流程（策略型）

## 4. 多级场景

### 何时使用多级场景

当一个场景内部还有子分支时，可以使用嵌套场景：

```
**场景 B：分类请求**

用户希望对资产进行分类。

**子场景 B1：自动分类**
- 用户说"帮我分一下"
- 执行：直接调用 Classifier

**子场景 B2：人工确认分类**
- 用户说"我来看看分类结果"
- 执行：调用 Classifier 后展示结果等待确认
```

### 设计原则

- **不超过两层**：场景 → 子场景。更深的嵌套说明需要拆分 Agent
- **子场景共享父场景的条件**：子场景的适用条件是父场景条件的细化
- **同层子场景互斥**：在父场景内部，子场景之间互斥

## 5. Standard Agent vs Super Agent 的场景设计差异

### Standard Agent

- 场景通常是**处理逻辑的分支**（如：有数据 vs 无数据，有质量问题 vs 无质量问题）
- 场景定义可以嵌入工作流程中，作为条件判断
- 简单的 Standard Agent 可能不需要显式的场景型 facet

### Super Agent

- 场景是**意图识别和路由**的核心（场景 A-F）
- 场景定义通常在"识别场景"阶段独立呈现
- 每个场景关联不同的 Sub Agent 组合
- 需要兜底场景

## 检查清单

### 意图识别
- [ ] 场景描述是否从用户目标出发（"用户希望..."）？
- [ ] 是否避免了关键词匹配？
- [ ] 每个场景是否包含三要素（目标 + 条件 + 操作）？

### 场景边界
- [ ] 场景之间是否互斥（或明确说明可组合）？
- [ ] 重叠区域是否有明确的处理策略？
- [ ] 是否有兜底场景？

### 耦合组织
- [ ] 场景定义和执行流程是否在同一 section？
- [ ] 共享逻辑是否提取到集中位置？

### 多级场景
- [ ] 嵌套不超过两层？
- [ ] 同层子场景互斥？

### 修改退化检测

修改场景型 facet 后，检查是否引入以下退化：

- [ ] **意图识别退化**：是否把语义理解改成了关键词匹配？（"用户希望..." → "当用户说'xxx'时触发"）
- [ ] **互斥性破坏**：修改场景条件后，两个场景的适用条件是否出现重叠但缺少处理逻辑？（应细化条件 / 设置优先级 / 合并场景）
- [ ] **兜底场景丢失**：是否删除或弱化了兜底场景？（Super Agent 几乎总需要）
- [ ] **三要素残缺**：修改后场景是否缺少用户目标、适用条件、可执行操作中的某个要素？
