# Prompt 结构与信息布局

## Part 1: Facet 核心概念与功能型分类

本部分为 Facet 概念的权威定义。

---

### Facet：核心概念

**Facet（任务侧面）是 prompt 设计的基本单位。**

Facet = 模型需要理解的任务的一个独立方面。缺少某个 facet，模型就会在对应类型的输入上犯错。

**设计 prompt 的流程**：

1. 识别任务需要哪些 facet
2. 决定 facet 如何打包成 section
3. 决定 section 在 prompt 中的布局

**Facet 独立性原则**（UniPrompt, ACL Findings 2025 实证）：

- 每个 facet 对模型效果有**相对独立的贡献**——移除任一 facet，对应类型的准确率下降
- 将所有 facet 合并为一段摘要，准确率显著下降
- **推论**：facet 应保持为可识别的独立单元。合并不等于简化，合并等于丢信息

---

### Facet 功能型分类

**分类依据**：去掉这段信息，模型会犯什么类型的错？

#### 六种功能类型

**定义型（Definition）**

模型需要理解自己是谁、做什么。缺失导致角色偏移、职责越界。

- 典型内容：角色定位、核心职责、职责边界、关键理解
- 实例：每个 Agent 的开头章节（角色定位、核心能力）

**知识型（Knowledge）**

模型需要领域背景才能正确处理。缺失导致领域错误、参数错误。

- 典型内容：术语对照、索引字段参考、数据格式、Agent 接口规范、Widget Schema
- 实例：Data_Discovery_Scout 的字段参考；Super Agent 的 Agent 接口规范

**场景型（Scenario）**

模型需要区分不同情况采取不同行动。缺失导致错误路由、场景混淆。

- 典型内容：意图识别、场景分支、处理方式判断（互斥逻辑）
- 实例：Super Agent V6 的场景 A-F；Suggestion Agent 的方式 1 vs 方式 2

**策略型（Strategy）**

模型需要知道具体怎么执行。缺失导致步骤遗漏、顺序错误。

- 典型内容：工作流程、执行步骤、处理逻辑、决策流程
- 实例：Classifier 的 5 步工作流程；Validator 的步骤 3.1-3.5

**边界型（Edge Case）**

模型需要避免在特定情况犯错。缺失导致边界处理错误、违规行为。

- 典型内容：禁止行为、特殊规则、空值处理、跳过规则、核心原则中的约束项
- 实例：Quality Checker 的空值处理；Classifier 的前缀匹配特殊规则

**规范型（Specification）**

模型需要知道输出长什么样。缺失导致格式错误、输出不可解析。

- 典型内容：输出格式、JSON 模板、进度格式、调用模板格式
- 实例：Validator 的 async_cui_callback JSON 格式；Guide Agent 的 Widget JSON Schema

#### 关键说明

- 不是每个 prompt 都必须包含全部六种类型。单任务 Agent 可能没有显式场景型 facet
- 知识密集型 Agent 的知识型 facet 可能占 60-70%，这是正常的

#### 归属维度

功能类型和归属是两个正交的维度：

- **专属 facet**：只和特定场景/Agent 相关。例：场景 A 的执行流程、Classifier 的跳过规则
- **共享 facet**：跨多个场景/Agent 通用。例：数据解析逻辑、JSON 精简步骤、语言传递规则

#### 识别 facet 的方法

问自己：如果去掉这段内容，模型会在哪种情况下犯错？

- 去掉后模型角色偏移 → 定义型
- 去掉后模型领域知识错误 → 知识型
- 去掉后模型走错分支 → 场景型
- 去掉后模型步骤遗漏 → 策略型
- 去掉后模型在边界 case 犯错 → 边界型
- 去掉后模型输出格式错误 → 规范型

## Part 2: 整体结构设计

### 核心理念

**Prompt 的结构决定了 AI 的理解质量。** 好的结构让每条信息都在正确的位置，AI 不需要在多处寻找同一概念的说明。

---

### Standard Agent 标准结构

```
# 角色
│  Agent 的身份和定位（1-2 句话）
│
# 核心任务
│  做什么 + 关键原则（业务背景、核心职责）
│
# 语言支持
│  中英文切换规则
│
# 输入格式
│  输入数据的格式和来源
│
# 工作流程
│  执行步骤（聚焦动作，引用定义而非重复）
│  ├── 步骤 1：...
│  ├── 步骤 2：...
│  └── 步骤 N：...
│
# 输出格式（如适用）
│  输出规范、JSON 示例、前端交互格式
│
# 核心原则
│  约束规则、禁止行为、边界说明
```

#### 各章节职责边界

**角色**：一句话说明 Agent 是谁、做什么。不包含具体规则或流程细节。

**核心任务**：说明 Agent 要解决什么问题、服务谁、在业务流程中的位置。"关键原则"列出 2-3 条最重要的行为准则。不包含执行步骤。

**工作流程**：聚焦"怎么做"。用自然语言描述执行步骤，引用定义而非重复解释。如果涉及多场景分支，场景定义应放在工作流程之前或第一个步骤中（具体选择见 Part 3）。

**输出格式**：定义输出的格式规范（JSON 结构、字段说明）。如果输出与处理逻辑紧耦合（多种条件输出），可以嵌入工作流程步骤中（具体选择见 Part 3）。

**核心原则**：放置全局性约束和禁止行为。核心原则可以重复强调其他章节中出现过的关键约束——这种有意重复有助于加强模型记忆，不算结构问题。

---

### Super Agent 标准结构

```
# 核心职责
│  协调器定位 + 语言支持 + 职责边界（✅ 做什么 / ❌ 不做什么）
│
# Sub Agent 接口
│  通用规则 + 每个 Sub Agent 的调用模板
│  ├── 通用规则
│  ├── Agent_A：用途 + 调用模板
│  └── Agent_B：用途 + 调用模板
│
# 核心工作流程
│  ├── 阶段 1：识别场景（场景定义 + 意图识别）
│  ├── 阶段 2：协调执行（信息加工 + Agent 调用）
│  └── 阶段 3：结果处理（分析 + 转发 + 建议）
│
# 核心原则
│  协调原则、忠实转发、语义理解等全局约束
```

#### 各章节职责边界

**核心职责**：明确"协调器"定位，列出做什么和不做什么的边界。

**Sub Agent 接口**：为每个 Sub Agent 提供调用规范。调用模板是内部传递格式，说明传递什么信息、格式是什么。如果 Agent 不需要 Super Agent 传递特定信息，可以省略调用模板。

**核心工作流程**：阶段式描述，通常分为识别、执行、结果处理三个阶段。场景定义通常放在"识别场景"阶段中。

**核心原则**：全局性约束，如"协调而非执行"、"忠实转发"等。如果 Super Agent 下辖 Sub Agent，常见的约束包括：
- Sub Agent 输出边界：明确 Sub Agent 允许输出什么（处理结果、错误信息）、禁止输出什么（操作建议、统计总结），防止 Sub Agent 越界
- 结果转发原则：Super Agent 将 Sub Agent 输出原样转发给用户，不重复、不总结、不修改

---

### 信息归属原则

**核心问题：这条信息应该放在哪个章节？**

**职责描述**（Agent 是什么、做什么）→ **角色** 或 **核心任务**
- 身份定位、业务背景、服务对象

**执行细节**（怎么做、步骤）→ **工作流程**
- 具体操作步骤、Tool 调用、数据处理逻辑

**格式规范**（输出长什么样）→ **输出格式**
- JSON 结构、字段说明、前端交互格式
- 如果与处理逻辑紧耦合 → 可以嵌入工作流程

**约束规则**（不允许做什么、必须遵守什么）→ **核心原则**
- 全局性禁止行为、数据完整性要求、边界说明
- 核心原则可以重复强调核心任务中的关键约束，用于加强模型记忆

**场景/概念定义**（是什么场景、什么概念）→ **工作流程之前** 或 **工作流程第一步**
- 取决于耦合度和作用域（见 Part 3）

**Sub Agent 调用规范** → **Sub Agent 接口**（仅 Super Agent）
- 调用模板、参数说明、传递格式

---

### 通用结构原则

#### 原则 1：定义与执行分离

**定义区说清楚 What（是什么）+ Why（为什么），执行区专注 How（怎么做）。**

```
Prompt 结构
    │
    ├── 定义区（前置部分）
    │     ├── 场景定义：是什么场景、适用条件、目标
    │     ├── 规则定义：什么规则、为什么有这个规则
    │     └── 概念定义：专有名词、业务概念
    │
    └── 执行区（工作流程）
          └── 只聚焦于动作
              ├── 用名称引用定义（不重复解释）
              ├── 说"怎么做"（不说"是什么"）
              └── 条件分支用名称区分（不重复定义）
```

#### 原则 2：一次定义，多处引用

**同一信息只在一个地方完整描述，后续用名称引用。**

✅ 好的做法：
```markdown
**如果是「发现」场景**：使用 multi_match 搜索
```

❌ 避免的做法：
```markdown
**如果是「发现」场景（用户想找数据资产，典型问题如"查找支付相关的表"）**：使用 multi_match 搜索
```

后者在执行区重复了定义区的内容。

#### 原则 3：耦合程度决定位置

**定义内容的位置取决于它与使用位置的耦合程度。**

```
耦合程度？
    │
    ├── 紧耦合（多步骤需要详细信息 / 内容由逻辑决定）
    │       └── 嵌入工作流程或就近放置
    │
    └── 松耦合（格式固定 / 只需名称引用）
            └── 独立章节
```

这三个原则在信息布局中有系统的应用方式（facet 分类、耦合度判据、作用域判据、section 模块化）→ 见 Part 3-6

---

### 结构健康判断

#### 健康信号

- 修改某个行为只需要改一处
- 每个章节的内容职责清晰，不存在"这段放哪里都行"的模糊地带
- 工作流程中不需要解释概念，只需要引用名称
- 核心原则可以重复强调关键约束以加强记忆，但措辞不应与其他章节矛盾

#### 不健康信号

- **同一规则多处描述**：修改一个行为需要在 2+ 个地方同步改动
- **核心任务和核心原则措辞矛盾**：对同一行为有不同说法（注意：内容重复不算问题，措辞矛盾才是问题）
- **工作流程中解释概念**：执行步骤中花大量篇幅解释"是什么"而非"怎么做"
- **信息位置模糊**：某段内容既像职责说明又像执行步骤，放哪里都不完全合适
- **章节之间矛盾**：不同位置对同一行为有不同说法

---

### 重构方法

当 Prompt 结构混乱时，按以下步骤梳理：

#### 步骤 1：识别重复

通读 Prompt，标记出所有描述同一概念/规则的位置。如果同一内容出现在 2+ 个地方，需要整合。

#### 步骤 2：确定归属

对每段内容问："这是职责描述、执行步骤、格式规范、还是约束规则？" 根据信息归属原则确定它应该在哪个章节。

#### 步骤 3：整合到一处

将分散的描述合并到正确的章节。选择最清晰的表述，删除其他重复。

#### 步骤 4：执行区清理

检查工作流程中是否有"解释性"内容（解释是什么、为什么），将这些移到定义区，工作流程只保留"怎么做"。

#### 步骤 5：验证引用

确保所有定义都能通过名称引用到，执行区不再重复定义内容。

## Part 3: Facet 到 Section 的布局决策

**适用范围**：Part 3-6 指导 prompt 中**信息的布局设计**，帮助决定：
- 一块信息属于什么功能类型（Facet 分类）
- 哪些信息应该打包在一起、哪些应该分开（布局决策）
- 打包在一起的信息如何组织内部顺序（见 Part 4）
- 如何设计 section 使其可独立增删改（见 Part 5）

整体结构设计（标准章节组成、信息归属）参考 Part 2。
示例值的写法（占位符、情境模板）参考 `guide-example.md`。
场景型 Facet 的深度写作指南参考 `guide-scenario.md`。
规范型 Facet 的深度写作指南参考 `guide-output.md`。

---

**核心问题**：哪些 facet 打包进同一个 section，哪些分开？

### 判据 1：facet 间耦合度

- 紧耦合（改一个 facet 必须改另一个）→ 打包进同一个 section
- 松耦合（可以独立变化）→ 分开放

**内聚性参考**（从合理到不合理）：

- **功能内聚**（一组 facet 为同一个功能构成 What → When → How 链条）→ 打包合理
  - What（定义型 + 知识型）：这个功能是什么、规则是什么
  - When（场景型 + 边界型）：什么时候适用、什么时候不适用
  - How（策略型）：具体怎么执行
  - 不要求三段都齐全（简单功能可能只有 What + How），但 facet 之间必须形成链条关系
- **顺序内聚**（一个 facet 的输出是另一个的输入）→ 打包合理
- **通信内聚**（操作同一组数据）→ 打包合理
- **逻辑内聚**（只是归类相似但本质不同）→ 不应打包，应分开

**内聚性区分方法**：

功能内聚 vs 逻辑内聚是最易混淆的一对。用以下问题区分：

- 功能内聚检验：
  - 依赖测试："去掉 facet A，facet B 还能独立完成它服务的功能吗？" 如果不能 → 功能内聚
  - 链条测试："这些 facet 能否为同一个功能构成 What → When → How 链条？" 如果能 → 功能内聚
- 逻辑内聚检验："去掉'相关/相似'这个标签，A 和 B 之间还有实际的数据流或功能依赖吗？" 如果没有 → 逻辑内聚
- 链条测试的典型反例：把所有知识型 facet 归在一个"规则汇总" section → 它们服务不同功能，无法构成链条 → 逻辑内聚

**关键澄清**：facet 功能类型（知识/策略/边界...）和耦合度是**正交维度**。不同类型的 facet 完全可以是功能内聚——判断依据是"能否为同一功能构成 What → When → How 链条"，而非"是否属于同一类型"。

### 判据 2：facet 作用域

- 专属 facet（只有 1 处使用）→ 嵌入使用位置
- 共享 facet（多处引用）→ 提取到集中位置

### 决策路径

```
这个 facet 是专属的还是共享的？
│
├── 共享 → 提取到集中位置（独立 section 或前置步骤）
│
└── 专属 → 和消费者的耦合度？
    │
    ├── 紧耦合（功能/顺序/通信内聚）
    │   → 打包进消费者的 section
    │
    └── 松耦合（逻辑内聚或无内聚）
        → 独立 section，用名称引用
```

### 常见误判模式

- **"facet 类型不同 → 松耦合"**：知识型规则 + 策略型流程，如果能为同一功能构成 What → How 链条，就是功能内聚。类型不同不等于松耦合
- **"section 太大 → 应该拆"**：大小不是拆分依据，耦合度和作用域才是。大而内聚的 section 比小而碎片化的多个 section 更健康
- **"知识型 facet 应该集中管理"**：只有**共享**知识型 facet 才需要集中。专属知识型 facet 应留在使用位置，保持功能自包含
- **"规则应从流程中分离"**：如果规则（What）和执行步骤（How）为同一个功能构成链条，分离就是打破功能内聚。例如把"术语匹配特殊规则"从"术语匹配流程"中拆出来单独成 section → 链条断裂，模型可能忽略优先级关系
- **"逐 section 检查通过 → 整体布局合理"**：每个 section 内部可以各自内聚良好，但不同 section 中服务同一功能的 facet 被按逻辑分类拆开仍是布局问题。需要跨 section 执行链条测试：不同 section 中的 facet 能否为同一功能构成 What → How 链条？若能 → 应合并

### 应用示例

以下示例均来自真实 prompt：

- **场景定义 + 执行流程**（Super Agent V6 场景 A）：场景型 + 策略型，功能内聚且专属 → 打包在同一个 section
- **Agent 接口规范**（知识型 + 规范型）：改 Agent 能力必须改调用模板 → 紧耦合打包
- **suggestionActions 处理**（混合归属）：解析逻辑（多处使用）→ 共享 → 提取到集中步骤；跳过规则（只服务 Classifier）→ 专属 → 留在 Classifier 接口。即使在同一个 section 中，也要检查是否混入了应提取的共享 facet
- **语言传递规则**（边界型）：所有调用都遵循 → 共享 → 集中在通用规则
- **"核心原则"章节**（多种边界型）：通常是多种**共享**边界型 facet 的集合 → 按共享 facet 规则集中放置（不是逻辑内聚，而是作用域驱动的集中）

## Part 4: Section 内部组织

**当多个 facet 打包在同一个 section 时**，用以下顺序组织内部内容。

### 内部顺序原则

```
1. 名称/标识 — 给 section 一个可引用的名字
2. What（定义型 + 知识型 facet）— 先说清楚"是什么"
3. When（场景型 + 边界型 facet）— 再说"什么时候适用"
4. How（策略型 facet）— 最后说"怎么做"
5. 输出/约束（规范型 + 边界型 facet）— 补充"输出什么、不做什么"
```

**顺序原则**：先理解 → 再判断 → 再行动。

**与 Part 3 的关系**：What → When → How 既是功能内聚的判定标准（能构成这条链 → 应打包），也是打包后的内部排列顺序。两者是同一个底层逻辑的两面。

### 示例

以 Super Agent V6 场景 A 为例，它在一个 section 中打包了多种 facet：

```
**场景 A：资产信息填写引导**        ← 名称
用户希望 AI 帮助检查或补充...       ← What（定义型）

**可协作的 Agent**：                ← What（知识型）
- Quality_Checker / Guide_Agent

**适用条件**：                      ← When（场景型）
- Request 状态为 DRAFT 或 REJECTED
- 用户请求引导、信息检查...

**执行流程**：                      ← How（策略型）
1. 先调用 Quality_Checker
2. 提取 JSON
3. 再调用 Guide_Agent
```

## Part 5: Section 间关系与模块化

### Section 间三种关系

- **前向引用**：定义 section 提供信息 → 执行 section 用名称引用（最健康）
- **查找引用**：执行 section 指向格式定义 section（松耦合时使用）
- **共置**：多个 facet 在同一个 section，无跨 section 引用（紧耦合时使用）

### 健康判断

- 单向引用 → 健康
- 无引用（共置）→ 紧耦合时健康
- 双向引用 → 不健康（facet 归属有问题）
- 内容重复（同一个 facet 出现在多个 section）→ 不健康（应提取为共享 facet）

### 模块化设计原则

使 section 可独立增删改：

1. **自包含**：紧耦合的 facet 打包在同一个 section，增加同类 section 时复制模式即可
2. **名称接口**：section 之间只通过名称引用，不引用具体 facet 内容
3. **模式化工作流程**：工作流程用通用模式描述，不枚举所有情况
4. **可预测影响**：增删改一个 section 时，能清楚知道还需要改哪里

### 操作指南

- **添加 section**：复制同类 section 的 facet 模式，填入新内容
- **添加 facet**：识别 facet 的功能类型和归属 → 按布局决策放到正确的 section
- **编辑**：自包含 section 只改该 section；检查名称引用是否仍然有效
- **删除**：删掉 section，检查是否有其他 section 通过名称引用了它

## Part 6: 自然语言输出描述维度

当 Prompt 需要输出自然语言时，规范型 facet 需要覆盖以下四个维度。缺少这些描述，模型会按默认方式输出——通常过于冗长和正式。

- **语气** — 告诉 agent 用什么口吻说话
- **内容边界** — 告诉 agent 说什么、不说什么
- **词汇约束** — 告诉 agent 哪些词可以用、哪些不能用
- **输出时机** — 告诉 agent 什么时候输出

每个维度的详细写法、好/差示例、以及混合输出说明参考 `guide-output.md`。

## Part 7: 检查清单

合并自 Part 2（整体结构设计）和 Part 3-6（信息布局）的检查清单，按逻辑分组。

### 章节完整性

- [ ] Standard Agent：是否包含角色、核心任务、工作流程？
- [ ] Super Agent：是否包含核心职责、Sub Agent 接口、核心工作流程？

### 信息归属

- [ ] 每段内容是否放在了正确的章节？
- [ ] 是否存在"放哪里都行"的模糊内容？

### 重复检查

- [ ] 核心任务的"关键原则"和"核心原则"章节是否存在措辞矛盾？（内容重复是允许的）
- [ ] 工作流程中是否在重复定义区的内容？

### 结构清晰度

- [ ] 工作流程是否只聚焦于动作，只用名称引用定义？
- [ ] 章节之间是否存在矛盾？

### Facet 识别

- [ ] 能否列出 prompt 包含的所有 facet？
- [ ] 每个 facet 的功能类型是否明确（定义/知识/场景/策略/边界/规范）？
- [ ] 每个 facet 的归属是否明确（专属/共享）？
- [ ] 去掉任一 facet，模型会犯什么类型的错？（验证必要性和分类）

### 布局决策

- [ ] 共享 facet 是否已提取到集中位置？（核心原则章节有意重复关键约束不算）
- [ ] 专属 facet 的打包是否基于功能内聚（而非逻辑内聚）？
- [ ] 打包的 facet 能否构成 What → When → How 链条？
- [ ] 通用 section 中是否混入了专属 facet？（应移到专属 section）
- [ ] 专属 section 中是否混入了共享 facet？（应提取到集中位置）
- [ ] 不同 section 中的 facet 能否为同一功能构成链条？（能 → 应合并）
- [ ] 同一 facet 是否只在匹配其作用域的 section 保留？
- [ ] 修改一个功能是否只需改一处？（需改多处 → facet 分散）

### Section 组织

- [ ] 多 facet 共置时，内部顺序是否遵循 What → When → How？
- [ ] Section 间引用方向是否健康（单向）？
- [ ] 有无同一 facet 重复出现在多个 section？

### 模块化

- [ ] 新增同类 section 是否只需复制模式？
- [ ] 工作流程是否模式化（非枚举）？
- [ ] 增删改一个 section 的影响是否可预测？
