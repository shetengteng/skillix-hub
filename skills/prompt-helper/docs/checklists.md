# 检查清单汇总

> **核心原则提醒**：
> - 六大 Facet 类型：定义型 / 知识型 / 场景型 / 策略型 / 边界型 / 规范型
> - 定义与执行分离：角色/职责在定义区声明，具体步骤在执行区编写

本文件汇总了所有子模块中的检查清单，按功能分类组织。每个检查清单标注了来源子模块，便于需要深度理解时查阅原文。

---

## 一、结构与布局检查（来源：guide-structure.md）

### 章节完整性

- [ ] Standard Agent：是否包含角色、核心任务、工作流程？
- [ ] Super Agent：是否包含核心职责、Sub Agent 接口、核心工作流程？

### 信息归属

- [ ] 每段内容是否放在了正确的章节？
- [ ] 是否存在"放哪里都行"的模糊内容？

### 重复检查

- [ ] 核心任务的"关键原则"和"核心原则"章节是否存在措辞矛盾？（内容重复是允许的）
- [ ] 工作流程中是否在重复定义区的内容？

### 结构清晰度

- [ ] 工作流程是否只聚焦于动作，只用名称引用定义？
- [ ] 章节之间是否存在矛盾？

### Facet 识别

- [ ] 能否列出 prompt 包含的所有 facet？
- [ ] 每个 facet 的功能类型是否明确（定义/知识/场景/策略/边界/规范）？
- [ ] 每个 facet 的归属是否明确（专属/共享）？
- [ ] 去掉任一 facet，模型会犯什么类型的错？（验证必要性和分类）

### 布局决策

- [ ] 共享 facet 是否已提取到集中位置？（核心原则章节有意重复关键约束不算）
- [ ] 专属 facet 的打包是否基于功能内聚（而非逻辑内聚）？
- [ ] 打包的 facet 能否构成 What → When → How 链条？
- [ ] 通用 section 中是否混入了专属 facet？（应移到专属 section）
- [ ] 专属 section 中是否混入了共享 facet？（应提取到集中位置）
- [ ] 不同 section 中的 facet 能否为同一功能构成链条？（能 → 应合并）
- [ ] 同一 facet 是否只在匹配其作用域的 section 保留？
- [ ] 修改一个功能是否只需改一处？（需改多处 → facet 分散）

### Section 组织

- [ ] 多 facet 共置时，内部顺序是否遵循 What → When → How？
- [ ] Section 间引用方向是否健康（单向）？
- [ ] 有无同一 facet 重复出现在多个 section？

### 模块化

- [ ] 新增同类 section 是否只需复制模式？
- [ ] 工作流程是否模式化（非枚举）？
- [ ] 增删改一个 section 的影响是否可预测？

## 二、场景设计检查（来源：guide-scenario.md）

### 意图识别

- [ ] 场景描述是否从用户目标出发（"用户希望..."）？
- [ ] 是否避免了关键词匹配？
- [ ] 每个场景是否包含三要素（目标 + 条件 + 操作）？

### 场景边界

- [ ] 场景之间是否互斥（或明确说明可组合）？
- [ ] 重叠区域是否有明确的处理策略？
- [ ] 是否有兜底场景？

### 耦合组织

- [ ] 场景定义和执行流程是否在同一 section？
- [ ] 共享逻辑是否提取到集中位置？

### 多级场景

- [ ] 嵌套不超过两层？
- [ ] 同层子场景互斥？

### 修改退化检测

- [ ] **意图识别退化**：是否把语义理解改成了关键词匹配？
- [ ] **互斥性破坏**：修改场景条件后，是否出现重叠但缺少处理逻辑？
- [ ] **兜底场景丢失**：是否删除或弱化了兜底场景？
- [ ] **三要素残缺**：修改后场景是否缺少用户目标、适用条件、可执行操作？

---

## 三、输出设计检查（来源：guide-output.md）

### 结构化输出（JSON）

- [ ] 必需字段和可选字段是否明确区分？
- [ ] 字段命名是否自解释？
- [ ] 空值处理策略是否明确？（`null` / 省略字段 / 空字符串）
- [ ] 输出位置是否合理？（嵌入 vs 独立）
- [ ] 流式 vs 批量选择是否合理？
- [ ] 示例是否使用 high-level 占位符？

### 自然语言输出

- [ ] 是否描述了语气？
- [ ] 是否描述了内容边界？
- [ ] 是否有词汇约束？
- [ ] 是否说明了输出时机？

### 混合输出

- [ ] NL 和 JSON 的关系是否说明？
- [ ] 两者的职责是否清晰？（NL 给人看，JSON 给系统用）

### 前端交互

- [ ] `async_cui_callback` 格式是否正确？（必需字段 + 标记）
- [ ] `async_cui_suggest` 格式是否正确？（字符串数组）
- [ ] 是否配合流式输出？

### Sub Agent 约束

- [ ] 是否限制了不该输出的内容？（总结、统计、建议）
- [ ] 是否只面向调用方输出结构化数据？

---

## 四、示例规范检查（来源：guide-example.md）

### 结构化示例（JSON）

- [ ] 是否用了具体数字/ID？（应该用 `"字段ID"`、`X`）
- [ ] 字符串值是否用占位符？（应该用 `"字段名"` 而非 `"user_email"`）
- [ ] JSON 示例是否语法正确？
- [ ] 是否体现边处理边输出？（单个对象，不是数组）
- [ ] 数组是否简洁（1-2 个元素）？

### 自然语言示例（情境模板）

- [ ] 是否覆盖了"有结果"的情况？
- [ ] 是否覆盖了"没问题/没结果"的情况？
- [ ] 是否覆盖了"边界情况"？
- [ ] 模板是否展示了期望的语气？

### 整体

- [ ] 示例是否保持 high-level 抽象？
- [ ] 具体例子是否在文字中说明？

## 五、信息传递检查（来源：guide-info-passing.md）

### Producer 设计

- [ ] JSON 结构清晰，字段命名合理？
- [ ] 必需字段和可选字段明确区分？
- [ ] 使用了正确的标记格式？
- [ ] 在 prompt 中说明了输出会被传递给谁？
- [ ] JSON 格式完整，可被解析？

### Coordinator 设计

- [ ] 正确识别和提取标记化信息？
- [ ] 保持 JSON 内容不变（不修改）？
- [ ] 精简了冗余数据？
- [ ] 使用相同标记格式转发？
- [ ] 添加了必要的上下文说明？

### Consumer 设计

- [ ] 在 prompt 中说明会接收什么信息？
- [ ] 正确识别标记化信息？
- [ ] 处理了数据缺失的情况？
- [ ] 根据接收的信息做出正确决策？
- [ ] 与其他数据源正确整合？

---

## 六、风格一致性检查（来源：workflow-audit.md）

- [ ] 是否避免使用表格？（Prompt 文件禁止使用 Markdown 表格）
- [ ] 术语是否一致？
- [ ] 语言是否自然流畅？
- [ ] 是否解释"为什么"而不只是"做什么"？

---

## 七、修改评审检查（来源：workflow-edit.md 阶段 3）

### 问题对应性

- [ ] 修改位置是否与用户问题相关？
- [ ] 修改的 facet 类型是否与问题匹配？
- [ ] 问题类别和修改范围是否匹配？

### 修改方式

- [ ] 是否优先使用了删除/重组而非新增规则？
- [ ] 修改是否最小化？
- [ ] 如果是结构问题，是否调整了结构而非打补丁？
- [ ] 如果是重组，是否遵循了布局决策原则？

### 副作用

- [ ] 修改处前后上下文是否仍然连贯？
- [ ] 是否引入了新的重复定义？
- [ ] 名称引用关系是否完整？
- [ ] 如果涉及输出/标记/调用模板格式，协作链上的其他 Prompt 是否受影响？

### Facet 完整性

- [ ] 没有意外合并应该独立的 facet？
- [ ] 没有意外拆分紧耦合的 facet？
- [ ] Section 内部顺序仍然合理？
- [ ] 删除的内容确认不需要？
- [ ] 如果属于同类 section 组，修改后模式是否仍然一致？
- [ ] 如果修改跨多个 section，功能级别的 facet 分布是否改善？

### 坏修改模式

- [ ] 没有"加规则代替重构"？
- [ ] 没有"范围膨胀"？
- [ ] 没有"信息重复"？
- [ ] 没有"表述矛盾"？
- [ ] 没有"过度具体化"？
- [ ] 没有"错误合并 facet"？
- [ ] 没有"场景退化"？
