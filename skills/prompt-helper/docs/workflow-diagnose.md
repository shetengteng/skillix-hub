# 行为诊断指南

> **核心原则提醒**：
> - 六大 Facet 类型：定义型 / 知识型 / 场景型 / 策略型 / 边界型 / 规范型
> - 诊断核心：从观察到的行为问题出发，定位到需要修改的 Facet 和 Section

---

## 核心问题

**模型做错了，问题出在 Prompt 的哪里？**

这个 Skill 从**观察到的行为问题**出发，定位到需要修改的 Facet 和 Section。它不检查 Prompt 的静态质量（那是 workflow-audit 的事），也不评估修改是否合理（那是 workflow-edit 阶段 3 的事）——它只负责**从症状找到病因**。

**与其他流程的分工**：

- **本流程**：有问题时用。从行为症状 → 定位根因 → 指导修复方向
- **workflow-audit**：日常检查。从 Prompt 文本 → 逐项检查质量
- **workflow-edit 阶段 3**：改完后用。从具体修改 → 评估修改是否合理

典型使用链路：本流程定位根因 → 输出诊断报告 + 创建 Trial 任务列表 → 逐 Trial 使用 workflow-edit 执行修复。

---

## 前置知识

本流程全程使用 Facet 概念（六种功能类型：定义/知识/场景/策略/边界/规范）进行错误分类和定位。Facet 类型的完整定义见 `docs/guide-structure.md` Part 1。

---

## 诊断输入

诊断需要以下信息：

- **错误行为**：模型实际做了什么（越具体越好）
- **期望行为**：模型应该做什么
- **触发输入**：导致错误的输入内容（如果有的话）
- **Prompt 文本**：当前的 Prompt 内容

---

## Step 1: 收集 + 快筛

先做「低挂果实」检查——很多问题不需要深度诊断，简单的结构性缺陷就是根因。

### 快筛清单

- **缺少步骤编号 / 执行顺序不明确**：工作流程没有清晰的步骤标号，模型容易跳步或乱序
- **执行区在解释"是什么"而非"怎么做"**：工作流程里混入了定义性内容，模型分不清"理解"和"执行"
- **同一规则在多处描述且措辞矛盾**：模型无法判断该遵循哪个版本
- **完全缺失某个必需 Facet**：如无输出格式（规范型）、无场景定义（场景型）、无工作流程（策略型）
- **无兜底场景**：Super Agent 缺少处理"不属于任何已知场景"的分支

**如果快筛命中** → 直接给出修复建议，引用对应的 guide。不需要进入深度诊断。

**如果快筛未命中** → 进入 Step 2。

---

## Step 2: 错误分类 + Facet 定位

这一步由**错误→Facet 映射表**驱动（见本文件末尾「错误类型 → Facet 映射表」）。但在开始分类之前，先做**诊断切分**。

### 前置：诊断切分（Trial Segmentation）

不要试图一次性诊断整个 Prompt 的所有问题。先把问题空间切成独立的诊断单元（Trial），形成一个 **Trial 队列**，然后按优先级逐个处理——每个 Trial 独立走完 Step 2→3→4 的完整流程。

**切分规则**：

- **多个症状** → 每个症状是一个独立 Trial，从影响最大的开始排入队列
- **多个场景的 Prompt** → 先隔离出问题场景，只诊断那个场景对应的 Section
- **长 Prompt（超过 200 行）** → 按 Section 分段，逐段进入阶段 B，不要一次性分析整个 Prompt
- **Super Agent + Sub Agent** → 先诊断 Super Agent 本身（一个 Trial），如果问题不在协调层，再诊断相关 Sub Agent（另一个 Trial）

**Trial 队列是动态的**：诊断过程中发现的新症状（如修复一个问题后暴露出另一个）作为新 Trial 追加到队列末尾。

**为什么切分**：TRAIL 论文发现诊断性能与轨迹长度反相关——分析的内容越多，准确率越低。切分后每个 Trial 的分析范围缩小，定位精度更高。

### 阶段 A：识别错误类型（全局视角）

从观察到的症状，对照映射表，识别属于哪种错误类型。映射表覆盖两大来源：

- **TRAIL 错误分类学**（20 种具体错误类型）：覆盖 Agent 执行层面的问题——推理错误、系统执行错误、规划与协调错误
- **PRISM 特有缺陷**（7 种）：覆盖 Prompt 设计层面的问题——场景互斥性破坏、信息传递断链等

**注意区分 Facet 问题和非 Facet 问题**：系统执行错误（API 报错、超时、配置错误等）不是 Prompt 的问题，不要去改 Prompt。映射表中这些类型标注为「非 Facet — 检查工具/环境配置」。

给出 1-2 个最可能的错误类型。不需要 100% 确定——后面用干预验证来确认。

### 阶段 B：从错误类型定位 Facet（局部检查）

映射表中每种错误类型指向 1-2 个最可能的 Facet 类型。定位到 Facet 类型后，在当前 Trial 对应的 Section 中逐段检查（切分范围见前置步骤）。

---

## Step 3: 假设 + 最小干预

### 形成假设

用一句话描述根因假设：

> 因为 **[Facet 类型]** 在 **[Section]** 中 **[具体问题]**，导致 Agent **[错误行为]**。

如果发现多个问题，按影响优先级排序。**先处理「决定性错误」**——修正它对行为改善影响最大的那个问题。不要试图一次修所有问题。

### 最小干预

对最高优先级假设做**最小化修改**来验证。这不是正式修复，只是测试假设是否正确。

**干预模板**：

- 怀疑**定义型** Facet → 临时加一句角色澄清或职责边界
- 怀疑**知识型** Facet → 临时补一条领域规则或术语定义
- 怀疑**场景型** Facet → 临时加一句场景边界或意图识别条件
- 怀疑**策略型** Facet → 临时加一个步骤或调整步骤顺序
- 怀疑**边界型** Facet → 临时加一条禁止规则或特殊处理
- 怀疑**规范型** Facet → 临时加一个输出格式说明或模板

**原则**：改动越小越好。只加 1-2 句话，不做结构性调整。

---

## Step 4: 验证 + Trial 流转

### 验证

用原始的问题触发输入测试干预后的行为。根据结果走三条路径：

**路径 A：假设成立（行为改善）** → 当前 Trial 标记为「已确认」，记录修复方案（见下），进入下一个 Trial。

**路径 B：假设不成立（行为未改善），还有候选 Facet** → 回到 Step 2 阶段 A，尝试下一个候选错误类型/Facet（仍在同一 Trial 内）。

**路径 C：假设不成立，已耗尽候选** → 当前 Trial 标记为「未确认——建议人工分析」，进入下一个 Trial。

**部分改善**：属于路径 A 的变体——最小干预改善了当前症状但暴露了新症状。当前 Trial 按路径 A 处理（标记为「已确认」），新症状作为新 Trial 追加到队列末尾。

### 停止条件

- **单 Trial**：同一 Trial 内最多尝试 3 个候选 Facet 类型。第 3 次仍不成立 → 走路径 C。
- **全局**：所有 Trial 处理完毕（无论已确认/未确认）→ 输出诊断报告。

### 记录修复方案（路径 A）

假设确认后，**撤销最小干预**，记录诊断结论和修复方向建议：

- 明确定位到的 Facet 类型和 Section
- 给出修复方向（应该怎么改，而非直接改）

### 行为测试

为修复的 Facet 类型设计 2-3 个测试输入：

1. **正常输入**：应该正确处理的典型输入
2. **边界输入**：之前出问题的那个输入
3. **混淆输入**：与其他场景/行为容易混淆的输入

### Trial 流转

当前 Trial 完成（已确认或未确认）后，检查队列中是否还有待处理的 Trial。如果有，从队列中取出下一个 Trial，回到 Step 2 开始新一轮诊断。如果没有，输出诊断报告。

---

## 输出格式

诊断完成后，先输出诊断报告，再创建 Trial 任务列表。

```
## 行为诊断报告

**总览**：共 [N] 个 Trial，[X] 个已确认，[Y] 个未确认

### 快筛结果
- [快筛发现 / 无明显结构问题]

---

### Trial 1: [症状简述]
**症状**：[Agent 的错误行为]
**期望**：[应有的正确行为]

**错误类型**：[从映射表识别的错误类型，如「推理错误 > 任务误解」]
**来源**：[TRAIL 分类 / PRISM 特有缺陷]

**Facet 类型**：[定位到的 Facet 类型]
**位置**：[具体 Section / 内容]
**根因假设**：因为 [X]，导致 Agent [Y]
**验证结果**：已确认 / 未确认（已尝试 N 个候选）

**修复方案**：
- **使用 Skill**：`docs/workflow-edit.md`
- **修复方向**：[具体建议]
- **行为测试**：
  1. [正常输入] → 期望 [X]
  2. [边界输入] → 期望 [Y]
  3. [混淆输入] → 期望 [Z]

### Trial 2: [症状简述]
...（同上结构）
```

**情境变体**：

- 如果只有 1 个 Trial，省略总览行，直接输出该 Trial 的内容
- 如果快筛阶段已全部命中（未进入 Step 2），不需要 Trial 结构——直接列出快筛发现和修复建议
- 如果所有 Trial 均未确认，仍输出完整报告，总览行标注全部未确认，并在末尾说明建议人工分析的原因

### Trial 任务列表

诊断报告输出后，使用 TodoWrite 将已确认的 Trial 转化为修复任务列表，驱动逐 Trial 修复。

**创建规则**：

- 每个已确认 Trial → 一个任务：`Trial N: [症状简述] — [修复方向摘要]`
- 第一个任务标记为 `in_progress`，其余标记为 `pending`
- 未确认的 Trial 不纳入任务列表（报告中已标注"建议人工分析"）
- 快筛直接命中的问题也作为独立任务纳入

**执行方式**：

创建任务列表后，按顺序逐个 Trial 使用 `docs/workflow-edit.md` 执行修复。完成一个 Trial 后标记为 `completed`，启动下一个。

**为什么用任务列表**：诊断可能产生多个 Trial，一次性全部修复容易遗漏或混乱。任务列表让模型聚焦当前 Trial，修完一个再修下一个，与 Trial 队列的"逐个处理"设计一致。

## 错误类型 → Facet 映射表

本映射表是行为诊断的核心参考。诊断 Step 2 中，从观察到的症状出发，在此表中查找对应的错误类型和 Facet。

**使用方法**：浏览症状描述，找到最匹配的错误类型 → 查看对应 Facet → 按诊断线索验证 → 按修复方向处理。

**来源说明**：

- Part 1 基于 TRAIL 论文的错误分类学（Deshpande et al., 2025），覆盖 Agent 执行层面的 20 种错误
- Part 2 是 PRISM 实践中识别的 Prompt 设计层面缺陷，TRAIL 未覆盖

---

### Part 1: TRAIL 错误 → Facet 映射

#### 一、推理错误（8 种，全部映射到 Facet）

##### 1.1 幻觉 — 工具相关

- **典型症状**：Agent 声称工具返回了实际上不存在的结果；编造工具能力（如"我已经调用了 XX 工具"但实际没有）
- **对应 Facet**：知识型（工具能力描述缺失或不准确）
- **诊断线索**：对比 Agent 声称的工具输出与实际工具输出；检查 Prompt 中是否有工具能力和限制的说明
- **修复方向**：在知识型 Facet 中补充工具的能力边界和输出格式说明（`docs/guide-structure.md`）

##### 1.2 幻觉 — 纯文本

- **典型症状**：Agent 编造不存在的事实、数据、规则；输出内容偏离事实
- **对应 Facet**：知识型（领域知识缺失，Agent 被迫"猜测"）
- **诊断线索**：检查 Agent 编造的内容是否属于 Prompt 应提供但未提供的领域知识
- **修复方向**：补充缺失的领域知识到知识型 Facet（`docs/guide-structure.md`）。如果 Agent 在不该回答的领域回答了，也可能是定义型 Facet 的职责边界不清

##### 1.3 信息处理 — 工具输出误解

- **典型症状**：Agent 正确调用了工具，但错误解读了返回结果（如把错误码当成功、把空结果当失败）
- **对应 Facet**：知识型（工具输出格式和字段含义未说明）
- **诊断线索**：查看 Prompt 中是否描述了工具返回值的格式和各字段含义
- **修复方向**：在知识型 Facet 中补充工具返回值的格式说明和关键字段解读（`docs/guide-structure.md`）

##### 1.4 信息处理 — 信息检索不佳

- **典型症状**：Agent 搜索了错误的内容，或用了不合适的查询条件，导致获取无关信息
- **对应 Facet**：策略型（检索策略未说明）或知识型（查询模式/关键词未提供）
- **诊断线索**：检查 Prompt 中是否有明确的检索策略（搜什么、怎么搜、搜到后怎么筛选）
- **修复方向**：在策略型 Facet 中补充检索步骤和筛选条件（`docs/guide-structure.md`）；或在知识型 Facet 中补充正确的查询模式（`docs/guide-structure.md`）

##### 1.5 决策 — 工具选择错误

- **典型症状**：Agent 在某个步骤选了错误的工具（如该调 A 工具时调了 B）
- **对应 Facet**：场景型（哪种情况该用哪个工具未说明）或知识型（工具各自的适用场景未描述）
- **诊断线索**：检查 Prompt 中是否有工具选择的决策条件（什么情况用什么工具）
- **修复方向**：在场景型 Facet 中补充工具选择的条件分支（`docs/guide-scenario.md`）；或在知识型 Facet 中补充各工具的适用场景（`docs/guide-structure.md`）

##### 1.6 决策 — 任务误解

- **典型症状**：Agent 对任务的理解有偏差，做的事情不是用户要求的
- **对应 Facet**：定义型（角色/任务定义模糊）或场景型（意图识别失败）
- **诊断线索**：检查 Prompt 中的角色定位和核心任务描述是否清晰；如果是多场景 Agent，检查意图识别是否能区分用户的实际需求
- **修复方向**：澄清定义型 Facet 中的角色和核心任务（`docs/guide-structure.md`）；或改进场景型 Facet 的意图识别（`docs/guide-scenario.md`）

##### 1.7 输出生成 — 指令不遵从

- **典型症状**：Agent 的输出不符合 Prompt 中明确要求的规则（如要求"不要输出建议"但还是输出了）
- **对应 Facet**：边界型（约束不够强或位置不够突出）或规范型（输出规则不清晰）
- **诊断线索**：检查对应规则在 Prompt 中的位置——是否在显眼位置？措辞是否足够明确和强硬？是否与其他规则矛盾？
- **修复方向**：加强边界型 Facet 的约束措辞，或将其移到更显眼的位置如核心原则章节（`docs/guide-structure.md`）；检查是否有矛盾的规则需要删除

##### 1.8 输出生成 — 格式错误

- **典型症状**：输出的结构化格式不正确（JSON 缺字段、字段名错、嵌套层级错）
- **对应 Facet**：规范型（输出格式模板缺失或不完整）
- **诊断线索**：检查 Prompt 中是否有完整的输出格式模板；对比模板和实际输出找差异
- **修复方向**：补充或修正规范型 Facet 中的输出格式模板（`docs/guide-output.md`）

---

#### 二、系统执行错误（8 种，非 Facet 问题）

> 以下错误类型**不是 Prompt 的问题**，不要去改 Prompt。列在此处是为了帮助排除误诊。

##### 2.1 配置 — 环境设置错误

- **典型症状**：环境变量配置不当，Agent 无法正常启动或连接服务
- **排除方法**：检查环境配置和部署设置，不是 Prompt 问题

##### 2.2 配置 — 工具定义不正确

- **典型症状**：工具的 schema/定义有误，Agent 传入错误的参数格式
- **排除方法**：检查工具的 function calling schema，不是 Prompt 问题（但如果 Prompt 中描述的工具用法与实际 schema 矛盾，需要同步修正 Prompt 的知识型 Facet）

##### 2.3 API — 404 资源未找到

- **典型症状**：Agent 请求了不存在的资源
- **排除方法**：检查 API 端点和资源路径，不是 Prompt 问题

##### 2.4 API — 500 服务器错误

- **典型症状**：后端服务报错
- **排除方法**：检查服务端日志，不是 Prompt 问题

##### 2.5 API — 401/403 认证错误

- **典型症状**：权限不足或认证失败
- **排除方法**：检查认证配置和权限设置，不是 Prompt 问题

##### 2.6 API — 429 速率限制

- **典型症状**：请求过于频繁被限流
- **排除方法**：检查速率限制配置，不是 Prompt 问题（但如果 Agent 反复重试导致限流，可能需要在策略型 Facet 中加入重试策略）

##### 2.7 资源管理 — 超时

- **典型症状**：Agent 陷入死循环或长时间无响应
- **排除方法**：检查超时配置和循环逻辑（但如果 Agent 因 Prompt 中缺少终止条件而无限循环，需要在边界型 Facet 中添加终止条件）

##### 2.8 资源管理 — 资源耗尽

- **典型症状**：Token 用完、内存不足等
- **排除方法**：检查资源配置和 Prompt 长度（如果 Prompt 过长导致可用 token 不足，需要精简 Prompt）

---

#### 三、规划与协调错误（4 种，全部映射到 Facet）

##### 3.1 上下文管理 — 资源滥用

- **典型症状**：Agent 重复调用同一工具，或在已有结果的情况下反复请求
- **对应 Facet**：边界型（缺少防重复规则）或策略型（重试/缓存策略不清晰）
- **诊断线索**：检查 Prompt 中是否有防重复调用的规则；是否说明了什么情况下可以重试
- **修复方向**：在边界型 Facet 中添加防重复/重试上限规则（`docs/guide-structure.md`）；或在策略型 Facet 中明确重试策略（`docs/guide-structure.md`）

##### 3.2 上下文管理 — 上下文处理失败

- **典型症状**：Agent 丢失了之前对话中的关键信息；忽略了用户之前提供的条件
- **对应 Facet**：策略型（信息保持/传递策略未说明）或边界型（上下文管理规则缺失）
- **诊断线索**：检查 Prompt 中是否有关于上下文信息保持和传递的指导；多轮对话中是否说明了哪些信息需要跨轮保持
- **修复方向**：在策略型 Facet 中补充上下文管理步骤（`docs/guide-structure.md`）；在边界型 Facet 中添加信息保持规则（`docs/guide-structure.md`）

##### 3.3 任务管理 — 任务编排错误

- **典型症状**：子任务执行顺序错误；该串行的并行了，该并行的串行了；某个子任务被跳过
- **对应 Facet**：策略型（工作流程/步骤顺序不清晰）
- **诊断线索**：检查 Prompt 中的工作流程是否有明确的步骤编号和执行顺序；依赖关系是否说明
- **修复方向**：重新组织策略型 Facet 中的工作流程，添加步骤编号和依赖说明（`docs/guide-structure.md`）

##### 3.4 任务管理 — 目标偏离

- **典型症状**：Agent 在执行过程中逐渐偏离原始目标，最终完成的不是用户要求的任务
- **对应 Facet**：定义型（目标定义不够清晰）或场景型（场景边界不清，导致执行途中"漂移"到其他场景）
- **诊断线索**：检查 Prompt 中的任务目标描述是否具体；长流程中是否有检查点让 Agent 校验方向
- **修复方向**：强化定义型 Facet 中的目标描述（`docs/guide-structure.md`）；在策略型 Facet 的工作流程中加入目标校验检查点

### Part 2: PRISM 特有缺陷 → Facet 映射

以下是 Prompt 设计层面的缺陷，TRAIL 的分类学未覆盖。

#### P1. 场景互斥性破坏

- **典型症状**：两个场景的适用条件有重叠，Agent 在交叉区域行为不稳定（有时走 A 有时走 B）
- **对应 Facet**：场景型
- **诊断线索**：列出所有场景的适用条件，检查是否存在同时满足两个场景条件的输入
- **修复方向**：细化条件消除重叠、设置优先级、或合并为一个场景（`docs/guide-scenario.md` 第 2 节"重叠处理"）

#### P2. 场景兜底缺失

- **典型症状**：遇到不属于任何已知场景的输入时，Agent 行为混乱（随机选一个场景、或完全不响应）
- **对应 Facet**：场景型
- **诊断线索**：检查是否有处理"其他/未知请求"的兜底场景
- **修复方向**：添加兜底场景（`docs/guide-scenario.md` 第 2 节"兜底场景"）

#### P3. 信息传递断链

- **典型症状**：Super Agent 调用 Sub Agent 后，Sub Agent 的输出信息在传递过程中丢失或变形；下游 Agent 收不到上游的数据
- **对应 Facet**：规范型（输出格式不匹配）+ 策略型（提取/转发逻辑缺失）
- **诊断线索**：追踪数据在 Agent 链条中的流转，检查每个节点的输出格式和下一个节点的期望输入是否匹配
- **修复方向**：对齐 Producer 的输出格式和 Consumer 的输入期望（`docs/guide-info-passing.md`）；补充 Coordinator 的提取转发逻辑

#### P4. 角色边界漂移

- **典型症状**：Agent 逐渐扩大自己的职责范围，做了不该做的事（如 Sub Agent 直接面向用户输出而非返回给 Super Agent）
- **对应 Facet**：定义型
- **诊断线索**：检查 Prompt 中的职责边界是否明确——"做什么"和"不做什么"是否都说了
- **修复方向**：在定义型 Facet 中强化职责边界，明确标注"不做什么"（`docs/guide-structure.md`）

#### P5. 步骤遗漏 / 顺序错乱

- **典型症状**：Agent 跳过了某些执行步骤，或以错误的顺序执行
- **对应 Facet**：策略型
- **诊断线索**：检查工作流程是否有清晰的步骤编号；步骤间的依赖关系是否说明；是否有可能被模型理解为"可选"的步骤
- **修复方向**：重新组织策略型 Facet，添加明确的步骤编号和强制顺序说明（`docs/guide-structure.md`）

#### P6. Facet 分散矛盾

- **典型症状**：Agent 行为不一致——对类似输入有时遵循规则 A，有时遵循规则 B
- **对应 Facet**：结构性问题（非单一 Facet，而是同一规则在多处表述不同）
- **诊断线索**：全文搜索与问题行为相关的关键词，检查是否在多个 Section 中有不同版本的描述
- **修复方向**：统一表述，将分散的 Facet 整合到一处（`docs/guide-structure.md`）。删除冲突的版本，保留最准确的一个

#### P7. 过度约束 / 不足约束

- **典型症状**：过度约束——Agent 对所有输入都给出相同的机械回应，缺乏灵活性；不足约束——Agent 在应该遵循规则的地方自由发挥
- **对应 Facet**：边界型
- **诊断线索**：检查约束规则的覆盖范围——是否太宽（限制了正常行为）或太窄（遗漏了需要限制的情况）
- **修复方向**：调整边界型 Facet 的约束范围和措辞强度（`docs/guide-structure.md`）。过度约束 → 缩小适用范围或放宽措辞；不足约束 → 补充缺失的规则

### Part 3: Facet 反向索引

按 Facet 类型反向索引。用于预防性审查——检查某类 Facet 时，了解它缺失或不足可能导致的错误。

#### 定义型 Facet 缺失可能导致

- 任务误解（1.6）
- 目标偏离（3.4）
- 角色边界漂移（P4）
- 纯文本幻觉（1.2，当 Agent 在不该回答的领域回答时）

#### 知识型 Facet 缺失可能导致

- 工具相关幻觉（1.1）
- 纯文本幻觉（1.2）
- 工具输出误解（1.3）
- 信息检索不佳（1.4，查询模式未提供时）
- 工具选择错误（1.5，工具适用场景未描述时）

#### 场景型 Facet 缺失可能导致

- 任务误解（1.6，意图识别失败时）
- 工具选择错误（1.5，工具选择的条件分支不清时）
- 目标偏离（3.4，场景边界不清时）
- 场景互斥性破坏（P1）
- 场景兜底缺失（P2）

#### 策略型 Facet 缺失可能导致

- 信息检索不佳（1.4，检索策略不清时）
- 任务编排错误（3.3）
- 上下文处理失败（3.2）
- 资源滥用（3.1，重试策略不清时）
- 步骤遗漏/顺序错乱（P5）
- 信息传递断链（P3，提取转发逻辑缺失时）

#### 边界型 Facet 缺失可能导致

- 指令不遵从（1.7）
- 资源滥用（3.1，缺少防重复规则时）
- 上下文处理失败（3.2，上下文管理规则缺失时）
- 过度/不足约束（P7）

#### 规范型 Facet 缺失可能导致

- 格式错误（1.8）
- 指令不遵从（1.7，输出规则不清时）
- 信息传递断链（P3，输出格式不匹配时）
